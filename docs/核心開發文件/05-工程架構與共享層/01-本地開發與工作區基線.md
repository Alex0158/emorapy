本文件承接 repo 根 package.json、各端 package.json/tsconfig 的真實啟動方式、工作區分工、本地聯調順序與共享 package 的正式工程口徑。

## 1. 工作區角色

| 目錄 | 角色 | 物理現狀 |
| --- | --- | --- |
| `frontend/` | 主 Web 端 | root workspace |
| `frontend-admin/` | 管理端 Web | root workspace |
| `mobile/` | Expo App | repo-local app；已預留共享 alias |
| `backend/` | API、jobs、health、metrics | repo-local service；獨立 package 與 tsconfig |
| `packages/contracts/` | 共享契約 | repo-local shared package；已被 `frontend/` 消費 |
| `packages/api-client/` | 共享 transport 與客戶端基線 | repo-local shared package；已建立但尚未形成穩定 app 消費面 |

## 2. 本地工具鏈

最低要求：

1. Node.js 20+
2. PostgreSQL
3. Redis
4. npm

本地若不啟 Redis，可以開發，但不等價於 production / staging。

## 3. 建議啟動順序

後端：

```bash
cd backend
npm install
cp .env.example .env
npm run dev:redis:up
npm run prisma:generate
npm run prisma:migrate
npm run dev
```

前端：

```bash
cd frontend
npm install
cp .env.example .env
npm run dev
```

管理端：

```bash
cd frontend-admin
npm install
cp .env.example .env
npm run dev
```

## 4. 本地聯調固定檢查

1. 後端 log 應能確認 Redis runtime 是否接通
2. `GET /health` 可用
3. `VITE_API_BASE_URL` 指向正確 API base
4. `frontend-admin` 能登入並成功讀取 `GET /admin/me`
5. 快速體驗能完成建 session / case / judgment
6. 訪談與聊天室能走通 SSE / stream 主鏈路

## 5. 共享 package 使用規則

`packages/contracts` 與 `packages/api-client` 是正式共享層，但不應因此假設所有類型與邏輯都已完全下沉，也不應假設所有 app 都已接入。

共享層最小責任：

1. 契約、enum、response shape
2. 純 transport 與共用客戶端行為
3. 純邏輯與可跨端重用的工具

不應下沉：

1. Web / RN UI
2. router / navigation 副作用
3. 平台 storage 實作
4. 原生能力與瀏覽器專屬能力

## 6. 本地與遠端環境

若需用遠端環境聯調：

1. staging 是日常推薦入口
2. production 僅限明確認知風險時使用
3. 本地聯調仍應優先先確認本地 `.env` 與依賴完整
