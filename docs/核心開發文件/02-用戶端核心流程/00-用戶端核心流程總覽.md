## 1. 本子域回答什麼

本子域只回答以下問題：

1. 快速體驗、正式案件、判決、Repair Journey、心理訪談、聊天室、配對如何分成不同主流程。
2. 這些流程各自由哪些頁面入口、哪些 API 族承接。
3. 各主流程在哪裡交接、何時需要回到認證、共用機制或接口主冊。

本子域不回答：

1. 某個 API 是否已使用、待承接、候選廢棄或已確認廢棄。
2. Admin Web、健康檢查、任務治理與後台報表。
3. `packages/*`、workspace、共享 package 的工程落點。
4. App 原生能力、Push、Deep Link、SecureStore 或 App lifecycle 的平台策略。

## 2. 主流程分組

`Emorapy` 的前台主流程不是單一路徑，而是至少六組穩定流程：


| 流程組            | 主功能 / 流程              | 主入口                                                 | 主輸出                                        |
| -------------- | --------------------- | --------------------------------------------------- | ------------------------------------------ |
| 快速體驗           | `F01` / `P01`         | `/quick-experience/create`                          | quick case、結果頁、升格承接                        |
| 協作聽證           | `F02`                 | `/quick-experience/collaborative`                   | collaborative case，判決讀取由 quick result 承接   |
| 正式案件與梳理結果     | `F03` + `F04` + `P02` | `/case/*`、`/judgment/:id`                           | formal case、可閱讀梳理結果與 backend 安全政策       |
| Repair Journey | `F05`                 | `/reconciliation/*`、`/execution/*`                  | plan bundle、repair track、checkin、dashboard |
| 引導訪談與個人觀察      | `F06` + `P03`         | `/profile/index`、`/interview/*`、`/profile/my-story` | interview session、result、可查看／更正的 profile 觀察 |
| 先聊再梳理          | `F07` + `P04`         | `/chat/room*`                                       | chat room、invite、message、judgment handoff  |
| 配對與關係檔案        | `F08`                 | `/profile/pairing`                                  | pairing、relationship profile               |
| 通知中心與前台提醒      | 通知 / reminder         | `/notifications`                                    | read / dismiss / snooze / act              |


其中 `F09` 認證與身份承接不在本子域裁決，需回 [../01-認證與會話/00-認證與會話總覽.md](../01-認證與會話/00-認證與會話總覽.md)。

## 3. 頁面入口族群

### 3.1 快速體驗與協作

1. `/quick-experience/create`
2. `/quick-experience/result/:id`
3. `/quick-experience/collaborative`

這一組允許訪客與 `Session` 用戶進入，是最低摩擦入口，但不等於正式案件流程。

### 3.2 正式案件與梳理結果

1. `/case/list`
2. `/case/create`
3. `/case/:id`
4. `/case/:id/review`
5. `/judgment/:id`

這一組屬登入後主流程，且建案前受 pairing gate 影響。

### 3.3 Repair Journey

1. `/reconciliation/:judgmentId`
2. `/reconciliation/:judgmentId/:id`
3. `/execution/dashboard`
4. `/execution/:planId/checkin`
5. `/execution/:planId/replan`

這一組只在已有 judgment 後成立，不應與梳理結果生成混寫成同一頁責任。進入後的有效 intent、伴侶邀請、共同修復與 solo-only 限制由 backend `repair_access` 裁決，前端 URL 或本地預設不得放寬。

### 3.4 引導訪談與個人檔案

1. `/profile/index`
2. `/profile/settings`
3. `/profile/pairing`
4. `/profile/my-story`
5. `/interview/:sessionId`
6. `/interview/:sessionId/result`

這一組涵蓋 consent、訪談、系統觀察與背景檔案，不等於案件主流程的附錄。當前現碼下，訪談 `start / respond / skip / end / cancel / retry / resume` 全部要求登入且通過 consent gate；使用者頁面不得以資料豐富度、信心分數或未探索主題製造完成壓力，且必須提供資料用途、來源、更正與刪除入口。

### 3.5 聊天室

1. `/chat/room`
2. `/chat/room/:roomId`

這一組是弱入口；能否發邀請、發起梳理、進入正式 judgment，仍取決於 user/session 身份、房間狀態與安全政策。當前現碼下，建房、房主邀請與公開邀請撤回可由 user 或合法 `session_id` 承接，但接受邀請必須登入，且 chat judgment 只允許 `roleA` 發起；安全狀態不得與一般 handoff CTA 並列競爭。

私人與共同內容的 flow 已由 `EMO-ADR-010` 決策：新建 room 先進本人 private channel；對方加入後提供獨立 `我與 AI` / `共同對話` lane，私人內容只可在本人授權下轉成無原因的 mediation controls，或由本人預覽批准為 Context Capsule。Web / App 的 lane、preference、capsule 與 exact approval，以及共同梳理 Decision Core、persisted Judgment recovery 與 shared Repair containment 已進入 `main@30c21bb`。本地全量測試及新資料庫 migration / backfill / audit 已通過；Production 狀態只以 exact main SHA 的正式 workflow、release evidence 與線上 canary 為準，詳見 [Chat 私密上下文待辦](../07-待處理問題與治理/待處理/Chat私密上下文與共同調解隔離重構待辦-2026-07-12.md)。

入口可直連不等於未登入常駐導航要展示聊天室。Header / BottomNav 的未登入態收斂到首頁、快速體驗與登入/註冊；登入後才把 `/chat/room` 作為主導航入口。

### 3.6 通知中心

1. `/notifications`

這一組屬登入後前台運營流，不是公開 feed；`/notifications` 是受保護前台路由。它直接承接 `read / dismiss / snooze / act`，並與 active repair journey、提醒卡片、動作跳轉保持一致。

## 5. 固定交接規則

以下交接規則屬正式正文，不應在頁面或 dated 報告中各自改寫：

1. quick flow、協作聽證、chat 房間與正式案件不是同一條業務流，只能在明確 handoff 點交接。
2. `/quick-experience/create` 會優先建立或恢復匿名 session，並用 `GET /api/v1/cases/by-session` 掃描 draft case；它不是正式案件入口。
3. `/quick-experience/collaborative` 是兩段式建案：第一段以 `plaintiff_statement` 建立 collaborative case 並返回 `phase = a_done + session_id`，第二段攜帶同一 `case_id + X-Session-Id` 補齊 `defendant_statement` 後提交，判決展示仍由 `/quick-experience/result/:id` 承接。
4. 正式案件的 `draft -> submitted -> in_progress -> completed/judgment_failed` 與 judgment 消費分開裁決；建案在 `/case/*`，梳理結果消費在 `/judgment/:id`。
5. 正式建案受 pairing gate 約束；當 pairing 非 `active` 時，`/case/create` 必須阻斷提交並導向 `/profile/pairing`。建案頁不得用 profile richness 或訪談 upsell 阻礙當前陳述任務。
6. remote draft 的回應方提交前必須 blind write：`GET /cases` 與 `GET /cases/:id` 不得返回發起方標題、分類、陳述、附件或 judgment，也不得替被隱藏附件簽發 URL；回應提交後才恢復一般當事人 view。
7. Repair Journey 只在 judgment 可用後成立；`reconciliation`、`execution`、`replan` 不得回頭改寫案件建立語義。responsibility ratio、有效 intent、invite/co-repair/solo-only 必須讀 backend policy。
8. 引導訪談的正式前台承接頁是 `result / my-story`；啟動與恢復以 `organic / pre_case / post_judgment / onboarding` trigger 和 `GET /api/v1/interview/resume` 為準，後者可同時回報 pending session 與最近一次 failed session。`pre_case` 仍可作 backend trigger taxonomy，但不再是建案頁的前置推銷入口。
9. chat judgment ready 只代表可以 handoff 到正式梳理結果消費；若當前仍是訪客，需先登入回跳。
10. 通知中心是受保護的 action inbox，不是旁路 feed；`read / dismiss / snooze / act` 必須與 repair / execution 等主鏈路跳轉保持一致。

## 6. 高風險流程點

以下內容應視為高風險前台鏈路：

1. quick experience 的 session 恢復、draft case 回訪與 result 回訪。
2. collaborative case 的 `phase = a_done -> submitted` 兩段式切換與 `session_id` 延續。
3. case review 生成 judgment 的高成本調用，以及 active safety state 覆蓋 stored route 的讀取一致性。
4. Repair Journey 的 `deferred / paused / replanning / resume`、有效 intent 校正及 partner invite/co-repair 限制。
5. 訪談 SSE、resume、retry、processing failed 與 abandoned / cancel 邊界。
6. chat invite、`request-judgment`、judgment handoff 的競態、冪等與安全 CTA 優先級。
7. remote draft 回應方 blind response 的列表、搜尋、詳情與附件簽名邊界。
8. 通知中心的 `read / dismiss / snooze / act`，尤其 active repair journey 的 `snooze` 邊界。

高風險證據入口在 [../90-證據與盤點/README.md](../90-證據與盤點/README.md)，活躍回歸入口在 [../測試/README.md](../測試/README.md)。
