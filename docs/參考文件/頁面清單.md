# JobPortal 頁面清單（主表 + 補充視圖）

本文件以實際前端代碼為準，採雙層結構維護：

- 主表區（必填）：作為路由頁面盤點的單一事實來源。
- 補充視圖區（分析）：保留統計快照、頁內分頁與入口判定脈絡，不替代主表。

## A. 主表區（必填）

## 1. 主表統計口徑（必填）

- `InCount`：可導航到該路由的顯式入口數（Header 菜單、程式內 `navigate/redirect`、流程中繼）。
- `OutCount`：該頁顯式可直接導航出去的路由數（`navigate/redirect/連結`），同路徑去重。
- 不納入：瀏覽器返回、隱式歷史回退、不可追蹤的外部手動輸入 URL。
- 頁面名字：採「路由語義名 + 元件名」雙標準。
- 路由入口：可直接命中路由的入口（菜單、跳轉、directUrl 中繼）。
- 功能入口：路由內可切換的功能子視圖入口（如 tab/subpage）。
- 操作入口：功能內觸發業務動作的操作入口（按鈕、表單提交、審核操作）。

## 6. 頁面內分頁與子視圖（分析補充）

### 6.1 `StudentProfile`（容器路由：`/profile`、`/student-applications`、`/favorites`）

- 路由分流規則：
  - `/profile` 預設 `personal`
  - `/student-applications` 預設 `applications`
  - `/favorites` 預設 `favorites`
- 主要頁內分頁：`personal`、`resume`、`coverLetter`、`applications`、`favorites`
- 備註：`dashboard`、`messages` 在型別中存在，但不作為目前主導航入口 tab。

### 6.2 `ApplicationManagement`（路由：`/employer-applications`）

- 主要頁內分頁：`processing`、`processed`、`all`
- 用途：僱主端申請狀態處理與明細審閱。

### 6.3 `UserManagement`（路由：`/user-management`）

- 主要頁內分頁：`users`、`employers`
- 管理層分頁：`management`（僅特定管理角色顯示）。

### 6.4 `UserSettings`（路由：`/user-settings`）

- 通用分頁：`profile`、`security`
- 管理員附加分頁：`privacyPolicy`、`sensitiveWords`、`systemSettings`。

### 6.5 `DataAnalysis`（容器路由：`/analytics-*`）

- `DataAnalysis` 目前由多個路由共用同一元件。
- 除 `/analytics-overview` 外，`/analytics-realtime`、`/analytics-pages`、`/analytics-behavior`、`/analytics-system` 目前無常規流程入口。
- 上述 4 條已統一歸類為「候選廢棄（保留路由）」，待業務確認後決定是否下線。

### 6.6 `CompanyCenter`（路由：`/company-center`）

- 主要頁內分頁：`company`、`account`
- 用途：企業資料維護與帳號管理切換。

### 6.7 `SystemSettings`（候選廢棄路由：`/system-settings`）

- 主要頁內分頁：`general`、`email`、`workflow`
- 用途：管理端系統設定分區管理（目前主入口已切至 `/user-settings/systemSettings`）。

### 6.8 `JobAudit`（路由：`/job-audit`）

- 主要頁內分頁：`pending`、`processed`、`jobs`
- 用途：職位審核待辦、已處理與職位清單視圖切換。

### 6.9 `CompanyAudit`（路由：`/company-audit`）

- 主要頁內分頁：`1`（待處理）、`2`（已處理）、`3`（企業列表）
- 用途：企業審核全流程視圖切換。

### 6.10 `MessagesPage`（路由：`/messages`）

- 非 Tabs 結構，但有頁面模式切換：`my`（我的消息）、`log`（操作/發送日誌，管理員）
- 用途：同一路由下以模式切換通知日誌視圖；目前未掛載會話聊天 UI。

## 8. 維護規範（防止再次過時）

1. 任何新增路由，必須同步補齊本文件主表與「頁內分頁/子視圖」內容。
2. 任何頁面調整，先更新「2. 新版主表（必填）」；若影響分析脈絡，再回寫第 5~7 節。
3. 若頁面僅能經通知、外部鏈接、或業務流程跳轉進入，必須標記為「弱入口」。
4. 僅存在路由但無主導航/流程證據者，標記為「無常規入口（待確認）」，禁止直接刪除。
5. 每次調整 Header 菜單、`handleNavigation`、`directUrl` 規則時，需回寫本文件。
6. 本文件聚焦頁面與可達性，不寫接口細節；接口統一維護在 `接口描述/` 與 `全接口清單-主文檔.md`。

## 9. 與映射文檔邊界

- 本文件僅維護「頁面視角」：路由、角色、頁面責任、可達性、頁內分頁。
- API × 功能 × 頁面的跨維度追溯，唯一來源為 `接口-功能-頁面-Mapping.md`。
- 若頁面新增或調整，先更新本文件；涉及 API/功能關聯時，再同步更新 `接口-功能-頁面-Mapping.md`。

## 10. 快速維護流程（操作清單）

1. 變更路由或頁面行為時，先更新「2. 新版主表（必填）」中的對應行。
2. 若涉及入口或跳轉規則，重新檢查 `InCount` 與 `OutCount` 並更新主表。
3. 若涉及 tab、模式切換或容器分流，回寫第 6 節「頁面內分頁與子視圖」。
4. 若入口性質改變（主入口/弱入口/無常規入口），同步更新第 3 節與第 7 節。
5. 若變更影響 API × 頁面關聯，最後同步更新 `接口-功能-頁面-Mapping.md`。

## 11. 變更記錄範本（可選）

每次較大調整建議在 PR 或提交說明附上以下資訊：

- 變更路由：`/example-a`、`/example-b`
- 主表更新：新增/修改/刪除的行
- 入口影響：`InCount`/`OutCount` 是否有變化
- 子視圖影響：第 6 節是否有更新
- 映射同步：是否已同步 `接口-功能-頁面-Mapping.md`
