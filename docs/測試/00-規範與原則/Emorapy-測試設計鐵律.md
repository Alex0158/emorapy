# Emorapy 測試設計鐵律

日期：2026-03-07  
最後更新：2026-03-07（深化 mock 邊界、場景覆蓋、層級策略、斷言品質、執行策略、審查清單）  
適用範圍：Emorapy 所有測試（後端單元/整合/流程測試、前端單元/元件測試、E2E/流程 spec）

## 一、核心原則

### 1.1 測試目的排序（不可顛倒）

1. 第一優先：徹底檢驗並發現業務邏輯與功能細節的問題
  - 包含邊界
  - 包含異常
  - 包含業務規則
  - 包含狀態轉移
2. 第二優先：檢查業務代碼的品質、可讀性與可維護性
3. 絕對禁止：
  - 為了讓測試通過而故意簡化測試
  - mock 過頭，導致測試偏離真實業務
  - 扭曲、迴避或弱化現行業務邏輯

**Mock 邊界說明（何謂「過頭」）：**

- **可接受 mock**：外部依賴（DB、第三方 API、檔案系統、網路、時間）、難以在單測環境重現的資源。
- **視為過頭**：mock 同專案內核心業務 service/store 超過一層；E2E 攔截整個 `/api/`** 並回假資料；多個 store 與 API 同時 mock 導致資料流全被替換。
- 若 mock 後測試通過但真實流程仍可能失敗，即為 mock 過頭。

### 1.2 質量高於數量

1. 質量永遠大於數量
2. 場景覆蓋率永遠大於測試案例數量
3. 寧可少寫，也必須保證每個測試都精準且真實反映業務
4. 嚴禁為了「提高覆蓋率」而寫無意義的重複測試

**場景定義與必備類型：**

- 場景 = 業務情境（前置條件 + 操作 + 預期結果）。
- 每個功能至少涵蓋：happy path、邊界（空值/極值/長度）、異常（網路失敗/權限不足/過期）、狀態轉移（A→B→C）。
- 審查時可依此檢查清單逐項確認。

### 1.3 必須 100% 貼合業務

1. 測試設計必須 100% 貼合當前業務與功能特性
2. 不得引入未經確認的假設
3. 若業務邏輯模糊，必須先澄清，不得自行猜測

## 二、可維護性與重用規則

1. 任何共用邏輯、依賴、測試資料、fixtures、輔助函式、mock 設定，一律提取成共用模組或 helper
2. 禁止在多個測試中重複撰寫相同 setup / teardown / input data；同一 setup 出現超過 2 處即應提取
3. 必須優先設計可重用的測試基礎設施，例如：
  - `TestDataFactory`
  - `BusinessRuleValidator`
  - `CommonTestSetup`
  - 共用 mock helper
4. 若現有共用基礎設施已能承接，優先擴充現有基礎設施，不重複造輪子

## 三、執行鐵律

每次產生或修改測試前，必須先在設計中明確列出：

1. 本次測試要驗證的業務規則 / 功能細節是什麼
2. 邊界與異常情境有哪些
3. 是否已使用共用模組避免重複

若出現以下情況，必須遵守對應規則：

1. 如果業務邏輯有模糊處：
  - 先提問澄清
  - 不得自行猜測
2. 如果發現業務代碼本身有問題：
  - 直接指出問題
  - 提出修正建議
  - 不得為了讓測試通過而扭曲測試
3. 如果某個測試設計違反本文件任一條：
  - 必須拒絕該設計
  - 明確指出違反哪一條
  - 提出符合鐵律的正確寫法

## 四、測試設計禁令

以下做法一律視為違規：

1. **過寬 mock**：攔整個 `**/api/`** 或同專案 service/store 多層，掩蓋真實資料流。E2E 禁止攔截全部 API 並回假資料。
2. **表面驗證**：只驗證「頁面有打開」，卻不驗證核心業務狀態、關鍵按鈕 enabled/disabled、錯誤提示、導頁或 payload。
3. **弱化斷言**：為求穩定而移除關鍵業務斷言；使用 `expect.anything()`、接受多種 status 的寬鬆斷言、或條件式 `if (result.data)` 跳過驗證。
4. **重複 setup**：在多個測試中複製貼上同一批資料建構與 mock 設定，未提取為 fixture 或 factory。
5. **過時對齊**：用與現行業務不一致的舊接口、舊路由、舊文案，卻不先修正測試設計。

## 五、推薦做法

### 5.1 驗證優先級

1. 優先驗證：業務規則、狀態轉移、權限邊界、錯誤分支
2. E2E 只保留最有價值的真實主流程與跨頁面鏈路
3. 複雜表單、角色組合、payload 細節，優先下沉到更穩定的元件測試或單元測試
4. 若已有測試僅是內容過時，優先重寫/優化現有測試，而不是新增重複測試

### 5.2 測試層級策略


| 層級  | 適用情境                   | 要求                           |
| --- | ---------------------- | ---------------------------- |
| 單元  | 純邏輯、無 I/O、可快速回歸        | 只 mock 外部依賴，不 mock 同專案核心業務模組 |
| 整合  | 跨模組、真實 DB/API、資料流與業務閉環 | 至少一條核心主鏈路須有可常態執行的整合測試        |
| E2E | 關鍵主流程、跨頁、真實瀏覽器         | 不得僅依環境變數或條件式 skip，而無替代執行計畫   |


### 5.3 排障導向

- 測試失敗時，應能產出缺陷報告級證據：觸發條件、預期行為、實際行為、根因位置。
- 斷言應驗證業務語義（狀態、錯誤碼、導頁、payload），而非僅驗證 i18n key 或 mock 呼叫次數。
- 關鍵流程測試應在描述或註解中標註對應業務規則編號或功能點，便於追蹤。

### 5.4 斷言品質

- 優先斷言業務結果（狀態、導頁、payload、錯誤碼），其次才是 UI 文案或呼叫次數。
- 避免斷言實作細節（內部函式名、私有變數、DOM 結構），除非該細節即為業務契約。
- 錯誤情境應驗證錯誤碼或錯誤類型，而不只是「有顯示某個 message」。

## 六、測試執行策略

- **必跑**：所有單元測試；至少一條核心主鏈路整合或 E2E。
- **可選跑**：需 DB、外部服務、長時間的整合測試，應在 CI 中明確標示，並有定期（如 nightly）執行計畫。
- **禁止**：以「環境未配置」為由長期 skip 核心流程測試，而不規劃補齊環境或替代方案。

## 七、與核心文檔的關係

所有 Emorapy 測試設計，必須優先對齊以下文檔：

1. `docs/核心開發文件/功能特性清單.md`
2. `docs/核心開發文件/業務流程整合.md`
3. `docs/核心開發文件/接口-功能-頁面-Mapping.md`
4. `docs/核心開發文件/全接口清單-主文檔.md`
5. 對應模組的 `docs/核心開發文件/接口描述/`*

如上述文檔與實際代碼不一致：

1. 以實際代碼為當前事實來源
2. 同時記錄文檔不一致點（於測試檔或專用清單）
3. 不得用過時文檔強行約束現行測試
4. 定期（如每迭代）檢視不一致清單，決定更新文檔或調整代碼；未處理前不得以過時文檔作為測試設計依據

## 八、落地要求

從本文件建立後開始：

1. 所有新測試都必須遵守本文件
2. 所有重構/優化/刪理測試都必須遵守本文件
3. 審查現有測試是否過時時，也必須以本文件為準

本文件屬於 Emorapy 測試工作的最高標準，後續所有測試相關設計與實作均不得違反。

## 附錄：審查檢查清單

審查測試設計與實作時，可依下列項目逐項確認：

**設計前**

- 是否列出本次要驗證的業務規則？
- 是否列出邊界與異常情境？
- 是否已使用或規劃共用模組？

**實作中**

- Mock 是否僅限外部依賴？同專案 service/store 是否 mock 過多？
- 斷言是否驗證業務語義，而非僅 i18n key 或 mock 次數？
- 是否有重複 setup 未提取？

**完成後**

- 測試失敗時能否產出缺陷報告級證據？
- 是否對齊核心文檔（或已記錄不一致）？
- 若為核心流程，是否有可常態執行的整合/E2E，而非長期 skip？
