## 1. 問題

`backend/src/services/chat.service.ts` 在聊天室轉正式 case 時固定產生 `聊天室轉判決-YYYY-MM-DD` 作為 `Case.title`。Web `Case/List`、Web `Case/Detail` 與 App `case/index` 會直接顯示 `case.title`，因此該字串會成為跨端可見文案。

此處同時違反兩條基線：

1. 不跟隨使用者當前 request locale；英文使用者仍會看到繁中標題。
2. 繁中術語使用「判決」，不符合核心術語表要求的「梳理結果 / Analysis」。

## 2. 影響

1. Web 案件列表與案件詳情會顯示錯誤語言與錯誤術語。
2. App Case screen 的 case card title 會顯示同一份 backend 儲存標題，因此也會漂移。
3. 因為 title 是資料欄位，不是 API response message，`responseFormatter` 與前端 i18n catalog 都不會自動修正。

## 3. 目標

1. Chat-to-case 自動生成 title 必須按 `requestJudgment` 的 `locale` 建立。
2. zh-TW title 使用「聊天室轉梳理結果-YYYY-MM-DD」；en-US title 使用「Chat to Analysis-YYYY-MM-DD」。
3. 既有手動輸入的 case title 不在本任務範圍內重寫；本任務只處理 backend 自動生成 title。
4. 同步補齊單元測試，避免「判決」再回流到該自動 title。

## 4. 邊界與注意事項

1. 不新增 DB 欄位與 migration；title 仍是 `Case.title`。
2. 不在 Web/App 端對儲存 title 二次翻譯，避免資料標題與詳情頁不一致。
3. 舊資料不在本任務中批量改寫；如需資料修復，應另建 migration / data repair 待辦。
4. route response message `已發起判決` 已有 backend i18n 英文映射，但繁中術語仍可後續另立專項收斂；本任務先修正已確認進入資料標題的缺口。
