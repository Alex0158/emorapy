## 1. 功能分層與編碼規則

### 1.1 主功能（F01-F10）

`F01-F10` 只用於表示**對使用者或運維角色可感知、可獨立驗收、可穩定映射到頁面/API/流程責任**的一級功能。

一個 `Fxx` 至少應滿足：

- 有清楚的功能目的與前置條件；
- 有明確入口頁或外部入口域；
- 能對應至少一組穩定 API；
- 能獨立驗收，不依賴讀者自行拼裝隱性知識。

### 1.2 保留能力（F11-F14）

`F11-F14` 保留既有編碼，僅作**歷史索引與候選/平台能力附錄**，不再與 `F01-F10` 同層解讀。

- `F11-F12`：候選保留能力
- `F13`：已啟用支撐能力，但仍不計入 `F01-F10` 前台主功能序列；通知中心以修復旅程、系統提醒與再進場 inbox 承接跨功能提醒，不獨立成為主流程。
- `F14`：平台/監控保留能力

後續若其中任一能力升格為主功能，必須先同步：

- 本文主表
- `頁面清單.md`
- `接口-功能-頁面-Mapping.md`
- `業務流程整合.md`
- 測試台帳與專項測試文檔

### 1.3 完成度口徑

- `已完成`：頁面可直接觸發核心 API，且已有前端調用證據。
- `跨功能依賴`：本功能會把責任交接給其他功能頁完成閉環，需明示承接方。
- `待驗證`：後端能力存在，但前端主流程或穩定入口證據不足。
- 上述完成度默認指 Web/Admin/Backend 當前實作；App 端不得從此欄位推斷為已完成，必須另看 `20-App端/` 與 `50-跨端Mapping與Parity/`，其中 App navigation / native adapter 以 `20-App端/01-App導航與平台Adapter基線.md` 為實作 gate，首輪能力對照以 `50-跨端Mapping與Parity/01-App首輪能力與工程落點Mapping.md` 為工程落點 gate，App 驗收與證據以 `08-測試規範與驗收/03-App測試與證據接入基線.md` 為進場 gate。

## 3. 主功能詳解（操作級）

### F01 快速體驗、結果與升格

- **目的**：在未登入條件下完成 session、quick case、結果頁判決與後續升格綁定。
- **主要角色**：訪客、Session 用戶、已登入後的升格用戶。
- **前置條件**：可匿名訪問；若無有效 `session_id`，需先建立或恢復 session。
- **功能邊界**：
  - 包含：session 建立/恢復、quick case、結果頁判決、補證據、內容推薦、`claim-session`。
  - 不包含：formal case 建立與正式案件狀態機，該部分由 F03 承接。
- **主要輸出/結果**：匿名案件、判決結果頁、註冊/登入後的 quick case 升格綁定。
- **跨功能依賴**：
  - 承接 F02 的協作 case 結果頁判決讀取。
  - `claim-session` 由 F09 成功登入/註冊後的隱式流程承接。
- **關鍵規則**：
  - `session_id` 失效時，前端應走 `清 session -> refresh -> 重試` 的恢復鏈路。
  - `GET /cases/by-session` 屬 quick case 回訪/恢復能力，不能只把 F01 理解為一次性建案。
  - `POST /auth/claim-session` 失敗時只能告警，不得阻斷 auth 主流程成功。
  - 證據上傳可走 `X-Session-Id`，不依賴 JWT。
  - `/quick-experience/result/:id` 使用 `AI Stream` phase 流：判決未完成時優先展示 `collecting_context -> analyzing_emotion -> building_responsibility -> drafting_judgment -> finalizing -> completed/persisted`，再回落既有判決查詢。
  - 前端 phase 累積邏輯由共享 `aiStreamState.ts` 承接，避免不同頁面各自維護 phase 去重與狀態轉換。
  - `QuickExperience/Result` 的 AI stream 訂閱由共享 `useAIStreamSubscription` 承接，頁面不自行管理 `after_seq`、重連與 snapshot/replay。
  - `QuickExperience/Result` 的 phase 展示使用共享 `AIPhaseTimeline`，頁面不手寫 phase 串接文案。
  - `QuickExperience/Result` 的錯誤、判決失敗與 timeout 殼層由共享 `AIErrorState` 承接，頁面不散落多段 `Alert` 結構。

### F02 協作聽證（同設備雙人）

- **目的**：提供同設備雙人輪流輸入的低摩擦協作建案模式。
- **主要角色**：訪客、Session 用戶。
- **前置條件**：無需登入；建議同場景使用。
- **功能邊界**：
  - 包含：A/B 交替輸入、phase 控制、協作 case 提交。
  - 不包含：協作結果頁判決查詢；提交後的結果頁由 F01 承接。
- **主要輸出/結果**：collaborative case、導向 result 的承接關係。
- **跨功能依賴**：`GET /cases/:id/judgment` 由 F01 結果頁承接。
- **關鍵規則**：
  - `phase=intro/role_a/handoff/role_b/submitting` 防止流程跳步。
  - 最小字數門檻：A 方 `>=30`、B 方 `>=10`，未達不發請求。
  - F02 的 `POST /cases/collaborative` 僅負責 session-bound 協作建案；`collaborative(session_id=null)` 屬 F03 正式建案能力。

### F03 正式案件管理

- **目的**：讓已登入用戶完成 formal case 的建立、編修、提交流程。
- **主要角色**：用戶。
- **前置條件**：已登入，且正式建案需 `pairing=active`；未配對應導向 F08。
- **功能邊界**：
  - 包含：案件列表、建立、詳情、證據、draft 編修、submit。
  - 不包含：judgment 本體的生成/接受/結果消費；該部分由 F04 承接。
- **主要輸出/結果**：formal case、`draft -> submitted` 狀態推進。
- **跨功能依賴**：
  - 依賴 F08 提供 pairing 與 relationship context。
  - submit 後由 F04 承接 judgment lifecycle。
- **關鍵規則**：
  - 只有 `draft` 可以 `update` 與 `submit`。
  - formal create 前的 `GET /pairing/status` 是硬性 gate，不是可選提示。
  - `POST /cases` 的 `mode` 現碼為 `remote|collaborative`；當 `mode=collaborative` 時需雙方陳述，且該案件為 `session_id=null` 的正式案件。
  - `POST /cases` 的 formal case safety assertion 目前是 backend additive support：route / service 可接受 `safety_assertion` / inline safety fields 並寫入 `Case.safety_metadata`，但 shared `CreateCaseDto` 與 Web / App create UX 尚未暴露；不得把基礎正式建案已完成外推為安全聲明 create path 已完成，缺口見 `正式案件安全聲明SharedDto未暴露待辦-2026-05-31.md`。
  - `GET /cases/:id` 與 `GET /cases/:id/judgment` 授權分流由 `backend/src/utils/case-classifier.ts` 裁決：`quick`/`collaborative(session_id 有值)` 走 session；`remote`/`collaborative(session_id=null)` 走當事人 JWT；產品來源另以 `product_flow` / `ChatToCaseLink` 區分 `chat_to_case`。

### F05 和好方案與執行追蹤

- **目的**：將 judgment 轉成一條可被承接、可共同參與、可再適配的 repair journey。
- **主要角色**：用戶。
- **前置條件**：已有 judgment。
- **功能邊界**：
  - 包含：方向選擇、生成主推薦/備選方案、單人承諾、邀請對方、開始今天的一小步、每日脈搏回報、dashboard。
  - 不包含：judgment 接受/拒絕本身；該部分屬 F04。
- **主要輸出/結果**：reconciliation plan bundle、repair track、participant commitment、repair checkins、journey dashboard。
- **跨功能依賴**：依賴 F04 提供可用 judgment。
- **關鍵規則**：
  - `POST /judgments/:id/reconciliation-plans` 需顯式接收 `intent` 與偏好，返回主推薦 bundle。
  - bundle 回應包含 `journey_entry + version_summary`；若已有 `solo_active / co_active / replanning / paused / completed / closed` 旅程，前端需優先給「回到這一輪」而不是一律重新生成。
  - F05 主要讀接口返回 `journey_context`，前端需以 `journey_context.primary_cta / secondary_cta / presentation_bucket` 作為主判斷，不再頁面各自猜 `track.status`。

  - `select` 不再代表全局唯一選定，而是「當前用戶承諾此方案」。
  - `respond` 補齊 invitee `viewed / committed / deferred / declined / paused` 回應閉環；`deferred` 是正式狀態，不再用 `declined` 假裝「之後再說」。
  - `execution/checkin` 不再只是 notes + photos，而是 `step_result + closeness + stress + needs_help` 的每日脈搏回報。
  - `replanning` 由獨立 `/execution/:planId/replan` 前台頁與 `/repair-tracks/:id/replan` 後端接口承接；重調不覆蓋舊 checkin 歷史。

  - 照片上傳失敗不應阻斷純文字回報。
  - 前端方案旅程頁必須兼容 legacy `plans[]` 陣列回應與缺失 bundle 字段，不得因 `recommended_plan_id/plans` 缺失而崩潰。
  - 每日一步頁必須兼容缺失 `recent_checkins/records` 的舊 shape，自動回退為空陣列。
  - 修復進展看板對未知 `journey_status` 必須忽略並回退空狀態，不得渲染空白頁；已知狀態需按 `draft / partner_invited / solo_active / co_active / replanning / paused / completed / closed` 分欄呈現。
  - reminder job 以狀態驅動：未選方向、已承諾未開始、partner invite 未查看/未回應、replanning、paused 都有對應通知模板與 deep link。
  - 通知中心已前台接線：Header bell 顯示未讀數，`/notifications` 作為 repair journey 的再進場 inbox，支持 `read / read-all / dismiss / snooze / act`；對 active repair journey 默認次操作應是 `snooze` 而不是直接 dismiss。

### F06 心理訪談、結果與心理畫像

- **目的**：以 consent 為前提建立 interview session，產出可被用戶回看與管理的結果與 psych profile。
- **主要角色**：用戶。
- **前置條件**：已登入，且同意心理資料使用。
- **功能邊界**：
  - 包含：consent、start/respond/skip/end、result 頁、resume/retry、my-story 的 psych profile 與 feedback、刪除心理資料。
  - 不包含：單獨宣稱「心理畫像已直接改寫 judgment 結果」的產品閉環；目前明確前台承接仍是 result / my-story。
- **主要輸出/結果**：interview session、訪談結果頁、psych profile、feedback history。
- **跨功能依賴**：
  - `/profile/index`、`/profile/pairing` 可作訪談入口。
  - `GET /psych-profile` 會被部分場景補充讀取，但主承接頁仍是 `/profile/my-story`。
- **關鍵規則**：
  - 預設限制：`maxTurns=25`、`turnIntervalMs=3000`、每小時 `3` 次、每日 `5` 次；可被 runtime config 覆蓋。
  - `end` 不代表一定產生畫像更新；內容不足時可直接進 `completed` 而不進 pipeline。
  - `respond/skip` 採提交式觸發，AI 可見輸出統一由 `GET /api/v1/streams/interview_session/:id` 驅動。
  - `/interview/:sessionId` 送出回答後立即顯示 AI placeholder；即使首 token 尚未到達，也需顯示 thinking bubble，避免空白等待。
  - 前端 AI 流狀態命名遵循統一 `thinking/streaming/persisting` 口徑。
  - `interviewStore` 已顯式維護 `streamingStatus`，使 `Interview`、`Chat Room`、`QuickExperience` 三類 AI 場景在前端共享同一套狀態語義。
  - `Interview` 與 `Chat Room` 的 streaming placeholder 已共用 `AIStreamingBubble` 骨架元件，避免再次分叉出第二套 draft bubble DOM 與可及性語義。
  - `Interview` 視圖層輸出共享 `AIStreamDraft` 形狀，不在頁面內手寫 `streamingText + streamingStatus` 的專用拼裝。
  - `Interview` 的 `cancelStream` 清除本地 draft 並以提示文案收口，不渲染可見 `cancelled` draft 氣泡。
  - `Interview` 後端會把提交後的 AI 任務發送到 `interview_session` 的統一 `AI Stream` scope，包含 `stream.started/delta/completed/persisted/cancelled`，使可見回覆、取消與落庫語義都走同一協議。
  - `AI Stream` runtime 在配置 `REDIS_URL` 時會自動啟用 Redis `INCR` seq、Pub/Sub 跨實例分發與短期 replay/snapshot；未配置時回退單機內存模式。
  - Redis-backed AI Stream 行為由 `npm run test:integration:ai-stream-redis` 與 `backend/tests/integration/ai-stream-redis.integration.test.ts` 承接；單次環境證據只作 `90-證據與盤點/` 回鏈，不作本文完成判定。
  - 後端 AI Stream 持久化模型包含 `ai_stream_sessions` 與 `ai_stream_events`：前者保存每條 stream 的權威狀態與最後快照，後者保存非 heartbeat 事件審計，用於排障與驗收。
  - `AI Stream` 治理能力包括：Admin 可透過 `GET /api/v1/admin/reports/ai-streams` 查看匯總，透過 `GET /api/v1/admin/reports/ai-streams/sessions` / `:streamId` 查看 session 與事件明細；後端亦會透過 `cleanup_ai_stream_persistence` 執行 archive-before-delete，將過期資料沉澱到 archive 表。
  - `frontend-admin` 的 `/admin/reports` 接入 AI Stream 治理接口，可直接查看匯總、session 列表與單條 stream 詳情。
  - `Interview/Chat` 頁面以 `GET /api/v1/streams/interview_session/:id` 作為主讀鏈路；當頁面重掛或連線中斷但仍有活動回覆時，頁面會根據 snapshot / replay 顯示 recovering draft，而不是回到空白狀態。
  - `Interview/Chat` 仍會處理 `stream.cancelled` 的 snapshot / replay 來完成狀態對齊，但視圖層不再顯示 cancelled draft 假氣泡。
  - `Interview/Chat` 在收到 `stream.persisted` 後會靜默回拉 canonical session，最終 turns 以後端已落庫結果為準，而不是只依賴本地 SSE 臨時拼裝。
  - `Interview/Chat` 為防 SSE 漏掉 terminal 事件，會在 streaming 期間啟用有界 canonical 自愈輪詢：每 `2500ms` 同步一次、最多 `24` 次，且以 lock 防止重入；terminal 收口後立即停止。
  - `AI Stream` 持久化以 Prisma model accessor 與真實 session/event 落庫為權威；避免出現 canonical turn 已生成但前端 draft 因缺 `stream.persisted` / replay 而停住的假完成狀態。
  - `Interview/Chat` 的 AI stream 生命週期由共享 `useAIStreamSubscription` 承接，頁面層不自行維護 `lastSeq/retry/cleanup`。
  - `Interview` 的 thinking 文案與增量 cursor 透過共享 `AIThinkingIndicator` / `AIStreamingText` 承接，不在 bubble 內直接手寫。
  - `Interview` 的 recovering 狀態透過共享 `AIRecoveryBadge` 呈現，初始載入失敗與流式錯誤殼層由共享 `AIErrorState` 承接。
  - `Chat Room` 頁面層命名固定為 `aiDraft`，不保留 `streamingAiDraft / streamingAiText` 這類歷史命名。

### F07 Chat 房間、邀請與轉判決

- **目的**：支撐 chat room、invite、message、SSE 與 chat -> judgment 的閉環。
- **主要角色**：訪客、Session 用戶、用戶。
- **前置條件**：弱入口可直達，但可用權限取決於 `Guest / Session / User` 身份與房間角色。
- **功能邊界**：
  - 包含：建房、房間讀取、邀請、進房、消息、SSE、leave/kick、request-judgment。
  - 不包含：judgment 詳情頁展示與接受；該部分由 F04 承接。
- **主要輸出/結果**：chat room、invite、message stream、judgment request 狀態。
- **跨功能依賴**：
  - judgment request 成功後由 `GET /chat/rooms/:roomId/judgment-status` 承接；若當前未登入，先進 `/auth/login`，再由 F04 消費 judgment 詳情。
- **身份矩陣（文檔口徑）**：
  - `Guest`：可進弱入口，但若無有效 session，無法穩定持有匿名房間能力。
  - `Session 用戶`：可匿名建房；若為匿名 A 房主，建立邀請需 `room.session_id` 與當前 `canonical session_id` 匹配。
  - `User`：可作為登入房主或被邀請方參與房間；登入 user 權限以 `owner_user_id` / room 角色判定。
  - `接受邀請`：收斂為 `User only`；B 方需先取得登入身份再承接 invite，不再以 Session 身份接受邀請。
- **關鍵規則**：
  - 訊息頻率限制：5 秒最小間隔，30 秒窗口最多 6 條。
  - `request-judgment` 僅 A 方可發起，且有鎖 + 冪等保護。
  - `history_visibility_mode` 與 `included_message_ids` 共同決定可被納入 judgment 的訊息範圍；`included_message_ids` 僅允許 `user_text + visibility_scope=all`，且在 `share_from_join_time/share_summary_only` 下需同時滿足 join-time 時間窗。
  - `request-judgment` 若提交超出白名單的 `included_message_ids` 應返回 `NOT_FOUND`；AI 生成不可用則返回 `AI_SERVICE_ERROR`，兩者都屬正式錯誤分支。
  - 匿名房主可建立 invite 的前提是 `room.session_id === 當前 canonical session_id`。
  - `canonical session_id` 指前端統一 session 管理層持有的有效匿名會話 ID；文檔口徑不再以任意 `localStorage/sessionStorage` 讀值作為身份判定基準。
- **UX 規則**：
  - **思考中反饋**：發送訊息且 `visibility_scope === 'all'` 時，發送成功後立即顯示「思考中...」；後續由 `AI Stream` 推動 `thinking -> streaming -> persisting`，15 秒內未收到任何 AI 事件則超時清除。
  - **建立後樂觀渲染**：建立聊天室成功後 `navigate` 帶 `state.room`，進入房間時若 `state.room.id === routeRoomId` 則走快速路徑（僅 `listChatMessages`，不呼叫 `getChatRoom`），減少首屏等待；無 state 或 listChatMessages 失敗時 fallback 至完整 `loadRoomInitial`。
  - **AI 流式回應**：聊天房間的 AI draft 由 `GET /api/v1/streams/chat_room/:roomId` 主驅動；前端以 `streamId` 維護草稿 AI bubble，首 token 前顯示 thinking，`stream.completed` 只切到 `persisting`，待 `stream.persisted` 與正式 message 落庫後才完成交接。
  - `/chat/rooms/:roomId/stream` 保留為房間級事件流，只負責 `message/invite/room_status` 之類 domain 事件；AI 文字增量與 persisted 狀態以 `AI Stream` 為準。
  - 後端暴露 `AI Stream` 指標，可對 TTFT、completed->persisted lag、terminal result 做驗收。
  - `AI Stream` 的持久化治理、archive 和 admin 查詢是平台共用能力，不由聊天頁自行維護。
  - `Chat Room` 的 AI stream 訂閱由共享 `useAIStreamSubscription` 承接，頁面不自行維護 `after_seq`、重連與 SSE 清理。
  - `Chat Room` 的 thinking indicator 與 streaming cursor 與 `Interview` 共用元件，不額外長出第三套 DOM 結構。

## 5. 例外流程與恢復矩陣


| 功能ID    | 例外情況                         | 觸發點                         | 預期恢復策略                                                                                  | 驗收標準                   |
| ------- | ---------------------------- | --------------------------- | --------------------------------------------------------------------------------------- | ---------------------- |
| F01     | session 過期/缺失                | quick create/result         | 清 session -> refresh -> 重試                                                              | 不出現無限 401 循環           |
| F01     | quick judgment 尚未生成          | result 頁                    | pending 輪詢/提示                                                                           | 可在就緒後自動顯示判決            |
| F01/F09 | `claim-session` 失敗           | register/login 成功後          | 保留 auth 成功，只提示綁定失敗                                                                      | 不阻斷登入/註冊成功             |
| F02     | A/B 文本不足字數                   | collaborative 提交            | 阻止提交並提示最小字數                                                                             | 不發出後端請求                |
| F03     | 非 `draft` 嘗試更新/提交            | case detail/review          | 顯示不可編輯/不可提交                                                                             | 狀態防線生效                 |
| F03     | 證據上傳失敗                       | FileUpload                  | 單檔重試，不丟已上傳成功項                                                                           | 成功項保留                  |
| F04     | 判決生成失敗                       | review 提交後                  | 顯示重試入口                                                                                  | 重試可恢復                  |
| F05     | 方案生成失敗                       | reconciliation list         | 保留頁面狀態 + 手動重試                                                                           | 二次請求可成功                |
| F05     | 打卡照片上傳失敗                     | execution checkin           | 提示警告後允許純文字打卡繼續提交                                                                        | checkin 仍可成功入庫         |
| F05     | 打卡提交失敗                       | execution checkin           | 保留填寫內容，允許重送                                                                             | 重送後進度累加                |
| F06     | consent 未授權                  | profile 啟訪談                 | 先引導 consent                                                                             | 未 consent 不應創建 session |
| F06     | 訪談回合過快                       | interview chat              | 返回 `TURN_TOO_FAST` 提示稍後                                                                 | 不應新增 turn              |

| F07     | 訊息頻率超限                       | chat room                   | 返回限流提示                                                                                  | UI 不崩潰，輸入框可恢復          |

| F07     | 轉判決競態                        | request-judgment 連點         | 走冪等/鎖，避免重複建案                                                                            | 最終只生成一個有效 case         |
| F08     | 邀請碼錯誤/過期                     | pairing join                | 提示並允許重新輸入                                                                               | 不改變既有 pairing          |

| F09     | 帳密錯誤                         | login                       | 留在頁內提示                                                                                  | 不應清理已存在的其他態            |
| F10     | 權限不足                         | admin 任意受限頁                 | 返回 403 並保留 admin 上下文                                                                    | 不應誤跳轉前台登入              |

## 7. 功能邊界裁決


| 項目         | 易混淆口徑                                 | 裁決                                                | 當前結論                     |
| ---------- | ------------------------------------- | --------------------------------------------------- | ------------------------ |
