## 2. 結果填寫規則

- 狀態只填：
  - `PASS`
  - `FAIL`
  - `BLOCKED`
- 若為 `FAIL` 或 `BLOCKED`，必填：
  - 問題類型：`代碼缺陷 / 環境問題 / 測試資料問題 / 非阻塞體驗問題`
  - 問題描述
  - 截圖或錄屏位置
- 每條流程至少執行：
  - 桌面 1 輪
  - 若流程涉及弱入口或視覺觀感，窄視口再補 1 輪

### 2.1 填寫後檢查命令

```bash
npm run manual-regression:check
```

- 若全部填完整，腳本應不報缺欄位。
- 若仍有缺欄位，腳本會直接指出對應 `P01-P05/record.md`。

```bash
npm run manual-regression:check:strict
```

- `manual-regression:check:strict` 會額外驗證「截圖/錄屏」路徑是否存在且包含實際檔案。

```bash
npm run manual-regression:summarize
```

- 會輸出：
  - [summary.md](/Users/alex/Desktop/CJ/mother-bear-court/docs/核心開發文件/發版前手動回歸證據/2026-03-17/summary.md)
- 建議在 `P01-P05` 全部回填後執行一次，作為最終人工回歸總覽。

```bash
npm run manual-regression:gate
```

- `manual-regression:gate` 為本地一鍵入口，會依序執行 `check -> check:strict -> summarize`。

```bash
npm run manual-regression:mark -- --date 2026-03-28 --flow P01 --status PASS --owner "QA-A" --time "2026-03-28 20:12" --device "Chrome Desktop" --account "manual-p01-20260328@example.com" --evidence "docs/核心開發文件/發版前手動回歸證據/2026-03-28/P01/P01-desktop-pass-01.png"
```

- `manual-regression:mark` 用於命令行回填單流程欄位（避免手動改 markdown 出錯）。

### 2.2 證據命名與落點（必填口徑）

- 每條流程至少一個可核對的檔案，命名需帶流程 ID：

### 2.3 CI Gate（GitHub Actions）

- Workflow：`Manual Regression Gate`
- 入口：`.github/workflows/manual-regression-gate.yml`
- 用法：
  - 不填 date：檢查最新批次目錄
  - 填 `date=YYYY-MM-DD`：檢查指定批次
- Gate 內容：
  - `manual-regression:check`
  - `manual-regression:check:strict`
  - `manual-regression:summarize`
- 產物：
  - 會上傳 `manual-regression-summary-<date>` artifact

## 4. P01 快速體驗閉環

### 4.1 入口

- 前台：`http://127.0.0.1:4173/quick-experience/create`

### 4.2 操作步驟

1. 以匿名身份建立 quick case。
2. 進入 result 頁，確認至少能看到：
   - waiting / long-wait / result 其中一種可理解狀態
   - `登入`、`註冊`、`再來一次` 等可行出口
3. 點 `登入` 或 `註冊`，完成身份建立。
4. 驗證 `claim-session` 完成後能回到原 `result` 頁。
5. 刷新一次頁面，確認不會掉到錯頁。

### 4.3 預期結果

- 能成功建立 quick case。
- result 頁不出現白屏或死頁。
- 認證後回跳到原 `result` 路徑，而不是 `/case/list` 等錯頁。
- 刷新後仍能看到可理解內容或出口。

## 6. P03 心理訪談閉環

### 6.1 入口

- 前台：
  - `http://127.0.0.1:4173/profile/index`
  - `http://127.0.0.1:4173/profile/pairing`

### 6.2 操作步驟

1. 從 profile 或 pairing 入口啟動訪談。
2. 通過 consent。
3. 完成至少一輪問答。
4. 進入 result 頁，確認 `processing / completed` 路徑可見。
5. 若出現慢處理或失敗態，確認有 `retry / back` 類出口。

### 6.3 預期結果

- 訪談可正常啟動，不會卡在 consent 或 pending 狀態。
- result 頁不會只有錯誤提示而無出口。
- `processing_failed` 或 timeout 時，至少有返回個人資料頁的出口。

## 7. P04 聊天轉判決閉環

### 7.1 入口

- 前台：`http://127.0.0.1:4173/chat/room/<roomId>`

### 7.2 操作步驟

1. 建立或進入 chat room。
2. 驗證 invite/accept 正常，或至少弱入口回訪有可理解提示。
3. 由 owner 發起 `request judgment`。
4. 若被要求登入，完成登入並確認回跳到 judgment detail。
5. 驗證 judgment detail 可正常讀取。

### 7.3 預期結果

- chat room 不應因 session/owner 問題直接白屏。
- invite 失敗時應有可理解錯誤。
- judgment handoff 應正確落到 `/judgment/:id`。
- 未登入時應先去 login，再回跳原目標。
