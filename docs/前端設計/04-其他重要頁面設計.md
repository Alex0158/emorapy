## 1. 範圍與口徑

本文件描述「非核心主流程」但仍承擔關鍵業務責任的頁面，重點是：
- 路由實際可達性（是否受 `ProtectedRoute`/`PublicRoute` 保護）
- 真實互動流程（API 呼叫、狀態更新、跳轉條件、錯誤分支）
- 當前實作細節（而非視覺稿假設）

核心頁面（首頁、快速體驗、判決詳情、訪談 Chat/Result）已在 `03-核心頁面詳細設計.md` 維護，本文件不重複。

## 2. 認證相關頁面

### 2.1 登錄頁 ` /auth/login `

- **路由保護**：由 `PublicRoute` 包裹，已登入用戶不應回到登入頁。
- **重定向白名單**：僅允許導向 `/case`、`/judgment`、`/reconciliation`、`/execution`、`/profile`、`/interview`、`/quick-experience` 或首頁，避免開放重定向。
- **記住我語義**：`rememberMe` 傳入 `authStore.login`，由 store 決定 token 持久化位置。
- **未驗證郵箱分支**：當錯誤碼或訊息命中未驗證情境時，前端會嘗試再次呼叫 `sendVerificationCode(email, "verify_email")`。
- **成功路徑**：登入成功提示後，`navigate(from, { replace: true })`。

### 2.2 註冊頁 ` /auth/register `

- **三步流**：目前是 3 步（填信箱 -> 驗證碼 -> 設密碼），不是 4 步完成頁。
- **驗證碼機制**：6 碼、可貼上、倒數 300 秒、倒數內禁止重發。
- **驗證 API**：`verifyEmail(email, code, "register")` 成功後才能進入密碼步驟。
- **註冊成功跳轉**：`register(...)` 成功後導向 ` /profile/pairing `，未做延遲完成頁。

### 2.3 忘記密碼頁 ` /auth/forgot-password `

- **三步流**：發送重置信 -> 輸入驗證碼 -> 新密碼提交。
- **重發策略**：與註冊頁一致，300 秒倒數控制。
- **成功收斂**：`confirmResetPassword` 成功後顯示完成狀態，2 秒後自動回 ` /auth/login `。

## 3. 案件與審理輔助頁

### 3.1 案件列表頁 ` /case/list `

- **保護路由**：必須登入。
- **資料來源**：`getCaseList`，帶分頁、狀態/類型篩選、排序、搜尋。
- **搜尋節流**：使用 `useDebounce`（500ms），避免每次輸入即打 API。
- **狀態口徑**：篩選含 `draft/submitted/in_progress/completed/judgment_failed/cancelled`，與後端枚舉對齊。
- **交互入口**：支援卡片點擊與按鍵（Enter/Space）進詳情。

### 3.2 案件詳情頁 ` /case/:id `

- **保護路由**：必須登入。
- **角色分支**：
  - 被告且 `draft + remote + 無 defendant_statement` 時，顯示「被告回應」輸入區。
  - 原告或其他狀態則不顯示此區塊。
- **提交路徑**：`submitCase(id)` 成功後進 ` /case/:id/review `。
- **錯誤分流**：對 401/403/404 有專門提示與導流（登入頁或列表頁）。

### 3.3 審理中頁 ` /case/:id/review `

- **輪詢判決**：案件 `submitted/in_progress` 時啟動輪詢 `getJudgmentByCaseId`。
- **結果收斂**：
  - 查到判決 -> 顯示成功提示並可跳 ` /judgment/:judgmentId `。
  - `judgment_failed` -> 顯示失敗原因，提供 `generateJudgment` 重試。
- **防錯導流**：`draft` 狀態訪問此頁時會提醒並返回案件詳情。

## 4. 和好方案頁

### 4.1 方案列表 ` /reconciliation/:judgmentId `

- **保護路由**：必須登入。
- **資料來源**：`getPlans(judgmentId, filters)`；空列表時可直接觸發 `generatePlans`。
- **篩選維度**：`difficulty`（easy/medium/hard）與 `type`（activity/communication/intimacy）。
- **選擇方案**：`selectPlan(planId)` 後直接導向詳情頁。
- **內容解析**：`plan_content` 透過 `safeParsePlanContent` 解析，避免 JSON 格式異常造成崩潰。

### 4.2 方案詳情 ` /reconciliation/:judgmentId/:id `

- **抓取策略**：優先 `getPlanById`，失敗時回退 `getPlans(judgmentId)` 再本地查找。
- **操作分支**：
  - 未被選中：可 `selectPlan`。
  - 已有任一方選中：顯示「開始執行」，呼叫 `confirmExecution`。
- **執行跳轉**：確認執行後進 ` /execution/:planId/checkin `。

## 5. 執行頁

### 5.1 執行看板 ` /execution/dashboard `

- **保護路由**：必須登入。
- **資料來源**：`getAllExecutionStatuses`。
- **分區展示**：將資料拆分為 `in_progress/pending` 與 `completed`。
- **空態導流**：無資料時提供回案件列表入口。
- **錯誤處理**：頁內保留 retry 按鈕，不依賴整頁刷新。

### 5.2 執行打卡 ` /execution/:planId/checkin `

- **保護路由**：必須登入。
- **打卡內容**：`notes` 必填；可選上傳照片（最多 3 張，前端不自動上傳）。
- **照片流程**：
  - 先從方案反查 `case_id`。
  - 再調 `uploadEvidence(caseId, files)` 取得 `file_url`。
  - 成功與否不阻斷打卡主流程（失敗會 warning 後繼續）。
- **提交 API**：`checkin({ plan_id, notes, photos })`。

## 6. 個人中心相關頁

### 6.1 個人資料首頁 ` /profile/index `

- **保護路由**：必須登入。
- **基本資料**：`getProfile` + `updateProfile`，並同步 `authStore.updateUser`。
- **頭像上傳**：走 `fetch /api/v1/user/avatar`（multipart），非透過統一 API wrapper。
- **心理畫像卡片**：
  - 讀取 `psychProfileStore.fetchProfile`。
  - 未同意時顯示 `ConsentModal` 與引導 CTA。
  - 已同意時顯示豐富度環、已探索領域、洞察摘要、繼續訪談入口。

### 6.2 配對頁 ` /profile/pairing `

- **保護路由**：必須登入。
- **主流程**：`createPairing`、`joinPairing`、`cancelPairing`、`getPairingStatus`。
- **邀請碼**：待配對狀態可複製 invite code。
- **訪談引導入口（觸發 A）**：
  - 未同意時顯示 banner，先開 `ConsentModal`。
  - 同意後走 `checkResume`（有 pending 則續聊）否則 `startSession("onboarding")`。

### 6.3 設定頁 ` /profile/settings `

- **保護路由**：必須登入。
- **當前功能聚焦**：通知開關 `notification_enabled` 讀寫。
- **失敗恢復**：初次載入失敗時頁內 `Alert + Retry`，不直接導回其他頁面。

### 6.4 我的故事頁 ` /profile/my-story `

- **保護路由**：必須登入。
- **定位**：心理畫像專頁（豐富度、域別內容、歷史訪談、資料管理）。
- **入口來源**：由 ` /profile/index ` 的相關 CTA 導入。
- **代碼狀態**：路由與頁面已存在並啟用。

## 7. 其他重要入口

### 7.1 聊天室頁 ` /chat/room `、` /chat/room/:roomId `

- 目前在路由層未包 `ProtectedRoute`；實際授權由頁面內與後端 API 再約束。
- 同一路由組件同時處理無 roomId（入口）與指定 roomId（房間詳情）。

### 7.2 前台到後台跳轉 ` /admin/* `

- 前台路由不承載後台頁本體，而是 `AdminRedirect`。
- 透過 `getAdminLoginUrl()` 轉向 admin 應用；無配置時回首頁。
