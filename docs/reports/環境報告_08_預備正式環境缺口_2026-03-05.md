# 報告 08：Staging / Preview 環境缺口報告

> 更新註記（2026-03-28）：
> 已新增 `scripts/smoke-staging.sh` 與 `.github/workflows/staging-smoke.yml`，
> 可用 workflow_dispatch + secrets 方式執行 staging smoke gate。
> 本報告中的「缺口」改為剩餘缺口，不再包含「完全無 staging smoke 入口」。

## 1. 環境定位

- 目標：在正式上線前，用「接近 production」配置做整合驗證與回歸測試。
- 現況：文件層有建議，但尚未形成獨立、可重複執行的制度化環境。

## 2. 目前現況（已存在）

- CI 有多層測試（unit/integration/flow/admin-e2e）。
- 生產配置驗證與保護機制完善（env strict checks、metrics 保護、DB fail-fast）。
- 前後台可獨立 build/run，具備落地 staging 的技術基礎。

## 3. 主要缺口（剩餘）

- 仍缺完整 `staging-deploy`（build -> deploy -> smoke）一體化流水線；目前僅有手動觸發 smoke gate。
- 無 staging 專屬 `DATABASE_URL` / `REDIS_URL` / OpenAI key 隔離策略文件。
- staging smoke 已有入口，但尚未覆蓋更完整場景（例如 claim-session 真 DB 預檢、admin 全鏈路、回退演練）。
- 無 staging 發版前後「準入/回退」標準化清單。

## 4. 建議的最小 staging 契約

- 與 production 同版號 build artifact。
- 獨立 secrets（不可共用 production 密鑰）。
- 必備服務：
  - PostgreSQL（獨立 DB）
  - OpenAI（獨立配額或受限 key）
  - Redis（建議）
- 必開保護：
  - `NODE_ENV=production`
  - `METRICS_TOKEN` 或 `METRICS_ALLOWED_IPS`
  - `ADMIN_JWT_SECRET` 合規

## 5. 推進順序（可直接落地）

1. 建齊 `STAGING_*` secrets 與環境變量模板（與 production 完整隔離）。
2. 將現有 `staging-smoke` workflow 納入 `staging-deploy` 後置 gate。
3. 將 `admin-e2e` 核心檢查與關鍵 flow 擴充到 staging smoke 套件。
4. 補 staging runbook（准入、回退、資料保護、監控）並與 CI 工件關聯。

## 6. 成效預期

- 可顯著降低「CI 綠但 production 行為不一致」風險。
- 能在上線前提前暴露配置/遷移/權限類問題。
- 將環境治理從「人記憶」轉為「流程契約 + 可重複驗證」。
