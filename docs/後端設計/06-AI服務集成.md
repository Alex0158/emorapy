# AI 服務集成（代碼對齊版）

**文檔版本**：v3.0  
**最後更新**：2026-03-05  
**對齊基準**：`backend/src/config/openai.ts`、`backend/src/services/ai.service.ts`、`judgment/interview/chat` 相關 service 與 routes

---

## 1. AI 基礎配置

## 1.1 OpenAI Client

- 檔案：`config/openai.ts`
- `OpenAI({ apiKey, maxRetries: 3, timeout: 60000 })`

## 1.2 三組配置

- `AI_CONFIG`：通用 AI（判決等）
- `INTERVIEW_AI_CONFIG`：訪談對話（較高溫度、較短輸出）
- `ANALYSIS_AI_CONFIG`：敘事/洞察分析（低溫度、長輸出）

## 1.3 主要環境變量

- `OPENAI_API_KEY`
- `OPENAI_MODEL`
- `OPENAI_INTERVIEW_MODEL`
- `OPENAI_ANALYSIS_MODEL`
- `OPENAI_MAX_TOKENS`
- `OPENAI_DAILY_LIMIT`
- `AI_MOCK`

---

## 2. 核心 AI 服務（`AIService`）

## 2.1 通用生成

- `generateText(prompt, options)`
- 內建：
  - quota 預留（每日限額，cache + lock）
  - retry/backoff
  - timeout + abort
  - error 映射（429/401/通用失敗）

## 2.2 業務能力（代表）

- `detectCaseType`
- `analyzeEmotionalDynamics`
- 判決文本生成與結構化處理（由 `judgment.service` 調用）
- 和好方案生成（`reconciliation` 流程調用）

## 2.3 Prompt 安全

- 對用戶輸入使用 `fenceUserInput(...)` 包裹
- system prompt 明確禁止被用戶文本覆寫角色/指令（prompt injection 防護）

---

## 3. AI 與業務流程耦合

## 3.1 判決流程

入口：

- `POST /api/v1/judgments/generate/:id`

鏈路（高層）：

1. `judgment.service` 整理案件與上下文
2. `aiService` 生成內容（含責任比例、摘要等）
3. `normalizeJudgment` 正規化輸出
4. 儲存與返回判決

## 3.2 訪談流程

入口：

- `POST /api/v1/interview/start`
- `POST /api/v1/interview/:id/respond`（SSE）
- `POST /api/v1/interview/:id/skip`（SSE）
- `POST /api/v1/interview/:id/end`

鏈路：

1. `interview.service` 用 `INTERVIEW_AI_CONFIG` 產生對話回覆
2. `end` 後進入 `async-pipeline.service`
3. pipeline 內調用 domain/narrative/insight/richness/feedback 相關 AI 步驟

## 3.3 聊天室 AI 協調

入口：

- 聊天訊息路徑（`/api/v1/chat/...`）觸發 orchestrator 邏輯

鏈路：

1. `chat-ai-orchestrator.service` 監聽訊息與狀態
2. 先走 `safetyRoutingService` 分流
3. 再調 `aiService.generateText` 產生 `ai_reflection` 或 `ai_mediation`

---

## 4. 安全分流與修復

## 4.1 Safety Routing

- `safety-routing.service`
- 輸出路由：
  - `standard`
  - `safety_support`（控制/威脅/暴力）
  - `crisis_support`（自傷/自殺）

## 4.2 聯盟破裂修復

入口：

- `POST /api/v1/judgments/:id/repair`

服務：

- `rupture-repair.service`
- 根據反饋判定修復類型：
  - `validation`
  - `apology_tone_fix`
  - `strategy_reset`

## 4.3 臨床品質指標

入口：

- `POST /api/v1/judgments/:id/metrics`

服務：

- `clinical-quality.service`
- 先寫入 cache 聚合（避免阻塞主流程）

---

## 5. 可靠性與成本控制

1. 日配額限制（`OPENAI_DAILY_LIMIT`）  
2. 失敗重試與回退（含 quota rollback）  
3. cache 命中（如案件類型）  
4. 多處分散鎖避免併發重入（lockService）  
5. `AI_MOCK` 支持本地/測試降級  

---

## 6. 目前邊界（務實）

- 尚非多供應商抽象層（以 OpenAI 為主）
- 臨床品質目前為 cache 聚合，非完整 BI pipeline
- 風險分流已接入判決/聊天，但外部人工介入流程仍主要靠運維規範而非全自動編排
