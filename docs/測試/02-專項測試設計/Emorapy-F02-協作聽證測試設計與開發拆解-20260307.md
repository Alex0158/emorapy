# Emorapy F02 協作聽證測試設計與開發拆解

日期：2026-03-07  
對應功能：`F02 協作聽證（同設備雙人）`  
對應流程：`P01 快速體驗閉環`（跨功能承接 F01 Result）

## 1. 本次測試目標

F02 的目標不是單純「A/B 各提一次文字」，而是要把同設備輪流陳述這條鏈路做成穩定、可回歸、能暴露缺陷的跨層流程。

本輪優先驗證的業務規則：

1. A 方第一次提交時，應建立 `collaborative` 模式 `draft` 案件，並把 case 綁到 session。
2. A 方與 B 方的最小字數門檻不同：
   - A 方 `>=30`
   - B 方 `>=10`
3. B 方第二次提交時，必須滿足：
   - `case_id` 存在
   - `sessionId` 存在且匹配
   - 案件仍是 `draft`
4. B 方提交成功後，案件應進入 `submitted`，並導向 `F01 Result` 頁。
5. F02 自身不負責拉判決；提交成功後的判決讀取、session continuity、result 顯示屬於 F01 承接責任。

## 2. 代碼真實規則（以實際代碼為準）

### 2.1 功能/頁面/API 基線

依據：

1. `docs/核心開發文件/功能特性清單.md`
2. `docs/核心開發文件/接口描述/03-case.md`
3. `docs/核心開發文件/接口-功能-頁面-Mapping.md`

當前 F02 主鏈路：

1. 訪客進入 `/quick-experience/collaborative`
2. A 方輸入並呼叫 `POST /api/v1/cases/collaborative`
3. 頁面進入 `handoff`
4. B 方輸入並再次呼叫 `POST /api/v1/cases/collaborative`
5. 成功後導向 `/quick-experience/result/:id`

### 2.2 前端 Collaborative 頁真實規則

核心位於：

1. `frontend/src/pages/QuickExperience/Collaborative/index.tsx`
2. `frontend/src/services/api/case.ts`
3. `frontend/src/utils/storage.ts`

可確認規則：

1. `phase` 僅有五種：`intro`、`role_a`、`handoff`、`role_b`、`submitting`
2. A 方提交成功後：
   - 會保存 `caseId`
   - 會保存 `sessionId`
   - 會 `sessionStorage.set(result.session_id)`
   - 進入 `handoff`
3. B 方提交成功後：
   - 會以既有 `caseId + sessionId` 再次送出
   - 會 `caseSessionMap.set(result.case.id, result.session_id)`
   - 會 `navigate(/quick-experience/result/:id)`
4. B 方提交失敗時，頁面會退回 `role_b`

### 2.3 後端 collaborative case 真實規則

核心位於：

1. `backend/src/services/case.service.ts`
2. `backend/src/controllers/case.controller.ts`
3. `backend/src/routes/case.routes.ts`

可確認規則：

1. Phase 1：
   - 若無有效 session，後端會新建 session
   - 驗證 `plaintiff_statement >= 30`
   - 建立 `mode=collaborative`、`status=draft` 案件
   - 把 case 綁回 session
   - 返回 `phase='a_done'`
2. Phase 2：
   - 必須帶 `case_id + defendant_statement`
   - 案件必須存在且 `mode=collaborative`
   - `sessionId` 必須匹配案件 session
   - session 不能過期
   - 案件必須仍是 `draft`
   - 驗證 `defendant_statement >= 10`
   - 成功後更新為 `status=submitted`
   - 返回 `phase='submitted'`
3. controller 在 `phase='submitted'` 時會異步觸發 `generateJudgment`，但 judgment lifecycle 本身由 F04 承接
