# 對外發布拓撲與上線方案（2026-03-05）

## 1) 先講結論：對外正式使用就走這套

你們目前最適合、且與現有 repo 最對齊的對外發布拓撲是：

- **前台（給一般使用者）**：`frontend` -> Vercel
- **管理台（給內部/運維）**：`frontend-admin` -> Vercel（獨立專案、獨立網域）
- **後端 API**：`backend` -> Railway
- **資料庫**：PostgreSQL（Railway PG / Supabase 均可）
- **Redis（建議）**：Upstash 或 Redis Cloud（給分散鎖/告警去重/metrics）

這是一套「前後端分離 + 管理台隔離」的 production 拓撲，符合你們現有程式碼與文件。

---

## 2) 為什麼這套最適合你們現在

- 你們已經有 Railway 配置：`railway.json`、`backend/railway.toml`
- 你們已經有 Vercel 配置：`vercel.json`（目前偏向 admin 輸出）
- 後端 production 防護已就位（`backend/src/config/env.ts`）：
  - DB 未就緒 fail-fast
  - `ADMIN_JWT_SECRET` 生產強制
  - `/metrics` 生產保護（token/IP allowlist）
- CI 已完整（lint/test/build/admin-e2e），雖是 CI 非 CD，但已具備上線前品質門檻

---

## 3) 建議的正式網域拆分

- `app.<你的主域名>` -> 前台（`frontend`）
- `admin.<你的主域名>` -> 管理台（`frontend-admin`）
- `api.<你的主域名>` -> 後端（`backend`）

後端 CORS 對應：

- `ALLOWED_ORIGINS=https://app.<域名>,https://admin.<域名>`
- `FRONTEND_URL=https://app.<域名>`

---

## 4) 各服務必填環境變數（production）

## 4.1 Backend（Railway）

最少必填（缺一不可）：

- `NODE_ENV=production`
- `PORT=3001`（或平台實際分配）
- `DATABASE_URL=postgresql://...`
- `JWT_SECRET=<至少32字元強隨機>`
- `ADMIN_JWT_SECRET=<至少32字元強隨機，且不可等於 JWT_SECRET>`
- `JWT_EXPIRES_IN=7d`
- `OPENAI_API_KEY=sk-...`
- `FRONTEND_URL=https://app.<域名>`
- `ALLOWED_ORIGINS=https://app.<域名>,https://admin.<域名>`

強烈建議：

- `REDIS_URL=redis://...`
- `METRICS_ENABLED=true`
- `METRICS_TOKEN=<強隨機>`（或改用 `METRICS_ALLOWED_IPS`）
- `ENABLE_SCHEDULED_JOBS=true`

可選（運維）：

- `ALERT_SLACK_WEBHOOK_URL=...`
- `ALERT_SLACK_DEDUP_WINDOW_SECONDS=600`

## 4.2 Frontend（Vercel，給一般使用者）

- `VITE_API_BASE_URL=https://api.<域名>/api/v1`
- `VITE_ADMIN_LOGIN_URL=https://admin.<域名>/admin/login`

## 4.3 Frontend-Admin（Vercel，給管理台）

- `VITE_API_BASE_URL=https://api.<域名>/api/v1`
- `VITE_ADMIN_LOGIN_URL=https://admin.<域名>/admin/login`

---

## 5) 上線順序（建議照這個執行）

1. **先上 Backend（Railway）**
   - 確認環境變數全填
   - 執行 migration：`npx prisma migrate deploy`
   - 驗證：`/health`、`/health/ready`、`/health/live`
2. **再上 Frontend（Vercel）**
   - 指向新 API 網域
   - 開站驗證主流程（Session -> 案件 -> 判決）
3. **最後上 Frontend-Admin（Vercel）**
   - 驗證 admin login、權限頁、報表/jobs
4. **上線後驗證**
   - `scripts/verify-deployment.sh`
   - `scripts/test-critical-paths.sh`
   - `scripts/health-check.sh`

---

## 6) 發版/回滾策略

發版前：

- backend：`npm run lint && npm run test:unit && npm run build:prod`
- frontend：`npm run lint && npm run test:run && npm run build`
- frontend-admin：`npm run lint && npm run build`
- 可加跑：CI `admin-e2e`

回滾：

- Vercel：Promote 上一個 stable deployment
- Railway：回退到上一個成功 deployment 或前一版 commit
- DB：遷移要先確認可回退策略（必要時 `prisma migrate resolve`）

---

## 7) 目前需要你們立即注意的兩個部署風險

- `vercel.json` 現在輸出是 `frontend-admin/dist`，代表若只建「一個」Vercel 專案，通常只會發 admin，不是主前台。
- `railway.json` 與 `backend/railway.toml` 的 build 命令口徑不完全一致（`build:prod` vs `build`），建議統一到同一策略，避免部署行為漂移。

---

## 8) 你們現在可以直接採用的最小落地版本（MVP）

- **Production 先一套**：`app/admin/api` 三子網域 + 一套 production secrets
- **Staging 下一步補上**：等 production 穩定後再加 `staging`（同配置、不同憑證與資料）

這樣最符合你們目前代碼與文件成熟度，也能最快對外安全提供服務。
