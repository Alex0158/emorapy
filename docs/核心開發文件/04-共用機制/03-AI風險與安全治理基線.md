## 1. 定位

本文把 CJ 的 AI / LLM 風險治理從「共用 NFR 的一部分」拆成可追溯規格。它參考 NIST AI RMF 1.0、NIST AI RMF Generative AI Profile、ISO/IEC 42001:2023 與 OWASP Top 10 for LLM Applications，但只作工程治理對標，不宣稱 CJ 已取得 AI 管理體系、合規或安全認證。

本文屬共用機制級。Web、App、Backend、Admin 只要新增或修改 AI runtime、prompt、AI stream、AI output downstream action、AI cost/ledger 或用戶可見 AI 定位，都必須回查本文。

## 2. 外部基線採用口徑

| 外部基線 | CJ 採用方式 | 不採用 / 不宣稱 |
| --- | --- | --- |
| NIST AI RMF 1.0 | 用 Govern / Map / Measure / Manage 檢查 AI 風險是否有 owner、場景、量測、處置 | 不宣稱 NIST 評估成熟度 |
| NIST AI RMF Generative AI Profile | 用於補強生成式 AI 特有風險：不實輸出、prompt injection、資料外洩、濫用與監控 | 不把 profile 當作法律或醫療安全認證 |
| ISO/IEC 42001:2023 | 用於要求 AI 能力有資產清單、責任邊界、變更管理與持續改善 | 不建立完整 AIMS 管理體系模板 |
| OWASP Top 10 for LLM Applications | 用於映射 prompt injection、sensitive information disclosure、improper output handling、overreliance 等技術風險 | 不宣稱已完成 LLM penetration test |

## 4. AI 風險控制矩陣

| 風險 ID | 對標 | 風險描述 | 現碼控制 | 最小驗證 | 狀態 |
| --- | --- | --- | --- | --- | --- |
| CJ-AI-RISK-001 | OWASP LLM prompt injection | 用戶輸入試圖覆寫 system prompt 或指示模型洩露內部規則 | `fenceUserInput()` 去除危險標籤並用 `<user_input>` 包裹；主 prompt 要求只把用戶內容視為資料 | `backend/tests/unit/utils/prompt.test.ts`；抽查所有新 AI prompt 是否使用 fencing 或明確免除理由 | 部分覆蓋 |
| CJ-AI-RISK-002 | OWASP LLM sensitive information disclosure | prompt、ledger 或 metadata 洩露用戶原文、系統提示或敏感上下文 | `AIRequestLedgerService` 不保存 prompt 原文，只保存 prompt_chars、scope、model、prompt_version、token、status、failure | `backend/tests/unit/services/ai-request-ledger.service.test.ts`；Admin report 不得展示 prompt 原文 | 已有核心控制 |

| CJ-AI-RISK-004 | Improper output handling | AI output 被直接下游執行、建案、邀請、通知、共同修復或責任比例展示 | `product-safety-policy.ts`、`repair-eligibility.service.ts`、chat request judgment safety gate | `backend/tests/unit/services/safety-routing.service.test.ts`、chat-to-case 回歸 | 部分覆蓋 |
| CJ-AI-RISK-005 | Safety / crisis misrouting | 自傷、暴力、控制或高風險關係被當普通衝突處理 | `SafetyRoutingService`、`ai-safety-signals.ts`、`RelationshipRiskState`、責任比例 visibility gate | IPV / crisis 單測與手動高風險回歸 | 已有核心控制；召回率未量化 |
| CJ-AI-RISK-006 | AI reliability / recovery | stream 斷線、provider timeout、retry 或持久化失敗導致永久卡死 | AI stream snapshot / replay / timeout / persisted events、ledger failure reason | AI stream 單測、手動回歸、Admin stream report | 部分覆蓋 |

## 5. 強制規則

1. 新增 AI runtime 必須登記 `CJ-AI-ASSET-*` 或在本文補新 asset；不得只在 service 內新增 prompt。
2. 新增或修改 AI prompt，必須同步 `AI_PROMPT_VERSIONS` 或說明為何不需要版本化；用戶輸入必須經 `fenceUserInput()` 或等效防 injection 包裹。
3. AI output 不能直接觸發伴侶邀請、共同修復、通知召回、責任比例展示或正式 case 狀態變更；必須先過 safety policy / service gate。
4. AI ledger 不得保存 prompt 原文、完整聊天文本或可直接重建私密敘事的 metadata。
5. 對外文案與 AI prompt 不得把 CJ / 模型描述為治療師、心理診斷、法律裁決或危機救援服務；`npm run ai:positioning:check` 會阻止 Web i18n 與 prompt 自我定位回流。安全 referral 可以建議尋求持牌專業支持，但語義必須是外部支持，不是 AI 自己扮演該角色。
6. App 新增 AI 能力前，必須先補 App lifecycle、SecureStore、stream reconnect、Push / Deep Link 和 evidence smoke；不得借 Web 測試宣稱 App AI 已覆蓋。

## 6. 缺口與後續治理

| 缺口 ID | 現狀 | 風險 | 處置規則 |
| --- | --- | --- | --- |
| CJ-AI-GAP-001 | AI 資產已分散在多個 service，過去缺少統一 inventory | 新增 AI 能力後 ledger / prompt version / safety gate 容易漏接 | 本文作為 AI asset inventory；新增 AI runtime 必須回寫 |
| CJ-AI-GAP-002 | prompt injection 有 fencing，但沒有全倉 prompt 覆蓋率守衛 | 新 AI prompt 可能繞過 fencing | 新增 AI prompt 時 PR 自檢必須查 `fenceUserInput` 或明確例外 |
| CJ-AI-GAP-003 | safety routing 以關鍵字與模型分析結果為主，缺少量化召回率 / 誤判率 | 高風險場景品質難以量化 | 指標文檔維持待建立基線，不得宣稱安全識別率 |
| CJ-AI-GAP-004 | production AI pricing / DB parity 仍需真環境證據 | 成本與 ledger 發布口徑可能只停留在 dev | 發布前由 release gate / evidence wrapper 留存 |
| CJ-AI-GAP-005 | App AI runtime 已建立本地 gate，但缺 release 級真機 / Push return evidence | 跨端文件容易把本地 App gate 誤導為 TestFlight / 真機完整能力 | App 條目保持部分覆蓋；strict release sign-off 前不得宣稱完整 |

## 7. 驗收口徑

1. 自動測試：prompt fencing、prompt version、AI ledger、safety routing、chat-to-case safety gate 必須有單測或集成測試入口。
2. 分析驗證：新增 AI flow 必須能追到 `request_id / stream_id / scope_type / product_flow / prompt_version / status`。
3. 檢查驗證：PRD / NFR / RTM / 接口文檔必須標註 AI output 的 downstream gate。
4. 演示證據：高風險關係、危機、stream 斷線、provider timeout、Admin report 至少需在回歸或 release evidence 中留入口。
5. 未量化的效果、召回率、安全識別率與成本節省，全部標為待建立基線。
