## 分層規則

後續維護 `核心開發文件/` 時，統一遵循以下分層規則：

1. 根層只放少量旗艦總覽文檔。
2. 細規格一律進對應正式子域。
3. 測試高位規範一律進 `08-測試規範與驗收/`；活躍案例、回歸包與執行入口進 `測試/`。
4. 活躍問題與治理狀態一律進 `07-待處理問題與治理/`。
5. 證據鏈、環境核驗、頁面快照與手動回歸實體一律進 `90-證據與盤點/`。
6. dated 驗收記錄不得兼任正式規格；若內含穩定真相，應回寫到正式子域後再保留為證據或歷史。
7. 歷史降級與不再作主依據的材料一律進 `99-歷史降級索引/`。
8. 變更追蹤、同步規則與遷移過程一律記在 `文件收斂/`，不在正式正文尾部堆疊「本輪更新」。
9. 營銷、品牌、對外敘事與投放素材一律不進本目錄，統一留在 `docs/核心營銷文件/`。
10. `00`、`01-05`、`10`、`20`、`50` 這些正式規格子域都應維持 `README.md + 00-*總覽.md` 的雙層入口；README 負責分工與跳轉，`00-*總覽.md` 負責穩定正文。

## Web / App 分層規則

從 App 版開發開始，核心文件採用三層 SSOT：

1. **跨端產品核心**：由 `00-跨端產品核心/` 承接，定義 Web 與 App 共用的產品語義、角色、流程、狀態與授權邊界。
2. **平台投影文件**：由 `10-Web端/` 與 `20-App端/` 承接，只描述該端如何實作或承接跨端核心，不得自行改寫產品規則。
3. **跨端映射與缺口**：由 `50-跨端Mapping與Parity/` 承接，追蹤 Web / App / Backend / API / DB / 共享層的一致性、差異與待處理任務。

Web 版本若被定格，`10-Web端/` 是「已實作基線」；App 版本開發時應以 `00-跨端產品核心/` 為最高產品裁決，以 `20-App端/01-App導航與平台Adapter基線.md` 管理 App navigation / native adapter 落地，以 `50-跨端Mapping與Parity/01-App首輪能力與工程落點Mapping.md` 管理 Web / App / Backend / API / DB / shared 差異，以 `08-測試規範與驗收/03-App測試與證據接入基線.md` 管理 App smoke / regression / CI / evidence 進場，不應把 Web UI 現狀或 Web/Admin 測試通過直接複製成 App 產品規則或 App 驗收結論。

## 維護口徑

1. 新增文檔前，先判定自己屬於正式規格、測試規範、活躍測試、證據、治理還是歷史材料，再決定目錄。
2. 根層不新增臨時排查報告、發版回歸記錄、營銷分析、環境收口稿或單次驗收結論。
3. 若現行正文與證據、歷史或治理材料衝突，以現碼、`00-跨端產品核心/`、根層旗艦文檔、平台投影子域與對應正式子域正文為準。
4. 若需要回溯調整過程與原因，查 `文件收斂/` 或 `99-歷史降級索引/`，不要把過程寫回正式正文。
