本文件承接 `CJ` repo 的工程分層、共享契約與跨端代碼落點規則。這裡的 `apps / services / packages` 同時是分類心智與現有物理樹的解讀方式，不等於現有目錄已全部進入同一 workspace 體系。

## 1. 本子域回答什麼

本子域只回答以下問題：

1. `frontend/`、`frontend-admin/`、`mobile/`、`backend/`、`packages/` 在 repo 中各自承擔什麼責任。
2. 共享契約、API client、純邏輯應下沉到哪裡，哪些內容不能下沉。
3. 修改共享層時，應同時考慮哪些 app/service 的影響面。

本子域不回答：

1. 某個功能流是否存在、由哪個頁面承接。
2. 某個接口是否已使用或候選廢棄。
3. 發版證據、單次驗收、排查快照。

## 2. 當前 repo 分層

`CJ` 是單一 monorepo，以 repo 根目錄為 git 邊界。當前正式分層如下：


| 目錄                | 角色       | 說明                              |
| ----------------- | -------- | ------------------------------- |
| `frontend/`       | 主 Web 端  | 面向普通用戶的 Web 應用                  |
| `frontend-admin/` | 管理端 Web  | 後台管理與治理界面                       |
| `mobile/`         | Expo App | 單一 codebase 支援 iOS / Android    |
| `backend/`        | API 服務   | 業務路由、服務、排程與運維節點                 |
| `packages/`       | 共享層      | contracts、api-client 與後續純邏輯下沉位置 |


## 3. 共享層落點規則

應放入 `packages/*` 的內容：

1. DTO、enum、response shape、error code。
2. 純 HTTP / SSE / upload transport。
3. 純函式商業邏輯、normalize、permission、query key、state machine。
4. 設計 token、常數、文案 key。

不應放入 `packages/*` 的內容：

1. React DOM / React Native UI 元件。
2. `react-router-dom`、Expo Router、導航副作用。
3. `localStorage`、`sessionStorage`、`SecureStore` 等平台儲存實作。
4. 相機、推播、檔案選擇、deep link、背景任務。
5. 任何直接依賴瀏覽器或原生執行環境的邏輯。

## 4. Web / Admin / Mobile / Backend 邊界

### 4.1 `frontend/`

承擔：

1. 主站 Web UI 與路由。
2. 瀏覽器環境下的 storage、導頁、頁面組裝。
3. 用戶端產品流程的具體頁面承接。

### 4.2 `frontend-admin/`

承擔：

1. 後台 UI、Admin token 守衛。
2. jobs、configs、reports、audit、health 等管理頁面。
3. 後台特有的操作流程與治理界面。

### 4.3 `mobile/`

承擔：

1. React Native / Expo UI。
2. 原生 storage、權限、相機、推播、deep link。
3. 與 Web 同一產品域在移動端的承接。

### 4.4 `backend/`

承擔：

1. API、SSE、jobs、健康探針、metrics。
2. 鑑權、限流、狀態機與持久化。
3. 對多端共享契約的實際供給。

## 5. 目前最重要的工程現實

以下是當前 repo 的正式工程現實，不應照搬理想化 monorepo 假設：

1. root npm workspaces 目前只有 `frontend/` 與 `frontend-admin/`。
2. `backend/`、`mobile/`、`packages/*` 都是 repo-local 目錄，各自維持獨立 `package.json` / `tsconfig` 或純源碼存在。
3. `frontend/` 已用 tsconfig alias 接入 `@cj/contracts`；`backend/` 與 `mobile/` 只完成 alias 預留，尚未形成穩定共享消費面。
4. `frontend-admin/` 目前只有 `@/*` alias，尚未接入 `@cj/contracts` 或 `@cj/api-client`。
5. `packages/contracts` 與 `packages/api-client` 是已建立的共享層入口，但不代表所有類型與 request 邏輯都已完全下沉，也不代表 `packages/api-client` 已被現行 app 穩定消費。
6. `backend` 仍有本地類型與 `rootDir` 限制，不能假設它已完全以共享原始碼作唯一來源。

## 6. 共享改動的影響規則

修改共享層時，至少要做以下影響判斷：

1. 改 `packages/contracts`，先檢查 `frontend`、`backend`、`mobile` 的 alias 消費面；若改動影響 Admin 對應型別，再額外檢查 `frontend-admin` 的本地型別與 API 適配。
2. 改 `packages/api-client`，先檢查其是否仍只是共享入口，還是已被某端實際消費；不要假設所有端都已接線。
3. 改產品流程頁，不要順手把 UI、storage、router 副作用下沉進共享包。
4. 新增純邏輯時，先判斷它是否真能脫離平台運行環境，再決定是否下沉。
