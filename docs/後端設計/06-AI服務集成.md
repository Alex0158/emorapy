## 1. AI 基礎配置

- 檔案：`config/openai.ts`

## 1.2 三組配置

- `INTERVIEW_AI_CONFIG`：訪談對話（較高溫度、較短輸出）

## 1.3 主要環境變量

- `OPENAI_API_KEY`
- `OPENAI_MODEL`
- `OPENAI_INTERVIEW_MODEL`
- `OPENAI_ANALYSIS_MODEL`
- `OPENAI_MAX_TOKENS`
- `OPENAI_DAILY_LIMIT`
- `AI_MOCK`

## 2. 核心 AI 服務（`AIService`）

## 2.1 通用生成

- `generateText(prompt, options)`
- 內建：
  - quota 預留（每日限額，cache + lock）
  - retry/backoff
  - timeout + abort
  - error 映射（429/401/通用失敗）

## 2.2 業務能力（代表）

- `detectCaseType`
- `analyzeEmotionalDynamics`
- 判決文本生成與結構化處理（由 `judgment.service` 調用）
- 和好方案生成（`reconciliation` 流程調用）

## 2.3 Prompt 安全

- 對用戶輸入使用 `fenceUserInput(...)` 包裹
- system prompt 明確禁止被用戶文本覆寫角色/指令（prompt injection 防護）

## 2.4 Repair Journey 2.0 補充

- `aiService.generateReconciliationPlans(...)`
  - 現已顯式接收 `intent`
  - 也會接收由前端偏好整理出的 `preferenceSummary`
- 和好方案輸出不再只有基本文案與成本欄位，還包括：
  - `fit_reason`
  - `do_not_use_when`
  - `first_step`
  - `fallback_step`
  - `pause_rule`
  - `risk_note`
- `reconciliation.service`
  - 會把 AI 輸出保存為 `reconciliation_plans + repair_tracks + repair_participant_states`
  - 並依偏好算出 `recommended_plan_id`

## 3. AI 與業務流程耦合

## 3.1 判決流程

入口：

- `POST /api/v1/judgments/generate/:id`

鏈路（高層）：

1. `judgment.service` 整理案件與上下文
2. `aiService` 生成內容（含責任比例、摘要等）
3. 透過 `aiStreamService` 發送 phase：
   - `collecting_context`
   - `analyzing_emotion`
   - `building_responsibility`
   - `drafting_judgment`
   - `finalizing`
4. `normalizeJudgment` 正規化輸出
5. 儲存後發送 `stream.completed` + `stream.persisted`

## 3.2 訪談流程

入口：

- `POST /api/v1/interview/start`
- `POST /api/v1/interview/:id/respond`
- `POST /api/v1/interview/:id/skip`
- `POST /api/v1/interview/:id/cancel`
- `POST /api/v1/interview/:id/end`

鏈路：

1. `interview.service` 用 `INTERVIEW_AI_CONFIG` 產生對話回覆
2. `end` 後進入 `async-pipeline.service`
3. pipeline 內調用 domain/narrative/insight/richness/feedback 相關 AI 步驟
4. `respond/skip` 採 `202 accepted` 提交式觸發，AI 可見輸出統一由 `GET /api/v1/streams/interview_session/:id` 提供
5. 客戶端主動取消或中途斷開時，`interview.service` 會發送標準 `stream.cancelled`
6. `GET /api/v1/streams/interview_session/:id` 提供統一 AI stream snapshot / replay 讀取入口

## 3.3 聊天室 AI 協調

入口：

- 聊天訊息路徑（`/api/v1/chat/...`）觸發 orchestrator 邏輯

鏈路：

1. `chat-ai-orchestrator.service` 監聽訊息與狀態
2. 先走 `safetyRoutingService` 分流（crisis_support / safety_support / standard）
3. 依 `hasRoleB` 與 `detectValidationSeeking` 選擇 system prompt：
   - 有 B：`MEDIATION_SYSTEM_PROMPT`（調解）
   - 無 B 且 validation-seeking：`SUPPORT_VALIDATION_SEEKING_PROMPT`（價值澄清）
   - 無 B 且一般：`SUPPORT_SYSTEM_PROMPT`（情感支持 + 價值澄清指引）
4. 以 `fenceUserInput('chat_context', ...)` 包裹對話脈絡，防 prompt injection
5. 調 `aiService.generateTextStream` 產生 `ai_reflection` 或 `ai_mediation`
6. 同步透過 `aiStreamService` 發送 `stream.started/delta/completed/persisted`
7. 房間事件流保留 `message/room_status/invite` 等 domain event；Chat AI 的文字增量、完成與 persisted 交接以 `aiStreamService` 對應的 `stream.*` 為主鏈路

## 3.4 統一 AI Stream Runtime

- 核心服務：`backend/src/services/ai-stream.service.ts`
- 指標服務：`backend/src/services/ai-stream-metrics.service.ts`
- 統一端點：`GET /api/v1/streams/:scopeType/:scopeId`
- 當前 scope：
  - `case_judgment`
  - `chat_room`
  - `interview_session`
- 核心能力：
  - 事件序號 `seq`
  - snapshot 聚合
  - `after_seq` replay
  - 對外標準 `stream.heartbeat`
  - `stream.completed` 與 `stream.persisted` 分離，支持前端草稿氣泡與正式實體交接
  - `REDIS_URL` 存在時自動啟用 Redis runtime：
    - 用 Redis `INCR` 作為 scope 級權威 `seq`
    - 用 Pub/Sub 做跨實例事件分發
    - 用短期 replay buffer + snapshot hash 支持重連與實例重啟後恢復
    - 無 Redis 時自動降級為單機內存 runtime，不阻塞開發環境
  - Prometheus 指標：
    - `ai_stream_events_total`
    - `ai_stream_terminal_total`
    - `ai_stream_time_to_first_delta_ms`
    - `ai_stream_complete_to_persist_ms`

相關測試：`backend/tests/unit/services/chat-ai-orchestrator.service.test.ts`（detectValidationSeeking、prompt 選擇）

## 3.5 與前端共享契約的對齊要求

- 後端 `stream.*` 事件必須支持前端共享 `AIStreamDraft` 契約的三態：
  - `thinking`
  - `streaming`
  - `persisting`
- `stream.completed` 只代表模型輸出完成，不代表可清除前端 draft。
- `stream.persisted` 才是前端把暫存氣泡交接為正式 message / judgment 實體的唯一完成信號。
- `Interview`、`Chat Room`、`QuickExperience` 的後端可見流輸出都已收斂到同一 `AI Stream` runtime。
- `AI Stream` 現已同時落地兩層持久化：
  - `ai_stream_sessions`：每條 stream 的權威狀態、最後 seq、最後文本、terminal 時間點
  - `ai_stream_events`：非 heartbeat 事件審計記錄，供排障與驗收
- `AI Stream` 治理能力已新增：
  - `GET /api/v1/admin/reports/ai-streams`：按時間窗口匯總 sessions / events / active / failures 的運維報表
  - `GET /api/v1/admin/reports/ai-streams/sessions`：按 source/status/scope 查 session 明細
  - `GET /api/v1/admin/reports/ai-streams/sessions/:streamId`：查看單條 stream 的 session + events 詳情
  - `cleanup_ai_stream_persistence`：每日 archive-before-delete 清理過期 `ai_stream_events` 與 terminal `ai_stream_sessions`
  - `ai_stream_session_archives / ai_stream_event_archives`：長期歸檔層，用於保留已清理 source 記錄的歷史審計資料
- 已補真實 Redis 多實例 integration test：`backend/tests/integration/ai-stream-redis.integration.test.ts`
- 可透過 `TEST_REDIS_URL=redis://... npm run test:integration:ai-stream-redis` 驗證跨節點分發、Redis replay 與 snapshot 恢復
- 已於 2026-04-04 在本機 Docker/Colima 啟動 `redis:7-alpine`，用 `TEST_REDIS_URL=redis://127.0.0.1:6380 npm run test:integration:ai-stream-redis -- --detectOpenHandles` 跑通正式實跑驗收
- 另據 2026-04-05 環境核對：
  - `production` 已配置 Railway 內網 `REDIS_URL`，`AI Stream` 以 Redis mode 為常態
  - `staging` 已補回 Railway 內網 `REDIS_URL`，並在啟動日誌中確認 `Redis connected for AI Stream runtime`
  - `staging` 已補齊 `METRICS_ENABLED=true` + metrics token 保護，`/metrics` 在無 token 時返回 `# metrics forbidden`，帶合法 token 時返回 Prometheus 文本
  - 當前 `staging` 與 `production` 的主要差異已不在 `AI Stream` / metrics / lock 主鏈路語義，而在域名、流量與部署角色

## 5. 可靠性與成本控制

1. 日配額限制（`OPENAI_DAILY_LIMIT`）  
2. 失敗重試與回退（含 quota rollback）  
3. cache 命中（如案件類型）  
4. 多處分散鎖避免併發重入（lockService）  
5. `AI_MOCK` 支持本地/測試降級

## 6. 目前邊界（務實）

- 尚非多供應商抽象層（以 OpenAI 為主）
- `AI Stream` 已具 Redis 級短期 replay / 跨實例分發，也已落地 `ai_stream_sessions / ai_stream_events` 持久化、archive 層、管理報表與清理任務；真 Redis 多實例實跑已完成，本地與 CI 僅需提供 `TEST_REDIS_URL`
- 臨床品質目前為 cache 聚合，非完整 BI pipeline
- 風險分流已接入判決/聊天，但外部人工介入流程仍主要靠運維規範而非全自動編排
