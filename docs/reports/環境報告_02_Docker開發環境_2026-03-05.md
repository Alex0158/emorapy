# 報告 02：Docker 開發/容器化環境（Docker Development）

## 1. 環境定位

- 目標：以容器方式執行後端，降低本機依賴差異。
- 目前範圍：後端容器化已提供；專案未提供 `docker-compose` 編排檔。

## 2. 主要配置來源

- `backend/Dockerfile`
- `docs/backend/DEPLOYMENT.md`（Docker 部署段落）

## 3. 建置與運行特徵

- 基底：`node:20-alpine`
- 重要步驟：
  - 安裝系統依賴（`openssl`, `libc6-compat`）
  - `npm ci --ignore-scripts`
  - `npx prisma generate`
  - `npm run build:prod`
  - `npm prune --production --no-save prisma @prisma/client`
- 啟動命令：`npm start`
- Dockerfile `EXPOSE`: `3000`

## 4. 必需/常用環境變量

- 必需：
  - `DATABASE_URL`
  - `JWT_SECRET`
  - `OPENAI_API_KEY`
- 常用：
  - `PORT`（建議顯式設置為 `3001` 或你實際映射的端口）
  - `FRONTEND_URL`, `ALLOWED_ORIGINS`
  - `RUN_MIGRATIONS`, `DB_*`, `REDIS_URL`, `METRICS_*`

## 5. 服務依賴（本環境）

- PostgreSQL：外部提供（容器外或另一容器）
- OpenAI：外部 API
- Redis/SMTP/Slack/成本 API：均可選，以 env 開關或憑證控制

## 6. 驗證清單

- 容器啟動後驗證：
  - `GET /health`
  - `GET /health/ready`
  - `GET /metrics`（若啟用）
- 若 `RUN_MIGRATIONS=true`，需確認容器可連上 DB 且 migration 成功。

## 7. 風險與改進

- 文件示例常見 `-p 3001:3001`，但 Dockerfile `EXPOSE 3000`，與 `env.ts` fallback `3000`/`.env.example` 建議 `3001` 存在口徑混用。
- 缺少 `docker-compose`（至少 backend + postgres + redis 的開發編排），容器化落地成本較高。
- 缺少 container `HEALTHCHECK` 指令，CI/CD 或平台探針可觀測性較弱。
