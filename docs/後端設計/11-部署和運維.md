當前部署事實：

| 環境 | Redis | 其他關鍵環境變量 | 當前運行口徑 |
|---|---|---|---|
| `production` | 已配置 Railway Redis + `REDIS_URL` | metrics 保護已開、admin JWT 已配置 | Redis-backed `lock/cache/metrics/AI Stream` |
| `staging` | 已配置 Railway Redis + `REDIS_URL` | `METRICS_ENABLED=true`、`METRICS_TOKEN` 已配置、`ADMIN_JWT_SECRET` / `ADMIN_JWT_EXPIRES_IN` 已配置 | Redis-backed `lock/cache/metrics/AI Stream`；`/health` 已恢復 `healthy` |
| `development` | 推薦本地 Redis | `backend/.env` 建議填 `REDIS_URL=redis://127.0.0.1:6379` | 若未配置 Redis，`AI Stream` / lock 會回退本地模式 |

- `production` 的 `REDIS_URL` 由後端統一復用於 `lock`、`cache`、`chat metrics`、`ops metrics` 與 `AI Stream`
- `staging` 已在 2026-04-05 完成 Redis + metrics 收口，當前與 `production` 的主要差異已不在核心 runtime 形態
- 公開狀態端點 `/health`、`/health/ready`、`/health/live`、`/version`、`/api/v1/version` 已對探活 / CLI 探測放寬；受保護接口仍按 `ALLOWED_ORIGINS` 拒絕跨站來源

### 部署平台

## 🏗️ 部署流程

### 1. 準備工作

```bash
# 1. 確保代碼已提交到Git
git add .
git commit -m "Ready for deployment"
git push origin main

# 2. 運行測試
npm run test

# 3. 構建項目（後端使用 build:prod）
cd backend && npm run build:prod
```

### 2. Railway部署

**步驟**：

1. 在Railway創建新項目
2. 連接GitHub倉庫
3. 配置環境變量
4. 部署

**Redis 補充**：

- Railway `production` / `staging` 都應掛載 Railway Redis，並在 `railway variables` 中可見 `REDIS_URL`
- 當前 staging 已補回 Redis；若未來 staging 因臨時環境需求再撤 Redis，才應回到 `ALLOW_SIMPLE_LOCK=true` 降級口徑
- `production` / `staging` 的 `/metrics` 都應啟用 token 或 allowed IP 保護；它是機器抓取端點，不作為公開狀態端點
- 健康檢查退化時，需先分辨是：
  - `REDIS_URL` 不可達導致的非預期降級
  - 還是某個臨時環境有意識的 simple-lock 降級
- `AI Stream` 啟動日誌若出現 `Redis connected for AI Stream runtime` 代表 Redis mode；若無 `REDIS_URL`，則不應把當前實例視為跨實例等價環境

**Railway配置**：

```json
{
  "build": {
    "builder": "NIXPACKS"
  },
  "deploy": {
    "startCommand": "npm start",
    "restartPolicyType": "ON_FAILURE",
    "restartPolicyMaxRetries": 10
  }
}
```

### 3. Render部署

**步驟**：

1. 在Render創建新Web服務
2. 連接GitHub倉庫
3. 配置構建命令：`npm install && npm run build`
4. 配置啟動命令：`npm start`
5. 配置環境變量
6. 部署

## 🚨 故障處理

### 常見問題

**1. 服務無法啟動**

```bash
# 檢查日誌
railway logs

# 檢查環境變量
railway variables

# 檢查數據庫連接
npx prisma db pull
```

**2. 數據庫連接失敗**

- 檢查DATABASE_URL是否正確
- 檢查數據庫是否運行
- 檢查網絡連接

**3. AI服務錯誤**

- 檢查OPENAI_API_KEY是否有效
- 檢查API限額
- 檢查網絡連接

**4. Redis / AI Stream / Lock 降級**

- 檢查 `REDIS_URL` 是否存在且可連線
- 檢查啟動日誌是否出現 `Redis connected for AI Stream runtime`
- 若出現 `AI Stream Redis unavailable, falling back to in-memory runtime`，代表當前實例已降級到 memory mode
- 若 `REDIS_URL` 本來就未配置，但 `ALLOW_SIMPLE_LOCK=true`，則屬有意識的 lock 降級；`/health` 可能顯示 `Lock backend degraded: simple-lock in production`
- 可從 `GET /api/v1/admin/reports/ai-streams` 的 `data.retentionPolicy.backendMode` 輔助確認 `AI Stream` 模式

**5. 公開狀態端點與 CORS**

- 以下端點應作為公開狀態端點處理：
  - `/health`
  - `/health/ready`
  - `/health/live`
  - `/version`
  - `/api/v1/version`
- 它們供 Railway / CLI / smoke gate 使用，不應因非白名單 `Origin` 被誤攔
- 但 `/api/v1/admin/*`、業務 API、需要權限的 SSE 仍必須嚴格受 `ALLOWED_ORIGINS` 約束

**實現對齊**：後端構建為 `npm run build:prod`；環境變量以 `backend/.env.example` 為準（含 UPLOAD_DIR、FILE_BASE_URL）；完整流程見 [發佈流程指引](../發布指引/發佈流程指引.md)、[ENVIRONMENT](../ENVIRONMENT.md)。

## 🧠 v2.0 部署注意事項

### 新增環境變量

| 變量 | 必需 | 說明 |
|------|------|------|
| `OPENAI_INTERVIEW_MODEL` | 否（默認 gpt-4o-mini） | 訪談對話模型 |
| `OPENAI_ANALYSIS_MODEL` | 否（默認 gpt-4o） | 分析/提取模型 |
| `INTERVIEW_MAX_TURNS` | 否（默認 25） | 訪談最大 turn 數 |
| `INTERVIEW_SOFT_TARGET` | 否（默認 15） | AI 建議結束 turn 數 |
| `INTERVIEW_TURN_INTERVAL_MS` | 否（默認 3000） | turn 最小間隔 |
| `INTERVIEW_START_RATE_LIMIT` | 否（默認 3） | start 端點每用戶每小時上限（express-rate-limit，防濫用，不計 turn 數） |
| `INTERVIEW_DAILY_SESSION_LIMIT` | 否（默認 5） | 每用戶每天最多 substantive session 數（業務邏輯，僅計 ≥ 3 輪） |
| `REDIS_URL` | 否 | 線上 Railway Redis 連接串；同時供 `lock`、`cache`、`chat metrics`、`ops metrics`、`AI Stream` 使用。本地未配置時才 fallback 到 memory / DB advisory lock |

### 數據庫遷移

```bash
npx prisma migrate deploy  # v2.0 新增 5 表 4 ENUM
```

### SSE 與反向代理

若使用 Nginx 反向代理，需禁用統一 `AI Stream` 端點的 buffering。`Interview` 已不再由 `/respond` 直接承載可見 SSE；當前主鏈路是 `GET /api/v1/streams/:scopeType/:scopeId`。建議在 Nginx 層顯式配置：

```nginx
location /api/v1/streams/ {
  proxy_buffering off;
  proxy_cache off;
  proxy_read_timeout 120s;
}
```

### 異步管線部署

異步管線（`AsyncPipelineService`）在 `POST /interview/:id/end` 後由同一進程內的異步任務執行（非獨立 worker queue）。部署注意事項：

- **無需獨立 worker**：管線以 `Promise` 形式在 Express 進程內非阻塞運行；不佔用 HTTP 線程，但會佔用 Node.js event loop
- **進程重啟容錯**：若進程在管線執行期間重啟，session 將停留在 `processing` 狀態。定時任務（每 30 分鐘）掃描 `processing` 超過 10 分鐘的 session 並標記為 `processing_failed`，用戶可從「我的故事」頁面觸發重試
- **水平擴展**：多實例部署時，`session-level mutex`（Redis 或 DB advisory lock）確保同一 session 的管線不會被多個實例同時處理
- **資源預估**：每個 session 的管線執行約 3-5 個 AI calls（GPT-4o-mini × 2 + GPT-4o × 1-2），預計耗時 15-45 秒，峰值記憶體增量 <50MB

### 監控要點

- **AsyncPipelineService**：監控 `processing_failed` session 數量（告警閾值：>5/hour）；監控 `processing` 超 10 分鐘的 session（可能是管線卡住）
- **SSE 超時**：監控 respond 端點的 P99 延遲（告警閾值：>30s）
- **AI 費用**：每月追蹤 OpenAI API 用量（預算：$10-50/月 基於 1000 session）
- **管線步驟耗時**：按 `pipeline_step` 維度追蹤各步驟耗時，識別瓶頸（通常是 step 3 摘要+洞察提取）
