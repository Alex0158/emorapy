本文件承接 `CJ` 的環境矩陣、部署形態、健康探針、metrics 保護與運行前提，作為平台治理層的穩定正文。

## 1. 環境分層

| 環境 | 角色 | 固定要求 |
| --- | --- | --- |
| `development` | 本地開發 | 本地 Postgres、建議啟 Redis、允許手動 smoke |
| `test` | 自動化與一次性驗證 | 以測試命令與 CI / 本機驗證為主，不承接正式發布口徑 |
| `staging` | 發版前收口與 smoke | 應在運維口徑上盡量鏡像 production，尤其是 Redis-backed runtime 與受保護 `/metrics` |
| `production` | 正式服務 | 必須使用正式 secrets、Redis-backed runtime、健康探針與保護後的 metrics |

## 2. 後端啟動前提

後端正式基線只承認以下前提：

1. 必填環境變量：`DATABASE_URL`、`JWT_SECRET`、`OPENAI_API_KEY`
2. `ADMIN_JWT_SECRET` 與 `ADMIN_JWT_EXPIRES_IN` 在 production 由代碼強制；staging 應比照 production 配置，不應依賴 development / test 的回退語義
3. `REDIS_URL` 是 production / staging 的正式 runtime 前提；本地若未提供，只能視為降級語義
4. production 啟用 `METRICS_ENABLED=true` 時，必須同時配置 `METRICS_TOKEN` 或 `METRICS_ALLOWED_IPS`；staging 應沿用同一保護口徑，但路由層目前只在 production 硬性攔截
5. 前後台拆分部署時，`ALLOWED_ORIGINS` 需同時包含主站與管理端來源
6. development 會把 `ALLOWED_ORIGINS` 與本地預設來源自動合併（`localhost/127.0.0.1` 的 `4173-4175`、`5173-5175`），避免平台注入正式白名單時阻斷本機 smoke
7. `AI_COST_PRICING_JSON` 是 AI request ledger 成本歸因配置；runtime 未配置時 ledger 仍記 request / token 且 `cost_usd=null`，但 release gate 會以 `ops:ai-pricing:check` 要求發布環境顯式配置；`ops:release:gate` 會設定 `CJ_RELEASE_GATE=1`，禁止 pricing 檢查回讀本機 `backend/.env`
8. `AI_COST_PRICING_JSON` 必須顯式包含 `source / version / models` 與每個模型 input / output USD per 1M token；`version` 必須以 `YYYY-MM-DD` 開頭、不能是未來日期，且預設不得超過 `AI_COST_PRICING_MAX_AGE_DAYS=30`；release gate 會檢查 `OPENAI_MODEL / OPENAI_INTERVIEW_MODEL / OPENAI_ANALYSIS_MODEL` 及 `AI_COST_REQUIRED_MODELS` 是否全被覆蓋，不得在代碼中硬編模型價格，也不得把未審核舊價格當作默認值

## 3. 公開狀態端點與保護端點

公開狀態端點：

1. `GET /health`
2. `GET /health/ready`
3. `GET /health/live`
4. `GET /version`
5. `GET /api/v1/version`

這些端點可供探活、smoke gate 與 CLI 使用，不等同於公開業務接口。

受保護機器端點：

1. `GET /metrics`

`/metrics` 不應裸露；抓取權限由 token 或 IP allowlist 裁決。當前路由層的硬性保護僅在 production 生效，因此 staging 需靠部署配置主動比照 production 收口。

全局 CORS 允許方法含 `PATCH`，以承接 admin 狀態更新與配置治理端點的 preflight（例如 `PATCH /api/v1/admin/users/:userId/status`）。

## 4. 版本與部署形態

當前正式版本口徑只承認兩種形態：

1. 本機開發版：Localhost 運行，用於開發、調試與測試；日常不以 Docker 作必要前提
2. 發布版：Vercel 承載前台與管理端，Railway 承載後端與 production runtime

Docker / 傳統服務器不再作正式版本口徑；若臨時使用，只能視為一次性運行形態，需自行確保遷移、探針與監控。

發布版是否閉環，不由單一平台狀態裁決，而由以下條件共同裁決：

1. secrets 完整
2. migration 已對齊
3. `health` / `ready` / `live` 正常
4. `/metrics` 受保護
5. smoke 與主鏈路驗證完成；完整 `ops:release:gate` 必須顯式設定 `RUN_MUTATING_RELEASE_SMOKE=true`，由 `ops:release:smoke` 執行 production-like quick / admin / claim-session smoke，Admin 頁與 Admin API 使用獨立 `ADMIN_BASE_URL / ADMIN_ORIGIN`，並在 smoke 後重新跑 active smoke/dev 帳號 hygiene；不能只用 health endpoint 替代

## 5. 本地與線上差異

本地開發允許與線上不同，但需明確承認差異：

1. 本地未啟 Redis 時，`AI Stream` 與 lock 可能退回 memory / simple-lock 語義
2. 當 Redis 在 runtime 失聯時，cache/lock 會在首次錯誤後主動關閉當前 Redis client 並固定降級到 memory / simple-lock；後續請求不再先做 Redis 重試，以避免每次操作都額外阻塞
3. 此類降級屬進程級行為；恢復 Redis-backed runtime 需重啟服務並重新建立連線
4. 這種本地退化不等價於 production，也不等價於 staging 的發版前收口口徑
5. 本地若要模擬線上，應直接啟 Redis 並跑同一條 smoke 鏈路
6. 本地 `.env` 的 `AI_COST_PRICING_JSON` 只證明本機開發版成本歸因可用；Railway release env 需獨立配置並通過 `ops:ai-pricing:check` / `ops:release:gate`，不能由本機設定推導發布版已精準記成本；release gate 模式下缺發布 pricing env 必須失敗，不得 fallback 到本機 `.env`

## 6. 環境核驗順序

1. 先驗 `.env` / 平台 secrets 是否完整
2. 再驗 migration 是否對齊
3. 再驗 `health` / `ready` / `live`
4. 再驗 `/metrics` 保護
5. 最後驗快速體驗、訪談、聊天室等主鏈路 smoke；發布版完整 gate 至少包含 `ops:release:smoke` 的 production-like quick / admin / claim-session smoke
