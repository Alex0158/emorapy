## 1. 定位

本文用 PRFAQ 方式固定 Emorapy 的產品意圖與邊界。它面向通用級產品裁決，不替代營銷文案，也不替代接口、測試或運維文檔。

## 2. Press Release 口徑

Emorapy 幫助親密關係中的雙方把反覆爭執轉化為可被理解、可回看、可行動的結構化梳理。用戶可以先匿名快速判斷，也可以登入後進入正式處理，或先在聊天室沉澱材料再明確轉入梳理結果。當內容涉及暴力、控制、自傷或其他高風險信號時，Emorapy 優先保護安全，不把危機當成普通吵架。

## 3. 客戶 FAQ

| 問題 | 回答口徑 |
| --- | --- |
| 我一定要註冊嗎？ | 不一定。快速判斷可用匿名 session 完成；若要保存、升格、正式處理或長期追蹤，需登入。 |
| Emorapy 是幫我判誰對誰錯嗎？ | 對外應表述為關係梳理與調整方向，不是法律或道德裁決。內部代碼仍保留 `judgment` 等歷史命名。 |
| 伴侶不在身邊也能用嗎？ | 可以。正式處理、chat、配對與關係檔案支持非同步場景；具體能力以當前 Web / App 承接狀態為準。 |
| 聊天會自動被拿去生成結果嗎？ | 不應自動轉換。chat -> judgment 必須是明確觸發的行為，且受後端授權與消息範圍限制。 |
| 這能替代心理諮商或法律建議嗎？ | 不能。Emorapy 不是臨床治療、法律代理或危機救援服務。高風險情境優先安全分流。 |
| App 和 Web 是兩套產品嗎？ | 不是。它們共用同一產品核心、backend、DB schema 和共享契約方向；App 只做移動端投影與原生能力承接。 |

## 4. 內部 FAQ

| 問題 | 裁決 |
| --- | --- |
| 是否可以為 App 新增 App-only case 狀態？ | 不可以。App 只能新增平台適配狀態；業務狀態機以 backend / DB / 跨端核心為準。 |
| Web 已完成的頁面是否等於 App 完成？ | 不等於。App 必須有 screen / adapter / API 消費 / smoke 或證據入口，且回寫 Parity。 |
| 指標沒有基線時能否填目標值？ | 不能。先標「待建立基線」，建立資料來源後再評估目標。 |
| 可否把營銷定位直接寫成工程事實？ | 只能摘要穩定產品意圖；具體功能、流程、接口、狀態必須回到核心開發文件和現碼。 |
| 可否用單次手動驗收替代正式規格？ | 不能。單次證據只能進 `90-證據與盤點/`；穩定真相需回寫正式子域。 |

## 5. 非目標

| 非目標 ID | 非目標 | 說明 |
| --- | --- | --- |
| CJ-NOGO-001 | 不做臨床診斷或治療承諾 | AI 可以做關係梳理，不可對外宣稱治療、診斷或醫療效果 |
| CJ-NOGO-002 | 不做法律裁決或責任判罰 | 梳理結果不是法律意見，也不是司法裁決 |
| CJ-NOGO-003 | 不做危機救援替代品 | 自傷、暴力、控制等高風險情境應走安全分流和外部資源提示 |
| CJ-NOGO-004 | 不做 App-only 業務分叉 | App 原生差異不能改寫核心產品語義 |
| CJ-NOGO-005 | 不做無證據的成功宣稱 | 指標、留存、轉化、效果提升需有資料來源或標為待建立基線 |
| CJ-NOGO-006 | 不做 Admin 普通 App 化 | 普通 App 不承接 Admin Web、Admin token 或 RBAC 管理台 |

## 6. 變更規則

1. 若 FAQ 答案改變產品承諾，必須同步更新 `01-產品PRD總章.md`。
2. 若 FAQ 涉及 Web / App 差異，必須同步更新平台投影與 Parity 文件。
3. 若非目標被重新納入產品範圍，必須先建立 PRD ID、驗證矩陣與待辦，不得只改頁面文案。
