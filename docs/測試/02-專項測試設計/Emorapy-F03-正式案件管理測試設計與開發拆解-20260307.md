# Emorapy F03 正式案件管理測試設計與開發拆解

日期：2026-03-07  
對應功能：`F03 正式案件管理（建案/列表/審閱/提交）`  
對應流程：`P02 正式案件閉環`

## 1. 本次測試目標

F03 的核心不是「頁面能打開」，而是要把登入後正式案件的狀態轉移與授權邊界固定住，避免出現半成品案件、錯誤提前提交或錯人可操作。

本輪優先驗證的業務規則：

1. 建案前必須已有有效配對。
2. `remote` 模式下，只有原告陳述時應建立 `draft`；有雙方陳述時才可直接進 `submitted`。
3. `collaborative` 模式下，正式案件不應允許只有單方陳述就直接建立 `submitted`。
4. 只有 `draft` 狀態允許更新或提交。
5. `Case Detail` 頁需正確區分：
   - 原告等待被告回覆
   - 可提交案件
   - 已提交後進 review

## 2. 代碼真實規則（以實際代碼為準）

### 2.1 功能/頁面/API 基線

依據：

1. `docs/核心開發文件/功能特性清單.md`
2. `docs/核心開發文件/接口描述/03-case.md`
3. `docs/核心開發文件/接口-功能-頁面-Mapping.md`

F03 主鏈路：

1. 用戶進入 `/case/create`
2. 檢查 pairing 是否 active
3. `POST /api/v1/cases` 建立正式案件
4. 進入 `/case/:id`
5. 視狀態決定：
   - 等待對方回覆
   - 直接提交
   - 跳轉 `/case/:id/review`

### 2.2 前端頁面真實規則

核心位於：

1. `frontend/src/pages/Case/Create/index.tsx`
2. `frontend/src/pages/Case/Detail/index.tsx`
3. `frontend/src/pages/Case/Review/index.tsx`

可確認規則：

1. `Case Create` 先調 `getPairingStatus()`，無 active pairing 不允許正式建案。
2. `remote` 模式下只要求原告陳述合法；`collaborative` 模式要求雙方陳述都合法。
3. `Case Detail` 中：
   - `remote + draft + 無 defendant_statement` 時，原告只能等待，被告才可回覆
   - `draft + 可提交` 時才出現 submit 按鈕
4. `submitCase` 成功後前端導向 `/case/:id/review`

### 2.3 後端正式案件真實規則

核心位於：

1. `backend/src/services/case.service.ts`
2. `backend/src/controllers/case.controller.ts`
3. `backend/src/utils/validation.ts`

本輪確認：

1. `createCase()` 會校驗：
   - pairing 存在且 active
   - user 屬於 pairing
   - 原告陳述合法
2. `remote` 模式：
   - 無被告陳述 -> `draft`
   - 有被告陳述 -> `submitted`
3. `submitCase()`：
   - 僅 `draft` 可提交
   - `remote` 若無被告陳述不可提交
4. 本輪新增明確規則：
   - `collaborative` 正式建案必須同時提供雙方陳述，否則拒絕請求

## 3. 本輪發現的真缺陷

### F03-BUG-001

修復前，`backend/src/services/case.service.ts` 對正式 `createCase()` 的 `collaborative` 分支只要 `mode=collaborative` 就會直接把案件標成 `submitted`，即使沒有 `defendant_statement`。

結果是：

1. 可繞過前端雙方陳述要求
2. 直接產生不完整正式案件
3. 提前觸發後續 review / judgment 流程

本輪已修復：

1. `createCaseSchema`：`collaborative` 必須帶 `defendant_statement`
2. `case.service.createCase()`：缺少被告陳述時直接拋 `VALIDATION_ERROR`
