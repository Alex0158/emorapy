缺陷編號：`F02-BUG-002`（註：F02-BUG-001 為匿名 Result 授權，已修復）

## 1. 缺陷摘要

目前 Emorapy 的「判決 / 責任分比例」鏈路存在明顯的產品語義錯位與算法收斂問題。

表面上，前端與對外文案將結果呈現為「判決結果」與「責任分比例」；但實際上，後端核心邏輯計算的並不是傳統意義上的「過錯責任」，而是「在當前互動循環中，哪一方更有調整空間」。

這一語義錯位會在一般衝突場景中造成理解偏差；在涉及暴力、動手、控制、威脅等安全風險場景中，則會進一步放大為高風險錯判。以實際案例描述為例：

1. 角色 A 因情緒不好摔碗
2. 角色 B 指出對方行為不妥
3. 角色 A 進一步對角色 B 動手
4. 最終結果卻可能輸出為角色 B 責任更高，接近 `70%`

這說明系統目前存在以下問題：

1. 暴力/動手場景未被穩定識別為安全場景
2. 責任比例算法存在向中間區間與固定點位收斂的傾向
3. 前端將「調整空間」包裝為「責任」，導致使用者將結果理解為道德或事實責任判定

## 2. 業務影響

此問題直接影響：

1. Emorapy 的核心輸出可信度
2. 使用者對產品「公正性」與「判決合理性」的信任
3. 涉及暴力、威脅、控制等敏感場景時的風險控制能力
4. 快速體驗主鏈路的口碑轉化
5. 後續和好方案、修復建議與安全建議的方向正確性

其中最嚴重的不是「比例偏一點」，而是：

1. 使用者會把這個結果理解為平台在進行責任裁定
2. 若暴力行為方沒有被明確識別，平台可能對受害方或非施暴方形成二次傷害
3. 一旦出現「施暴方較輕、被動應對方較重」的輸出，會嚴重損害產品可信度

## 3. 問題現象

基於當前代碼與流程分析，已確認存在以下現象：

1. 責任比例經常落在 `50/50`、`55/45`、`60/40`、`65/35`、`70/30` 等少數固定點位
2. 快速體驗、一般建案、聊天室轉判決等入口最終都會流向同一套責任比例計算邏輯
3. 暴力/控制/威脅的安全場景與一般衝突場景，雖然 prompt 有做語義區分，但最終責任比例仍可能被混合算法拉回中間區間
4. 若結構化責任比例評估失敗，系統會回退到「文案抽取 + 規則校準」，這條回退路徑本身就偏向中間值
5. 前端只負責顯示，沒有重新計算，但顯示文案會強化使用者對「責任」的理解

## 4. 當前流程梳理

目前判決責任比例主鏈路如下：

1. 使用者在前端提交 `plaintiff_statement` 與 `defendant_statement`
2. 後端 `case.controller` 建案成功後，非同步觸發 `judgmentService.generateJudgment()`
3. `judgment.service` 先做情感分析，再做安全分流
4. `ai.service.generateJudgment()` 先生成一篇長篇「判決 / 關係分析」文案
5. 隨後 `ai.service.computeResponsibilityRatio()` 再另外計算責任比例
6. 最終結果以 `plaintiff_ratio` / `defendant_ratio` 寫入資料庫
7. `backend/src/utils/judgment.ts` 將其補為 `responsibility_ratio`
8. 前端結果頁與詳情頁直接讀取顯示，不參與重算

換句話說，真正決定責任比例的核心，不在前端展示層，而在：

1. `backend/src/services/ai.service.ts`
2. `backend/src/services/judgment.service.ts`
3. `backend/src/services/safety-routing.service.ts`

## 5. 根因分析

### 5.1 產品語義錯位

這是最核心的根因。

後端 prompt 已明確說明「調整比重」代表的是：

1. 在這段互動中，可以率先做出不同選擇的餘地
2. 不是道德責任
3. 不是過錯歸屬

但前端對外呈現使用的卻是：

1. `判決結果`
2. `責任分比例`
3. `基於 AI 分析的公正判決`

這導致系統內部與使用者感知的語義完全不一致。

系統實際想做的是「互動改變建議排序」，使用者看到的卻是「誰責任更大」。這不是純技術 bug，而是產品定義與展示層之間的結構性錯位。

### 5.2 安全風險識別詞表過窄

目前安全分流依賴 `IPV_SIGNAL_REGEX`、`CRISIS_SIGNAL_REGEX` 與分析結果中的 `safetyFlags`。

但實際正則主要覆蓋的是：

1. `控制行為`
2. `暴力`
3. `威脅`
4. `權力不對等`
5. `經濟控制`
6. `人身威脅`
7. `貶低人格`
8. `孤立社交`

這裡的主要缺口是：對使用者真實敘述中更常見的詞彙覆蓋不足，例如：

1. `打我`
2. `打了他`
3. `動手`
4. `推我`
5. `扇巴掌`
6. `砸東西`
7. `掐`
8. `拉扯`
9. `摔東西後衝過來`

因此，像「摔碗後又打男朋友」這種已經非常明確的人身侵害場景，未必能穩定進入 `safety_support`。

### 5.3 責任比例算法天然收斂到中間區間

責任比例的最終值不是單一來源，而是三條來源混合：

1. `assessResponsibilityRatio()`：要求 LLM 輸出 JSON 比例
2. `extractResponsibilityRatio()`：從文案中用 regex 抽取百分比
3. `deriveRatioFromSignals()`：基於情感分析與語句特徵做 heuristic 推估

然後再經過：

1. `blendRatios()` 混合
2. `normalizeRatio()` 歸一化
3. `Math.round()` 整數化

此外，`deriveRatioFromSignals()` 還存在明確的區間鉗制：

1. 對方未發言時：`45~55`
2. 一般場景：`30~70`
3. 有 safety flags 但非 IPV：`35~75`
4. 有 IPV：`30~85`

這些機制會讓結果天然收斂到少數中間區間點位，從而形成「比例幾乎固定」的體感。

### 5.4 fallback 與預設值會再次強化固定比例

目前流程中存在多個會把結果往固定值推回去的 fallback：

1. `useMock` 時直接固定輸出 `55/45`
2. `routeType === crisis_support` 時直接固定 `50/50`
3. `extractResponsibilityRatio()` 抽不到時預設 `50/50`
4. `normalizeRatio()` 遇到非法值或總和異常時回退 `50/50`
5. `assessResponsibilityRatio()` 失敗時，會回退到 `extract + heuristic`，混合權重固定 `0.55`

這代表即便上游分析有一定差異，只要中間任一節點失敗，最終結果仍會被拉回幾個常見點位。

### 5.5 文案格式與抽取規則存在對齊風險

prompt 要求輸出格式是：

1. `角色 A：[X]% 調整空間`
2. `角色 B：[Y]% 調整空間`

但 `extractResponsibilityRatio()` 的 regex 更偏向抓：

1. `角色 A：70%`
2. `角色 B：30%`
3. `原告：70%`
4. `被告：30%`

若模型輸出不完全吻合抽取規則，就會進入默認值與混合邏輯，間接放大固定比例現象。

### 5.6 快速體驗入口存在「自動代寫被告」的敘事污染風險

前端快速體驗頁提供 `自動代寫`，且其模板本身偏向和解語氣：

1. `我理解這件事讓你不舒服`
2. `我願意調整`
3. `希望找到折衷方案`

如果使用者未真正提供 B 方陳述，而是依賴這段模板，案件就會被系統以「雙方都有調整空間」的關係衝突語境去理解，進一步稀釋暴力、指責、威脅等非對稱信息。

## 6. 為何會出現「施暴方較輕、非施暴方較重」的結果

以本次案例類型推演，最可能的錯判路徑如下：

1. `打人/動手` 沒有被正確命中安全風險詞表
2. 案件被當成一般伴侶衝突而非 `safety_support`
3. AI 會優先分析「互動循環」而不是「誰有明確有害行為」
4. 若角色 B 有批評、指責、講道理、追問等語氣，heuristic 可能將其歸入高壓、批評、追逐等模式
5. 最終計算的是「誰更有調整空間」，而不是「誰做了更嚴重的不當行為」
6. 再經過混合、鉗制、四捨五入後，比例被穩定收斂到 `60/40` 或 `70/30`

因此，這類錯判不是單純的模型幻覺，而是：

1. 安全分類不足
2. 算法目標錯置
3. 顯示語義錯置

三者共同造成的結構性輸出偏差。

## 7. 風險等級判定

本問題應定義為高嚴重度，原因如下：

1. 涉及產品核心輸出
2. 使用者高度依賴其作為「第三方判斷」
3. 一旦落入暴力/動手/威脅場景，將構成明顯的安全與公信力風險
4. 問題不是個別頁面錯誤，而是跨入口共用服務層邏輯問題

若後續要做正式分級，建議按兩層處理：

1. 產品層：`高`
2. 安全/風控層：`高`

## 8. 優化建議

以下建議按優先級排序。

### 8.1 P0：立即修正對外語義

在未重構算法前，最先要做的是避免把「調整空間」誤導為「責任」。

建議：

1. 前端將 `責任分比例` 改為 `調整方向比例`、`互動調整空間`、或 `優先調整建議`
2. 將 `判決結果` 改為 `關係分析結果`、`互動分析結果`、或 `溝通修復建議`
3. 在比例模組下方明確增加說明：
   - 這不是法律責任
   - 不是事實裁定
   - 不是道德評分
   - 而是根據當前敘述推估，哪一方更適合率先改變互動方式
4. 在暴力/控制/威脅場景中，不顯示對稱式比例，改為非對稱安全提示

這一項是止血措施，成本最低，但能立即降低誤解風險。

### 8.2 P0：擴充安全風險詞表與路由規則

建議立即補強 `IPV_SIGNAL_REGEX` 與相關 safety flags 規則，至少納入：

1. `打`
2. `打人`
3. `動手`
4. `推`
5. `扇`
6. `掐`
7. `毆`
8. `砸`
9. `摔東西`
10. `拉扯`
11. `抓住`
12. `揍`
13. `傷害`
14. `出手`

同時建議：

1. 區分 `情緒性抱怨` 與 `明確人身侵害`
2. 只要出現明確肢體暴力，就強制進入 `safety_support`
3. 在 `safety_support` 下禁止輸出對稱式責任比例
4. 對受害方輸出「保護自己 / 求助支持」方向，對有害行為方輸出「停止傷害 / 承擔行為責任」方向

### 8.3 P0：將「安全場景」與「一般比例算法」徹底解耦

目前安全場景雖然有 route，但比例仍走共享邏輯，這是不夠的。

建議：

1. `standard` 路由才允許輸出雙方比例
2. `safety_support` 路由不輸出 `X/Y` 對稱比例
3. `crisis_support` 路由不輸出責任比例，只輸出安全與支持建議
4. 如業務仍要求安全場景輸出結構化信息，應改為：
   - `有害行為方需要立即停止的模式`
   - `受影響方的保護與支持方向`

不要再沿用同一個 `plaintiff_ratio / defendant_ratio` 模型。

### 8.4 P1：重構責任比例算法，降低固定化收斂

若仍保留一般衝突場景的雙方比例，建議：

1. 取消或放寬過度鉗制的區間限制
2. 降低 fallback 對 `50/50` 的依賴
3. 將 `structured`、`extracted`、`heuristic` 的權重改為可觀測、可配置
4. 當結構化評估失敗時，不應默默用 `0.55` 混合，而應記錄明確 failure reason
5. 文案抽取與 prompt 輸出格式必須做嚴格對齊
6. 將比例精度從整數提升到至少一位小數，內部保留更高精度，再由 UI 決定是否四捨五入
7. 若雙方資訊極不對稱，應顯示「資訊不足，不宜輸出比例」而非硬算

### 8.5 P1：移除或限制 mock 對正式流程的污染

目前 `useMock` 條件過於寬鬆：

1. `AI_MOCK=true`
2. API key 包含 `sk-dev-`
3. API key 包含 `your-openai-api-key`

建議：

1. 僅允許 `AI_MOCK=true` 時啟用 mock
2. 對開發 key 或示例 key 不要自動靜默切換固定判決
3. 在所有 mock 輸出上打明確標記，例如 `mock_judgment=true`
4. 前端在讀到 mock 結果時，明確顯示「示範數據」或禁止作為正式分析展示

否則在開發、測試、演示過程中，很容易誤把固定 `55/45` 當成真實算法輸出。

### 8.6 P1：限制快速體驗的被告自動代寫對高風險場景的干擾

建議：

1. 若使用者原告陳述中已包含暴力、控制、威脅等關鍵詞，禁用自動代寫被告
2. 若使用者仍選擇單方敘述，系統應顯示「基於單方資訊，結果僅供初步參考」
3. 自動代寫模板不應默認帶入和解性語句，例如 `我願意調整`
4. 自動代寫產物應只作草稿提示，不應直接作為高權重輸入送入責任計算

### 8.7 P2：建立觀測性與審計能力

建議在後端保留更清晰的責任比例審計欄位，例如：

1. `route_type`
2. `structured_ratio`
3. `structured_confidence`
4. `extracted_ratio`
5. `heuristic_ratio`
6. `final_ratio`
7. `fallback_reason`
8. `safety_flags`
9. `mock_used`

這樣後續一旦出現錯判，可以快速回答：

1. 是沒進安全路由？
2. 是結構化輸出失敗？
3. 是 regex 沒抽到？
4. 是 heuristic 把結果往中間拉了？

## 9. 建議驗證方案

優化後應至少補充以下測試護欄：

1. `打人/動手/推/扇巴掌/拉扯` 等語句必須進入 `safety_support`
2. `safety_support` 場景不得輸出對稱式責任比例
3. 單方陳述且資訊不足時，不得輸出高置信度雙方比例
4. 非安全的一般衝突場景，比例不應長期收斂在少數固定點位
5. `extractResponsibilityRatio()` 必須完整覆蓋 prompt 規定格式
6. `useMock` 場景在 API 與前端展示層都必須可識別
7. 快速體驗被告自動代寫不得污染安全場景判斷

此外，建議增加一批真實業務語料導向測試矩陣，至少覆蓋：

1. 一般日常摩擦
2. 雙方互相指責
3. 單方長文、另一方缺席
4. 暴力 / 動手
5. 控制 / 威脅
6. 自傷 / 自殺風險
7. 分手/離開傾向

## 10. 建議修復順序

建議按以下順序落地：

1. 先修正前端與對外文案語義，避免繼續把「調整空間」呈現為「責任」
2. 立即補強安全詞表與 `safety_support` 強制路由
3. 將安全場景從雙方比例算法中完全拆出
4. 重構比例計算與 fallback，降低固定化收斂
5. 補齊測試矩陣與審計字段

## 11. 結論

本問題不是單一案例「AI 判錯」這麼簡單，而是現有判決鏈路在以下三點上同時存在缺陷：

1. 產品語義錯位：系統算的是「調整空間」，前端展示成「責任比例」
2. 安全風險識別不足：`打人/動手` 等明確暴力敘述未被穩定納入安全路由
3. 算法收斂過強：多重 fallback、鉗制、混合與整數化使比例過度集中

因此，像「施暴方較輕、非施暴方較重」這類輸出，不應被視為偶發，而應被視為當前系統設計下可預期的高風險缺陷。

若不優先修正，Emorapy 的核心判決輸出將持續面臨：

1. 公信力下降
2. 使用者誤解
3. 敏感場景下的安全風險與二次傷害風險
