本文件定義 `CJ` 文檔體系的活躍治理口徑，用於確保核心文檔持續與代碼同步。

本文件不是產品/工程正文 SSOT；它只裁決“文檔應如何組織、同步與守衛”。

## 1. SSOT 層級規則

1. 一級權威：`docs/核心開發文件/`
2. 補充測試入口：`docs/核心開發文件/測試/` 與 `docs/核心開發文件/08-測試規範與驗收/`
3. 證據與核驗：`docs/核心開發文件/90-證據與盤點/`
4. 歷史回看：`docs/核心開發文件/99-歷史降級索引/` 與根級 `歸檔/`
5. 營銷、品牌、對外敘事：`docs/核心營銷文件/`，不屬於工程真相裁決層

整個 `/Users/alex/Desktop/CJ` 只允許三個 Markdown 存在區：

1. `mother-bear-court/docs/核心開發文件/`
2. `mother-bear-court/docs/核心營銷文件/`
3. 根級 `歸檔/`

若發生衝突，以現碼、`00-跨端產品核心/`、根層旗艦文檔、平台投影子域與對應正式子域為準。

現行核心文件採用三層 SSOT：

1. `00-跨端產品核心/`：跨 Web / App 的產品意圖、PRD、成功指標、產品能力、角色、流程、狀態與授權邊界。
2. `10-Web端/`、`20-App端/`：平台投影與端側差異，不重新定義產品核心。
3. `50-跨端Mapping與Parity/`：Web / App / Backend / API / DB / shared packages 的一致性、差異與待辦映射。

其中 `00-跨端產品核心/01-產品PRD總章.md`、`00-跨端產品核心/02-用戶場景與假設台帳.md`、`00-跨端產品核心/03-成功指標與產品健康.md`、`00-跨端產品核心/04-PRFAQ與非目標.md` 是通用級 PRD 入口；`00-跨端產品核心/05-工程級PRD對標與治理缺口台賬.md` 承接工程級需求屬性、外部標準對標與治理缺口；`04-共用機制/02-信任安全與非功能需求.md` 承接共用 NFR；`04-共用機制/03-AI風險與安全治理基線.md` 承接 AI / LLM 風險、prompt、ledger、prompt version 與 output gate；`04-共用機制/04-資料治理與隱私風險基線.md` 承接資料分類、consent/delete、retention、archive、log / evidence 最小化與 App storage / telemetry；`04-共用機制/05-威脅建模與安全需求基線.md` 承接 trust boundary、threat model、`CJ-SEC-*`、ASVS / MASVS 參考與安全需求缺口；`04-共用機制/06-狀態機與業務不變式治理基線.md` 承接核心狀態、transition、guard、side effect、非法轉移與恢復任務；`04-共用機制/07-可訪問性本地化與內容設計治理基線.md` 承接 Web / Admin / App 可訪問性、本地化、`lang`、screen reader 與高敏內容設計缺口；`05-工程架構與共享層/02-架構決策與ADR治理基線.md` 承接架構視圖、`CJ-ADR-*`、架構取捨、驗證與重審 gate；`05-工程架構與共享層/03-資料模型SchemaMigration與相容性治理基線.md` 承接 Prisma schema、migration history、release DB parity、backfill、棄用與版本相容性；`03-管理端與平台治理/06-SLO可觀測性與事故治理基線.md` 承接 SLI / SLO、incident、postmortem、release gate 與可觀測性；`06-接口描述/11-API契約與OpenAPI缺口台賬.md` 承接 API contract / OpenAPI 缺口；`06-接口描述/12-錯誤模型與ProblemDetails缺口台賬.md` 承接 API 錯誤模型、HTTP status、`error.code`、request id、retry policy 與 Problem Details 缺口；`08-測試規範與驗收/04-需求驗證矩陣.md` 承接 `CJ-PRD-*` / `CJ-NFR-*` 到功能、流程、API、頁面、測試與證據的追溯；`08-測試規範與驗收/05-可訪問性本地化驗收基線.md` 承接 keyboard、ARIA、axe / WCAG gap、i18n completeness、screen reader 與 App native accessibility 驗收缺口；`08-測試規範與驗收/06-SchemaMigration與相容性驗收基線.md` 承接 migration status、release DB parity、schema compatibility、backfill 與 App parity 驗收缺口；`08-測試規範與驗收/07-SLO可觀測性與事故演練驗收基線.md` 承接 health / metrics / ops alerts / request id / logs / release evidence / incident drill / App telemetry 的驗收缺口。

App 相關變更有兩個中層 gate：

1. `20-App端/01-App導航與平台Adapter基線.md`：裁決 Expo Router、App navigation、SecureStore、Push、Deep Link、upload、telemetry 等平台承載與 adapter 邊界。
2. `50-跨端Mapping與Parity/01-App首輪能力與工程落點Mapping.md`：裁決 App screen 到 Web / Backend / API / DB / shared packages 的映射、缺口與待辦。

因此，App 文件不能只在 `20-App端/` 自洽；凡涉及 API、DB、shared package、狀態、授權或 Web-App 一致性，都必須能從 `20/01` 追到 `50/01`，再追到待辦或已落地代碼。

## 3. 編輯流程

1. 先改代碼，再改文檔，不允許先寫未落地能力為已實作。
2. 每條新增接口至少補齊主註冊表、對應接口子文檔與 Mapping。
3. 每個新增頁面至少補齊 `頁面清單.md`、`功能特性清單.md` 與 Mapping。
4. 每個新增或改名的 `CJ-PRD-*` / `CJ-NFR-*` 必須至少有一條 `CJ-RTM-*`，並追到功能、流程、平台、工程落點與驗收狀態。
5. 判斷修改屬於通用級、Web級、App級、共用機制級或驗收級；通用級先回寫 PRD，再投影到 Web / App，不允許端側文件反向定義產品核心。
6. 需求若使用 MUST / MUST NOT，必須能說明安全、資料歸屬、發布、互操作或跨端契約理由；否則降為 SHOULD / MAY 或普通描述。
7. 外部標準只能作參考對標；未經正式審計不得宣稱 ISO / IEEE / OWASP 合規等級。
8. AI / LLM 風險不得因單點控制存在就寫成已完整治理；新增 AI runtime 必須回查 AI asset inventory、prompt fencing、prompt version、ledger、output gate 與 App 待承接口徑。
9. API contract 不得因 `docs:check:truth` 通過而宣稱 OpenAPI 或 schema contract 完成；未有 `openapi.yaml/json` 前只能標 OpenAPI gap。
10. 資料治理不得因 consent flag、單一 delete endpoint、cleanup job 或 log masking 存在就寫成完整隱私合規；涉及高敏資料必須標出資料級別、保留/刪除/證據口徑與缺口。
11. SLO / incident 不得因 `/health` 回 200、Prometheus metrics 存在、request id 存在或 release gate 單次通過就寫成 SLA、error budget、distributed tracing 或 incident drill 已完成；涉及事故時必須能追到 SEV、signals、mitigation、recovery evidence 與 follow-up。
12. 威脅建模不得因 CORS、Helmet、JWT、RBAC、rate limit 或 media auth 存在就寫成完整安全治理；涉及高風險流程必須能追到 trust boundary、`CJ-SEC-*`、現有控制與缺口。
13. 架構治理不得把 repo 目錄說明當作 ADR；涉及 workspace、shared package、AI runtime、安全邊界、App adapter、DB/API/shared 互操作或 release gate 的變更必須記錄架構取捨。
14. 狀態治理不得因 enum、contract union 或 service guard 存在就寫成完整狀態機；新增或修改狀態必須補 trigger、guard、side effect、illegal transition、recovery 與測試/inspection 入口。
15. 錯誤契約不得因 `AppError`、error handler 或 response envelope 存在就寫成 RFC 9457 Problem Details；新增或修改錯誤碼、status、validation details 或 retry policy 必須同步錯誤模型與接口文檔。
16. 可訪問性與本地化不得因 i18n catalog、Radix / shadcn 元件、局部 `aria-label`、role tests 或 focus hook 存在就寫成 WCAG 合規、雙語完整或 App accessibility 完成；涉及可見文案、`lang`、ARIA、keyboard/focus、screen reader 或 App native accessibility 時必須同步治理與驗收基線。
17. Schema migration 不得因 `schema.prisma` 更新、migration commit、Dev DB 套用、`db push` 成功或 release gate 單項通過就寫成 production-safe、backfill 完成、舊 client 相容或完整 data dictionary；涉及 schema、migration、enum、DTO、API response、App storage 或棄用時必須同步治理與驗收基線。
18. dated 驗收記錄不得直接升格成正式規格；若包含穩定真相，必須先回寫到正式子域。
19. 任何與 AI 流式或高風險鏈路相關的調整，若保留證據，應回到 `90-證據與盤點/`。
20. 若屬封板級、跨子域或大口徑重構，先更新本目錄台賬，再回寫正式正文。
21. 除上述三個區域外，不保留根 README、模組 README、repo-local skill 文檔或任何跳轉 stub。
22. `手動回歸證據/<latest-date>/summary.md` 與 `P01-P05/record.md` 的狀態必須逐條一致；不一致視為文檔漂移，`docs:check` 必須失敗。
23. 若某項差異必然要求 Web / App / Backend / DB / API 兩邊統一，但本輪未實作，必須主動新增 `07-待處理問題與治理/待處理/` 任務，並在 `50-跨端Mapping與Parity/` 可追溯。
24. 執行全量文檔 metadata / 台賬同步前，必須先跑 `npm run docs:audit:dry-run` 查看 current / non-current 與子域影響面。日常正式正文維護優先跑 `npm run docs:audit:dry-run:current`；處理證據/歷史/治理層時才跑 `npm run docs:audit:dry-run:non-current` 或全量 dry-run。只需同步局部子域時，優先使用 `node scripts/sync-core-docs-metadata.mjs --dry-run --prefix <path>` 先確認再寫入。若需要逐文件清單，再加 `--list`，避免平時輸出過長。
25. `npm run docs:audit:sync` 是全量寫入入口，必須設置 `DOCS_AUDIT_SYNC_CONFIRMED=1` 才能執行；未確認時應拒絕寫入並回到 dry-run / prefix 流程。
26. App 若新增 screen、adapter、Deep Link、Push token、SecureStore session restore、upload 或 App-only error / telemetry context，必須先更新 `20-App端/01-App導航與平台Adapter基線.md` 與 `50-跨端Mapping與Parity/01-App首輪能力與工程落點Mapping.md`；若對應 Backend / API / DB / shared package 尚未完成，同步新增待處理任務。
