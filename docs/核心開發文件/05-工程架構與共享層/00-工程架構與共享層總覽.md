本文件承接 `Emorapy` repo 的工程分層、共享契約與跨端代碼落點規則；當前物理路徑 / GitHub repo 仍保留 legacy `mother-bear-court`，等待 P4 repo / deployment migration。這裡的 `apps / services / packages` 同時是分類心智與現有物理樹的解讀方式，不等於現有目錄已全部進入同一 workspace 體系。

## 1. 本子域回答什麼

本子域只回答以下問題：

1. `frontend/`、`frontend-admin/`、`mobile/`、`backend/`、`packages/` 在 repo 中各自承擔什麼責任。
2. 共享契約、API client、純邏輯應下沉到哪裡，哪些內容不能下沉。
3. 修改共享層時，應同時考慮哪些 app/service 的影響面。
4. App navigation / native adapter / shared package 接線的工程前置 gate 應回到哪份 App / Parity 中層文件。
5. 哪些架構性變更需要 ADR，架構決策如何追溯到 PRD、NFR、RTM、接口、Parity 與威脅建模文件。
6. Prisma schema、migration history、release DB parity、backfill、棄用與版本相容性應如何治理。

本子域不回答：

1. 某個功能流是否存在、由哪個頁面承接。
2. 某個接口是否已使用、待承接或候選廢棄。
3. 發版證據、單次驗收、排查快照。
4. 單一安全控制是否達成外部認證；安全需求與 threat model 回 [../04-共用機制/05-威脅建模與安全需求基線.md](../04-共用機制/05-威脅建模與安全需求基線.md)。

## 2. 當前 repo 分層

`Emorapy` 是單一 monorepo，以 repo 根目錄為 git 邊界；legacy `CJ` 只作歷史別名或 package / PRD / env 兼容期識別符。當前正式分層如下：


| 目錄                | 角色       | 說明                              |
| ----------------- | -------- | ------------------------------- |
| `frontend/`       | 主 Web 端  | 面向普通用戶的 Web 應用                  |
| `frontend-admin/` | 管理端 Web  | 後台管理與治理界面                       |
| `mobile/`         | Expo App | 單一 codebase 支援 iOS / Android    |
| `backend/`        | API 服務   | 業務路由、服務、排程與運維節點                 |
| `packages/`       | 共享層      | contracts、api-client 與後續純邏輯下沉位置 |

`mobile/` 當前仍是 repo-local App，不是 root workspace。其現有代碼已從 Expo Router 模板進入 Emorapy App 普通用戶主流程：`mobile/app` 承接 `(public)` / `(app)` route group、quick、auth、case、chat、profile、notifications、repair、modal 等 M0-M5 screen；`mobile/src/platform` 承接 API、SecureStore、SSE / AI stream subscription、upload、notifications、linking、lifecycle、telemetry runtime adapter；`mobile/package.json` 以 `file:../packages/*` dependency + Metro alias 消費 shared packages。

因此任何改動若要新增 App route、改變 App navigation、改動 SecureStore / Push / Deep Link / upload / SSE / telemetry adapter，或調整 M6 release evidence 口徑，必須先對齊 [../20-App端/01-App導航與平台Adapter基線.md](../20-App端/01-App導航與平台Adapter基線.md)。若同時牽動 shared contracts、api-client、Backend API、DB schema、Deep Link、notification、upload 或 telemetry，還必須同步對齊 [../50-跨端Mapping與Parity/01-App首輪能力與工程落點Mapping.md](../50-跨端Mapping與Parity/01-App首輪能力與工程落點Mapping.md)。


## 3. 共享層落點規則

應放入 `packages/*` 的內容：

1. DTO、enum、response shape、error code。
2. 純 HTTP / SSE / upload transport。
3. 純函式商業邏輯、normalize、permission、query key、state machine。
4. 設計 token、常數、文案 key。

不應放入 `packages/*` 的內容：

1. React DOM / React Native UI 元件。
2. `react-router-dom`、Expo Router、導航副作用。
3. `localStorage`、`sessionStorage`、`SecureStore` 等平台儲存實作。
4. 相機、推播、檔案選擇、deep link、背景任務。
5. 任何直接依賴瀏覽器或原生執行環境的邏輯。

## 4. Web / Admin / Mobile / Backend 邊界

### 4.1 `frontend/`

承擔：

1. 主站 Web UI 與路由。
2. 瀏覽器環境下的 storage、導頁、頁面組裝。
3. 用戶端產品流程的具體頁面承接。

### 4.2 `frontend-admin/`

承擔：

1. 後台 UI、Admin token 守衛。
2. jobs、configs、reports、audit、health 等管理頁面。
3. 後台特有的操作流程與治理界面。

### 4.3 `mobile/`

承擔：

1. React Native / Expo UI。
2. 原生 storage、權限、相機、推播、deep link。
3. 與 Web 同一產品域在移動端的承接。

### 4.4 `backend/`

承擔：

1. API、SSE、jobs、健康探針、metrics。
2. 鑑權、限流、狀態機與持久化。
3. 對多端共享契約的實際供給。

## 5. 目前最重要的工程現實

以下是當前 repo 的正式工程現實，不應照搬理想化 monorepo 假設：

1. root npm workspaces 目前包含 `frontend/`、`frontend-admin/` 與 `packages/*`。
2. `backend/`、`mobile/` 仍是 repo-local 目錄，各自維持獨立 `package.json` / `tsconfig`；`packages/*` 已進入 root workspace，但不能因此假設所有端都已全量消費共享層。
3. `frontend/` 已用 tsconfig alias 接入 `@cj/contracts`，並已用 tsconfig alias 與 root workspace dependency 接入 `@cj/api-client`；`frontend/src/services/api/*` 已消費 M1-M5 shared domain client；`backend/` 已改用共享 package declaration artifact，避免直接引用 `packages/*/src` 觸發 `rootDir` 邊界問題；`mobile/` 已以 repo-local `file:../packages/*` dependency + Metro alias 消費 `@cj/contracts` / `@cj/api-client`。
4. `frontend-admin/` 已接入 `@cj/contracts` 與 `@cj/api-client` alias；目前只下沉 admin job stats、system config 與 interview runtime config 等穩定 DTO，並只使用 `@cj/api-client` 的 request transport baseline。
5. `frontend/` package manifest 已聲明 `@cj/contracts` workspace dependency，並已聲明 `@cj/api-client` workspace dependency；`frontend-admin/` package manifest 已聲明 `@cj/contracts` 與 `@cj/api-client` workspace dependencies。
6. `frontend/` 與 `frontend-admin/` 不再保留 app-local package-lock；CI 前端 jobs 必須從 root `package-lock.json` 執行 `npm ci` 並用 `--workspace` 跑 app scripts。
7. root package manifest 已聲明 root-owned script 需要的 `playwright` devDependency；root script 不應依賴 app-local `node_modules` 物理布局。
8. `packages/contracts` 與 `packages/api-client` 是已建立的共享層入口；主站 Web 已完成 M1 auth / session / quick case、M2 interview / psychProfile、M3 chat REST、M4 pairing / formal case / judgment / reconciliation / execution、M5 notifications / media upload 與 AI stream pure helper 的 shared consumer 收斂，但不代表 Web request side effect、Admin domain API 或 backend service logic 已完全下沉。
9. `backend` 仍有本地類型與 `rootDir` 限制，不能假設它已完全以共享原始碼作唯一來源。
10. `mobile/src/platform` 已建立 API、SecureStore、SSE、upload、notifications、linking、lifecycle、telemetry runtime adapter 與平台副作用邊界；這代表 M0-M5 App 工程 baseline 已收斂。release DB parity 是 release audit 證據槽，不能替代 EAS / TestFlight / physical device / provider delivery / native crash runtime 的 M6 strict release sign-off；後續若新增 release-blocking migration，仍必須刷新 release DB parity evidence。
11. `backend/prisma/schema.prisma`、`backend/prisma/migrations` 與 release DB `_prisma_migrations` 是三個不同層面的證據；schema 更新或 migration commit 不代表 release DB 已套用。

## 6. 共享改動的影響規則

修改共享層時，至少要做以下影響判斷：

1. 改 `packages/contracts`，先檢查 `frontend`、`backend`、`mobile` 的 alias 消費面；若改動影響 Admin 對應型別，再額外檢查 `frontend-admin` 的本地型別與 API 適配。
2. 改 `packages/api-client`，先檢查其是否仍只是共享入口，還是已被某端實際消費；不要假設所有端都已接線。
3. 改產品流程頁，不要順手把 UI、storage、router 副作用下沉進共享包。
4. 新增純邏輯時，先判斷它是否真能脫離平台運行環境，再決定是否下沉。
5. 改 App 端 shared 消費、native adapter 或 navigation 時，必須同時判斷是否需要更新 `20-App端/01-App導航與平台Adapter基線.md`、`50-跨端Mapping與Parity/01-App首輪能力與工程落點Mapping.md`，以及 `07-待處理問題與治理/待處理/` 下的 parity 待辦。
6. 改 Prisma schema、migration、DB constraint、enum、shared DTO 或 API response shape 時，必須同時判斷是否需要更新 [03-資料模型SchemaMigration與相容性治理基線.md](./03-資料模型SchemaMigration與相容性治理基線.md)、release DB parity gate、RTM 與待辦。

### 6.1 主站 Web 消費 `packages/api-client` 的裁決

主站 Web shared API client 收斂已完成歸檔；子方案細節與完成證據保留在 `07-待處理問題與治理/已處理/` 的 Web shared API client 相關任務文件中。本文只裁決 `packages/api-client` 可下沉的長期責任與不得下沉的端側副作用。

`packages/api-client` 可以承接：

1. API response envelope、error / response normalize、request id 與 retry 語義等純 transport contract。
2. auth、session、quick、profile / interview、case、chat、judgment、reconciliation、execution、notification、media upload 等 M1-M5 domain method 的 platform-neutral API shape。
3. 可由 Web / Admin / App 共用的 DTO、query parameter、path builder、AI stream reducer / snapshot selector / terminal helper 與 pure domain helper。

`packages/api-client` 不得承接：

1. `toast`、`react-router-dom` navigation、`window.location`、`localStorage`、`sessionStorage`、`SecureStore`。
2. Web request cancel registry、頁面級 loading / error UI、Admin login redirect、quick session store mutation。
3. Browser FormData boundary 處理以外的 DOM-only side effect；平台差異應留在 Web adapter 或 App platform adapter。

遷移策略固定為 adapter / strangler pattern：保留 `frontend/src/services/api/*` 對頁面層的 export，再把內部 implementation 切到 `@cj/api-client` domain method。當前 M1-M5 domain client 與 AI stream pure helper 已有 shared consumer baseline；驗收入口由 `@cj/api-client` domain tests、主站 API wrapper tests、主站 build 與 diff hygiene 承接。本裁決不包含 Admin domain API 下沉、Mobile release sign-off、Web A11Y / P0 true-service 後續批次或 release 完成宣稱。
