## 問題位置與現象

全局語言排查下一輪確認，`frontend/src/utils/errorMessages.ts` 仍以 `ERROR_MESSAGES: Record<string, string>` 保存一整張繁中文字串表，包括 auth、session、case、interview、system 類錯誤碼。深入復核後確認，該 legacy helper 目前沒有 production import，主要是維護性與防回流風險，不是當前頁面的直接渲染入口。

真正的可見缺口在 `frontend/src/pages/Interview/Chat/index.tsx` 的頁內錯誤 title resolver：頁面以 `getErrorMessage(error, errorCode)` 形式把 store 中的可見 `error` 當作 code、把真正的 `errorCode` 當作 fallback。當 `errorCode` 是 `NOT_FOUND`、`MAX_TURNS_REACHED`、`NETWORK_ERROR` 等 code 時，頁面可能顯示原始 code，未按使用者目前語言顯示本地化錯誤文案。

## 影響範圍

- Web Interview Chat：錯誤狀態 title、rate limit / not found / max turns / consent required 等 action state。
- 前台 legacy helper：`frontend/src/utils/errorMessages.ts` 及其單測，作為防止硬編文案回流的維護性修復。
- 前台 i18n catalog：需為 helper 仍保留的錯誤碼補齊 zh-TW / en-US key parity。
- Backend / App / Admin：本輪不改；後續仍需按全局掃描繼續逐輪處理。

## 目標行為與方案

1. `Interview/Chat` 頁內 resolver 明確改為 `resolveVisibleErrorTitle(errorCode, error)`，讓可識別錯誤碼跟隨目前 locale，未知錯誤保留已正規化可見 message。
2. `interviewChatUtils` 補齊 `NETWORK_ERROR -> common.networkError`，避免 network 類錯誤直出 code。
3. 將 hardcoded `ERROR_MESSAGES` 改為 `ERROR_MESSAGE_KEYS`，錯誤 code 只映射到 i18n key。
4. legacy `getErrorMessage(code, fallbackMessage)` 以 code 對應 key 為優先；未知 code 才顯示非空 fallback message；兩者都不可用時使用 `common.unknownError`。
5. 優先復用既有 `common.*`、`message.*`、`interview.error.*` key，只為 legacy code helper 缺少的 code 補最小必要 `errorCode.*` key。
6. 不改 store error contract、不改 API contract、不新增 page-level ad hoc 中文 map。

## 邊界與注意事項

- 後端或 store 已提供的具體非空 message 仍可作 fallback，避免丟失 domain-specific 訊息。
- `JUDGMENT_*` legacy code 的可見文案需按當前產品術語顯示為「梳理結果 / Analysis」，不得回寫成「判決」。
- Catalog parity 必須保持 zh-TW / en-US key 完全一致。
- 單測需覆蓋 zh-TW 與 en-US 切換，避免再把中文字串當固定契約。
