# 管理端與平台治理總覽

本文件承接 `CJ` 的 Admin Web、平台治理、運維觀測與後台配置能力。

## 1. 本子域回答什麼

本子域只回答以下問題：

1. Admin 端的身份建立、權限矩陣與頁面責任如何劃分。
2. 健康檢查、任務治理、配置治理、審計與報表屬於哪一組正式能力。
3. 什麼屬於後台治理，什麼不應混入前台產品正文。

本子域不回答：

1. quick、formal case、Repair Journey、interview、chat 的前台主流程正文。
2. 品牌、投放、對外敘事與營銷材料。
3. 共享 package 工程落點與 workspace 設計。

## 2. 能力分組


| 分組             | 主要頁面 / 入口                             | 核心責任                                      |
| -------------- | ------------------------------------- | ----------------------------------------- |
| Admin 身份與 RBAC | `/admin/login`、`/admin/me`            | admin token 建立、權限恢復、後台守衛                  |
| 運維觀測           | `/admin/health`、`/health*`、`/metrics` | 健康檢查、就緒探針、Prometheus 指標                   |
| 任務治理           | `/admin/jobs`、`/admin/ops/jobs`       | jobs 查詢、觸發、統計                             |
| 配置治理           | `/admin/configs`、`/admin/settings`    | runtime config、feature flags、alerts rules |
| 用戶與審計治理        | `/admin/users`、`/admin/audit-logs`    | 用戶狀態、審計查詢與導出                              |
| 報表治理           | `/admin/reports`                      | overview、funnel、costs、AI stream sessions  |


## 3. 入口與權限邊界

### 3.1 前台與 Admin 的邊界

主站的 `/admin/*` 只負責轉導，不承接後台正文；真正的後台頁責由 `frontend-admin` 承接。

### 3.2 Admin 身份

後台身份與前台 JWT 分離：

1. 前台用戶登入不等於 Admin 登入。
2. Admin token 不應用來承接前台產品流程。
3. 無 admin token 直連 `/admin/*`（除登入頁）時，需回 `/admin/login`。

### 3.3 RBAC

後台權限按能力域分層，不以「是否是管理員」作粗糙判斷。當前正式能力至少包括：

1. `ops:read`
2. `ops:execute`
3. `config:read`
4. `config:write`
5. `users:read`
6. `users:write`
7. `reports:read`
8. `admin:all`

## 5. 固定規則

以下規則屬後台治理正文：

1. `/admin/login` 是獨立身份建立入口，不與前台 `/auth/*` 混用。
2. `health`、`ready`、`live`、`metrics` 屬運維探針，不等於前台功能頁。
3. jobs、configs、alerts、feature flags、reports 都屬治理層能力，不應回寫到前台功能文檔作主流程描述。
4. Admin 報表中的 AI stream 統計是治理能力，不直接替代前台對 AI stream 的產品語義。
5. `/admin/bootstrap` 這類初始化接口不等於長期正式前台入口；其狀態仍由接口主冊裁決。

## 6. 目前最重要的邊界

`03-管理端與平台治理/` 現階段同時承接 Admin 產品頁與平台運維治理，閱讀時要先分清兩個層面：

1. 如果你在改 `frontend-admin` 的實際頁面或操作流程，先看 Admin 身份、RBAC、對應 API。
2. 如果你在改健康檢查、報表、metrics、jobs、alerts，先把它視為平台治理而不是普通產品頁。

後續若這一層再變大，應優先在本目錄內分成「管理端產品域」與「平台治理域」，而不是把治理內容散回根層。
