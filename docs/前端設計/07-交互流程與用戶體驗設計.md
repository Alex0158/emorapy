# 熊媽媽法庭 - 交互流程與用戶體驗設計

**項目名稱**：熊媽媽法庭（Mother Bear Court）  
**設計階段**：MVP開發階段  
**文檔版本**：v3.6
**最後更新**：2026-04-05

## AI 回覆統一體驗（2026-04 補充）

- 所有 AI 回覆統一遵循 `queued -> thinking -> streaming -> persisting -> final/error`。
- `Interview`：送出後立即顯示 AI bubble；首 token 前展示 thinking 文案；token 到達後無縫切換為流式文本。
- `Interview Store`：內部狀態已與共享流契約對齊為 `thinking -> streaming -> persisting`，避免訪談頁維護另一套命名。
- `Interview` 視圖層應先轉成共享 `AIStreamDraft` 形狀，再交給共享 bubble 元件，不允許頁面自行拼接另一套 draft 結構。
- `Chat Room` 頁面層同樣只允許使用 `aiDraft: AIStreamDraft | null` 作為暫存 AI 回覆入口，不再保留 `streamingAi*` 歷史命名。
- `Chat Room`：`ai_end` 不再等同 UI 結束；AI draft 進入 `persisting`，直到 `message(streamId)` 落庫後才替換為正式消息。
- `QuickExperience Result`：不再只顯示固定 loading 文案，而是展示 `collecting_context / analyzing_emotion / building_responsibility / drafting_judgment / finalizing` 的實時 phase。
- 所有 AI 氣泡都應避免「空白等待後整段跳出」與「暫存 bubble 先消失、正式消息再刷新出現」。
- 這三個場景的前端流狀態轉換與連線生命週期已共用 `useAIStreamSubscription` + `aiStreamState.ts`，不允許頁面自定義另一套 phase/status/retry/replay 語義。
- `Interview` 與 `Chat Room` 的 draft AI 氣泡骨架應共用 `AIStreamingBubble`，只允許外層視覺皮膚差異，不允許分叉出第二套 streaming DOM 語義。
- `thinking` 文案、增量文字 cursor、phase timeline、recovering badge、error/timeout 殼層都已拆成共享元件，不再允許頁面直接內聯拼接第二套 DOM 語義。

## F05 Repair Journey 2.3（2026-04-05 補充）

- 判決頁不再直接主推「生成和好方案」，而是先承接為 4 向分流：
  - 修復
  - 先降溫
  - 體面結束 / 拉開距離
  - 安全支持
- `/reconciliation/:judgmentId`
  - 從純列表頁升級為「方向 + 偏好 + 主推薦」
  - 不只讀 `plans`，還要先讀 `journey_entry.status`
  - 優先展示 1 個主推薦 + 最多 2 個備選；若已有活動旅程，主 CTA 改為續接 / 重調 / 恢復
- `/reconciliation/:judgmentId/:id`
  - 從純詳情頁升級為「共同承諾工作台」
  - 支持單人先行、邀請對方、invitee viewed/accept/deferred/decline、暫停旅程、恢復旅程
- `/execution/:planId/checkin`
  - 從「執行打卡」升級為「今天的一小步」
  - 主問題先問 `done/partial/skipped + closeness + stress + needs_help`
- `/execution/:planId/replan`
  - 新增正式重調頁
  - 只處理同一 intent 內的降壓 / 降速 / 單人先行，不覆蓋歷史
- `/execution/dashboard`
  - 從「打卡進度儀表板」升級為「修復進展看板」
  - 主看 `journey_status + relationship_mode + pulse_summary`
  - 版面分為 `等你決定 / 等對方回應 / 今天的一小步 / 需要重新調整 / 暫停中 / 已完成`
- F05 主要頁面現已優先消費後端 `journey_context`
  - 判斷主 CTA 時，不再前端各頁自己猜 `track.status`
  - `journey_context` 需統一控制 `title / body / primary_cta / secondary_cta / presentation_bucket`

## 🎯 用戶體驗設計原則

### 核心原則

1. **零門檻進入**：快速體驗模式無需註冊，立即使用
2. **清晰引導**：每個步驟都有明確的引導和說明
3. **即時反饋**：每個操作都有明確的視覺反饋
4. **情感化設計**：通過動畫、插畫增強情感連接
5. **錯誤預防**：通過驗證和提示預防用戶錯誤
6. **優雅降級**：網絡錯誤時提供友好的錯誤提示
7. **用戶激勵**：通過成就、積分、徽章激勵用戶持續使用
8. **轉化優化**：在關鍵節點引導用戶註冊和使用更多功能

## 🔄 核心交互流程設計

### 1. 快速體驗流程

#### 1.1 流程圖

```
首頁
  ↓
點擊「立即開始」
  ↓
快速體驗頁面
  ↓
選擇布局（左右/上下分屏）
  ↓
填寫角色A陳述
  ↓
填寫角色B陳述（可選）
  ↓
（可選）上傳證據（判決完成後關閉）
  ↓
點擊「提交案件」
  ↓
顯示「AI正在分析中...」
  ↓
等待30-60秒
  ↓
跳轉到判決結果頁面
  ↓
查看判決結果
  ↓
（可選）選擇下一步方向（需註冊）
  ↓
（可選）保存記錄（需註冊）
  ↓
引導註冊
```

#### 1.2 詳細交互說明

**步驟1：進入快速體驗**

**觸發**：首頁點擊「立即開始」按鈕

**交互**：
- 按鈕點擊動畫（縮放0.98）
- 顯示加載狀態（可選）
- 跳轉到快速體驗頁面
- 頁面加載動畫（漸入，0.5秒）

**步驟2：選擇布局**

**觸發**：進入快速體驗頁面

**交互**：
- 默認選擇「左右分屏」（桌面端）或「上下分屏」（移動端）
- 用戶可點擊切換布局
- 切換時平滑過渡動畫（0.3秒）
- 保持輸入內容不變

**步驟3：填寫陳述**

**觸發**：用戶開始輸入

**交互**：
- **聚焦輸入框**：
  - 邊框顏色變為溫暖橘
  - 邊框寬度變為2px
  - 顯示小陰影（溫暖橘色調）
  - 顯示引導問題（可選）
- **輸入過程**：
  - 實時更新字數統計
  - 不足最低字數時顯示警告提示（警告黃）
  - 超過2000字時禁止輸入（或截斷）
  - 每30秒自動保存到localStorage
- **失焦**：
  - 恢復正常邊框樣式
  - 隱藏引導問題（如果顯示）

**步驟4：提交案件**

**觸發**：點擊「提交案件」按鈕

**前置條件**：
- 角色A必填（至少30字）
- 角色B可選；若填寫需 ≥10字
- 字數不超過限制

**交互**：
- **提交前驗證**：
  - 檢查角色A是否完成（≥30字），角色B可選（≥10字或留空）
  - 檢查字數是否超過限制
  - 驗證通過後啟用提交按鈕
- **提交過程**：
  - 顯示加載動畫（旋轉圖標）
  - 按鈕文字變為「AI正在分析中...」
  - 禁用所有輸入
  - 顯示進度提示：「預計等待時間：30-60秒」
- **提交成功**：
  - 顯示成功提示（Toast）
  - 跳轉到判決結果頁面
  - 平滑過渡動畫
- **提交失敗**：
  - 顯示錯誤提示（Toast）
  - 提供重試按鈕
  - 保持輸入內容不變

**步驟5：查看判決結果**

**觸發**：判決生成完成

**交互**：
- **頁面加載**：
  - 顯示骨架屏（判決書區域）
  - 漸入動畫（0.5秒）
- **責任分比例動畫**：
  - 圓餅圖/進度條從0%到目標比例（1秒動畫）
  - 平滑過渡
- **判決書展示**：
  - 章節導航固定頂部
  - 滾動到對應章節時，導航自動高亮
  - 平滑滾動
- **操作按鈕**：
  - 「看看最適合你們的下一步」：未註冊時顯示鎖圖標，懸停顯示提示
  - 「保存記錄」：未註冊時顯示鎖圖標
  - 「分享判決」：點擊後顯示分享選項彈窗
  - 「語音播放」：點擊後使用TTS播放判決書

**步驟6：引導註冊**

**觸發**：判決結果頁面，未註冊用戶

**交互**：
- **註冊引導區域**：
  - 背景：漸變背景（溫暖橘到柔和藍）
  - 圖標：母熊法官形象（小尺寸）
  - 標題：「想要保存記錄和獲得更多功能？」
  - 功能列表：
    - 「✓ 保存判決記錄」
    - 「✓ 選擇最適合你們的下一步」
    - 「✓ 持續修復旅程」
    - 「✓ 關係健康監測」
  - 「立即註冊」按鈕：主要按鈕樣式（白色背景，溫暖橘文字）
  - 關閉功能：右上角關閉按鈕（可關閉，但會在其他位置再次顯示）
- **點擊註冊**：
  - 跳轉到註冊頁面
  - 保存當前判決ID到sessionStorage
  - 註冊成功後自動關聯當前判決
  - 顯示「註冊成功，已保存判決記錄」提示
- **A/B測試點**：
  - 引導文案（「想要保存記錄」vs「解鎖更多功能」）
  - 按鈕樣式（主要按鈕vs次要按鈕）
  - 顯示時機（立即顯示vs延遲3秒顯示）

### 3. 和好方案流程

#### 3.1 流程圖

```

  ↓
選擇方向（修復 / 先降溫 / 體面結束 / 安全支持）

  ↓

  ↓

  ↓

  ↓

  ↓
```

#### 3.2 詳細交互說明

**方向選擇**：

**觸發**：判決結果頁面完成閱讀後

**交互**：
- 先承接情緒，再展示 4 個方向卡片。
- 卡片文案聚焦「下一步想怎麼走」，不是立刻要求承諾。
- 高風險或安全信號場景，安全支持方向需優先高亮。

**方案旅程頁**：

**交互**：
- 頂部固定旅程條：`理解問題 → 選方向 → 選下一步 → 一起開始 → 持續修復`
- 固定展示 1 個主推薦 + 最多 2 個備選，避免選擇疲勞。
- 每張卡片都需展示：
  - 適配理由
  - 主要在修什麼
  - 現在不適合的情況
  - 卡住時的低壓替代
- 旅程頁必須容忍 legacy `plans[]` 回應與缺失 bundle 字段，不得因 `recommended_plan_id` 缺失而崩潰。

**承諾工作台**：

**觸發**：用戶點擊主推薦或備選的承諾 CTA

**交互**：
- 清楚標示目前是 `單人先行` 還是 `雙方共修`。
- 主 CTA 動態切換：
  - `我願意先開始`
  - `邀請對方一起試`
  - `開始今天的第一步`
- 次 CTA 保留：
  - 換成更低壓版本
  - 先暫停，不等於放棄
  - 重新選方向
- `select` 只代表當前用戶承諾，不得阻斷另一方後續承諾。

### 4. 修復旅程流程

#### 4.1 流程圖

```
承諾方案
  ↓
（可選）邀請對方 / 對方查看與回應
  ↓
開始今天的一小步
  ↓
完成 done / partial / skipped
  ↓
回報 closeness / stress / needs_help
  ↓
如有需要，切到 replanning / pause
  ↓
更新修復進展看板
  ↓
進入下一步或完成本輪旅程
```

#### 4.2 詳細交互說明

**每日一步頁**：

**觸發**：點擊「開始今天的第一步」或從看板續接

**交互**：
- 一次只顯示 1 個 active step，不攤開整份方案。
- 步驟卡需展示：
  - 今天做什麼
  - 低壓替代版本
  - 不舒服時怎麼暫停
- 表單主問題固定為：
  - `done / partial / skipped`
  - `closer / same / farther`
  - `low / medium / high`
  - `needs_help`
- `recent_checkins/records` 缺失時必須回退空陣列，不得因舊 shape 失敗。
- 照片上傳僅作可選佐證，不得搶走主流程焦點。

**重調頁**：

**觸發**：每日一步結果導向 `replanning`，或用戶主動要求換更低壓版本

**交互**：
- 頁面固定說清楚「不是從頭來過，而是把這一輪調整得更適合現在的狀態」。
- 僅提供 3 個模式：
  - `lower_pressure`
  - `slower_pace`
  - `solo_first`
- 提交後不可假裝同步完成，必須切到 AI 等待態：
  - phase timeline
  - recovering badge
  - failed 時明確說明「原版本仍保留」
- 重調後需保留舊歷史與舊版本，不得刪除舊記錄。
- 若要改變大方向，應返回方向頁重新選 intent，而不是在 replan 頁硬改。

**修復進展看板**：

**交互**：
- 主視角改為 `journey_status + relationship_mode + pulse_summary`。
- 看板分為：
  - `等你決定`
  - `等對方回應`
  - `今天的一小步`
  - `需要重新調整`
  - `暫停中`
  - `已完成`
- `journey_status` 未知時，應忽略該卡並回退空狀態，不得渲染空白頁。
- 完成態卡片需提供「下次再卡住時先做哪一步」的回看信息。
- 每張卡都應使用後端返回的 `primary_cta / secondary_cta`，避免前端自行猜測跳轉。

**執行提醒**：

**觸發**：以旅程狀態 + 時間組合驅動

**交互**：
- 未選方向：提醒「先選一個方向，不用急著修好」
- 已承諾但未開始：提醒「今天只做一小步就好」
- 已邀請對方但未查看：提醒對方先看一眼，不催促承諾
- 已邀請對方但未回應：提醒發起方目前仍可單人先行
- 兩次高壓或距離惡化：提醒「建議重新調整方案」
- 暫停後回訪：提醒「可以回來，不代表之前白費」

**通知中心**：

**入口**：Header bell + `/notifications`

**交互**：
- bell 只顯示未讀數，不在 header 內展開複雜菜單。
- `/notifications` 必須按語義分區：
  - 需要你現在處理
  - 等對方回應
  - 重新調整 / 恢復
  - 稍後提醒我
  - 較早通知
- 每條通知卡支持：
  - 點卡片：mark read + 進入 path
  - 主 CTA：mark acted + 進入後端返回的 target.path
  - 對 repair journey actionable：次 CTA 應為 snooze
  - 對歷史 / 較早通知：次 CTA 才是 dismiss
- 通知中心的責任是「跨時間再進場 inbox」，不是取代 dashboard。
- dashboard 與通知中心若同時暴露同一狀態，CTA 語義必須一致。

- Repair Journey 主鏈路不使用積分、勳章、成就來驅動修復決策，避免「交作業感」。
- 如需保留成就或積分，只能放在個人中心或社區層，不得出現在方向選擇、承諾工作台、每日一步、修復進展看板的主 CTA 周邊。

## 🎨 動畫設計

### 動畫原則

1. **自然流暢**：動畫時長適中，過渡自然
2. **有意義**：動畫傳達信息，增強理解
3. **不干擾**：動畫不影響用戶操作
4. **性能優先**：使用CSS動畫，避免JavaScript動畫

### 動畫類型

**過渡動畫**：
- **淡入淡出**（Fade）：
  - 時長：0.3秒
  - 用途：頁面切換、彈窗顯示
- **滑動**（Slide）：
  - 時長：0.3秒
  - 用途：側邊欄、抽屜
- **縮放**（Scale）：
  - 時長：0.2秒
  - 用途：按鈕點擊、彈窗顯示

**狀態動畫**：
- **加載動畫**：
  - 旋轉圖標：1秒一圈
  - 骨架屏：閃爍動畫
- **進度動畫**：
  - 進度條：平滑過渡
  - 責任分比例：從0%到目標比例（1秒）

**反饋動畫**：
- **按鈕點擊**：縮放0.98（0.1秒）
- **卡片懸停**：上移4px（0.3秒）
- **輸入框聚焦**：邊框顏色變化（0.2秒）

### 動畫實現

**CSS動畫**：
```css
/* 淡入動畫 */
@keyframes fadeIn {
  from {
    opacity: 0;
  }
  to {
    opacity: 1;
  }
}

/* 滑入動畫 */
@keyframes slideInRight {
  from {
    transform: translateX(100%);
  }
  to {
    transform: translateX(0);
  }
}
```

**React動畫庫**：
- **Framer Motion**：複雜動畫
- **React Spring**：物理動畫
- **CSS Transitions**：簡單動畫（優先使用）

## 🧠 心理畫像交互流程（v3.0 新增）

> 設計依據：`UPGRADE_PLAN_PERSONALIZED_JUDGMENT.md` v10
> 最高原則：**用戶做越少越好**。用戶只做一件事——和 AI 聊天。分類、分析、推斷、結構化全部由系統背後完成。

### 訪談觸發時機（4 種）

觸發與後端 `interview_trigger_enum` 對應如下：

```
觸發 A（trigger: onboarding，註冊後 → 配對頁面 /profile/pairing）
  條件：psych_consent_given === false（從未同意過）
  → 頂部引導卡片：「想讓未來的判決更懂你嗎？花 5 分鐘和我聊聊」
  → [好啊] → POST /interview/start(trigger: onboarding) → /interview/:id
  → [之後再說] → 隱藏卡片，繼續配對

觸發 B（trigger: pre_case，首次案件前）
  → 「你還沒告訴我關於你自己的事。判決會更準確如果我了解你。」
  → [花 5 分鐘聊聊] / [直接提交]

觸發 C（trigger: post_judgment，判決後）
  → 「想讓下次判決更個人化嗎？」
  → [好啊] / [下次吧]

觸發 D（trigger: organic，個人頁面）
  → 「我的故事」卡片 + 畫像豐富度進度環
```

**關鍵 UX 原則**：所有觸發都有「跳過」選項，**永遠不阻擋用戶的主要操作**。

### 完整訪談流程

```
觸發 → 知情同意（僅首次） → AI 對話（5-15 輪）
  → AI 自動結束（should_end: true，用戶零操作）
  → 轉場動畫 → 後端異步處理 → polling
  → 洞察反饋卡片 → 按觸發上下文回流
```

**知情同意**（進入 `/interview/*` 時由頁面 mount 檢查，⚠️ 無獨立 ConsentGuard 組件）：
- 進入訪談路由時，頁面檢查 `psych_consent_given`（後端 `requireConsent` 中間件為最終防線）
- 未同意 → 展示同意彈窗（僅首次）
- 「你可以隨時跳過、隨時清除所有資料」
- 同意後設置 `psych_consent_given = true`（POST /psych-profile/consent）
- 刪除畫像時重設為 `false`，下次需重新同意

**對話中的 UX 設計**：
- AI 回應逐字流式顯示（降低心理等待感）
- 跳過按鈕：跳過後 AI 退回更輕鬆的話題（v9：不跳到另一個深層話題）
- 暫停按鈕：session 保持 `in_progress`，下次進入提示續聊
- 連點防護：發送後 disable 按鈕，SSE 完成後恢復

**結束判斷（雙層設計）**：
- **AI 軟性目標（~15 輪）**：AI prompt 中設定約 15 輪自然收尾（v9：感知用戶投入度，正在傾訴時不打斷），到達目標輪數後 AI 傾向設置 `should_end: true`
- **第 16-25 輪**：若 AI 在第 15 輪未設 `should_end`，後端仍接受 respond/skip 請求，前端不禁用輸入框。AI 在 16-25 輪範圍內仍可繼續對話，直到設置 `should_end: true` 或到達硬限
- **後端硬限制（25 輪）**：第 25 輪 respond/skip 後，後端返回 `MAX_TURNS_REACHED`（**422**），前端自動觸發結束流程
- **前端不以 15 輪為判斷依據**，完全以後端的 `should_end` metadata 和 `MAX_TURNS_REACHED` 為準

**結束後的用戶旅程**：
- `should_end: true` → 前端自動隱藏輸入框 → 自動調 POST /end → 用戶零操作
- 轉場動畫（10-30 秒）→ polling 結果 → 60 秒超時非阻塞提示
- 洞察反饋卡片展示（觀察性語言，不用專業標籤）
- 主按鈕按觸發動態切換：pre_case→[繼續提交案件]、post_judgment→[查看判決]

### 錯誤處理

| 情境 | 錯誤碼 / 觸發 | 前端處理 | 可重試 |
|------|---------------|---------|--------|
| SSE 連線中斷 | 網絡斷開 | 後端繼續完成 AI call；前端顯示「連線中斷，正在恢復…」；resume 時調 `GET /interview/:id` 載入完整對話 | 自動恢復 |
| Polling 處理中 | **200** + `status: 'processing'` | 繼續輪詢（每 3 秒）。⚠️ 設計曾規劃 202 碼，但代碼統一返回 200 + session 對象。✅ 60 秒超時提示**已實現**（顯示「處理時間較長」+ [繼續等待] 按鈕） | 自動繼續 |
| 後端異步處理失敗 | **200** + `status: 'processing_failed'` | 「分析遇到問題」→ 展示「重試」按鈕 → 調 `POST /interview/:id/retry`（對 processing_failed 重試管線）→ 重新 polling。⚠️ 設計曾規劃 500 碼，代碼為 200 | 是 |
| AI 回應無 metadata | 解析失敗 | 整段視為可見文本，turn 的 intent/domains 設為 null，不影響後續對話 | — |
| 安全警報 | SSE `safety_alert` | ✅ **已實現**：interviewStore 設置 `safetyAlert` 狀態，Chat 頁面渲染 SafetyAlert 組件（含 severity 分級、critical 時顯示危機資源熱線），用戶可手動關閉。 | — |
| 知情同意未完成 | `CONSENT_REQUIRED` (403) | 頁面 mount 時檢查或後端 `requireConsent` 攔截 → 展示 `ConsentModal`；同意後重新調 start（⚠️ 無獨立 ConsentGuard 組件） | 同意後是 |
| Session 不存在或非本人 | `NOT_FOUND` (404)（⚠️ 代碼合併為 `NOT_FOUND`，不區分「不存在」與「無權限」） | Toast「訪談不存在或無權限」→ 導回個人頁面 | 否 |
| Session 已結束 | `SESSION_COMPLETED` (409) | 導向 `/interview/:id`（適用 `completed` 和 `abandoned`）。⚠️ `abandoned` 由後端定時任務在 `in_progress` 超 24h 時自動標記（若已 ≥ 3 輪則先觸發 pipeline 保存數據） | 否 |
| 達到最大輪數 | `MAX_TURNS_REACHED` (**422**) | 自動觸發結束流程（同 `should_end: true`），顯示「感謝你的分享」 | 否 |
| 併發請求 | `CONCURRENT_REQUEST` (409) | 「請等待上一條回應完成」→ 前端 disable 已在處理。鎖在上一輪 turn 持久化 + SSE 流關閉後釋放，前端等待 3 秒後可重試 | 自動恢復 |
| 發送太快 | `TURN_TOO_FAST` (429) | 「請稍等幾秒再發送」→ 3 秒後自動恢復輸入框 | 自動恢復 |
| 開始訪談頻率超限 | `RATE_LIMIT_EXCEEDED` (429) | Toast「你已開始過多次訪談，請稍後再試」→ 導回觸發頁面（不阻塞其他操作）。⚠️ 代碼統一使用 `RATE_LIMIT_EXCEEDED`（設計中的 `START_RATE_LIMIT` 未被使用）。雙層：①中間件每小時 3 次；②業務邏輯每天 5 個 substantive session（僅計 ≥ 3 輪） | 稍後是 |
| AI 調用失敗 | SSE `error` (`AI_CALL_FAILED`) | 「暫時無法回應，請稍後再試」→ 恢復輸入框 | 是 |
| 回應超時（30s 無 token） | 前端計時：respond 後 30 秒內無 token 事件 | 「回應超時」→ 恢復輸入框，可重新發送 | 是 |
| 連線超時（60s 無 SSE 事件） | 前端計時：SSE 連線 60 秒完全靜默 | 「連線超時，正在重試」→ 重試 1 次 → 仍失敗則提示刷新 | 自動重試 |
| 401/403（token 過期） | HTTP 401/403 | 由全局攔截器處理 → 刷新 token 或導向登錄 | — |

### 畫像與判決的關係

```
用戶有畫像 → 判決自動注入個人化分析（用戶不需做任何事）
用戶沒畫像 → 判決使用現有邏輯，品質不降級
```

**用戶永遠不會因為沒完成訪談而被懲罰。**
