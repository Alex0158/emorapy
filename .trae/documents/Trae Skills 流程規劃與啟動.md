## 目標與範圍
- 建立可重複的端到端流程：從需求分解→實施→驗證→沉澱，以 Trae Skills 驅動。
- 聚焦品質閾值：安全、可維護性、國際化、測試覆蓋、文檔一致性。

## 階段一：規劃與開始（develop / improve）
- develop（入口）
  - 收集上下文與約束，分解任務與風險；輸出子任務清單與執行策略。
  - 明確輸入/輸出與品質關卡（lint、測試、審計、文檔同步）。
- improve（既有代碼優化）
  - 鎖定大文件/耦合模組，先分拆再重構；保持行為不變，小步提交。
  - 產出：前後對比、重構說明、影響面與回退策略。

## 階段二：實施與功能開發（i18n + 測試並行）
- add-dict / add-translation / translate（i18n）
  - 在實施中段執行：掃描新增/硬編碼文本、補齊資源字典、輸出腳本與風險清單。
  - 目標：避免後期漏翻，確保全球化準備就緒。
- add-unit-test（測試補強）
  - 與實施並行或緊跟：用 AI 輔助生成初稿，覆蓋邊界情形，逐步逼近 100% 覆蓋。
  - 產出：新增測試代碼 + 覆蓋報告 + 信心指標。

## 階段三：除錯與驗證（fixbug / test-unit / audit / check）
- fixbug（深度修復）
  - 以 Find References + linter 定位根因；小步修補，最多重試 2 次；記錄影響面。
- test-unit（運行所有測試）
  - 小步執行全套測試與覆蓋率，收斂失敗；必要時回到 fixbug。
- audit（原則與安全審核）
  - 檢查 SOLID/DRY/KISS + DevSecOps；量化合規率，低於 90% 提建議與修復清單。
- check（文檔一致性）
  - 比對代碼與設計/規格/API/安全文檔，更新不一致處，記錄修改。

## 技術品質關卡（Quality Gates）
- 測試覆蓋：目標 100%，關鍵模組不得低於 90%。
- 安全：不允許硬編碼憑證；Helmet/CORS/限流等安全中介必須存在；依賴無嚴重漏洞。
- Lint：無 error；warning 逐步清零。
- 文檔：與代碼/規格同步，重要變更須有修改記錄。

## 執行策略
- 小步迭代：每步輸出可驗證成果（代碼、測試、審計報告、文檔更新）。
- 並行化：i18n 與單元測試可與實施並行；審計與文檔在驗證階段串行。
- 回退：每步保留回退路徑（git 分支/補丁），風險變更先灰度。
- 不主動提交：除非你要求，僅在工作區更新並提供可審閱改動。

## 輸入/輸出清單
- develop：任務拆解文檔 + 子任務清單 +風險與關卡。
- improve：重構說明 + 前後對比 + 受影響模組。
- i18n（三技能）：新增條目清單 + 字典更新預覽 + 多義風險清單。
- add-unit-test：新增測試 + 覆蓋率報告 + 信心。
- fixbug：修復詳述 + 根因/影響 + 預防建議。
- test-unit：測試摘要 + 詳細 log + 覆蓋率。
- audit：合規率報告 + 修復建議。
- check：更新清單 + 文檔差異報告。

## 啟動建議（可選）
- 以 develop 啟動：鎖定當前優先任務，生成子任務與時間序列。
- 若涉及既有頁面/服務優化，緊接使用 improve 完成首輪拆分與可視化檢查。

請確認以上流程規劃。確認後，我將以 develop 作為入口，開始執行並在每一階段產出對應成果與報告。
