# Reference

## Evidence Matrix

每輪至少整理出這張矩陣：

| 類型 | 來源 | 關鍵結論 | 狀態 |
|---|---|---|---|
| 文檔 | 產品/流程/接口/測試文檔 | 文檔聲稱什麼 | 已讀 |
| 代碼 | 前端/後端/工具/狀態管理 | 真實行為是什麼 | 已驗證 |
| API | 路由/參數/錯誤碼/字段 | 對外契約是什麼 | 已驗證 |
| 測試 | 單元/整合/E2E | 有無護欄與缺口 | 已驗證 |
| 歷史缺陷 | 缺陷報告/修復報告 | 是否曾出問題、現在是否已解 | 已核對 |

## Issue Status

只使用以下結論分級：

- `已確認問題`：至少兩類證據一致，且能指出真實影響鏈路。
- `高疑似問題`：有明確線索，但還缺一塊關鍵證據。
- `已修復歷史問題`：當前代碼與測試已覆蓋，歷史文檔僅供背景參考。
- `文檔或認知過時`：舊說法與當前真實行為不一致。
- `需要運行驗證`：靜態分析不足，需要瀏覽器、接口或真實資料驗證。

## Cross-Role Prompts

### Marketing Expert

- 這個點會不會讓新用戶失去信任？
- 會不會影響第一分鐘內的轉化？
- 用戶是否會因結果不可信而停止分享或回訪？

### Product Manager

- 規則是否前後一致？
- 狀態轉移是否完整？
- 這個設計是在解真正問題，還是在補表象？

### Real User

- 用戶看到這一步時會怎麼理解？
- 會不會覺得自己做錯了但其實是系統問題？
- 會不會覺得產品在責怪、說教或不公？

### Risk And Abuse

- 是否能被濫用、刷量、繞過或誤觸？
- 是否存在權限、資料、風險或公平性漏洞？
- 是否會產生錯誤但沒有可理解的恢復路徑？

## Optimization Scoring

方案可用以下 4 維快速評估：

| 維度 | 問題 |
|---|---|
| 影響 | 對轉化、信任、穩定性影響有多大 |
| 確定性 | 是否有足夠證據支持這是值得做的 |
| 成本 | 前後端、測試、文檔改動成本高低 |
| 風險 | 是否可能引入新的誤導、回歸或運營風險 |

預設優先順序：

1. 高影響、高確定性、低到中成本
2. 高影響、高確定性、高成本
3. 中影響但能修復核心信任問題

## Analysis Notes

- 若一個問題同時涉及「內容真實性」與「顯示方式」，先拆成生成、保存、回傳、顯示、感知五層。
- 若流程牽涉 session、token、case 映射、輪詢、重試，務必畫出憑證流轉與錯誤恢復路徑。
- 若某問題曾被報告過，先核對是否已有修復文檔、測試與代碼對照，再決定是否列為現存問題。

## Analysis To Implementation Boundary

分析層到此為止時，應只交付：

- 已確認問題
- 已排除問題
- 推薦方案與替代方案
- 風險與待驗證項
- 是否允許進入落地層的判定

不應在分析層內混入：

- 直接修改哪些檔案的具體 patch
- 未經確認就開始的業務代碼改動
- 尚未穩定就被當成既定方案的實作細節

若要開始真正落地，改用 `business-flow-implementation-handoff`。
