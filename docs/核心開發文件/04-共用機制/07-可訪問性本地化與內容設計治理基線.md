## 1. 定位

本文把 CJ 的「可訪問性、本地化與內容設計」從單條 NFR 提升為可治理基線。頂級工程級 PRD / SRS 通常不只描述功能可用，還會說明用戶能否用鍵盤、輔助技術、不同語言、不同設備和低認知負擔完成完整流程。

本文不宣稱 CJ 已通過 WCAG、平台商審核或任何外部無障礙認證。當前 Web / Admin 有 i18n catalog、ARIA / focus / role 測試、static A11Y contract、route-level axe smoke 與 manual evidence contract；App 已有 M0-M5 screen、shared UI accessibility contract、TextInput label / hint static gate 與 copy contract baseline。這些都只構成工程基線，仍不能宣稱 App accessibility 或 localization 已完成，因 native screen reader、Dynamic Type、VoiceOver / TalkBack、真機或平台級 accessibility evidence 尚未閉環。

## 2. 外部基線參考

| 基線 | 採用原因 | CJ 採用方式 |
| --- | --- | --- |
| WCAG 2.2 | Web content accessibility 的 W3C Recommendation，提供可測成功準則 | 用於校準 perceivable / operable / understandable / robust 與 A / AA / AAA，不宣稱 conformance |
| WAI-ARIA / APG | 補充複雜互動元件的 role、name、state、keyboard pattern | 用於校準 dialog、menu、tabs、select、tooltip、status message 等元件語義 |
| ISO 9241-210 | human-centred design for interactive systems | 用於要求以目標用戶、使用情境、可用性與迭代證據約束內容設計 |
| W3C Internationalization Authoring | Web 語言標記、方向、字元編碼、locale authoring | 用於校準 `lang`、文本方向、編碼、切換語言與 fallback |
| Unicode CLDR | locale data 與國際化資料來源 | 用於提醒日期、數字、複數、區域格式不得靠臨時字串拼接長期治理 |
| Android Accessibility / React Native Accessibility | Native App 可訪問性、label、role、touch target、screen reader 與平台測試 | 用於 App 級 native accessibility gate；Web 證據不能替代 App 證據 |

## 7. 維護規則

1. 新增可見文案、錯誤提示、AI waiting / failure、刪除/保留、危機/安全分流文案時，必須同步 i18n catalog 與術語表，不能只寫中文硬編字串。
2. 新增 icon-only button、dialog、menu、select、toast、AI stream status 或 upload 控件時，必須有 accessible name、keyboard/focus 口徑與測試或 inspection。
3. 修改 locale、語音朗讀、日期/數字/時間/百分比格式時，必須明確使用 `getLocale()` 或等價 locale policy，不得各頁自行拼接。
4. Web / Admin / App 的 locale catalog 若已支持某語系，該語系 known key 必須在 `t()` 可被呼叫前可用；不得用非同步空 catalog 讓 selected locale 短暫回退到另一語系文案。
5. App 端新增 screen 或 native adapter 時，必須同步 accessibility label / role / hint、screen reader、touch target、Deep Link / Push failure 文案與 App 測試證據。
6. Backend 新增會進入 API response / stream payload 的自有成功、錯誤、validation 或 status message 時，必須同步 `backend/src/i18n/index.ts` 的 en-US 翻譯與 focused test；server log、ops alert 與第三方 raw detail 只有在進入 UI 時才納入。
7. Backend 新增會寫入資料庫、之後由 Web/App 直接顯示的自動文案時，必須在 backend 生成點承接 locale 與核心術語；前端不得為同一資料欄位建立臨時翻譯表。
8. 未完成 WCAG / screen reader / axe / App native 測試前，不得宣稱「可訪問性合規」或「App accessibility 已完成」。

## 8. Web 十項修復裁決

Web / Admin 十項問題已按「測試地基、可訪問性 / 本地化語義、真服務證據、shared API client」四層完成工程治理歸檔，子方案與完成證據保留在 `07-待處理問題與治理/已處理/`。本文只保留可復用裁決；橫向可訪問性證據仍不得由局部 gate 或單次測試替代。

正式裁決如下：

1. icon-only button 必須同時具備 `type="button"` 與明確 accessible name；關閉、清除、刪除、返回、稍後再說不得共用含糊 label。
2. `div role="button"` 只可作為無 native HTML 替代時的最後手段；通知卡片 open 行為與 Chat reply preview action 應改用 native `<button>` 或 explicit link / button，並確保 action buttons 不被 nested interactive 包住。
3. Web / Admin 的 `html[lang]` 必須與 default / selected locale 一致；locale switch 成功後必須同步 `document.documentElement.lang`。
4. 日期、時間、數字與百分比格式不得散落在頁面用裸 `toLocaleString()`；應進 locale-aware helper 並有測試或 inspection 證據。
5. i18n catalog completeness / fallback gate 建立前，不得因 zh-TW / en-US catalog 存在而宣稱雙語完整。
6. Input、Textarea、Select、file input 與 credential field 不得只靠 placeholder 或鄰近文字；必須能以 `getByLabelText` 或 `getByRole(..., { name })` 定位，credential field 必須有合適 autocomplete。
7. accessibility / i18n 修復完成前，必須先清理或明確登記相關 skipped tests；總測試 pass 不能覆蓋 skip 仍存在的事實。
