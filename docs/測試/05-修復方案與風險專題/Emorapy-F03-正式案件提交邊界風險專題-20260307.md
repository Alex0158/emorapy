# Emorapy F03 正式案件提交邊界風險專題

日期：2026-03-07  
關聯功能：`F03`、`F04`  
關聯流程：`P02 正式案件閉環`

## 1. 專題背景

F03 的風險不只在能不能建案，而在「什麼時候算可以提交」。只要提交邊界鬆掉，後面的 review、judgment、reconciliation 都會建立在不完整資料上。

本輪確認，正式案件至少有三條高風險邊界：

1. pairing 是否已就緒
2. `draft -> submitted` 是否只在條件滿足時發生
3. `Case Detail` 是否把等待/提交/跳 review 三種狀態分清

## 2. 本輪確認的風險

### 2.1 風險 A：正式 collaborative 可繞過雙方陳述要求

修復前表現：

1. 前端 `Case Create` 要求雙方陳述
2. 但後端 `POST /cases` 可直接收只有原告陳述的 `mode=collaborative`
3. 案件直接進 `submitted`

當前狀態：`已修復，降為回歸風險`

### 2.2 風險 B：remote draft 的 detail 頁若誤顯示 submit，會提前送案

當前規則應為：

1. 被告尚未回覆時
2. 原告只能看到 waiting 提示
3. 不應看到 submit 按鈕

本輪已補前端頁面測試固定這條規則。

### 2.3 風險 C：Review 入口依賴前序狀態正確

只要 F03 前面把狀態錯標：

1. review 頁就會錯進輪詢
2. judgment 生成會過早觸發
3. 後續 F04/F05 觀察到的錯誤其實根源在 F03

## 3. 已完成修復方案

### 3.1 後端

1. `createCaseSchema` 對 collaborative 補必填 `defendant_statement`
2. `case.service.createCase()` 對 collaborative 補業務防線

### 3.2 測試護欄

1. `case.service.test.ts` 固定缺少被告陳述時必須失敗
2. `validation-schemas.test.ts` 固定 schema 層不可放行
3. `CaseDetail.index.test.tsx` 固定：
   - 原告等待被告時不可 submit
   - 可 submit 時必須導向 review

## 4. 建議的後續回歸清單

1. `Case Create` collaborative 模式前端提交護欄
2. `Case Review` 的 draft redirect 與 judgment_failed retry
3. `Case List` 的狀態/排序/搜尋矩陣
4. 正式 evidence upload/delete 在 draft/submitted 下的權限與恢復

## 5. 結論

F03 的核心風險不是頁面數多，而是「提交邊界一旦鬆掉，後續整條正式案件鏈都會在錯誤前提上運行」。`F03-BUG-001` 已修復，但 F03 後續迭代仍需持續把 `draft/submitted/review` 的狀態邊界當作最高優先級護欄。
