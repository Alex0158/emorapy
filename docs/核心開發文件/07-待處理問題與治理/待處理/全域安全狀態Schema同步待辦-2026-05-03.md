## 背景

後端已新增 migration：

- `backend/prisma/migrations/20260503235500_add_safety_assessment_models/migration.sql`

此 migration additive 新增：

- `SafetyRiskLevel`
- `SafetyAssessmentSubjectType`
- `SafetyAssessmentSource`
- `safety_assessments`
- `relationship_risk_states`
- `relationship_risk_states_one_active_scope_idx` partial unique index，確保每個 scope 同時最多一筆 active risk state

同時 `backend/src/utils/product-safety-policy.ts` 已新增 route snapshot helper，可把 `standard / safety_support / crisis_support` 映射為可直接寫入安全狀態表的欄位快照；`backend/src/services/safety-assessment.service.ts` 已提供單一讀寫入口，封裝 `SafetyAssessment` 建立、active `RelationshipRiskState` 刷新、active state 讀取與 route fallback snapshot，但暫未接入既有 runtime 路徑。

## 必須同步的兩邊

1. Supabase Dev DB：本機開發版使用的資料庫。
2. Release / Production DB：Railway backend 實際連接的 Supabase/Postgres 資料庫。

## 驗證命令

在對應環境配置好 `DATABASE_URL` 後執行：

```bash
npm run ops:db:status
cd backend && npx prisma migrate status
```

套用後至少驗證：

```bash
cd backend && npm test -- --runInBand tests/unit/utils/product-safety-policy.test.ts
cd backend && npm test -- --runInBand tests/unit/services/safety-assessment.service.test.ts
cd backend && npm run build
npm run docs:check
```

## 後續接入要求

1. 先把 `chat-to-judgment` 的 route / safety notice 寫入 `SafetyAssessment`。
2. 再把 `Case.safety_metadata` / `Evidence.safety_metadata` 產生的 assertion 同步寫入 `SafetyAssessment`。
3. `RelationshipRiskState` 只能由明確來源 assessment 或人工 admin review 更新；不得由前端單方隱式覆蓋。
4. repair eligibility、notification、responsibility ratio visibility 後續應透過 `SafetyAssessmentService.getEffectiveRouteSnapshot` 優先讀 active `RelationshipRiskState`，再 fallback 到現有 route-based helper。
5. 完成 dev / release DB migration 與最小 smoke 後，把本文件移入 `已處理/` 或改寫為具體接入任務。

## Release 注意事項

- 已提交 migration 不代表 release DB 已套用。
- 發布 backend 前，必須確認 release DB migration 狀態。
- 若 release DB 尚未套用，後續任何寫入 `SafetyAssessment / RelationshipRiskState` 的 runtime 變更都會失敗。
- 此 migration 目前只新增表、enum 與 index，不改既有 API 行為；可先行部署 schema，再分批接入讀寫路徑。
