## 2. 文件服務 `file.service.ts`

### 2.1 實際能力

- Multer 落盤上傳（`upload`），限制檔案類型與大小，預設最多 3 檔。
- 檔案雙重驗證：副檔名 + MIME + Magic Number（內容簽名）校驗。
- 圖片壓縮：`sharp` 旋轉校正、縮放、轉 JPG。
- 影片處理：使用 `fluent-ffmpeg` + `ffmpeg-static` 產生縮圖。
- 下載保護：支援 URL 簽名（JWT token + nonce + 檔案特徵）。

### 2.2 安全重點

- 拒絕非法副檔名與 MIME 偽造。
- Magic Number 不匹配時刪除檔案並報錯。
- 僅對同源 `uploads` URL 進行簽名，避免 token 泄露到第三方網域。
- `ALLOW_PUBLIC_UPLOADS=true` 時可跳過簽名（明確風險開關）。

## 3. 郵件服務 `email.service.ts`

### 3.1 實際能力

- 若 SMTP 配置完整（`SMTP_HOST/USER/PASS`），啟用 Nodemailer；否則進入「記錄警告並跳過發送」模式。
- 支援郵件類型：
  - 驗證碼（註冊/重置/驗證郵箱）
  - 配對成功通知
  - 判決完成通知

### 3.2 防護與容錯

- 對 email 做換行與控制字元清理（header injection 防護）。
- 發信失敗寫 logger，部分通知場景採「不阻斷主交易」策略（記錄錯誤不拋出到上層）。

---

## 4. 內容推薦服務 `content.service.ts`

- `listContent`：依內容類型、標籤、語言、啟用狀態查詢內容庫。
- `getRecommendations(caseId)`：查案件關聯內容（`case_content_links`）。
- `linkContent`：案件與內容建立關聯，使用 upsert 保證幂等。
- 會先驗證案件與內容是否存在，不存在直接回 `NOT_FOUND`。

---

## 5. 通知服務 `notification.service.ts`

- `list`：取用戶通知（可按狀態篩）。
- `create`：建立 pending 通知（template + payload + channel + dedup_key）。
- `markAsSent` / `markFailed`：通知投遞結果回寫。
- `getPending`：批量拉待發通知（含用戶通知開關）。
- `createIfEnabled`：先檢查 `notification_enabled`，未啟用則跳過建立。

## 6. 後台管理服務群

## 6.1 `admin.service.ts`

- 管理員初始化（bootstrap）：要求 `ADMIN_BOOTSTRAP_TOKEN`，並做 timing-safe 比對。
- 管理員登入：驗證密碼後簽發 admin JWT。
- 角色與權限：保證預設角色存在（super_admin/ops/marketing/support）。
- 稽核審計：敏感操作寫入 `audit_logs`。
- 管理員帳號維護：列表、建立、更新、刪除（軟刪或狀態變更）。

## 6.2 `system-config.service.ts`

- `getConfigValue/getBooleanConfig/getNumberConfig` 三種讀取接口。
- 配置讀取失敗不使系統崩潰，回退 fallback 並記錄 warning。
- 典型用途：運行時開關、閾值與策略調優。

## 7. 聊天支撐服務

## 7.1 即時事件匯流 `chat-events.service.ts`

- 以 room 為維度維護 listener 集合（記憶體 pub/sub）。
- 事件型別：`message`、`invite`、`room_status`、`system`。
- 單房監聽上限（200）超出即回限流錯誤，避免 SSE/連線爆量拖垮節點。

## 7.2 指標統計 `chat-metrics.service.ts`

- Prometheus counter：消息量、限流命中、AI 觸發、安全命中、判決成功/失敗。
- Redis 分鐘桶累計（TTL 3 天），支撐運維告警與趨勢分析。
- Redis 不可用時降級為僅本進程計數（不阻斷主流程）。

---

## 8. 運維告警服務

## 8.1 告警檢查 `ops-alerts.service.ts`

- 拉取 `/health` 檢查鎖後端健康（`lock` check）。
- 讀 Redis HTTP 分鐘桶，計算 5xx 比例與 409 比例。
- 支援樣本數門檻，避免低流量時誤報。
- 可選 Slack 發送，且有 Redis 去重窗口避免告警風暴。

## 8.2 依賴服務

- `slack.service.ts`：統一 Slack webhook 發送封裝。
- `ops-metrics.service.ts` + middleware `opsMetrics.ts`：HTTP 狀態碼分鐘桶寫入來源。

## 9. 服務層共通實作規則（本倉庫現況）

- **實例模式**：多採「class + singleton export」（例如 `export const emailService = new EmailService()`）。
- **事務策略**：資料一致性要求高的流程由核心服務用 Prisma transaction；支撐服務多做「最佳努力」。
- **錯誤策略**：
  - 核心交易（如資料不存在、權限不合法）拋出業務錯誤。
  - 邊際能力（郵件、告警、次要指標）傾向記錄後降級，不反向打斷主交易。
- **配置讀取**：優先 `env`，必要時讀 `system_config` 做 runtime 覆蓋。
