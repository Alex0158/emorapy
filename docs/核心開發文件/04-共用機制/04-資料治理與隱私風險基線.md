## 1. 定位

本文把 Emorapy 的資料治理與隱私風險從 NFR 中拆成可審計基線。它參考 NIST Privacy Framework、ISO/IEC 27701:2025、NIST CSF 2.0 與既有安全/NFR 文件，但只作工程治理對標；不宣稱 Emorapy 已完成 PIMS、GDPR、等保、HIPAA 或任何外部隱私合規認證。

本文屬共用機制級。Web、App、Backend、Admin、AI runtime 或 release evidence 只要新增、保存、輸出、刪除或導出個人資料、親密關係資料、心理推斷資料、安全風險資料或運維日誌，都必須回查本文。

## 2. 外部基線採用口徑

| 外部基線 | Emorapy 採用方式 | 不採用 / 不宣稱 |
| --- | --- | --- |
| NIST Privacy Framework 1.0 | 用 Identify-P / Govern-P / Control-P / Communicate-P / Protect-P 檢查資料處理是否有 inventory、owner、choice、communication 與 protection | 不宣稱已完成 privacy framework profile |
| ISO/IEC 27701:2025 | 用 PII controller / processor、PIMS、accountability、risk treatment 與 continuous improvement 校準隱私治理欄位 | 不建立完整管理體系模板，也不宣稱 ISO 認證 |
| NIST CSF 2.0 | 用 Govern / Identify / Protect / Detect / Respond / Recover 把隱私事件接入平台事故治理 | 不把資安框架替代產品隱私政策或法律條款 |
| 既有 Emorapy NFR / AI 基線 | 把資料最小化、AI ledger caller metadata 不傳 prompt 原文、output gate、App storage 差異納入同一套資料治理 | 不把單點控制寫成完整隱私合規 |

## 3. Emorapy 資料分類

| 資料級別 | 類型 | 典型來源 | 風險 | 當前證據 | 狀態 |
| --- | --- | --- | --- | --- | --- |
| EMO-DATA-0 | 公開或非個人資料 | 靜態文檔、公開設定、版本資訊 | 低 | `GET /version`、文檔與靜態配置 | 已覆蓋 |
| EMO-DATA-1 | 帳號與身份資料 | email、password hash、login 狀態、token version、admin token | 高 | `User`、`AdminUser`、auth routes | 部分覆蓋 |
| EMO-DATA-2 | 親密關係與案件資料 | case statement、evidence、chat message、judgment、repair journey、notification payload | 高 | `Case`、`Evidence`、`Chat*`、`Judgment`、`Repair*`、`Notification` | 部分覆蓋 |
| EMO-DATA-3 | 心理訪談與推斷資料 | interview turns、profile narrative、profile insight、feedback card、richness score | 高 | `InterviewSession`、`InterviewTurn`、`ProfileNarrative`、`ProfileInsight`、`ProfileSnapshot` | consent / delete 部分覆蓋 |
| EMO-DATA-4 | 安全風險與危機資料 | safety assessment、relationship risk state、reasons、metadata | 極高 | `SafetyAssessment`、`RelationshipRiskState` | 部分覆蓋 |
| EMO-DATA-5 | AI 與運維證據資料 | AI request ledger、AI stream events/archive、logs、metrics、release evidence | 中高到高 | `AIRequestLedger`、`AIStream*`、ops logs / metrics | 部分覆蓋 |

任何資料若同時落入多級，以更高級別處理。心理推斷、危機、安全風險、親密關係原文不能因保存於 metadata、archive、log 或 evidence 中而降級。

## 5. 強制規則

1. 新增資料表、JSON metadata、archive、log 欄位或 evidence payload 時，必須標記其最高資料級別。
2. 涉及 `EMO-DATA-2` 以上資料時，MUST 明確資料 owner、讀寫入口、保留期限、刪除/撤回口徑與測試/審查入口。
3. 心理畫像 delete 只能說「刪除心理畫像相關資料並撤回 consent」；在 `ProfileSnapshot` 保留前，不得對外寫「全量刪除所有心理相關紀錄」。
4. AI ledger callers 不得傳入 prompt 原文、完整聊天文本或可重建私密敘事的 metadata；新增 metadata 必須使用白名單思路，而不是任意透傳。補齊 service-level guard 前，正式文件只能說現有 callers 遵守最小化，不得宣稱 `AIRequestLedgerService` 已強制白名單。
5. AI stream archive 若保存文本或完整 error，按 `EMO-DATA-5` 高敏資料處理；不得因其是「運維證據」而降低保護。Admin 查詢必須由 backend 強制 redaction；普通報表只能取得 error 的低敏 `code/retryable`，只有 `admin:all` 或 `reports:sensitive:read` 可取得原文與完整 error，且敏感 read 必須可稽核。
6. 生產日誌不得輸出 session id 明文、JWT、admin token、verification code、DB URL、API key、prompt 原文或完整私密敘事。
7. Web storage、SecureStore、Push payload、Deep Link query、telemetry event 不得保存 `EMO-DATA-2` 以上資料，除非先在 App / Parity 文件中完成明確例外裁決。
8. release evidence、manual regression、bug report 若需要引用用戶資料，必須先匿名化、最小化，並優先使用 synthetic data。
9. `EMO-DATA-2` 以上資料的 consent 必須綁定 subject、purpose、audience、target、exact version/hash、granted/revoked/expired timestamps；全域 consent 或由另一方提交的 boolean 不得當作共同披露、正式梳理或跨案件重用授權。
10. 私人原文、心理推斷與安全理由不得因轉成 summary、snapshot、AI stream、diagnostic context 或 Repair personalization 而擴大 audience；若需要共同使用，必須建立由資料擁有人預覽及批准的獨立 Context Capsule。
11. 私人原文送往 external AI 前必須完成不含原文的 durable requested audit；audit 失敗即 fail closed 且 provider call 為零，provider 後只可追加低敏 outcome。
12. Chat 訊息是否可進入 AI context 必須由持久、deny-by-default 欄位裁決；display visibility 或 legacy channel 分類不能隱式授權後續 AI 重用。

## 6. Web / App 分層

| 維度 | Web級口徑 | App級口徑 |
| --- | --- | --- |
| 本地保存 | Browser storage 只保存身份、session 或必要 UI 狀態；不得把心理/關係原文當持久快取 | SecureStore 只承接 token / session 類資料；AsyncStorage 不得保存高敏原文 |
| 通知 | Web in-app notification payload 需最小化 | Push notification payload 不得含關係原文、心理推斷、危機細節或安全評估理由 |
| Deep Link | 只傳 route / id / action，不傳敏感正文 | 同 Web，且需考慮系統通知、剪貼板、recent apps 暴露 |
| AI stream | Web 可顯示 stream text，但持久化與 replay 由 backend 裁決 | App background/reconnect 不得自行保存完整 stream 文本作補償 |
| Telemetry | backend metrics / logs 與 App telemetry 最小化摘要分離 | App telemetry 已建立 safe event schema、backend 二次清洗、30d event retention 與 Admin 聚合報表；仍不得保存關係原文、心理推斷、prompt payload、push token、JWT、session id 或完整 device fingerprint |

## 8. 驗收口徑

1. Test：branch 已有 private/shared projection、purpose authorization、participant stream、capsule hash/revoke、exact approval、ProfileSnapshot no-raw、Decision Core/Renderer 與 Repair audience focused tests；production-like malicious echo、Redis/archive、migration/backfill、legacy inventory 與跨端真服務仍需 release-blocking evidence。AI ledger metadata 白名單與 cleanup dry-run 分支亦仍待補。
2. Analysis：schema / service / route 新增資料欄位時，必須能映射到 `EMO-DATA-*`。
3. Inspection：PRD / NFR / RTM / 接口文檔新增資料處理時，必須標明 Web / App / Backend / Admin 的資料責任。
4. Demonstration：release evidence、手動回歸、事故記錄使用真實資料前，必須先完成匿名化或使用 synthetic data。
5. Baseline Pending：DSAR、帳號刪除、全域 retention、native crash / OTel telemetry、archive purge 均保持待建立基線；App telemetry safe event retention baseline 不得被寫成完整隱私治理完成。
