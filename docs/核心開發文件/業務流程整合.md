## 0) 跨端定位

- `P01-P05` 是產品流程 ID，不是單一 Web 路由 ID；本文以 Web/Admin 現碼描述當前流程閉環、狀態機與回歸腳本。
- 上游產品需求以 `00-跨端產品核心/01-產品PRD總章.md` 的 `CJ-PRD-*` 為準；本文只描述這些需求在 Web/Admin 現碼中的流程閉環。
- 跨 Web / App 的產品語義以 `00-跨端產品核心/00-跨端產品核心總覽.md` 為上位依據；App 是否承接、如何承接、與 Web 有何差異，以 `20-App端/00-App端總覽.md` 與 `50-跨端Mapping與Parity/00-跨端Parity總覽.md` 為總覽入口。
- 進入 App 實作時，App navigation、Deep Link、session restore、SecureStore / Push / upload adapter 的落地 gate 以 `20-App端/01-App導航與平台Adapter基線.md` 為準；能力到 Backend / API / DB / shared package 的 App 工程對照以 `50-跨端Mapping與Parity/01-App首輪能力與工程落點Mapping.md` 為準。
- 進入 App 驗收或證據留存時，App smoke / regression / CI / evidence 的進場 gate 以 `08-測試規範與驗收/03-App測試與證據接入基線.md` 為準；不得用本文 Web/Admin 流程已完成推斷 App 流程已驗收。
- 本文中的路由例子均為 Web/Admin 實作例子；不得把 `/case/*`、`/quick-experience/*`、`/admin/*` 直接視為 App 導航設計。
- `P05 Admin 運維閉環` 只屬 Web Admin / 平台治理，不作為一般用戶 App 端能力承接項。

## 1.2) 流程步驟完成度口徑（v2.1）

- `已完成`：步驟可在當前流程路徑直接觸發，且有前端/後端對應實作證據。
- `跨功能依賴`：本步驟需要跳轉到其他功能頁或借用他功能能力完成閉環。
- `待驗證`：接口存在但流程入口或前端調用證據不足（本文件 P01~P05 目前無此狀態）。
- P05 的 `已完成` 只裁決當前 `frontend-admin` 已有頁面 / service 可直接操作的流程；backend-only Admin API 若尚無 Admin Web service / page，仍以接口主冊與治理待辦的 `待承接` 狀態為準。

## 1.3) Case 詳情/判決授權分流（全流程共用）

- `GET /cases/:id`、`GET /cases/:id/judgment` 在現碼不是單一 session 規則。
- `quick` 與 `collaborative(session_id 有值)`：必須匹配 `session_id`。
- `remote` 與 `collaborative(session_id=null)`：必須是案件當事人 JWT（`plaintiff_id/defendant_id`）。
- 以上是授權結果口徑；後端實作必須使用 `backend/src/utils/case-classifier.ts`，不得在流程頁或 service 內重新手寫 `mode` 分支。產品流另由 `product_flow` / `ChatToCaseLink` 裁決，`chat_to_case` 不可被 `remote` / `collaborative` 表層模式覆蓋。

## 7) 關鍵狀態機與分支規則

## 7.1 Case 狀態機

| 狀態 | 進入條件 | 可轉移 | 禁止操作 |
|---|---|---|---|
| `draft` | 建案成功 | `submitted`、`update`、`cancelled` | 直接執行/和好 |
| `submitted` | submit 成功 | `in_progress`、`judgment_failed`、`cancelled` | 再次編輯 draft 字段 |
| `in_progress` | 判決生成或聊天室轉判決進行中 | `completed`、`judgment_failed`、`cancelled` | 重複生成需走顯式重試/恢復邏輯 |
| `judgment_failed` | 生成判決失敗 | `in_progress`、`cancelled` | 不應直接視為完成 |
| `completed` | 判決/主鏈路收口完成 | 後續由 judgment / repair journey 消費 | 不再回退為 draft |
| `cancelled` | 用戶或系統主動關閉 | 終止主鏈路 | 不再進入判決生成 |

## 7.2 Interview 狀態機

| 狀態 | 進入條件 | 分支規則 |
|---|---|---|
| `in_progress` | start 成功 | 超過最大輪次或主動 end 才離開 |
| `processing` | end 且內容充分 | 進 async pipeline |
| `completed` | end 且內容不足或 pipeline 成功完成 | 結束並可進 result / my-story |
| `processing_failed` | pipeline 失敗 | 可 `retry` 回 `processing` |
| `abandoned` | 長時間無更新被清理任務收口 | 提示恢復或重新開始 |

## 7.3 ChatRoom 狀態機

| 狀態 | 典型事件 | 分支規則 |
|---|---|---|
| `solo_active` | 建房、B 方離開 | 可發邀請、可發訊息 |
| `invite_pending` | 發邀請後 | 接受 -> `invite_accepted`；邀請全拒/過期 -> `solo_active` |
| `invite_accepted` | B 方已接受邀請 | 進一步收口到 `group_active` | 房間雙方角色初始化完成 |
| `group_active` | 雙方都在房 | A 方可 request judgment |
| `judgment_requested` | 發起轉判決 | 成功 -> `judgment_completed`；失敗 -> `judgment_failed` |
| `judgment_completed` | 判決建立完成 | 可查狀態、跳轉判決 |
| `judgment_failed` | 判決流程失敗 | 允許重試 request judgment |

## 9) 高風險回歸腳本（QA 基線）

> 說明：本節保留穩定的主流程回歸基線；具體執行證據、批次結果與 dated 驗收記錄應留在 `08-測試規範與驗收/`、`測試/` 或 `90-證據與盤點/`。
> 標記：`P0` 必測，`P1` 建議每版測，`P2` 變更觸發測。

### 9.1 P01 快速體驗腳本

#### Script P01-S01（P0）快速建案到結果頁

- 前置：未登入、清空本地 `token/session`。
- 步驟：
  1. 進入 `/quick-experience/create`。
  2. 輸入合規陳述並提交。
  3. 觀察是否跳轉 `/quick-experience/result/:id`。
  4. 在結果頁等待判決拉取完成。
- 預期：
  - 已發送 `POST /sessions/quick`、`POST /cases/quick`、`GET /cases/:id`、`GET /cases/:id/judgment`。
  - 結果頁可顯示判決或 pending 态提示，不白屏。
- 失敗判定：
  - 任一 API 401 循環重試超過 2 次。
  - result 路由存在但核心區塊無資料且無錯誤提示。

#### Script P01-S02（P0）Session 過期恢復

- 前置：已有 quick case，手動清除或篡改 `X-Session-Id`。
- 步驟：
  1. 直接訪問 `/quick-experience/result/:id`。
  2. 觸發案件讀取。
- 預期：
  - 前端清除舊 session，嘗試 `POST /sessions/refresh`。
  - 恢復成功後可重拉案件；失敗則導回 `/quick-experience/create`。
- 失敗判定：
  - 出現無限 toast 或重導循環。

### 9.2 P02 正式案件腳本

#### Script P02-S01（P0）建案-提交-判決

- 前置：已登入且 `pairing=active`。
- 步驟：
  1. 進入 `/case/create`，提交案件。
  2. 進 `/case/:id` 點提交。
  3. 進 `/case/:id/review` 觸發判決生成。
  4. 跳轉 `/judgment/:id`。
- 預期：
  - case 狀態按 `draft -> submitted -> in_progress -> completed/judgment_failed` 推進。
  - `POST /judgments/generate/:id` 成功後可讀到判決詳情。
- 失敗判定：
  - submitted 狀態仍可編輯 draft 字段（違規）。

#### Script P02-S02（P1）證據上傳與刪除

- 前置：已有 case（draft 或 quick）。
- 步驟：
  1. 上傳 1 張合法圖片。
  2. 刪除該證據。
  3. 嘗試上傳超過限制或錯誤 MIME 文件。
- 預期：
  - 成功時 evidence 列表即時變更。
  - 非法文件返回 `FILE_TOO_LARGE/INVALID_FILE_TYPE` 並保留頁面。
- 失敗判定：
  - 成功與失敗文件狀態混淆（成功項被回滾）。

### 9.3 P03 訪談腳本

#### Script P03-S01（P0）Consent 閘門

- 前置：新帳號，`psych_consent_given=false`。
- 步驟：
  1. 於 `/profile/index` 直接啟動訪談。
  2. 完成 consent 後再次啟動。
- 預期：
  - consent 前 `POST /interview/start` 被拒絕（`CONSENT_REQUIRED`）。
  - consent 後可正常建立 session 並進入 `/interview/:sessionId`。
- 失敗判定：
  - 未 consent 仍能建立訪談。

#### Script P03-S02（P0）SSE 穩定性

- 前置：訪談中。
- 步驟：
  1. 連續回答 2-3 輪。
  2. 人為中斷網路再恢復。
  3. 在一輪內快速連續提交兩次。
- 預期：
  - 收到 `token/metadata/complete` 事件。
  - 網路中斷後可透過 resume 恢復。
  - 快速提交觸發 `TURN_TOO_FAST`。
- 失敗判定：
  - 中斷後 session 消失且無恢復入口。

### 9.4 P04 聊天轉判決腳本

#### Script P04-S01（P0）雙方入房與轉判決

- 前置：A 可建房；B 有可用帳號可接受邀請。
- 步驟：
  1. A 建房並建立邀請碼。
  2. B 以邀請碼加入。
  3. A/B 各發送多條訊息。
  4. A 發起 `request-judgment`。
- 預期：
  - room 狀態 `solo_active -> invite_pending -> invite_accepted -> group_active -> judgment_requested -> judgment_completed`。
  - 最終可導航到 `/judgment/:id`。
- 失敗判定：
  - 同一房間生成多筆 case/link（冪等失效）。

#### Script P04-S02（P0）訊息限速

- 前置：房間已建立。
- 步驟：
  1. 5 秒內連續發送多條消息。
  2. 30 秒內超過 6 條消息。
- 預期：
  - 返回 `RATE_LIMIT_EXCEEDED`，頁面保持可互動。
- 失敗判定：
  - 輸入框被鎖死或頁面崩潰。

### 9.5 P05 Admin 腳本

#### Script P05-S01（P0）權限矩陣

- 前置：準備兩個管理員（高權限/低權限）。
- 步驟：
  1. 低權限訪問 `PUT /admin/configs`、`PUT /admin/feature-flags`。
  2. 高權限執行同操作。
- 預期：
  - 低權限 403。
  - 高權限成功並可回讀配置。
- 失敗判定：
  - 低權限可修改高風險配置。

#### Script P05-S02（P1）審計與報表導出

- 前置：已有審計事件。
- 步驟：
  1. 查 `GET /admin/audit-logs`。
  2. 下載 `GET /admin/audit-logs.csv`、`GET /admin/reports/overview.csv`。
- 預期：
  - 列表與 CSV 有一致的時間區間與核心字段。
- 失敗判定：
  - CSV 空文件或內容與列表明顯不一致。
