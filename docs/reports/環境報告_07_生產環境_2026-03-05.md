# 報告 07：生產環境（Production）

## 1. 環境定位

- 目標：正式對外服務、穩定性與安全性優先。
- 主要入口：`backend/src/index.ts`（啟動前先驗證 env，待 DB ready 後才 listen）。

## 2. 啟動序列

1. `validateEnvConfig()` 驗證配置
2. `databaseReady` 初始化（含 migration 策略與連線重試）
3. DB 未就緒則 `process.exit(1)`（fail-fast）
4. 服務 listen 後啟動 `startJobs()`，並對齊 runtime jobs 配置

## 3. 必需環境變量（重點）

- 核心必需：
  - `DATABASE_URL`
  - `JWT_SECRET`
  - `OPENAI_API_KEY`
- 生產嚴格要求（`backend/src/config/env.ts`）：
  - `NODE_ENV=production`
  - `JWT_EXPIRES_IN` 必須顯式設置
  - `ADMIN_JWT_SECRET` 必填且不可等於 `JWT_SECRET`
  - `JWT_SECRET`/`ADMIN_JWT_SECRET` 需長度、熵值、格式合規
  - 若 `METRICS_ENABLED=true`，必須設 `METRICS_TOKEN` 或 `METRICS_ALLOWED_IPS`

## 4. 服務依賴（本環境）

- PostgreSQL（Prisma）：必需；migration/連線失敗會阻斷啟動
- OpenAI：必需（AI 主功能）
- Redis：強建議（分散鎖、ops metrics/alerts，無 Redis 在 production 會受限制）
- SMTP：可選（通知流程）
- Prometheus `/metrics`：可選但建議啟用，且必須保護
- Slack webhook：可選（ops 告警通知）
- Railway/OpenAI Billing API：可選（管理端成本報表）

## 5. 安全與保護機制

- CORS allowlist（`ALLOWED_ORIGINS`）
- `helmet` + rate limit（middleware）
- `/metrics` 生產保護（token 或 IP allowlist）
- JWT 密鑰強度與污染檢查
- DB 啟動 fail-fast，避免帶病上線

## 6. 驗證清單

- `GET /health`
- `GET /health/ready`
- `GET /health/live`
- `GET /metrics`（帶 `X-Metrics-Token` 或允許 IP）
- `npm run ops:alerts:check`
- `npm run ops:migration:report`

## 7. 風險與改進

- 文件仍有少量端口與遷移口徑漂移（需統一一份 production SSOT）。
- 成本報表依賴外部 API，建議設定 timeout/錯誤告警閾值與運維 runbook。
- 建議加上生產配置基線檢查腳本（preflight）並納入發版門檻。
