# 快速體驗優化說明（Quick Experience）

本文件用於統一「快速體驗」的產品流程、權限規則與邊界條件，避免文檔與代碼不一致。

---

## 🎯 目標
- **零門檻**：不註冊即可完成案件 → 判決閉環
- **穩定可用**：Session 不被誤刷新導致案件不可訪問
- **準確可控**：證據上傳、判決生成、重試與錯誤提示清晰

---

## ✅ 標準流程（P0）
1. 進入 `/quick-experience/create`
2. 自動建立 Session（`POST /api/v1/sessions/quick`）
3. 填寫角色A（必填 ≥30字）；角色B可選（≥10字或留空）
4. 提交案件（`POST /api/v1/cases/quick`）
5. （可選）上傳證據（`POST /api/v1/cases/:id/evidence`）
6. 結果頁輪詢判決（`GET /api/v1/cases/:id/judgment`）
7. 判決完成 → 展示結果 → 引導註冊

---

## 🔐 Session 規則
- Session 格式：`guest_{timestamp}_{random}`
- 初始有效期：24 小時
- **判決成功後自動延長至 7 天**
- 前端避免在有效期內自動覆蓋 Session（避免舊案件不可訪問）

---

## 🧾 證據上傳規則
- 允許狀態：`draft`、`submitted`、`in_progress`
- **判決完成後不再允許上傳**
- 若上傳失敗，結果頁可補傳（判決完成前）

---

## ♻️ 判決重試
- 若判決失敗：顯示「重試」按鈕
- 重試使用 `POST /api/v1/judgments/generate/:id`
- 冷卻時間由後端控制（`JUDGMENT_RETRY_COOLDOWN_MS`）

---

## 🔍 主要風險與對策
- **Session 覆蓋導致案件無法訪問** → 不主動刷新、由後端返回新的 Session
- **證據補傳與判決狀態衝突** → 前後端統一「判決完成後關閉上傳」
- **輪詢過長** → 5 分鐘上限 + 指數退避 + 手動重試

---

## 🧠 與訪談系統的關係

快速體驗（Session 模式）**不包含、不依賴** v2.0 訪談系統。訪談僅供**已登入且同意知情同意**的用戶使用，兩條路徑互不接軌。快速體驗用戶註冊轉正後方可使用訪談功能。

---

## ✅ 驗收標準
- 快速體驗流程可完整走通（Session → 案件 → 判決）
- 判決失敗可重試
- Session 延長邏輯一致
- 證據補傳僅在判決完成前開放
