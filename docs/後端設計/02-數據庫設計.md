**最後更新**：2026-02  
**適用環境**：本地開發 / 預發 / 生產（PostgreSQL 15+）

## 🎯 設計目標與改進重點
- 覆蓋快速體驗 + 完整模式全流程：案件提交流程、AI 判決、和好方案、執行追蹤、背景/關係檔案、內容推薦、郵件驗證、通知。  
- **v3.0 新增**：深度心理畫像系統——AI 引導式訪談、多維敘事存儲、結構化洞察提取、判決時畫像快照凍結。詳見 `UPGRADE_PLAN_PERSONALIZED_JUDGMENT.md`（v10）。
- 嚴格防髒數據：核心字段 ENUM 化，責任比例 CHECK=100，快速模式/完整模式行為用約束明確。  
- 企業級運維：軟刪除、審計、過期清理、部分索引、生成列全文索引預留、冷熱分離預案。  
- 可持續擴展：單庫優先，預留分區、對象存儲遷移、讀寫分離升級路徑。  

---

## 🏗️ 選型與環境
- **DB**：PostgreSQL 15+（Supabase/Railway/RDS 均可）。  
- **本地/CI**：優先 Docker Postgres；需要極簡可用 SQLite，但功能差異（ENUM/GIN）需降級策略。  
- **字符集/排序**：UTF8，`en_US.utf8`；全部時間用 `TIMESTAMPTZ` 存 UTC。  

---

## 📚 基礎約定
- snake_case；主鍵 `id` UUID；時間欄位 `*_at`; 重要表加 `deleted_at`（軟刪）。  
- 審計字段：`created_by` / `updated_by` 可空（快速模式或系統作業）。  
- 文件：只存簽名 URL，保留 `mime_type`、`checksum`；不存裸鏈。  
- PII 加密預留：email/電話/頭像可接 KMS 對稱加密；分析環脫敏。

  sub_type VARCHAR(50),

  defendant_statement TEXT,

### 16) interview_sessions（訪談會話）

```sql
CREATE TABLE interview_sessions (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  status interview_status_enum NOT NULL DEFAULT 'in_progress',
  trigger interview_trigger_enum NOT NULL DEFAULT 'organic',
  ai_model_used VARCHAR(50),
  total_user_words INT NOT NULL DEFAULT 0,
  total_ai_words INT NOT NULL DEFAULT 0,  -- AI 累計字數（用於成本追蹤）
  domains_touched psych_domain_enum[] DEFAULT '{}',
  feedback_card TEXT,             -- AI 生成的洞察反饋（純文字）
  pipeline_step INT NOT NULL DEFAULT 0, -- 異步管線已完成到第幾步（0=未開始，1-6=步驟編號），用於 retryFailed 從失敗步驟恢復
  started_at TIMESTAMPTZ,        -- 首次 respond 時間（用於計算對話時長）
  ended_at TIMESTAMPTZ,          -- endSession 時間
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX idx_interview_sessions_user_status ON interview_sessions(user_id, status);
CREATE INDEX idx_interview_sessions_user_created ON interview_sessions(user_id, created_at);
-- 部分唯一索引：每用戶同時只能有 1 個 in_progress session
CREATE UNIQUE INDEX ux_interview_sessions_user_in_progress
  ON interview_sessions(user_id) WHERE status = 'in_progress';
```

**設計要點**：
- `status` 使用 v10 的五態狀態機：`in_progress → processing → completed`（正常）/ `processing_failed`（異步失敗）/ `abandoned`（超時或被取代）。
- `total_ai_words` 在每次 AI 回應後累加，用於成本追蹤和分析。
- `started_at` 在用戶首次 respond 時記錄，`ended_at` 在 endSession 時記錄，兩者用於計算實際對話時長。
- `domains_touched` 在後端異步處理完成後回填，非即時寫入。
- `feedback_card` 在異步處理的最後一步由 AI 生成。
- 每個用戶同時只能有 1 個 `in_progress` session；新 session 自動將舊的標記為 `abandoned`。**由 `ux_interview_sessions_user_in_progress` 部分唯一索引在 DB 層保證**。
- **abandoned 行為定義**：`startSession` 只將 `in_progress` 的舊 session 標為 `abandoned`，不影響 `processing`/`completed`/`processing_failed` 的 session。若用戶有一個 `processing` 中的 session 又 start 新的，兩者共存（processing 繼續跑完管線，新 session 獨立進行）。
- **processing_failed 重試**：對 `processing_failed` 的 session 調用 `POST /interview/:id/retry`，從 `pipeline_step + 1` 恢復管線執行。`POST /end` 僅接受 `in_progress` 狀態（非冪等），**不可用於重試**。
- **pipeline_step 語義**：0=未開始、1=敘事提取(NARRATIVE_EXTRACTION)、2=敘事摘要(NARRATIVE_SUMMARY)、3=洞察提取(INSIGHT_EXTRACTION)、4=豐富度計算(RICHNESS_CALCULATION)、5=反饋卡片生成(FEEDBACK_GENERATION)、6=完成(COMPLETED)。每步成功後更新此欄位。`retryFailed` 讀取此值，從 `pipeline_step + 1` 開始恢復執行。

### 17) interview_turns（訪談輪次）

```sql
CREATE TABLE interview_turns (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  session_id UUID NOT NULL REFERENCES interview_sessions(id) ON DELETE CASCADE,
  turn_order INT NOT NULL,
  ai_message TEXT NOT NULL,
  ai_intent TEXT,                    -- AI 問這個問題的臨床目的（用戶不可見）
  ai_target_domains psych_domain_enum[] DEFAULT '{}',
  user_response TEXT,                -- NULL=尚未回答，空字串=跳過
  skipped BOOLEAN NOT NULL DEFAULT false,
  safety_flag BOOLEAN NOT NULL DEFAULT false,  -- AI 偵測到安全風險
  safety_detail TEXT,              -- 安全風險詳情（審計用，不展示給用戶）
  response_word_count INT,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE UNIQUE INDEX ux_interview_turns_session_order ON interview_turns(session_id, turn_order);
```

**設計要點**：
- `(session_id, turn_order)` 具有 UNIQUE 約束，在 DB 層防止併發寫入重複 turn。
- `ai_target_domains` 是 AI 提問意圖的近似標記（精確分類在 session 結束後批次完成）。
- `safety_flag` 由 AI metadata 中的 `safety_flag: true` 觸發，用於審計自傷/自殺/家暴等安全事件。
- `turn_order` 在 transaction 中基於當前最大值 +1 計算；UNIQUE 約束作為最後防線。

### 18) profile_narratives（域級敘事）

```sql
CREATE TABLE profile_narratives (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  domain psych_domain_enum NOT NULL,
  raw_narrative TEXT NOT NULL,       -- 合併的用戶原始文字
  ai_summary TEXT,                   -- AI 摘要（200-500字）
  word_count INT NOT NULL DEFAULT 0,
  completeness FLOAT NOT NULL DEFAULT 0 CHECK (completeness BETWEEN 0 AND 1),
  source_sessions TEXT[] DEFAULT '{}',   -- Prisma String[] 映射為 PostgreSQL TEXT[]（非 UUID[]）
  is_latest BOOLEAN NOT NULL DEFAULT true,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX idx_profile_narratives_user_domain ON profile_narratives(user_id, domain, is_latest);
-- 部分唯一索引：每用戶每域只有一個 is_latest=true
CREATE UNIQUE INDEX ux_profile_narratives_user_domain_latest
  ON profile_narratives(user_id, domain) WHERE is_latest = true;
```

**設計要點**：
- **updated_at**：Prisma 使用 `@updatedAt`，會在任意修改時自動更新此欄位；DDL 中需包含 `updated_at TIMESTAMPTZ NOT NULL DEFAULT now()` 以與 schema 對齊。
- 從多次訪談中按域合併用戶回答。新訪談產生新敘事時，舊的標記 `is_latest: false`，在同一 transaction 中操作。**`ux_profile_narratives_user_domain_latest` 在 DB 層保證每域只有一個最新版本**。
- `completeness` 由後端 heuristic 計算：`0.5 × min(1, wordCount/200) + 0.5 × min(1, turns/10)`（非 AI 評估），用於 richness 計算。
- `raw_narrative` 單域上限 5000 字（應用層驗證 + `CHECK (char_length(raw_narrative) <= 5000)` 可選）；超出時先壓縮舊文本為摘要再追加。

### 19) profile_insights（AI 洞察）

```sql
CREATE TABLE profile_insights (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  narrative_id UUID REFERENCES profile_narratives(id) ON DELETE SET NULL,
  domain psych_domain_enum NOT NULL,
  insight_type insight_type_enum NOT NULL,
  key VARCHAR(100) NOT NULL,         -- "attachment_style", "defense_mechanism"
  value TEXT NOT NULL,               -- "anxious-preoccupied", "intellectualization"
  confidence FLOAT NOT NULL DEFAULT 0.5 CHECK (confidence BETWEEN 0 AND 1),
  evidence TEXT,                     -- 引用的用戶原文（僅內部審計，不入判決 prompt）
  clinical_note TEXT,                -- AI 的臨床推理
  is_active BOOLEAN NOT NULL DEFAULT true,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX idx_profile_insights_user_domain ON profile_insights(user_id, domain, is_active);
CREATE INDEX idx_profile_insights_user_key ON profile_insights(user_id, key, is_active);
```

**設計要點**：
- `confidence < 0.4` 的洞察不注入判決 prompt，僅存儲供未來交叉驗證。
- `evidence` 欄位僅用於內部審計和模型改進，**絕不進入判決 prompt**（防止隱私洩露）。
- 多次訪談合併時：同 key 覆蓋（舊 `is_active: false`）、新 key 新增、消失 key 保留不刪除。

### 20) profile_snapshots（畫像快照）

```sql
CREATE TABLE profile_snapshots (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  case_id UUID NOT NULL REFERENCES cases(id) ON DELETE RESTRICT, -- 快照必定關聯案件；案件刪除前需處理快照
  snapshot_data JSONB NOT NULL,      -- { insights: [{key,value,confidence,domain}], richness_score }
  richness_score FLOAT NOT NULL DEFAULT 0 CHECK (richness_score BETWEEN 0 AND 1),
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  UNIQUE (case_id, user_id)          -- 同一案件、同一用戶只有一個快照
);
CREATE INDEX idx_profile_snapshots_user ON profile_snapshots(user_id);
```

**設計要點**：
- `case_id` 不可 NULL（快照只在判決時建立，必定關聯案件）。`@@unique([case_id, user_id])` 確保每案每人一個快照。
- `snapshot_data` 只存 key/value/confidence，不存原始敘事 → 隱私最小化。
- 用戶刪除心理畫像時（DELETE /psych-profile），snapshots 處理方式：✅ **設計與代碼一致**——snapshots **保留**（判決紀錄完整性的合法利益依據）。代碼註釋 `// ProfileSnapshot 保留：判決紀錄完整性（consent.point3 已告知用戶）`。

### richness_score 計算算法

```typescript
const DOMAIN_WEIGHTS: Record<PsychDomain, number> = {
  attachment: 2.0, family_origin: 2.0,
  life_events: 1.5, relationship_history: 1.5,
  belief_values: 1.0, cultural_background: 1.0,
  personality: 0.8, education_cognition: 0.5,
};
// totalWeight = 10.3

function calculateRichness(narratives: ProfileNarrative[]): number {
  const totalWeight = Object.values(DOMAIN_WEIGHTS).reduce((a, b) => a + b, 0);
  let weightedSum = 0;
  for (const n of narratives) {
    if (!n.is_latest) continue;
    weightedSum += (DOMAIN_WEIGHTS[n.domain] ?? 1.0) * n.completeness;
  }
  return Math.min(1, weightedSum / totalWeight);
}
```

**注入門檻**（基於 richness_score）：

| richness | 等級 | 注入策略 |
|---|---|---|
| < 0.05 | L0 空白 | 完全不注入，使用現有判決邏輯 |
| 0.05-0.2 | L1 微量 | 僅人格傾向，1 句 |
| 0.2-0.5 | L2 中等 | 人格 + 初步依附 + 文化，1-2 句 |
| >= 0.5 | L3 深度 | 完整心理畫像段落 |

## 🧹 數據生命周期與清理
- `quick_sessions`：24h 失效；案件完成可延長至 7 天；批量清理並連帶 `pairings(pairing_type=quick,temp)`。  
- `pairings`：`pending` 超 24h → `cancelled`；`temp` 超 30 天清理。  
- `email_verifications`：過期即刪；`used=true` 保留 24h。  
- 🧠 `interview_sessions`（v2.0）：`in_progress` 超過 24 小時 → 定時任務自動標記 `abandoned`（若 ≥3 輪先觸發異步管線保存數據）。`processing_failed` 保留不清理（等待用戶手動重試）。`completed` 永久保留。  
- 軟刪表：每月歸檔至冷庫或外部儲存，避免主表膨脹；冷數據不參與 OLTP 索引。

## 🔐 安全與合規
- 最小權限：應用連接角色最小化；只讀/寫分離；禁止 superuser。
