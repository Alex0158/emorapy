本文件承接正式發布、部署後驗證與回滾口徑，不再以分散的平台指引或根層說明稿作裁決。

## 1. 發布前檢查

必做：

1. 環境變量檢查
2. backend / frontend 構建
3. 單元與必要集成測試
4. 核心文檔守衛（`npm run docs:check`）
5. 關鍵主鏈路 smoke 預演

建議命令：

```bash
./scripts/validate-env.sh
./scripts/ci-local.sh
npm run docs:check
./scripts/test-critical-paths.sh
```

## 2. 建議發布順序

1. 合併到正式分支
2. 打版本 tag 或記錄發布 commit
3. 若本次含 schema 變更，先確認 production migration 計畫與回退方案
4. 部署 Railway 後端
5. 部署 Vercel 前台與管理端
6. 執行 smoke、健康驗證與 version endpoint 核對
7. 觀察 logs、alerts、metrics

## 3. 部署後驗證

最低驗證集：

1. `GET /health`
2. `GET /health/ready`
3. `GET /health/live`
4. `POST /api/v1/sessions/quick`
5. 主站 `/version.json`
6. 管理端 `/version.json`
7. 後端 `/version` 或 `/api/v1/version`
8. `DATABASE_URL=<prod-url> npm --prefix backend run ops:smoke-accounts:check` 確認 release DB 無 active smoke/dev 帳號污染
9. `RUN_MUTATING_RELEASE_SMOKE=true BACKEND_BASE_URL=<backend-url> DATABASE_URL="<prod-url>" RELEASE_SMOKE_ADMIN_EMAIL=<admin-email> RELEASE_SMOKE_ADMIN_PASSWORD=<admin-password> npm run ops:release:smoke` 通過；此命令會建立 production-like quick case / claim-session 測試資料，必須用專用 smoke 帳號並保留執行證據

建議腳本：

```bash
RUN_MUTATING_RELEASE_SMOKE=true BACKEND_BASE_URL=https://<backend-domain> DATABASE_URL="<prod-url>" RELEASE_SMOKE_ADMIN_EMAIL=<admin-email> RELEASE_SMOKE_ADMIN_PASSWORD=<admin-password> npm run ops:release:gate
```

手動補充抽測：

1. 快速體驗：session -> case -> judgment
2. 訪談：start -> respond / skip -> end -> result
3. 聊天室：建房 -> 發訊息 -> 邀請 -> 接受 -> request-judgment

## 4. 回滾口徑

前台：

1. 回到上一個穩定部署並 promote
2. 或回退到上一個 tag 重新部署

後端：

1. 回退到上一個成功部署
2. 必要時按 commit 重新部署

數據庫：

1. 優先要求 migration 前後兼容
2. 必要時以審慎方式修正 migration state

## 5. 高風險項

本輪若涉及以下能力，發布後必須加驗：

1. 訪談同意與 retry
2. 聊天室 request-judgment 權限與 room 狀態收斂
3. `/metrics` 保護
4. Redis-backed runtime 與 `AI Stream` 恢復
