## 1. 本子域回答什麼

本子域只回答以下問題：

1. 哪些規則不是單一功能流獨有，而是跨頁、跨流程、跨端共同遵守。
2. AI stream、通知、身份承接、證據上傳、錯誤恢復、冪等語義應如何統一理解。
3. 哪些共用語義應回寫到正式正文，而不是留在 dated 驗收記錄裡。

本子域不回答：

1. 單一產品流程的完整頁面與交互正文。
2. 某個 API 是否在用。
3. `packages/*` 的代碼落點與工程分層。

## 2. 共用機制分組

| 分組 | 影響範圍 | 核心內容 |
| --- | --- | --- |
| 身份承接 | quick、chat、formal case | `session_id`、JWT、claim-session、回跳 |
| Case 分類與授權 | quick、collaborative、remote、chat-to-case | session-bound vs user-bound、正式 collaborative、mixed credential |
| 安全與修復資格 | judgment、chat、repair journey、notification | safety/crisis route、partner invite、co-repair、partner notification |
| 路由與守衛 | Web、Admin | `ProtectedRoute`、`PublicRoute`、`AdminRedirect`、admin token redirect |
| AI stream | quick judgment、repair replan、interview、chat | phase、replay、timeout、shared stream state |
| 通知與再進場 | Repair Journey、個人中心 | unread、dismiss、snooze、act、notifications inbox |
| 證據與上傳 | quick case、formal case、execution checkin | multipart、可選照片、失敗降級 |
| 錯誤恢復與冪等 | session refresh、request-judgment、重調流程 | refresh/retry、競態、冪等保護 |

當前現碼並不是把所有共用機制都抽到單一 package。主 Web 與 Admin 仍各自維護 request / token 攔截與頁面組裝；共享真相主要落在 route guard 語義、部分 common 元件、`packages/contracts` 與既有 request / SSE / AI stream helper。

## 3. 身份承接與守衛

以下內容屬跨流共用規則：

1. `canonical session_id` 是匿名身份唯一正式依據，不以任意 storage 臨時值作準。
2. `ProtectedRoute`、`PublicRoute` 是前台守衛語義；`AdminRedirect` 與 admin token 守衛是後台語義，兩者不能混成一套。
3. quick、chat、evidence 上傳都可能同時受 session 與 JWT 影響；承接規則先看身份，再看具體功能。
4. `claim-session` 是登入後補綁，不是 auth 主結果本身。
5. 主 Web 的 `frontend/src/services/request.ts` 與 Admin 的 `frontend-admin/src/services/request.ts` 目前是兩條正式請求堆棧，不存在一條完全統一的全端 token/request 中樞。

### 3.1 Case 分類與授權 invariant

以下規則是跨 case、judgment、evidence、chat-to-case、repair 的正式共用 invariant：

1. `quick` 一律是 session-bound。
2. `collaborative + session_id 有值` 是快速雙人 / 匿名協作，走 session-bound。
3. `remote` 走 user/JWT participant 授權，即使請求帶了過期或無關的 `X-Session-Id`，也不得覆蓋 JWT 授權。
4. `collaborative + session_id = null` 走 user/JWT participant 授權，可承接正式雙方陳述或 chat-to-case，不得被誤判成匿名協作。
5. 後端不得在各 service/controller 中重新手寫 `mode` 分支；應使用 `backend/src/utils/case-classifier.ts`。
6. 產品入口、分析來源與轉換來源不能只靠 `Case.mode` 推斷；`ChatToCaseLink` 仍是當前識別「先聊再判」的正式來源。若未來新增 `entry_point / source_channel / product_flow` 等字段，必須建立 schema migration 待辦並同步 dev / release DB。

### 3.2 安全路由與修復資格 invariant

安全路由不是單頁提示，而是會影響判決、修復旅程、伴侶邀請與通知召回的硬 gate：

1. case scope active `RelationshipRiskState` 是修復旅程、伴侶邀請、伴侶通知與 execution dashboard 的優先來源；沒有 active state 或讀取失敗時，才回退到判決 route policy。
2. 判決生成階段的 `emotional_analysis.route` 是 route fallback 的首要來源，允許值為 `standard / safety_support / crisis_support`。
3. 若 stored route 缺失，可由 `judgment_content` 透過 `SafetyRoutingService.decideRoute` 做 fallback 判斷。
4. `safety_support` / `crisis_support` 一律視為 high-risk product route：
   - 默認修復 intent 為 `safety_support`。
   - 不允許生成一般 `repair` 共同修復方案。
   - 不允許伴侶邀請。
   - 不允許進入 `co_active` 共同修復。
   - 不觸發伴侶修復通知。
5. `standard` 才允許一般 repair、co-repair、partner invite 與 partner notification；但若 active `RelationshipRiskState` 已把同一 case 標為高風險，必須以 active state 為準。
6. 以上規則由 `backend/src/services/repair-eligibility.service.ts` 聚合 `SafetyAssessmentService` 與 product safety policy；其他 service、controller、前端不得自行重寫高風險判斷。
7. Chat invite 屬安全邊界，不是純 UX 行為。同一聊天室短時間重複邀請需由後端限流，避免撤回/重發造成騷擾。

## 4. AI Stream 共用語義

以下 AI stream 規則不應各頁自行再定義一套：

1. quick judgment、repair replan、interview、chat 都屬統一 AI stream 家族。
2. phase、`after_seq`、重連、snapshot/replay、timeout fallback 應優先沿用既有 SSE / AI stream helper，不應再新增第二套協議語義。
3. 前台的 `thinking / streaming / persisting` 等命名應維持共享語義，不為單頁重命名。
4. UI 展示層目前仍保留頁面級組裝，但錯誤命名、phase 映射與重試語義不得分叉成互不兼容的第二套口徑。

## 5. 通知、再進場與 CTA

通知中心不是裝飾層，而是正式再進場機制：

1. Header bell、`/notifications`、未讀數、單條 read/read-all、dismiss、snooze、act 構成完整通知入口。
2. active repair journey 的次操作應優先 `snooze`，不應把 `dismiss` 當成通用「稍後再說」。
3. `journey_context.primary_cta / secondary_cta / presentation_bucket` 屬跨頁共用判斷依據，不應每頁重猜一套。

通知接口主文在 [../06-接口描述/08-content-notification.md](../06-接口描述/08-content-notification.md)。

## 6. 證據上傳與恢復語義

以下規則屬跨功能共用：

1. `POST /cases/:id/evidence` 同時服務 quick case、formal case、execution 補充照片。
2. 證據上傳失敗時，應允許保留頁面上下文與單檔重試，不得把整個流程判為失敗。
3. execution checkin 的照片是可選資料；照片失敗時，應可降級成純文字打卡。
4. 文件大小、類型、下載授權屬共用風險點，實際字段與錯誤碼回接口文檔。
5. Web request 攔截器遇到 `FormData` 必須移除預設 `Content-Type`，由瀏覽器自動注入 multipart boundary；這是頭像與證據上傳的共用前提。
6. 簽名文件 URL 的 same-site 判斷在開發環境視 `localhost/127.0.0.1/::1` 為同站，避免本地跨 loopback hostname 時頭像/證據簽名被誤判失效。

## 7. 冪等、限流與錯誤恢復

以下規則屬共用防線：

1. `request-judgment`、AI 重調、訪談回合提交都應按冪等與限流來理解，不以頁面按鈕狀態代替後端保護。
2. session refresh、chat 邀請過期、SSE 中斷、AI 失敗 retry 都屬正式恢復分支，不是「偶發例外」。
3. 弱入口與分享鏈路必須假設 URL 直訪、過期、狀態變更隨時會發生。
4. AI 超時與鎖時間屬跨功能防線：單次 OpenAI 請求 `OPENAI_REQUEST=90000ms`、判決流程總超時 `JUDGMENT_GENERATION=180000ms`、判決生成鎖 `LOCK_TTL.JUDGMENT_GENERATION=300s`；三者需同步調整，避免中途鎖過期導致重入。
5. Redis 失聯時，cache / lock 會在首次錯誤後直接停用當前 Redis client 並降級到 memory / simple-lock，後續不再每次請求都先做 Redis 重試；要恢復 Redis-backed runtime 需由運維重啟進程。
