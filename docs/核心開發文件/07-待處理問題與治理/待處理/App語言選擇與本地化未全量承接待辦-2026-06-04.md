## 問題位置與現象

App 端目前沒有使用者可切換的 locale state / provider / catalog。`mobile/src/config/runtime.ts` 只從 `EXPO_PUBLIC_LOCALE`、Expo extra 或預設值讀取固定 `locale`；`mobile/src/platform/api/client.ts` 與 `mobile/src/platform/sse/client.ts` 以該固定值送出 `X-Locale`；`mobile/app` 與 `mobile/src/ui` 多數可見 UI、accessibility label / hint、錯誤與空狀態仍直接硬編繁體中文。

因此使用者即使在 Web / Admin 已有 `cj_locale` 語言偏好，App 也不會全量按使用者所選語言顯示，且 backend 回應語言也可能只跟 build-time locale 走。

## 影響範圍

- Web：既有 `frontend/src/utils/i18n.ts` 與 catalog parity 已建立，本待辦不直接改 Web。
- Admin：既有 `frontend-admin/src/utils/i18n.ts` 與 catalog parity 已建立，本待辦不直接改 Admin。
- App：受影響。涵蓋 public/auth/quick/case/chat/profile/notification/repair/modal/root error boundary、shared UI 預設 accessibility hint。
- Backend：間接受影響。App API/SSE `X-Locale` 目前不是 runtime user-selected locale。
- Shared：間接受影響。`@cj/api-client` 消費的 request context 有 `locale` 欄位，但 App adapter 尚未接 runtime user locale。

## 目前語言處理缺口

1. App 缺少 `zh-TW` / `en-US` catalog parity。
2. App 缺少 `getLocale` / `setLocale` / listener / provider / hook。
3. App 缺少使用者可操作的語言切換入口。
4. App API / SSE adapter 仍使用 build-time locale，而不是使用者 runtime 選擇。
5. App 可見文案、錯誤、空狀態、accessibility label / hint 仍大量硬編繁中。
6. App 缺少本地化 completeness / placeholder parity gate，無法防止新增 hardcoded 或漏 key。

## 目標行為

1. App 使用 runtime user-selected locale 作為 UI 與 API/SSE `X-Locale` 的唯一來源。
2. App 提供至少 `zh-TW` 與 `en-US` catalog，且 key 與 placeholder parity 可測。
3. App root / public landing / auth / quick / main authenticated flows 的可見文案逐步改為 `t()`，不再依賴 hardcoded UI string。
4. 語言切換後當前畫面重新 render，按鈕、空狀態、錯誤、accessibility label / hint 同步切換。
5. 未全量完成前，不宣稱 App 雙語完整；剩餘 hardcoded string 必須由本待辦或後續拆分待辦持續追蹤。
