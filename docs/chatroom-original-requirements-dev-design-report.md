# 聊天室原始需求：開發設計級拆解報告

## 1. 原始需求還原（以你最初描述為準）

你對聊天室的原始要求可還原為以下一句話：

`A 先單聊訴苦（A+AI） -> 可隨時 Invite B 入房 -> A/B/AI 三方持續對話 -> 可隨時發起判決 -> AI 以完整會話歷史梳理後，銜接正式案件流程`

對應你當時的關鍵意圖：

- 單人訴苦/吐槽模式（先做情緒承接與分析）
- AI 要提供傷心點洞察、建議、支持與情緒價值
- 不是固定雙人起手，而是可中途升級到群聊
- 升級後 AI 仍在場，且能在群聊中參與
- 可在任意時點把聊天轉成正式判決流程
- 體驗上接近群組型聊天產品

---

## 2. 原始需求的本質（產品層）

這不是單一「聊天室功能」，而是一條**可遞進的關係處理管線**：

1. **Support（支持）**：先情緒支持與理解
2. **Mediation（調解）**：再把單聊升級成雙方對話協調
3. **Adjudication（判決）**：最後才進正式判決流程

核心設計原則：

- 先支持、再調解、後判決（避免直接法律化）
- AI 在三階段扮演不同角色，不可混用語氣
- 任何轉判決都必須可追溯、可解釋、可審計

---

## 3. 需求拆分（功能級）

## 3.1 階段 A：單聊訴苦（A+AI）

**必備能力**

- A 建立私密聊天室（僅 A 與 AI）
- A 可持續輸入情緒與事件
- AI 持續回應，不是一次性摘要
- AI 輸出至少包含：
  - 情緒命名（你正在經歷什麼）
  - 傷心點/觸發點
  - 未被滿足需求
  - 可執行的小步驟建議

**設計邊界**

- 此階段 AI 不做最終責任裁定
- 對危機語句（自傷、暴力）需先安全分流

## 3.2 階段 B：隨時 Invite B 入房

**必備能力**

- A 可在任何時間邀請 B
- B 入房前必須處理歷史可見性策略：
  - 全量共享
  - 摘要共享
  - 只看加入後

**設計邊界**

- 邀請、接受、拒絕、過期有明確狀態機
- 不可因重複邀請造成房間狀態漂移

## 3.3 階段 C：A/B/AI 三方對話

**必備能力**

- A、B 均可發言，AI 可插話
- AI 在群聊中做主持與重述，而非終局裁判
- AI 需要具備最少 3 種介入能力：
  - 去火降溫（de-escalation）
  - 觀點翻譯（reframing）
  - 議題結構化（issue framing）

**設計邊界**

- AI 不能把單方敘事直接視為事實
- 對高風險互動模式必須優先安全提示

## 3.4 階段 D：隨時發起判決

**必備能力**

- 聊天室可一鍵轉正式案件
- 轉換依據是「可審計的對話快照」
- 判決仍沿用既有正式判決流程（非聊天室內直接定案）

**設計邊界**

- 必須明確標示哪些訊息被納入判決
- 需有冪等與去重，避免重複建案

## 4. 現況對照（與原始需求的差距）

> 這段是依目前程式行為做的對照結論。

- 已具備：
  - 房間、邀請、訊息、轉判決主流程
  - SSE 同步與狀態流轉
  - 安全路由前置（危機/安全命中會寫入 `safety_notice`，並帶冷卻）
  - 常態化 AI 回覆編排（`ChatAIOrchestrator`，support/mediation，房級節流）
  - 轉判決前預覽 + 手動勾選納入訊息（前端提交 `included_message_ids`，後端校驗並記錄 snapshot）
  - 訊息回覆引用（`reply_to_message_id`）與錨點定位（`#msg-<id>`）
  - 歷史訊息分頁載入（cursor-based pagination）
  - 房級訊息限流 + Chat metrics（Prometheus `/metrics` + Redis minute buckets）
- 主要缺口：
  - **摘要共享（`share_summary_only`）缺少「自動生成中立摘要」能力**：目前 `summary_only` 需使用者手動輸入，B 加入前只會看到被標記為摘要共享的訊息
  - AI 主持策略仍屬「最小可用」：目前以單一短 system prompt + 簡單策略切換（support vs mediation），尚未引入可配置規則/場景模板（例如去火、重述、議題樹、輪流發言提醒）
  - AI 回覆未流式化、未提供「thinking/blocked」等可見子狀態；安全命中時雖會提示但 UI 尚可更強（如固定置頂、行為引導）
  - 可觀測性目前以總量 counters 為主，尚未落地告警門檻到實際監控系統（Prometheus rule 僅提供範例）

一句話：**已完成「單聊→邀請→群聊→轉判決」的可用主流程，AI 也已能常態介入，但在「摘要共享自動化」與「主持策略產品化」仍有升級空間。**

## 5. 開發設計級方案（目標態）

## 5.1 角色與狀態機（建議最終形）

- 角色：`roleA`, `roleB`, `aiMediator`
- 房間狀態：`solo_active -> invite_pending -> group_active -> judgment_requested -> judgment_completed|judgment_failed`
- AI 子狀態（新增）：`idle | thinking | responded | blocked_by_safety`

## 5.2 訊息模型（建議擴充）

在既有 `chat_messages` 基礎上，明確區分：

- `user_text`
- `ai_reflection`（同理回應）
- `ai_mediation`（雙方翻譯/降溫）
- `ai_summary`（階段性結構化）
- `safety_notice`
- `system_event`

必要欄位補充建議：

- `reply_to_message_id`（回覆關聯）
- `ai_strategy`（此次 AI 採用的介入策略）
- `ai_confidence`（低信心時限制措辭）

## 5.3 AI 回覆編排器（新模組）

新增 `ChatAIOrchestrator`：

1. 監聽新訊息事件（SSE/event bus）
2. 判斷是否觸發 AI（節流、防洗版、角色輪替）
3. 讀取上下文（最近 N 輪 + 必要摘要）
4. 選擇策略（support/mediation/safety）
5. 呼叫 LLM，回寫 `chat_messages`
6. 廣播事件給前端

關鍵控制：

- 節流：每房間每 X 秒最多一次 AI 回覆
- 去重：同一觸發鍵不重複生成
- 中斷：進入 `judgment_requested` 後停止一般 AI 對話

## 5.4 Prompt 分層（開發必須明確）

- `P1 Support Prompt`：單聊同理與情緒支持
- `P2 Mediation Prompt`：群聊翻譯與降溫
- `P3 Conversion Prompt`：轉判決時的結構化提取
- `P4 Safety Prompt`：高風險場景優先安全介入

每層都要有：

- 輸入欄位定義
- 輸出 JSON schema
- 失敗回退策略

## 5.5 前端互動規格

- 聊天泡泡明確顯示角色（A/B/AI/System）
- AI 回覆中顯示「介入類型」標籤（支持/調解/安全）
- 發起判決前顯示「納入判決摘要預覽」
- 若 AI 暫停（節流/安全/狀態限制）需有可解釋提示

## 6. 非功能需求（NFR）

## 6.1 安全與隱私

- 歷史可見性策略必須可審計
- 敏感段落需可標記不共享
- 危機訊號一律先安全分流，不可直接責任化

## 6.2 可觀測性

至少追蹤：

- AI 回覆觸發率、成功率、平均延遲
- 被節流次數、被安全攔截次數
- 聊天轉判決成功率、重試率

## 6.3 穩定性

- SSE 斷線可回補（polling fallback）
- AI 回覆失敗不影響用戶發言
- 轉判決全程冪等（避免重複建案）

---

## 7. 驗收標準（Definition of Done）

- A 單聊時，AI 可持續回覆並提供情緒支持與洞察
- B 中途加入後，AI 仍可在群聊中主持式介入
- 任意時點可發起判決，且能基於對話快照轉正式流程
- 高風險對話可被正確分流與提示
- 全鏈路可追蹤（message -> conversion snapshot -> case -> judgment）

---

## 8. 建議里程碑（實作順序）

1. **M1：AI 回覆編排器最小版**
   - 先支援 `solo_active` 的 `ai_reflection`
2. **M2：群聊 AI 主持**
   - 支援 `group_active` 的 `ai_mediation`
3. **M3：轉判決前預覽與確認**
   - 前端顯示 conversion snapshot
4. **M4：安全與運營強化**
   - 風險分流、指標、告警、回放工具

---

## 9. 最終結論

你的原始聊天室需求本質上是「**支持 -> 調解 -> 判決**」的一體化系統，而不是普通訊息房。  
目前基礎骨架（邀請、狀態、轉判決）已具備，但要完全達到你的原始要求，下一個關鍵完成點是：**把 AI 從「判決前工具」升級為「聊天室內常態參與者」**，並用可審計、可控風險的方式落地。
