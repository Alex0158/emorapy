## 1. 當前狀態

開發版 backend 目前使用 Supabase Dev Postgres，但 Redis 未接入：

1. `REDIS_URL=` 為空。
2. `ALLOW_SIMPLE_LOCK=true`。
3. `/health` 可回 200，但 development runtime 會以 simple lock / memory fallback 運行。

發布版與 staging 口徑則要求 Redis-backed runtime，以覆蓋 lock、cache、metrics、AI Stream replay / snapshot 等能力。

## 2. 風險

開發版不接 Redis 時，仍可做一般功能開發，但不等價於發布版行為。以下能力可能無法在本機完整暴露：

1. Redis-backed lock。
2. AI Stream Redis replay / snapshot。
3. cache / metrics / ops cost signals。
4. 多進程或接近發布版 runtime 的一致性驗證。

## 3. 建議處置

優先方案：

1. 為本機開發版接入一個獨立 dev Redis，不共用 production Redis。
2. `backend/.env` 設置 `REDIS_URL=<dev-redis-url>`。
3. 設置 `ALLOW_SIMPLE_LOCK=false`，讓 Redis 失聯可以明確暴露。
4. 重啟 backend dev service。
5. 驗證 `/health` 中 lock/cache/AI Stream 相關訊號不再依賴 simple fallback。

Redis 來源需二選一：

1. 本機 Redis：最便宜、最快，但依賴本機服務存活。
2. 遠端 dev Redis：更接近 Supabase Dev DB 的持久共享模式，但需管理成本與憑證。

## 4. 驗證命令

```bash
ENV_FILE=backend/.env npm run ops:db:status
node -e "fetch('http://127.0.0.1:3001/health').then(r=>r.json()).then(j=>console.log(JSON.stringify(j.data.checks,null,2)))"
```

處理完成前，不得把「本機開發版已完全等價發布版 Redis runtime」作為結論。
