# 文檔封板後 - 代碼與測試對齊清單

**文檔版本**：v1.1  
**最後更新**：2026-03-17  
**定位**：當核心 SSOT 已完成一輪封板或功能邊界重構後，用此清單把文檔口徑落回代碼、測試與驗收證據，避免再次出現「文檔已修正，但代碼/測試仍沿用舊規則」。

---

## 1. 適用時機

以下任一情況成立，都應執行本清單：

- `功能特性清單.md` 的功能邊界、角色、前置條件或 handoff 規則有變動。
- `頁面清單.md` 的路由守衛、入口責任、弱入口/ProtectedRoute 定義有變動。
- `接口-功能-頁面-Mapping.md` 的 API 歸屬、頁面責任或流程節點有重排。
- `業務流程整合.md` 的主流程步驟、失敗恢復、身份矩陣或狀態機有變動。
- `術語表.md` 新增了影響權限判定、流程收斂或測試前提的新術語。

---

## 2. 執行原則

1. 先以核心 SSOT 為準，再驗證代碼與測試是否仍符合該口徑。
2. 若文檔與代碼衝突，不得直接改測試去接受錯誤行為。
3. 若測試揭露真實業務缺陷，需進入 `docs/測試/04-業務缺陷報告/` 或 `05-修復方案與風險專題/`，不能只在清單上打勾。
4. 若發現 SSOT 仍有歧義，先回修 SSOT，再繼續對齊代碼與測試。

| 檢查面向 | 必查代碼區域 | 必答問題 | 完成證據 |
|---|---|---|---|

| API 契約 | `frontend/src/services/api`、`backend/src/routes`、`backend/src/services` | API 權限、錯誤碼、輸入輸出與文檔是否一致？新 handoff 是否停在正確邊界？ | route/service 取證、接口測試、接口描述同步 |

| 狀態機與恢復 | 前端 store、後端 service、輪詢/SSE 邏輯 | 新狀態轉移、重試、回跳、失敗恢復是否落地？ | store/service 測試、流程驗證、錯誤分支記錄 |

## 5. 測試對齊檢查

### 5.1 最小覆蓋要求

- [ ] 至少補到本輪變更直接命中的前端單元/元件測試。
- [ ] 至少補到一層 API 或 service 級測試，確認新規則不是只存在 UI 文案。
- [ ] 若變更涉及跨頁 handoff、權限切換、session、回跳、SSE 或 polling，需補整合測試或 E2E。
- [ ] 若變更涉及 `P0` 主流程，需重跑對應 `回歸執行腳本`，不得只看局部測試。

### 5.2 測試設計檢查點

| 類型 | 必問問題 | 常見漏點 |
|---|---|---|
| 單元/元件測試 | 新條件是否真的影響按鈕可用性、頁面守衛、錯誤訊息或 fallback？ | 只測 happy path，沒測 disabled / retry / fallback |
| API/service 測試 | 新增權限或狀態規則是否有明確輸入輸出與錯誤碼驗證？ | 僅 mock 成功，不測 `FORBIDDEN` / `NOT_FOUND` / `CONFLICT` |
| 整合測試 | 頁面、store、service 串接後是否仍能正確 handoff？ | 只測單點，不測跨頁回流或共享 store |
| E2E | 使用者是否能沿著真實入口走完整鏈路？ | 跳過登入回跳、session 恢復、房間失效、手動重試 |
| 手動回歸 | 高風險流程是否仍可被人實際操作完成？ | 只看自動化通過，沒驗證真實瀏覽器行為 |

## 6. 本輪高優先對齊項（2026-03-09 封板）

### 6.1 F04 判決能力收斂為登入後消費

- [ ] 驗證 `/judgment/:id` 仍為 `ProtectedRoute`，且未登入直連確實先進 `/auth/login`。
- [ ] 驗證 chat 流程只負責推進到 `judgment ready` / `judgment-status` handoff，不直接宣稱匿名可消費 F04。
- [ ] 補或重跑 `F03 -> F04`、`F07 -> F04` 的 handoff 測試。
- [ ] 驗證登入後回跳目標正確，且不會掉回首頁或錯頁。

### 6.2 `/profile/pairing` 收斂為 F08 主責、F06 次責入口

- [ ] 驗證 `/profile/pairing` 的主流程仍是 create/join/status/cancel pairing 與 relationship profile。
- [ ] 驗證配對成功態若可觸發 `POST /interview/start`，其行為被測試覆蓋為 F06 次責入口，而不是把整頁誤當 F06 主頁。
- [ ] 檢查頁面、store、API 命名與註解是否仍把 pairing 頁寫成純訪談頁或純配對頁。

### 6.3 F07 匿名房主規則收斂為 `canonical session_id`

- [ ] 驗證匿名 owner 判定只依賴統一 session 管理層的 `canonical session_id`，不再散落多個 storage 直讀口徑。
- [ ] 補 invite 建立前的 owner 判定測試，覆蓋 match / mismatch / 缺 session 三分支。
- [ ] 驗證測試工具、mock data、E2E helper 使用的 session 寫入方式與現行 canonical 口徑一致。
- [ ] 驗證 chat room 失效、重整、重新進頁後，owner 能力不會因 session 來源不一致而漂移。
