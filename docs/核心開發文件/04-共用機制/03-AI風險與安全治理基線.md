## 1. 定位

本文把 Emorapy 的 AI / LLM 風險治理從「共用 NFR 的一部分」拆成可追溯規格。它參考 NIST AI RMF 1.0、NIST AI RMF Generative AI Profile、ISO/IEC 42001:2023 與 OWASP Top 10 for LLM Applications，但只作工程治理對標，不宣稱 Emorapy 已取得 AI 管理體系、合規或安全認證。

本文屬共用機制級。Web、App、Backend、Admin 只要新增或修改 AI runtime、prompt、AI stream、AI output downstream action、AI cost/ledger 或用戶可見 AI 定位，都必須回查本文。

## 2. 外部基線採用口徑

| 外部基線 | Emorapy 採用方式 | 不採用 / 不宣稱 |

| OWASP Top 10 for LLM Applications | 用於映射 prompt injection、sensitive information disclosure、improper output handling、overreliance 等技術風險 | 不宣稱已完成 LLM penetration test |

## 5. 強制規則

1. 新增 AI runtime 必須登記 `EMO-AI-ASSET-*` 或在本文補新 asset；不得只在 service 內新增 prompt。
2. 新增或修改 AI prompt，必須同步 `AI_PROMPT_VERSIONS` 或說明為何不需要版本化；用戶輸入必須經 `fenceUserInput()` 或等效防 injection 包裹。
3. AI output 不能直接觸發伴侶邀請、共同修復、通知召回、責任比例展示或正式 case 狀態變更；必須先過 safety policy / service gate。
4. AI ledger callers 不得傳入 prompt 原文、完整聊天文本或可直接重建私密敘事的 metadata；補齊 service-level metadata 白名單前，不得把「現有 callers 未傳 prompt」寫成 `AIRequestLedgerService` 已強制阻止任意敏感 metadata。
5. 對外文案與 AI prompt 不得把 Emorapy / 模型描述為治療師、心理診斷、法律裁決或危機救援服務；`npm run ai:positioning:check` 會阻止 Web i18n 與 prompt 自我定位回流。安全 referral 可以建議尋求持牌專業支持，但語義必須是外部支持，不是 AI 自己扮演該角色。
6. App 新增 AI 能力前，必須先補 App lifecycle、SecureStore、stream reconnect、Push / Deep Link 和 evidence smoke；不得借 Web 測試宣稱 App AI 已覆蓋。
7. 任何 AI output 的 audience 大於其 raw context audience 時，必須由集中式 policy resolver 證明只使用 shared/approved context；不能以「prompt 要求不要洩漏」替代資料隔離。
8. 私密上下文只能輸出 schema allowlist 的程序控制或本人可見內容；共同事實、診斷、可信度、責任、正式梳理及 joint Repair 不得 materially rely on private-only context。正式梳理必須把 evidence-only Decision Core 與 delivery personalization 分開。

## 7. 驗收口徑

1. 自動測試：prompt fencing、prompt version、AI ledger schema/caller no-prompt、safety routing、chat-to-case safety gate、兩角色 private projection、participant-scoped stream、malicious echo、capsule versioned consent、Decision Core input equality 與 joint Repair context boundary 必須有單測或集成測試入口；AI ledger metadata 白名單 / redaction gate 仍待補。
2. 分析驗證：新增 AI flow 必須能追到 `request_id / stream_id / scope_type / product_flow / prompt_version / status`。
3. 檢查驗證：PRD / NFR / RTM / 接口文檔必須標註 AI output 的 downstream gate。
4. 演示證據：`a685db36` 已有 exact Production release evidence；高風險關係、危機、stream 斷線、provider timeout、雙身份 side-channel與 App native仍需各自保留回歸或 release evidence，不能由部署成功外推。
5. 未量化的效果、召回率、安全識別率與成本節省，全部標為待建立基線。
