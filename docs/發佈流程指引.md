# 發佈流程指引（開發 → 生產）

## 目標
- 提供從開發環境到生產環境的標準化發佈流程
- 覆蓋前端（Vercel）與後端（Railway/PM2）部署與驗證
- 保證隱私合規與未成年人保護，使用可解釋與可回滾的流程

## 範圍
- 說明與腳本化步驟：分支管理、構建、部署、驗證、回滾
- 涵蓋現有基礎設施與配置文件：
  - 前端：Vercel（單頁應用）
  - 後端：Railway（Nixpacks）或傳統 PM2
- 不修改應用程式碼與接口、只新增指引文檔

## 不動區
- 不更動任何後端/前端業務邏輯與API
- 不更動資料庫 schema 與遷移
- 不更動現有環境變量命名與安全策略

## 驗收標準
- 前後端生產部署均健康可用，健康檢查 200
- 關鍵路徑測試全綠（Session → 案件 → 判決）
- 無敏感資料泄露，日誌不含 PII
- 支援回滾；部署驗證腳本通過

## 優先級/時限
- 優先級：P0
- 可即時執行；建議在每次合入主分支後觸發

---

## 環境與基礎設施
- 前端（Vercel）
  - 設定於根目錄的 `vercel.json`
  - 建置命令：`npm install && npm run build`
  - 輸出目錄：`dist`（Vite）
  - 單頁應用重寫至 `/index.html`
- 後端（Railway）
  - 專案級配置：`railway.json`
  - 服務級配置：`backend/railway.toml`
  - 建置：`cd backend && npm ci && npm run build:prod`
  - 啟動：`cd backend && npm start`
- 本地/PM2 傳統部署
  - 後端部署指南：`backend/DEPLOYMENT.md`
  - 部署腳本：`scripts/deploy.sh`

參考環境分離與校驗：
- 前端環境變量校驗：`frontend/src/config/env.ts`
- 後端環境變量校驗：`backend/src/config/env.ts`
- 全局環境說明：`docs/ENVIRONMENT.md`

---

## 分支與版本策略
- 建議分支模型
  - `main`：生產分支，合入即觸發部署（Vercel/Railway）或人工部署
  - `develop`：開發分支，日常疊代
  - feature 分支：`feature/<short-name>`
- 版本與標記
  - 每次生產發佈打 tag：`vX.Y.Z`
  - 更新後端變更記錄：`backend/CHANGELOG.md`

---

## 發佈前檢查清單
- 代碼與品質
  - 前端：`cd frontend && npm run build && npm run lint`
  - 後端：`cd backend && npm run build:prod && npm run lint && npm run test:unit`
- 環境變量與安全
  - 驗證環境：`./scripts/validate-env.sh`（必要變量/格式/HTTPS/密鑰長度）
- 關鍵路徑驗證（本地或預發）
  - 流程測試：`./scripts/test-critical-paths.sh`
  - 健康檢查：`./scripts/health-check.sh`
- 隱私與合規
  - 不在日誌或檢查腳本中輸出真實個資
  - 測試資料採用固定、可公開模板（不含未成年人敏感資訊）

## 發佈步驟（推送 → 自動部署）
1. 合入 `develop` → `main`，撰寫發佈備註，打 `vX.Y.Z` tag
2. 前端（Vercel）
   - 連接 Git 倉庫後，`main` push 觸發自動建置與部署
   - 生產環境必填：`VITE_API_BASE_URL = https://<backend-domain>/api/v1`
   - 單頁重寫與資產快取已在 `vercel.json` 配置
3. 後端（Railway）
   - 連接 Git 倉庫後，`main` push 觸發 Nixpacks 建置與部署
   - 必要環境變量：
     - `DATABASE_URL`, `JWT_SECRET`, `OPENAI_API_KEY`（必填）
     - `FRONTEND_URL`, `ALLOWED_ORIGINS`（建議使用 HTTPS）
   - 配置文件：`railway.json`、`backend/railway.toml`
4. 驗證部署（生產域名）
  - 統一驗證：`./scripts/verify-deployment.sh`
   - 手動抽查：
     - 後端健康：`GET /health`
    - 快速體驗：`POST /api/v1/sessions/quick` → `POST /api/v1/cases/quick` → 輪詢判決

## 替代步驟（人工/PM2 部署）
1. 伺服器安裝 Node.js 18+（建議 20/22）
2. 拉取最新代碼，配置 `.env`（勿提交版本控制）
3. 生成與遷移
   - `npm run prisma:generate`
   - `npm run prisma:migrate` 或 `npx prisma migrate deploy`
4. 構建與啟動
   - `npm run build:prod`
   - `pm2 start dist/index.js --name mother-bear-court-backend && pm2 save`
5. 參考文檔與腳本：`backend/DEPLOYMENT.md`、`scripts/deploy.sh`

## 回滾計劃
- 前端（Vercel）
  - 於 Vercel 控制台將上一個穩定部署「Promote」為 Production
  - 或切回上一個 tag 的 commit 重新觸發部署
- 後端（Railway）
  - 切換至上一個成功部署的 build 或回退 commit 重新部署
  - PM2：保留上一版本，`pm2 restart <prev_process>` 快速切回
- 資料庫
  - Prisma 遷移需保持可逆；必要時使用 `prisma migrate resolve` 標記已應用狀態

## 風險與防護
- 功能開關
  - 後端：`AI_MOCK` 可作為真實 AI 流量開關（開發/預發使用 true，生產使用 false）
  - 建議引入「RELEASE_GUARD」旗標門控高風險能力（策略層處理），以便分階段放量
- 安全與隱私
  - 生產強制 HTTPS；密鑰長度 ≥ 32；禁止示例值
  - 日誌聚合不含 PII；必要遮罩（Email/Session/Token 前綴+脫敏）
- 效能
  - 前端資產以 CDN 長快取（vercel.json 已配置 assets Cache-Control）
  - 後端限流與健康檢查已內建，建議加上外部監控與告警

## 常見問題與差異提示
- 本地端口與代理
  - Vite 代理指向 `http://localhost:3001`：參見專案根目錄下 `frontend/vite.config.ts`
  - 範例 `.env` 以 `3001` 為示例，實際以運行配置為準（前端 `VITE_API_BASE_URL`、後端 `PORT`）
- 環境分離與校驗
  - 生產環境變量請使用平台的環境管理，不使用 `.env`
  - 變量校驗錯誤在生產會拋出異常（避免隱性錯誤）
