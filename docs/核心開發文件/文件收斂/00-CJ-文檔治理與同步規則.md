本文件定義 `CJ` 文檔體系的活躍治理口徑，用於確保核心文檔持續與代碼同步。

本文件不是產品/工程正文 SSOT；它只裁決“文檔應如何組織、同步與守衛”。

## 1. SSOT 層級規則

1. 一級權威：`docs/核心開發文件/`
2. 補充測試入口：`docs/核心開發文件/測試/` 與 `docs/核心開發文件/08-測試規範與驗收/`
3. 證據與核驗：`docs/核心開發文件/90-證據與盤點/`
4. 歷史回看：`docs/核心開發文件/99-歷史降級索引/` 與根級 `歸檔/`
5. 營銷、品牌、對外敘事：`docs/核心營銷文件/`，不屬於工程真相裁決層

整個 `/Users/alex/Desktop/CJ` 只允許三個 Markdown 存在區：

1. `mother-bear-court/docs/核心開發文件/`
2. `mother-bear-court/docs/核心營銷文件/`
3. 根級 `歸檔/`

若發生衝突，以現碼、`核心開發文件/` 旗艦文檔與對應正式子域為準。

## 3. 編輯流程

1. 先改代碼，再改文檔，不允許先寫未落地能力為已實作。
2. 每條新增接口至少補齊主註冊表、對應接口子文檔與 Mapping。
3. 每個新增頁面至少補齊 `頁面清單.md`、`功能特性清單.md` 與 Mapping。
4. dated 驗收記錄不得直接升格成正式規格；若包含穩定真相，必須先回寫到正式子域。
5. 任何與 AI 流式或高風險鏈路相關的調整，若保留證據，應回到 `90-證據與盤點/`。
6. 若屬封板級、跨子域或大口徑重構，先更新本目錄台賬，再回寫正式正文。
7. 除上述三個區域外，不保留根 README、模組 README、repo-local skill 文檔或任何跳轉 stub。
8. `手動回歸證據/<latest-date>/summary.md` 與 `P01-P05/record.md` 的狀態必須逐條一致；不一致視為文檔漂移，`docs:check` 必須失敗。

## 4. 文檔命名規則

1. 正式規格文件不用日期。
2. dated 文件只允許進 `文件收斂/`、`測試/`、`90-證據與盤點/`、`99-歷史降級索引/`。
3. 根層不新增 dated 報告、營銷材料、一次性排查稿或環境核驗稿。
