`CJ` 的測試文檔固定分為四層：

1. `核心開發文件/08-測試規範與驗收/`
   - 長期有效的策略、分層、覆蓋與驗收口徑
2. `核心開發文件/測試/`
   - 與 SSOT 緊密配套的活躍案例、回歸包與執行入口
3. 根級 `歸檔/`
   - 歷史測試設計、失敗分析、缺陷報告與已退出活躍維護的專題稿
4. `核心開發文件/90-證據與盤點/`
   - 已核實的環境驗證、頁面快照與回歸證據

使用順序固定如下：

1. 先讀 `08-測試規範與驗收/`
2. 再讀 `核心開發文件/測試/`
3. 如需更細的歷史設計與缺陷分析，再讀根級 `歸檔/`
4. 如需已核實的證據鏈，再讀 `90-證據與盤點/`

現行自動化與門禁入口以 repo 內腳本與測試樹為準：

1. 前端頁面/狀態測試：`frontend/src/**/*.test.tsx`
2. 前端 chat 類 Playwright：`frontend/e2e/**/*.ts`
3. Admin 與跨站真服務 Playwright：`e2e/**/*.ts`
4. 手動回歸門禁與匯總：`manual-regression:*`（發版前建議直接跑 `npm run manual-regression:gate`）
5. staging smoke / 關鍵 e2e 防回退：`npm run smoke:staging`、`npm run --workspace frontend test:e2e:critical-guard`

當前列表不代表 App 端已驗收。`mobile/` 仍缺 CJ App smoke / regression / CI 入口；新增 App 測試時，應先判斷其對應的是 App navigation、platform adapter、shared contracts / api-client、Backend API 還是 DB schema，並同步回寫 `20-App端/01-App導航與平台Adapter基線.md`、`50-跨端Mapping與Parity/01-App首輪能力與工程落點Mapping.md` 或對應待辦。

App 測試進入正式測試體系前，需先按 [03-App測試與證據接入基線.md](./03-App測試與證據接入基線.md) 判斷最小分層、證據落點與不得替代邊界。該文件只定義進場規則，不表示 App smoke / regression / CI 已存在。

禁止反向使用：

1. 不要用根級 `歸檔/` 裡的歷史分析稿裁決當前產品真相
2. 不要用 `90-證據與盤點/` 的單次驗收結果替代正式測試規範
3. 不要把 dated 回歸執行記錄直接升格成正式規範
4. 不要用 Web/Admin 測試通過替代 App smoke / native adapter / Deep Link / Push / Upload 驗收
