## 4. Web / Admin / App 差異

| 平台 | 驗收口徑 |
| --- | --- |
| Web | 以 WCAG 2.2 參考、semantic HTML、ARIA APG、keyboard/focus、route smoke、i18n catalog 為主 |
| Admin Web | 以表格、篩選、配置、token、danger action、audit log、報表的可理解性與鍵盤操作為主 |
| App | 以 React Native accessibility props、native screen reader、touch target、dynamic type、Deep Link / Push landing failure 為主 |

Web 通過不代表 App 通過；Admin 通過不代表普通用戶 Web 通過。若 App 仍是 template，所有 App accessibility / l10n 驗收狀態必須是待承接。

## 5. 不得宣稱

1. 沒有全量 axe/WCAG 掃描與人工審查前，不得宣稱 WCAG 2.2 AA 或任何外部等級；route-level axe smoke 只能證明所列路由無自動化 axe 違規。
2. 沒有 screen reader smoke 前，不得宣稱輔助技術已覆蓋。
3. 沒有 i18n key completeness / fallback 檢查前，不得宣稱雙語完整。
4. 沒有 native App screen reader / dynamic type / platform smoke 前，不得宣稱 App accessibility 已完成；RNTL accessibility props gate 只能證明 shared UI contract。
5. 內容設計不能只用「已有翻譯」代表安全；高敏關係與 AI 文案需按術語表、PRFAQ、AI 風險、安全分流逐項審查。

## 6. 維護規則

1. 新增頁面、主流程或高風險 modal 時，至少補一條 role/focus/keyboard 或 inspection 證據。
2. 新增 i18n key 時，zh-TW / en-US 都需同步；若刻意不翻譯，需寫明平台或發布原因。
3. 新增 form field / credential input / file input / select / textarea 時，必須可用 `getByLabelText` 或 `getByRole` 定位；credential input 需檢查 autocomplete。
4. 新增 App screen 時，必須同步 `20-App端/01-App導航與平台Adapter基線.md`、`50-跨端Mapping與Parity/01-App首輪能力與工程落點Mapping.md` 與本文。
5. 若後續引入 axe、Playwright accessibility scan、i18n completeness script 或 App native smoke，必須回寫本文、RTM 與文件治理規則。
