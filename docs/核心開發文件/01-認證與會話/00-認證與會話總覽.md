本文件承接 `CJ` 前台身份建立、匿名會話、登入後承接與入口守衛的正式規格總覽。

## 1. 本子域回答什麼

本子域只回答以下問題：

1. 訪客、`Session` 用戶、登入用戶之間如何建立與承接身份。
2. Web 主站的 `/auth/*`、`ProtectedRoute`、`PublicRoute` 如何決定進出。
3. `session_id`、JWT、`claim-session` 與 chat/judgment handoff 的正式邊界。

本子域不回答：

1. 正式案件、Repair Journey、心理訪談、聊天室本身的完整業務正文。
2. Admin token、RBAC、運營治理與健康探針。
3. `packages/*`、workspace、共享 package 的工程落點。

## 2. 角色與身份邊界

| 身份 | 憑證 | 主要能力 | 典型入口 |
| --- | --- | --- | --- |
| 訪客 | 無 | 可進首頁、快速體驗、chat 頁面弱入口，但未必能完成建房或邀請處理 | `/`、`/quick-experience/*`、`/chat/room*` |
| Session 用戶 | `X-Session-Id` | 可承接 quick / collaborative case、匿名證據上傳、session-owned chat 房主能力 | `POST /api/v1/sessions/quick`、`POST /api/v1/sessions/refresh` |
| 用戶 | JWT | 可進保護頁、正式案件、判決、Repair Journey、心理訪談，並可接受 chat 邀請 | `/auth/login`、`/auth/register` 後的保護頁 |

關鍵術語以 [../術語表.md](../術語表.md) 的 `訪客`、`Session 用戶`、`canonical session_id`、`claim-session / 升格` 為準。

## 3. 承接的功能與流程

本子域直接承接：

1. `F09 身份認證與個人資料管理` 的身份建立與回跳規則。
2. `P01 快速體驗閉環` 中 session 建立、刷新與登入後升格。
3. `P04 聊天轉判決閉環` 中未登入手續與登入後回跳判決頁的身份承接。

本子域與其他流程的交界固定如下：

1. quick case 是否存在、如何提交、如何展示判決，回 [../02-用戶端核心流程/00-用戶端核心流程總覽.md](../02-用戶端核心流程/00-用戶端核心流程總覽.md)。
2. chat 房間邀請、消息、轉判決的房間規則，回 [../06-接口描述/07-chat.md](../06-接口描述/07-chat.md)。
3. Admin 登入與後台權限，回 [../03-管理端與平台治理/00-管理端與平台治理總覽.md](../03-管理端與平台治理/00-管理端與平台治理總覽.md)。

## 4. 入口與守衛族群

### 4.1 前台認證入口

固定入口頁如下：

1. `/auth/login`
2. `/auth/register`
3. `/auth/forgot-password`

它們都屬 `PublicRoute`。已登入用戶不得停留在這些頁；若 `location.state.from.pathname` 為 `/`，或以 `/case`、`/judgment`、`/reconciliation`、`/execution`、`/profile`、`/interview`、`/quick-experience`、`/chat` 開頭，應優先回到原始目標，否則回跳 `/case/list`。

### 4.2 前台保護頁守衛

以下能力一律屬 `ProtectedRoute` 範圍：

1. `/case/*`
2. `/judgment/:id`
3. `/reconciliation/*`
4. `/execution/*`
5. `/profile/*`
6. `/interview/*`

若未登入直連保護頁，應先進 `/auth/login`，再按原始意圖回跳。

### 4.3 匿名與弱入口

以下頁面可由訪客或 `Session` 用戶進入，但不代表可完成全部操作：

1. `/quick-experience/*`
2. `/chat/room`
3. `/chat/room/:roomId`

弱入口的產品責任以 [../頁面清單.md](../頁面清單.md) 為準；能否執行動作仍取決於後端對 user/session 身份與房間角色的判定。當前現碼下，`POST /api/v1/chat/rooms`、`POST /api/v1/chat/rooms/:roomId/invites`、公開邀請撤回可由合法 `session_id` 承接，但 `POST /api/v1/chat/invites/:inviteCode/accept` 必須是已登入帳號。

## 5. 接口族與正式規則

本子域的主接口族只看 [../06-接口描述/01-auth-session.md](../06-接口描述/01-auth-session.md)，核心邊界如下：

1. `POST /api/v1/sessions/quick`：建立匿名 `session_id`。
2. `POST /api/v1/sessions/refresh`：匿名會話恢復，不依賴 JWT。
3. `POST /api/v1/auth/register`、`POST /api/v1/auth/login`：建立正式登入身份。
4. `POST /api/v1/auth/send-verification-code`、`POST /api/v1/auth/verify-email`：處理驗證碼與郵件驗證。
5. `POST /api/v1/auth/reset-password`、`POST /api/v1/auth/reset-password-confirm`：處理密碼重置。
6. `POST /api/v1/auth/claim-session`：把 quick/session 下的資料綁到登入帳號。

固定規則：

1. `POST /api/v1/sessions/refresh` 若收到合法且可查到的舊 `X-Session-Id`，後端會原子旋轉 session，遷移關聯 `case_id`、`pairing_id` 與 `session_data`；若舊值缺失、格式錯誤或查無可旋轉 session（例如已被清理），則直接建立新 session。
2. 前端 refresh 後必須以新 `session_id` 覆寫本地存儲，並同步替換 `caseSessionMap` 等依賴映射，不得繼續持有舊 session。
3. `session_id` 與 JWT 並存時，必須顯式處理身分承接，不得以任意暫存值取代 `canonical session_id`。
4. `claim-session` 是登入/註冊成功後的後置承接；失敗可告警，但不得反向宣告 auth 主流程失敗。
5. `GET /api/v1/cases/by-session`、匿名證據上傳、chat 房間 owner/session 校驗都依賴合法 `session_id`，不能把 session 理解成一次性 quick case 票據。
6. chat 邀請接受屬登入能力；訪客若要最終消費 `/judgment/:id`，需先完成登入回跳。
7. `GET /api/v1/cases/:id`、`GET /api/v1/cases/:id/judgment` 的授權邊界必須按 mode 分流：
   - `quick` 與 `collaborative(session_id 有值)`：走 `session_id` 校驗。
   - `remote` 與 `collaborative(session_id=null)`：走當事人 JWT（`plaintiff_id/defendant_id`）校驗。

## 6. 高風險邊界

以下情況一律按高風險鏈路處理：

1. `session_id` 失效後的恢復：前端應以舊 `X-Session-Id` 先嘗試 refresh；若後端無法旋轉則建立新 session，並用新值替換本地映射後再決定是否回到 create / result。
2. `claim-session` 與 quick/chat handoff：不得讓 quick 升格阻斷正常登入。
3. `ProtectedRoute` hydration 逾時：前端應按未登入處理，不得誤顯示保護內容。
4. 匿名 chat 房主建立 invite：`room.session_id` 必須與當前 `canonical session_id` 對齊。
