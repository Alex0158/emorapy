> 入口語義口徑（跨文件統一）：
> - 路由入口：可直接進入某路由的入口（菜單、程式跳轉、directUrl 中繼）。
> - 功能入口：路由內某功能子視圖的入口（tab/subpage/分頁）。
> - 操作入口：功能內觸發業務行為的入口（提交、審核、啟停用、置頂等操作）。

## 流程 1：註冊與登入

| 階段 | 角色 | 關鍵節點 | 相關接口/頁面 |
|------|------|----------|----------------|
| 學生註冊 | 公共 | 填寫資料、郵箱驗證、AIS 學號同步 | `/register`、`/register/student`、`POST /api/jp/auth/register/student` |
| 學生帳號建立 | 後端 | 寫入 jp_student（f_status_code=pending）、觸發學生審核 | `applyForAudit(auditType="student")` |
| 僱主註冊 | 公共 | 填寫企業資料、驗證碼、上傳 Logo、建立 record；企業識別採 BRN/Tax ID 二選一（有 BRN 時執照必傳，無 BRN 時執照可選） | `/register`、`/register/employer`、`POST /api/jp/auth/register/employer`（註：`no-invite` 已列候選廢棄，前端未使用） |
| 僱主審核觸發 | 後端 | 寫入 jp_employer_informationrecord、觸發企業審核 | `applyForAudit(auditType="employer_info")` |
| 登入 | 公共 | 帳密登入、Azure SSO 登入 | `/login`、`/sso-callback`、`/auth/azure/callback`、`GET /api/oauth/Login`、`GET /api/oauth/CurrentUser` |
| 忘記密碼 | 公共 | 登入頁「忘記密碼？」→ 輸入 `username + email` 發送驗證碼 → 重置密碼（後端二次校驗 username/email 綁定） | `/login`、`ForgotPasswordModal`、`GET /api/openApi/qwgw/forgotPasswd/sendMail`、`GET /api/openApi/qwgw/ResetAuthentication` |
| 登出與會話清理 | 前端 | 無後端 logout API；清除登入態、角色與字典快取，導回登入頁 | Header 用戶菜單登出、`clearDictionaryCache` |
| 會話過期處理 | 前端/API 攔截器 | 回應 code=600 或 HTTP 401 時，清除 token 與用戶狀態，跳轉 `/login` | 全域 response interceptor、`messageUtils` |
| 字典快取 | 前端 | 未登入用公開字典、登入用全集；快取 24 小時，登出時清除 | `GET /api/jp/dictionary/public`、`GET /api/system/DictionaryData/All` |
| 登入失敗鎖定與 SSO 解鎖 | 平台層/後端 | 密碼錯誤達門檻後依 lockType 鎖定；Azure SSO 成功可自動重置鎖定狀態 | `passwordErrorsNumber`、`lockType`、`unlockTime` |
| 登入分流 | 用戶 | 依角色導向學生首頁/僱主儀表板/管理儀表板 | 儲存 userRole 後導向 |
| 帳號狀態檢查 | 學生/僱主 | 登入後讀取帳號可用性、顯示 Banner 提示 | `GET /api/jp/student/{userId}`、`GET /api/jp/auth/employer/info`、`StudentAccountStatusBanner`、`EmployerAccountStatusBanner` |

| 職位評估 | 學生 | 查看職位與公司詳情 | `/job-detail/:id`、`/company/:id`、`GET /api/jp/jobs/{id}`、`GET /api/jp/employer-information/{id}` |
| 職位收藏 | 學生 | 添加/取消收藏、收藏列表管理 | `/job-detail/:id`、`StudentProfile.favorites`、`/favorites`、`POST /api/jp/favorites`、`DELETE /api/jp/favorites`、`GET /api/jp/favorites` |
| 投遞準備 | 學生 | 選擇簡歷/求職信（須審核通過；僅 PLATFORM 職位需此步） | `StudentProfile.resume`、`StudentProfile.coverLetter`、`GET /api/jp/resume/page`、`GET /api/jp/cover-letters` |
| 提交申請（平台內） | 學生 | `applyMethod=PLATFORM`：選簡歷/求職信、發起投遞、觸發申請審核 | `/job-detail/:id`、`POST /api/jp/applications` → `applyForAudit(auditType="application")` |

| 學生端審核操作 | 學生 | 接受/拒絕面試、接受/拒絕 Offer、撤回申請（僅平台內申請） | `studentApi.executeAuditAction`（CONFIRM/DECLINE/ACCEPT_OFFER/REJECT_OFFER/WITHDRAW）、`auditApi.executeEntityAuditAction`（無 auditId 時） |

## 流程 5：Audit 機制完整梳理

本節按「審核域」逐一梳理觸發時機、審核入口、操作與 API、狀態流轉及業務邏輯。

### 5.1 學生審核（auditType=student）

| 維度 | 說明 |
|------|------|
| **觸發時機** | 學生註冊：`POST /api/jp/auth/register/student` → `StudentServiceImpl.registerStudent` → 寫入 jp_student（f_status_code=pending）→ `auditService.applyForAudit(auditType="student", businessId=jp_student.f_id)` |
| **操作入口（審核）** | 頁面 `/user-management`，UserManagement 學生列表；按鈕條件：`statusCode === "pending"` 且 `hasUserId` |
| **操作與 API** | 通過/拒絕：`POST /api/jp/audit/entity/action`（businessId=studentId, auditType=student, action=PASS/REJECT, comment）；因列表以 businessId 維度查詢，使用 entity/action 而非 auditId 路徑 |
| **狀態流轉** | pending → approved（通過）或 rejected（拒絕） |
| **通過後業務** | `jp_student.f_status_code=approved`；`base_user.F_ENABLEDMARK=1`（啟用帳號） |

### 5.2 企業審核（auditType=employer_info）

| 維度 | 說明 |
|------|------|
| **觸發時機** | ① 僱主註冊：`EmployerServiceImpl`（`POST /api/jp/auth/register/employer`）→ 寫入 jp_employer_informationrecord → `applyForAudit(auditType="employer_info", businessId=recordId)`；② 企業中心保存：`PUT /api/jp/employer/employer-information/{id}`（含 FormData 上傳 logo/registrationNo）→ `EmployerInformationServiceImpl.updateEmployerInformation` → 新建 record → `applyForAudit` |
| **操作入口（審核）** | 頁面 `/company-audit`，CompanyAudit 分頁（待處理/已處理/企業列表） |
| **操作與 API** | 通過/拒絕：`POST /api/jp/audit/{auditId}/action`（action=PASS/REJECT, comment）；REJECT 時 comment 必填 |
| **識別規則** | `jp_employer_information` 以 BRN/Tax ID 作機構唯一識別：建立與更新均需保證至少一個有效值，不允許雙空；有 BRN 走 BRN 驗重，無 BRN 走 Tax ID 驗重 |
| **設計邏輯** | 記錄表（jp_employer_informationrecord）為主，主表暫不更新；審核期間對外顯示主表舊內容 |
| **通過後業務** | `syncBusinessEntityStatus` → `handleEmployerAuditStatus`（先校驗記錄識別合法性，再搬運待審附件）→ 記錄表覆寫主表 |
| **多階段** | `twoStageOptimized`；SAOAdmin 可同請求內連續兩次 PASS 自動通過 |

### 5.3 職位審核（auditType=JOB / JOB_PROXY_PUBLISH）

| 維度 | 說明 |
|------|------|
| **觸發時機** | 僱主發布職位：`POST /api/jp/employer/job/publish` → `JobServiceImpl.publishJob` → 職位狀態 pending → `applyForAudit(auditType=JOB 或 JOB_PROXY_PUBLISH, businessId=jobId)`；ADMIN_PROXY 且 SAO_ADMIN 時可同請求內自動通過 |
| **操作入口（審核）** | 頁面 `/job-audit`，JobAudit 分頁（pending/processed/jobs） |
| **操作與 API** | 通過/拒絕：`POST /api/jp/audit/{auditId}/action`（action=PASS/REJECT, comment）；職位操作：`POST /api/jp/admin/jobs/{jobId}/activate`、`deactivate`、`pin`、`unpin`；違規：`POST /api/jp/admin/jobs/{jobId}/violation`、`violation/cancel` |
| **狀態流轉** | pending → approved（上架）或 rejected |
| **通過後業務** | 職位可被搜索與申請 |

### 5.4 申請審核（auditType=application）

| 維度 | 說明 |
|------|------|
| **觸發時機** | **僅 applyMethod=PLATFORM 的申請**：`POST /api/jp/applications` → `ApplicationServiceImpl.submitApplication` → 寫入申請記錄 → `applyForAudit(auditType="application", businessId=applicationId)`；**applyMethod=EXTERNAL_LINK 不觸發**，僅寫入 jp_application（external_redirected） |
| **操作入口（審核）** | 僱主端 `/employer-applications`（ApplicationManagement）；學生端 `StudentProfile.applications`、`/student-applications` |

**雙端操作與狀態機**：

| 當前狀態（auditStatus） | 僱主端操作 | 學生端操作 | 備註 |
|------------------------|-----------|-----------|------|
| pending | INTERVIEW（邀面試）、REJECT、CANCEL | WITHDRAW（撤回） | 待初篩 |
| interview_invited | CANCEL | CONFIRM、DECLINE、WITHDRAW | 面試邀請已發 |
| interview_confirmed | SEND_OFFER、REJECT、CANCEL | WITHDRAW | 學生已確認面試 |
| offer_sent | CANCEL | ACCEPT_OFFER、REJECT_OFFER、WITHDRAW | Offer 已發 |
| offer_accepted | ENROLL、CANCEL | WITHDRAW | 學生已接受 Offer |

**終態**（無操作按鈕）：enrolled（已入職）、withdrawn（已撤回）、cancelled（已取消）、interview_declined（已拒絕面試）、offer_rejected（已拒絕 Offer）、rejected（已拒絕）。

**API 對應**：

| 端別 | API | action 取值 |
|------|-----|-------------|
| 僱主端 | `POST /api/jp/audit/{auditId}/action` | INTERVIEW、REJECT、CANCEL、SEND_OFFER、ENROLL |
| 學生端 | `studentApi.executeAuditAction(auditId, action)` 或 `auditApi.executeEntityAuditAction`（無 auditId 時） | CONFIRM、DECLINE、ACCEPT_OFFER、REJECT_OFFER、WITHDRAW |

## 附錄 C：審核對照（類型/狀態/操作）

### C.1 auditType 對照

| auditType | 說明 | businessId |
|-----------|------|------------|
| `student` | 學生註冊審核 | `jp_student.f_id` |
| `employer_info` | 企業註冊/企業資料變更審核 | `jp_employer_informationrecord.f_id` |

### C.2 常用審核狀態

`pending`、`approved`、`rejected`、`interview_invited`、`interview_confirmed`、`interview_declined`、`offer_sent`、`offer_accepted`、`offer_rejected`、`enrolled`、`withdrawn`、`cancelled`、`external_redirected`。

### C.3 常用審核操作

`PASS`、`REJECT`、`INTERVIEW`、`CONFIRM`、`DECLINE`、`SEND_OFFER`、`ACCEPT_OFFER`、`REJECT_OFFER`、`ENROLL`、`WITHDRAW`、`CANCEL`。

## 附錄 D：關鍵業務錯誤碼（節選）

| 錯誤碼 | 說明 | 常見場景 |
|--------|------|----------|
| `45001` | 發布上限已達 | 職位激活/發布額度不足 |
| `45226` | 違規職位不可激活 | 職位已標記違規 |
| `45219` | 職位已過期 | 截止後不可激活/申請 |
| `45003` | 內容包含敏感詞 | 發布或檢測未通過 |
| `45033` | 學生審核未通過 | 平台內投遞受阻 |
