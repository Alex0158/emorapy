## 1. 定位

本文承接通用級用戶問題與產品假設。它只記錄跨 Web / App 共同成立的場景，不定義某一端 UI。Web / App 的具體 screen、route、Deep Link、Push、SecureStore 或 upload 承接，分別回到平台投影文件裁決。

## 2. 目標用戶場景

| 場景 ID | 場景 | 用戶意圖 | 通用級成功條件 | 平台注意 |
| --- | --- | --- | --- | --- |
| CJ-SCENE-001 | 單人低門檻試用 | 訪客想快速描述爭執並看到一次 AI 梳理 | 不登入也能完成 quick case、看到結果、可回訪或升格 | App 需先解決匿名 session restore，不能複製 Web storage |
| CJ-SCENE-002 | 同設備雙人協作 | 雙方在同一設備輪流輸入觀點 | A/B 視角清楚分離，結果仍由 quick flow 承接 | App 需確認觸控交互與本地 session 不混淆 |
| CJ-SCENE-003 | 正式處理一件事 | 登入用戶想留下正式記錄並進入修復旅程 | pairing gate、formal case、梳理結果、reconciliation、execution 可閉環 | Web 已落地；App 應分階段承接 |
| CJ-SCENE-004 | 先聊天再決定 | 用戶不想先填表，希望先自然對話 | 聊天材料只在明確 request judgment 後轉入 case / judgment | App 可優先承接，但 handoff 授權不得改 |
| CJ-SCENE-005 | 長期個性化 | 用戶願意提供背景以提升後續建議 | interview、psych profile、relationship profile 可形成可回用上下文 | App 適合承接訪談節奏，但 schema 不分叉 |
| CJ-SCENE-006 | 高風險關係訊號 | 用戶文本中出現控制、暴力、自傷或危機信號 | 先觸發安全分流，不把危機當普通吵架處理 | Web / App 都不得繞過 backend safety policy |
| CJ-SCENE-007 | 運維治理 | 平台需要追蹤 AI 成本、失敗、版本、DB parity 與發布狀態 | Admin / health / metrics / release gate 可提供證據 | App 只提供必要 telemetry，不承接 Admin Web |

## 3. 核心產品假設

| 假設 ID | 假設 | 對應主線 | 反證信號 | 驗證方式 |
| --- | --- | --- | --- | --- |
| CJ-HYP-001 | 用戶願意先用匿名快速判斷降低試用門檻 | 快速判斷 | quick 入口高跳出、結果頁低停留、claim 極低 | 觀察 quick funnel、結果頁回訪與 claim-session |
| CJ-HYP-002 | 正式處理需要配對和資料歸屬 gate，否則容易造成授權混亂 | 正式處理 | 未配對直建案需求強、pairing gate 導致大量放棄 | 觀察 pairing -> case create 轉化與 support/錯誤 |
| CJ-HYP-003 | Chat 是高頻入口，但轉梳理必須是顯式行為 | 先聊再判 | 用戶誤以為所有聊天都被自動判斷，或不理解 handoff | 觀察 request judgment 使用、錯誤提示與登入回跳 |
| CJ-HYP-004 | 個人/關係背景能提升梳理結果的可接受性 | 讓系統更懂你 | 訪談完成率低、完成後不提升正式處理回訪 | 觀察 interview completion、profile reuse 與後續 flow |
| CJ-HYP-005 | 安全分流比生成完整梳理更優先 | 平台治理 / 全主線 | 高風險內容仍進普通梳理、用戶被誤導為治療或法律服務 | 審查 safety-routing、prompt、文案與高風險回歸 |
| CJ-HYP-006 | App 的價值不是複製 Web，而是提升聊天、通知、訪談與回訪頻率 | App 投影 | App 首輪只做 Web 截圖式搬運，無 Push/Deep Link/session restore 優勢 | App parity、App smoke、Push/Deep Link 待辦閉環 |

## 4. 假設信心與處置規則

| 假設 ID | 當前信心 | 已有依據 | 尚缺依據 | 反證後動作 |
| --- | --- | --- | --- | --- |
| CJ-HYP-001 | 中 | Web quick flow、session refresh、claim-session 已有工程承接 | quick funnel、result revisit、claim rate 基線 | 調整 quick onboarding 或登入升格策略 |
| CJ-HYP-002 | 中 | pairing / formal case / auth gate 已有現碼 | pairing -> case submit 流失原因 | 檢討 pairing gate 是否需要更清晰提示或替代路徑 |
| CJ-HYP-003 | 中 | chat room 與 request judgment handoff 已有現碼 | 活躍房間轉 judgment 的使用行為與誤解率 | 調整 chat -> judgment 文案、授權提示或消息範圍 |
| CJ-HYP-004 | 低到中 | interview / profile API 已存在 | profile reuse 對後續梳理可接受性的量測 | 若無提升，降低 profile 前置權重 |
| CJ-HYP-005 | 高 | safety routing、product safety policy 與高風險文案已有現碼依據 | 持續高風險回歸與誤分流監測 | 安全路由優先，必要時阻斷普通梳理 |
| CJ-HYP-006 | 中 | App M0-M5 screen、adapter、Push token lifecycle、Deep Link 與 SecureStore 已有工程承接，release audit 可列出外部 sign-off blocker | EAS/TestFlight、physical device、provider delivery、production native crash runtime、長期 App 指標 | 若 release 後 App 原生價值不足，回到 App parity 待辦重新裁決範圍 |

信心等級不是成功結論，只表示目前能支撐假設的證據強度。低信心假設不得升格為 P0 產品承諾；若確需升格，必須先補 RTM、NFR 或待辦。

## 5. 需求狀態口徑

| 狀態 | 定義 |
| --- | --- |
| 已有工程承接 | Web / Backend / Admin 或共享層已有可核驗實作 |
| 待建立基線 | 需要量測或證據，但尚未有穩定指標基準 |
| 待承接 | 通用級需求成立，但某端尚未實作 |
| 待裁決 | 需要產品或工程決策，不能由單一文件自行決定 |
| 不承接 | 已明確裁決該端不提供此能力 |

## 6. 回寫規則

1. 場景或假設若改變產品語義，先更新本文與 `01-產品PRD總章.md`。
2. 若改變 Web 頁面、route、guard 或 Admin 投影，更新 Web 端與根層旗艦文檔。
3. 若改變 App screen、Deep Link、Push、SecureStore、upload 或 telemetry，更新 App 端與 Parity 文件。
4. 若假設需要量測，更新 `03-成功指標與產品健康.md` 與 `08-測試規範與驗收/04-需求驗證矩陣.md`。
