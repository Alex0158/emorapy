# 聊天室過度同理優化 — 設計審查報告

**日期**：2026-03-15  
**範圍**：chat-ai-orchestrator 優化（方案 A/B/C/D）  
**結論**：設計方向正確，能解決問題；已修正偵測誤報風險，並釐清既有設計邊界。

---

## 1. 問題與解決方案對齊

### 1.1 原始問題

- 用戶描述「和另一個男生親嘴」後詢問「你說我有問題嗎」，AI 過度同理、未做價值澄清
- 用戶進一步辯護「我愛和誰親嘴搞關係有什麼問題」，AI 仍以 support 策略一味支持

### 1.2 實作方案

| 方案 | 內容 | 對問題的對應 |
|------|------|---------------|
| A+D | 強化 System Prompt，加入價值澄清指引 | 直接修正：同理≠認同，邀請思考伴侶感受 |
| B | 情境偵測 + 動態策略（validation-seeking） | 在尋求認同時切換至更強價值澄清 prompt |
| C | fenceUserInput 防護 | 安全強化，防 prompt injection |

**對齊結論**：三方向均對應問題根因（prompt 過短、策略單一、無安全防護），設計合理。

---

## 2. 業務設計審查

### 2.1 單方 vs 雙方模式

- **單方（無 roleB）**：使用 support 策略，可觸發 validation-seeking 偵測
- **雙方（有 roleB）**：使用 mediation 策略，不偵測 validation-seeking

**合理性**：mediation 是翻譯雙方、降溫，不偏袒任一方；validation-seeking 僅在單方情感支持時有意義。✓

### 2.2 與發起方偏誤的關係

- 聊天室為「即時情感支持」，非判決流程
- 發起方偏誤分析主要針對「判決 prompt、責任比例、敘事順序」
- 聊天室優化針對「單方陳述時 AI 過度認同發起方」——與偏誤分析中的「單方陳述時易採信 A 視角」一致
- 本次優化在**支持階段**就加入價值澄清，可減少後續判決時「A 已被 AI 背書」的錨定效應

**結論**：與發起方偏誤緩解方向一致，互補。✓

### 2.3 與情感陪伴測試案例的關係

情感陪伴案例為「A 需陪伴、B 迴避」的追逐-迴避型，預期 B 較多調整空間。聊天室若僅 A 在場，會走 support 模式；若用戶非 validation-seeking（如「他都不回我訊息，我好難過」），則用一般 SUPPORT_SYSTEM_PROMPT，保持同理與需求探索。✓

## 3. 代碼設計審查

### 3.1 detectValidationSeeking — 誤報修正（已實施）

**原風險**：以下 pattern 過於寬鬆，易誤傷一般支持情境

- `你說呢`、`對吧` → 可能匹配「我們該溝通對吧」「你覺得呢」
- `那咋了|那又怎樣|怎樣了` → 可能匹配「他遲到那又怎樣」（A 抱怨 B，非辯護自己）
- `有什麼問題$` → 可能匹配「我想約他談談，有什麼問題？」（詢問計劃可行性）

**修正後**：

- 移除單獨的 `你說呢`、`對吧`、`那咋了|那又怎樣`
- `那咋了|那又怎樣` 僅在與爭議行為（親嘴/出軌/劈腿等）同句時觸發
- `有什麼問題` 須搭配爭議行為或「我愛和誰…」辯護脈絡

**仍可正確觸發的範例**：

- 「你說我有問題嗎」✓
- 「你也覺得沒有問題吧」✓
- 「我愛和誰親嘴搞關係有什麼問題」✓
- 「親嘴有什麼問題」「那咋了，我愛和誰親嘴…」✓

### 3.2 僅偵測最新一則訊息

- 目前只對 `message.content`（本輪用戶訊息）做偵測
- 多輪對話中，若第一輪已描述爭議行為、第二輪才問「你說我有問題嗎」，可正確觸發 ✓
- 若用戶分多輪鋪陳（先抱怨、再問「對吧」），且「對吧」已移除，可避免誤報 ✓

### 3.3 fenceUserInput 使用

- 對 `buildPrompt` 產出的完整對話脈絡做 fence
- `fenceUserInput` 會 sanitize `user_input`、`system`、`instruction` 等標籤
- 與 system prompt 中的 FENCE_SAFETY 規則一致 ✓

### 3.4 safety_support 流程（既有設計，非本次引入）

**現況**：

```
safety_support 時：
1. 建立 safety_notice 訊息（「偵測到可能的安全風險…」）
2. 不 return，繼續執行
3. 仍呼叫 AI 產生一般 support/mediation 回應
```

**落差**：safety_notice 寫「將優先以安全為核心回應」，但實際傳給 AI 的仍是標準 support/mediation prompt，未切換為安全導向 prompt。

**建議**（後續優化）：safety_support 時可傳入 safety-focused 的 system prompt，或至少加入安全注意事項片段，使 AI 回應與通知一致。

## 4. 代碼品質

### 4.1 結構

- Prompt 常數置頂，易維護
- `detectValidationSeeking` 為純函數，易測試
- 策略選擇邏輯清晰：`hasRoleB → mediation`，否則 `isValidationSeeking → validation prompt`，否則 `support`

### 4.2 型別與錯誤處理

- `OrchestratorContext`、`ChatMessageType` 等型別正確
- `fenceUserInput` 對空字串有處理
- `buildPrompt` 對 `sender_participant` 為 null 時 fallback 為 `unknown`，tag 為 `AI`

### 4.3 可測試性

- `detectValidationSeeking` 可獨立做單元測試（建議後續補上）
- 可透過 mock `aiService.generateTextStream` 驗證傳入的 `systemPrompt` 是否符合預期

## 5. 風險與邊界

| 項目 | 風險 | 緩解 |
|------|------|------|
| 偵測漏報 | 新口語表達未覆蓋 | 以 heuristics 為主，可依上線後案例擴充 pattern |
| 偵測誤報 | 一般支持被當成 validation-seeking | 已收緊 pattern，移除高誤報項 |
| 非一對一關係 | prompt 假設「伴侶關係」 | 目前產品定位為伴侶衝突，可接受 |
| safety_support 與 AI 回應不一致 | 既有設計 | 已記錄，建議後續加入 safety prompt 變體 |
