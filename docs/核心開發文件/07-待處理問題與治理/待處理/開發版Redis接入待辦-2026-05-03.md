## 1. 當前狀態

開發版 backend 目前使用 Supabase Dev Postgres，並已接入本機 Redis：

1. `REDIS_URL=redis://127.0.0.1:6379`。
2. `ALLOW_SIMPLE_LOCK=false`。
3. `/health` 回 200，lock check 顯示 `Lock backend: redis`。
4. backend log 顯示 Redis 已接入 ops metrics、lock、cache、chat metrics、AI Stream runtime。

發布版與 staging 口徑則要求 Redis-backed runtime，以覆蓋 lock、cache、metrics、AI Stream replay / snapshot 等能力。

決策：本機開發版固定以本機 Redis 作正式基線，不再要求 Railway `development` Redis 才能視為開發版閉環。已建立 Railway `development` environment，但新增 Railway dev Redis 時 CLI 回 `Unauthorized`。2026-05-03 重新完成 `railway login --browserless` 後，`railway add -d redis -s Redis-dev --json --verbose true` 仍在 fetching template details 階段返回 `Unauthorized`；因此 Railway dev Redis 只保留為可選 parity 增強，不作日常開發阻塞項。

## 2. 風險

開發版目前已接本機 Redis，可覆蓋 Redis-backed runtime 的主要本機驗證。Redis 資料可接受隨本機進程與工作區重建；真正需要跨機器持久化的業務資料由 Supabase Dev DB 承載。

1. Redis-backed lock。
2. AI Stream Redis replay / snapshot。
3. cache / metrics / ops cost signals。
4. 多進程或接近發布版 runtime 的一致性驗證。

## 3. 已採納處置

已完成：

1. 啟動本機 Redis tmux session：`cj-redis-dev`。
2. `backend/.env` 設置 `REDIS_URL=redis://127.0.0.1:6379`。
3. `backend/.env` 設置 `ALLOW_SIMPLE_LOCK=false`。
4. 重啟 backend dev service。
5. 驗證 `/health` 中 lock check 顯示 `Lock backend: redis`。

可選增強，不阻塞本機開發版：

1. 釐清 Railway 帳號對該 workspace/project 的 template/resource creation 權限，或改由 Railway dashboard 建立 dev Redis 後再回填環境變數。
2. 在 Railway `development` environment 建立獨立 Redis。
3. 只在需要跨機器共享 Redis runtime 或接近 Railway runtime parity 時，才將本機 `backend/.env` 從本機 Redis 切到遠端 dev Redis public URL。
4. 切換後重新驗證 `/health`、backend logs 與 Redis connectivity。

## 4. 驗證命令

```bash
ENV_FILE=backend/.env npm run ops:db:status
node -e "fetch('http://127.0.0.1:3001/health').then(r=>r.json()).then(j=>console.log(JSON.stringify(j.data.checks,null,2)))"
redis-cli -h 127.0.0.1 -p 6379 ping
```

正式口徑：開發版 Redis = 本機 Redis-backed runtime；發布版 Redis = Railway Redis-backed runtime。不得把本機 Redis 說成遠端化，也不得把 Railway dev Redis 視為日常開發必需品。
