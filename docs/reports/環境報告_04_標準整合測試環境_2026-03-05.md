# 報告 04：標準整合測試環境（Integration Test - Standard）

## 1. 環境定位

- 目標：驗證 API 端點與流程契約，成本低於全真實 E2E/flow。
- 入口：`backend/package.json` -> `npm run test:integration`

## 2. 執行特徵

- `jest --testPathPatterns=tests/integration --runInBand`
- 包含多個 integration 檔；其中 flow 類測試預設由環境變量 gate 控制（未開啟即 skip）。
- CI 額外有 `Admin integration tests`（`admin-api-flow.test.ts`）作為輕量 gate。

## 3. 關鍵變量

- 常見：
  - `NODE_ENV=test`
  - `SKIP_DB_INIT=true`
  - `DATABASE_URL=file:./test.db`（CI）
  - `JWT_SECRET=test-*`
  - `OPENAI_API_KEY=sk-dev-test-placeholder`
- `backend/tests/setup.ts` 會補齊預設 test env。

## 4. 服務依賴（本環境）

- PostgreSQL：多數情境不依賴真實 DB（部分測試模擬資料層）
- OpenAI：mock/placeholder
- Redis/SMTP/Slack：通常不參與

## 5. 驗證清單

- `npm run test:integration`
- CI `Admin integration tests` 綠燈

## 6. 風險與改進

- 命名上是 integration，但部分案例（如 admin API flow）實際仍偏 mock-based contract test，與「真實依賴整合」有落差。
- 建議在命名和分層上更清晰：`integration-mock` / `integration-real-db` / `e2e` 三層並行。
