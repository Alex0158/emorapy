## 🛡️ 安全設計概述

### 安全原則

1. **最小權限原則**：用戶只能訪問授權的資源
2. **深度防禦**：多層安全防護
3. **輸入驗證**：所有輸入都必須驗證
4. **輸出編碼**：防止XSS攻擊
5. **敏感數據加密**：密碼、Token等敏感數據加密存儲

### 安全層級

```
┌─────────────────────────────────┐
│   應用層安全（認證、授權）        │
├─────────────────────────────────┤
│   數據層安全（加密、驗證）        │
├─────────────────────────────────┤
│   網絡層安全（HTTPS、CORS）      │
├─────────────────────────────────┤
│   基礎設施安全（防火牆、DDoS）   │
```

## 🔐 認證中間件

### JWT認證中間件

**文件**：`src/middleware/auth.ts`

```typescript
import { Request, Response, NextFunction } from 'express';
import jwt from 'jsonwebtoken';
import { AppError } from '../utils/errors';

interface UserPayload {
  id: string;
  email: string;
}

declare global {
  namespace Express {
    interface Request {
      user?: UserPayload;
      sessionId?: string; // 快速體驗模式的Session ID
    }
  }
}

export const authenticate = async (
  req: Request,
  res: Response,
  next: NextFunction
): Promise<void> => {
  try {
    // 1. 從Header獲取Token
    const authHeader = req.headers.authorization;
    if (!authHeader || !authHeader.startsWith('Bearer ')) {
      throw new AppError(401, 'UNAUTHORIZED', '未提供認證Token');
    }
    
    const token = authHeader.substring(7);
    
    // 2. 驗證Token
    const decoded = jwt.verify(token, process.env.JWT_SECRET!) as UserPayload;
    
    // 3. 檢查用戶是否存在且激活
    const user = await userRepository.findById(decoded.id);
    if (!user || !user.is_active) {
      throw new AppError(401, 'UNAUTHORIZED', '用戶不存在或未激活');
    }
    
    // 4. 將用戶信息附加到請求對象
    req.user = {
      id: user.id,
      email: user.email
    };
    
    next();
  } catch (error) {
    if (error instanceof jwt.JsonWebTokenError) {
      next(new AppError(401, 'INVALID_TOKEN', 'Token無效'));
    } else if (error instanceof jwt.TokenExpiredError) {
      next(new AppError(401, 'TOKEN_EXPIRED', 'Token已過期'));
    } else {
      next(error);
    }
  }
};

// 可選認證（快速體驗模式）
export const optionalAuthenticate = async (
  req: Request,
  res: Response,
  next: NextFunction
): Promise<void> => {
  try {
    const authHeader = req.headers.authorization;
    if (authHeader && authHeader.startsWith('Bearer ')) {
      const token = authHeader.substring(7);
      const decoded = jwt.verify(token, process.env.JWT_SECRET!) as UserPayload;
      const user = await userRepository.findById(decoded.id);
      if (user && user.is_active) {
        req.user = {
          id: user.id,
          email: user.email
        };
      }
    }
    next();
  } catch (error) {
    // 認證失敗不影響請求繼續
    next();
  }
};

// Session驗證中間件（快速體驗模式）
export const validateSession = async (
  req: Request,
  res: Response,
  next: NextFunction
): Promise<void> => {
  try {
    const sessionId = req.query.session_id as string || req.headers['x-session-id'] as string;
    
    if (!sessionId) {
      throw new AppError(400, 'SESSION_ID_REQUIRED', 'Session ID是必需的');
    }
    
    // 驗證Session ID格式
    if (!sessionId.startsWith('guest_')) {
      throw new AppError(400, 'INVALID_SESSION_ID', '無效的Session ID格式');
    }
    
    // 驗證Session是否存在且未過期
    const session = await sessionService.getSession(sessionId);
    if (!session) {
      throw new AppError(401, 'SESSION_EXPIRED', 'Session已過期或不存在');
    }
    
    // 將Session ID附加到請求對象
    req.sessionId = sessionId;
    
    next();
  } catch (error) {
    next(error);
  }
};
```

## 🔒 授權中間件

### 資源所有權驗證

```typescript
export const authorizeResource = (resourceType: 'case' | 'judgment' | 'pairing') => {
  return async (req: Request, res: Response, next: NextFunction): Promise<void> => {
    try {
      if (!req.user) {
        throw new AppError(401, 'UNAUTHORIZED', '未認證');
      }
      
      const resourceId = req.params.id;
      let hasAccess = false;
      
      switch (resourceType) {
        case 'case':
          hasAccess = await caseRepository.hasAccess(resourceId, req.user.id);
          break;
        case 'judgment':
          hasAccess = await judgmentRepository.hasAccess(resourceId, req.user.id);
          break;
        case 'pairing':
          hasAccess = await pairingRepository.hasAccess(resourceId, req.user.id);
          break;
      }
      
      if (!hasAccess) {
        throw new AppError(403, 'FORBIDDEN', '無權限訪問此資源');
      }
      
      next();
    } catch (error) {
      next(error);
    }
  };
};
```

```typescript
router.get(
  '/cases/:id',
  authenticate,
  authorizeResource('case'),
  caseController.getCaseById
);
```

## ✅ 請求驗證中間件

### Joi驗證中間件

**文件**：`src/middleware/validator.ts`

```typescript
import { Request, Response, NextFunction } from 'express';
import Joi from 'joi';
import { AppError } from '../utils/errors';

export const validate = (schema: {
  body?: Joi.ObjectSchema;
  params?: Joi.ObjectSchema;
  query?: Joi.ObjectSchema;
}) => {
  return (req: Request, res: Response, next: NextFunction): void => {
    const errors: string[] = [];
    
    // 驗證請求體
    if (schema.body) {
      const { error } = schema.body.validate(req.body, { abortEarly: false });
      if (error) {
        errors.push(...error.details.map(d => d.message));
      }
    }
    
    // 驗證路徑參數
    if (schema.params) {
      const { error } = schema.params.validate(req.params, { abortEarly: false });
      if (error) {
        errors.push(...error.details.map(d => d.message));
      }
    }
    
    // 驗證查詢參數
    if (schema.query) {
      const { error } = schema.query.validate(req.query, { abortEarly: false });
      if (error) {
        errors.push(...error.details.map(d => d.message));
      }
    }
    
    if (errors.length > 0) {
      next(new AppError(400, 'VALIDATION_ERROR', errors.join('; ')));
      return;
    }
    
    next();
  };
};
```

## ⚠️ 錯誤處理中間件

### 統一錯誤處理

**文件**：`src/middleware/errorHandler.ts`

```typescript
import { Request, Response, NextFunction } from 'express';
import { AppError } from '../utils/errors';
import logger from '../config/logger';

export const errorHandler = (
  err: Error | AppError,
  req: Request,
  res: Response,
  next: NextFunction
): void => {
  // 記錄錯誤
  logger.error('Error occurred', {
    error: err.message,
    stack: err.stack,
    url: req.url,
    method: req.method,
    userId: req.user?.id
  });
  
  // 處理自定義錯誤
  if (err instanceof AppError) {
    res.status(err.statusCode).json({
      success: false,
      error: {
        code: err.code,
        message: err.message,
        details: err.details || {}
      }
    });
    return;
  }
  
  // 處理未知錯誤
  res.status(500).json({
    success: false,
    error: {
      code: 'INTERNAL_ERROR',
      message: process.env.NODE_ENV === 'production' 
        ? '服務器內部錯誤' 
        : err.message
    }
  });
};
```

```typescript
export const validatePasswordStrength = (password: string): { valid: boolean; message?: string } => {
  // 至少8位
  if (password.length < 8) {
    return { valid: false, message: '密碼長度至少8位' };
  }
  
  // 最多128位（防止DoS攻擊）
  if (password.length > 128) {
    return { valid: false, message: '密碼長度不能超過128位' };
  }
  
  // 包含字母
  if (!/[a-zA-Z]/.test(password)) {
    return { valid: false, message: '密碼必須包含字母' };
  }
  
  // 包含數字
  if (!/[0-9]/.test(password)) {
    return { valid: false, message: '密碼必須包含數字' };
  }
  
  // 檢查常見弱密碼
  const commonPasswords = ['password', '12345678', 'qwerty', 'abc123'];
  if (commonPasswords.includes(password.toLowerCase())) {
    return { valid: false, message: '密碼過於簡單，請使用更複雜的密碼' };
  }
  
  return { valid: true };
};
```

### Session安全驗證

```typescript
// 驗證Session ID格式和有效性
export const validateSessionId = (sessionId: string): boolean => {
  // 格式：guest_timestamp_random
  const pattern = /^guest_\d+_[a-z0-9]{6,}$/;
  if (!pattern.test(sessionId)) {
    return false;
  }
  
  // 提取時間戳，檢查是否在合理範圍內（不超過24小時）
  const parts = sessionId.split('_');
  if (parts.length !== 3) {
    return false;
  }
  
  const timestamp = parseInt(parts[1]);
  const now = Date.now();
  const maxAge = 24 * 60 * 60 * 1000; // 24小時
  
  if (timestamp < now - maxAge || timestamp > now + 60000) { // 允許1分鐘誤差
    return false;
  }
  
  return true;
};

// 防止Session枚舉攻擊
```

## 🛡️ 安全審計與加固（2026-02）

### 帳戶鎖定機制

- `User` 模型新增 `login_failed_attempts`（Int）、`locked_until`（DateTime?）欄位
- 連續登入失敗達閾值後，帳戶臨時鎖定
- 鎖定到期後自動重置失敗計數
- `login_failed_attempts` 使用 Prisma 原子 `{ increment: 1 }` 操作，防止併發 race condition 導致計數不一致

### Token 版本化（Session 失效機制）

- `User` 模型新增 `token_version`（Int）欄位
- JWT payload 攜帶 `token_version`
- 密碼重設時 `token_version` 自增，所有現有 token 立即失效
- `authenticate` 中間件驗證 JWT 中的 `token_version` 與資料庫一致

### JWT 安全加固

- `jwt.sign` 和 `jwt.verify` 明確指定 `algorithm: 'HS256'`，防止算法混淆攻擊
- `JWT_SECRET` 在生產環境強制 ≥ 32 字元，並檢查熵值
- `JWT_EXPIRES_IN` 縮短至 `24h`

### Prompt Injection 防禦

- `fenceUserInput()` 函式將用戶輸入包裹在 `<user_input>` 定界符中，並清除輸入中的定界符
- `SYSTEM_PROMPT` 明確指示模型忽略 `<user_input>` 標籤內的角色扮演或指令

### AI 輸出消毒

- `sanitizeAIOutput()`：移除 `<script>`、`<iframe>` 標籤，截斷過長字串
- `sanitizePlanStrings()`：遞迴清理和好方案 JSON 中的所有字串值
- 在 `judgmentService` 和 `reconciliationService` 存入資料庫前統一調用

### 電子郵件安全

- `EmailService.sanitizeEmail()`：移除 CRLF 和控制字元，防止 email header injection
- 所有發送郵件的呼叫點（驗證碼、配對通知、判決通知）均經過消毒

### 檔案與路徑安全

- `FileService.deleteFile()` 使用 `path.basename()` + `path.resolve()` 確保刪除操作嚴格限制在 `UPLOAD_DIR` 內
- 防止 `../../etc/passwd` 類型的路徑穿越攻擊

### 資源授權強化

- `reconciliationService.getPlansByJudgmentId` 新增 `userId` 參數，驗證用戶為案件當事人
- `executionService.checkin` 對 `notes`（移除 script/iframe、限制 2000 字）和 `photos_urls`（驗證 https 協議、限制 20 張）進行消毒
- `caseService.getCases` 的 `sort_by` 參數使用白名單驗證

### 併發安全

- `lockService.withLock()` 用於案件建立和判決生成，防止重複建立
- 登入失敗計數使用 Prisma atomic increment

### 前端安全

- Login 頁面 `from` 路徑驗證，防止 open redirect
- `ErrorBoundary.componentDidCatch` 僅記錄安全的錯誤屬性（name, message, componentStack）
- `ErrorFallback` 在生產環境隱藏具體錯誤訊息
- Zustand persist 不持久化 `token`，`rememberMe=false` 時 token 存入 `sessionStorage`，關閉瀏覽器即失效
- `register()` 存 token 到 `localStorage` 時同步清除 `sessionStorage` 殘留 token，與 `login()` 保持一致的互斥清理
- `request.ts` 中自定義 `sessionStorage`（快速體驗 Session 管理）使用別名 `quickSessionStorage` 導入，避免遮蔽全域 `window.sessionStorage`，確保 auth token 讀寫使用正確的 Storage API
- Register / ForgotPassword 的 catch 區塊統一使用 `getErrorMessage()` 從 `@/utils/apiError` 提取後端回傳的具體錯誤訊息，避免 `error instanceof Error` 對 request interceptor 拋出的普通物件永遠返回 false、導致用戶只看到通用 fallback 訊息
- Register 步驟 3 的自動跳轉 timer 在用戶手動導航時正確清除，防止意圖被覆蓋
- ForgotPassword 成功後自動跳轉 timer 同樣使用 `useRef` 管理並在 unmount 時清除，保持一致性

### 國際化（i18n）

- 後端：`localeMiddleware` 從 `X-Locale`/`Accept-Language` 標頭偵測語言
- 後端：`translateBackendMessage()` / `translateErrorByCode()` 將錯誤訊息翻譯為對應語言
- 後端：`directEnUSMap` 覆蓋所有 rate limiter 自定義訊息（驗證碼/重設密碼/配對限流）、分散式鎖、Session 相關訊息
- 前端：634 個 zh-TW key 全部有 en-US 對應翻譯
- 前端：「記住我」標籤已移除不準確的時長描述（JWT 過期為 24h，標籤不應承諾 7 天）

## 🧠 心理畫像安全機制（v2.0 新增）

> 設計依據：`UPGRADE_PLAN_PERSONALIZED_JUDGMENT.md` v10

### 訪談端點 Rate Limiting

| 端點 | 限制 | 層級 | 說明 |
|------|------|------|------|
| `POST /interview/start` | 每用戶每**小時** 3 次 | 中間件（`interviewStartLimiter`，express-rate-limit） | 防濫用，不計 turn 數，純 HTTP 請求計數 |
| `POST /interview/start` | 每用戶每**天** 5 個 substantive session | 業務邏輯（`interview.service.ts`） | 僅計 ≥ 3 輪的 session；< 3 輪的 abandoned 不計入（防止連線問題懲罰用戶） |
| `POST /interview/:id/respond` | 每 session 最多 25 輪 | 業務邏輯 | 硬限（`INTERVIEW_MAX_TURNS` 默認 25）；AI 軟目標 ~15 輪自然收尾 |
| `POST /interview/:id/respond` | 每用戶每分鐘 20 次 | 中間件（`interviewRespondLimiter`） | 防自動化 |
| `POST /interview/:id/respond` | 每輪最少間隔 3 秒 | 業務邏輯 | `INTERVIEW_TURN_INTERVAL_MS`（默認 3000） |

### 知情同意機制（Consent）

**後端邏輯**（`requireConsent` 中間件）：
- 所有訪談路由（`/interview/start`、`/:id/respond`、`/:id/end`、`/:id/skip`、`/resume`、`/:id`、`/:id/retry`）使用 `requireConsent` 中間件
- 中間件檢查 `User.psych_consent_given`；未同意 → 返回 `403 CONSENT_REQUIRED`
- `POST /psych-profile/consent` 設置 `psych_consent_given = true`、`psych_consent_at = now()`
- 刪除畫像時 → `DELETE /psych-profile` 重設 `psych_consent_given = false`（下次需重新同意）

**前端檢查流程**（與前端 `02-路由.md` 對齊）：
- 前端進入 `/interview/*` 路由時，訪談頁面 mount 時檢查 `consent_given`（⚠️ 無獨立 `ConsentGuard` 路由守衛組件，由頁面級邏輯處理）
  - `consent_given = true` → 放行
  - `consent_given = false` → 展示 `ConsentModal`
  - 用戶同意 → 調 `POST /psych-profile/consent` → 放行
- 即使前端未攔截，後端 `requireConsent` 中間件仍會返回 403，確保雙重保護

**知情同意文案要點**：
- 「你可以隨時跳過、隨時清除所有資料」
- 「清除資料後，過去的判決紀錄仍會保留簡化的分析摘要（不含原始敘述）」
- 文案保持 3 句話以內，避免法律文書語氣

### 併發控制

- `POST /interview/:id/respond`：session 級 mutex lock（Redis advisory lock 或 DB advisory lock）
  - 進入時獲取 `interview:respond:{id}`（鍵格式：`interview:respond:${sessionId}`）
  - 已被佔用 → 返回 `409 CONCURRENT_REQUEST`
  - SSE 完成後釋放
- `turn_order`：在 transaction 中基於 `MAX(turn_order) + 1` 計算

### SSE 分隔符注入防護

- 後端取 AI 輸出的**最後一個** `---METADATA---`
- 前端同理：buffer 全部 token，最後分隔符為準
- `user_response` 存儲前 strip `---METADATA---` 字串
- AI prompt 指令：嚴禁在可見文字中輸出分隔符，只在末尾輸出一次
- ⚠️ **實現說明**：以上防護針對「設計方案」中的 `---METADATA---` 雙通道格式。當前實現為 AI 回傳單一 JSON（`text`/`intent`/`target_domains`/`should_end`/`safety_flag`/`safety_message`），由服務解析後轉為 SSE 事件，故當前程式碼中不存在分隔符注入風險。

### 安全偵測（Safety Flag）

- 訪談 AI 的 system prompt 包含安全關鍵詞偵測（自傷、自殺、家暴）
- 觸發時：AI 在 metadata 中設 `safety_flag: true` → 前端暫停訪談並展示危機資源
- `InterviewTurn.safety_flag` 記錄事件供審計

### 判決注入隱私保護

- 判決 prompt 只注入 AI 摘要和 key/value 洞察，**絕不注入用戶原始引述**
- `evidence` 欄位僅用於內部審計，不進入判決 prompt
- 判決輸出使用泛化表述，不引用具體事件，不透露敏感敘事給伴侶

### SSE 斷線恢復

- 用戶在 AI 回應流式傳輸中斷開連線 → 後端**繼續完成** AI call 並存儲 turn
- 用戶 resume 時從 `GET /interview/:id` 載入完整對話（含 turns，包含斷線時的 AI 回覆）

### 遺忘權（Right to Erasure）

**觸發端點**：`DELETE /api/v1/psych-profile`

**級聯刪除順序**（在**單一 batch transaction** 中完成，失敗全回滾）：

```
1. profile_insights（FK → profile_narratives → users）
2. profile_narratives（FK → users）
3. interview_turns（FK → interview_sessions → users）
4. interview_sessions（FK → users）
5. users.psych_consent_given = false, psych_consent_at = null
```

**ProfileSnapshot 保留策略**：
- 已凍結的 `ProfileSnapshot`（`case_id` 關聯的）**不刪除**
- 原因：判決紀錄完整性（`profile_snapshots.case_id` 有 `ON DELETE RESTRICT`）
- Snapshot 僅含 `key/value/confidence` + `richness_score`，**不含原始敘述、不含 evidence**
- 此行為在知情同意文案中已告知用戶
- 若用戶需要連 snapshot 一起刪除，需通過客服通道手動操作（需同時刪除關聯判決的個人化段落）
- 與 `02-數據庫設計.md` 的 `profile_snapshots` FK `ON DELETE RESTRICT` 對齊

**事務保證**：
```typescript
await prisma.$transaction([
  prisma.profileInsight.deleteMany({ where: { user_id: userId } }),
  prisma.profileNarrative.deleteMany({ where: { user_id: userId } }),
  // ProfileSnapshot 保留：判決紀錄完整性（consent 文案已告知用戶）
  prisma.interviewTurn.deleteMany({ where: { session: { user_id: userId } } }),
  prisma.interviewSession.deleteMany({ where: { user_id: userId } }),
  prisma.user.update({
    where: { id: userId },
    data: { psych_consent_given: false, psych_consent_at: null },
  }),
]);
```
