# 後端架構設計（代碼對齊）

**最後更新**：2026-03-05  
**對齊基準**：`backend/src/app.ts`、`backend/src/index.ts`、`backend/src/config/*`、`backend/src/routes/*`

---

## 1. 架構總覽

當前後端是 **Express + Prisma + PostgreSQL** 的單體 API 服務，並帶有：
- OpenAI 能力（判決、訪談、分析）
- SSE 長連線（訪談與聊天室）
- 後台管理 API（RBAC）
- 定時任務與運維監控（health/metrics/alerts）

不是 API Gateway + 多微服務拆分架構；主要以模組化單體維護。

---

## 2. 啟動與生命週期

## 2.1 啟動序列（`index.ts`）

1. `validateEnvConfig()`：啟動前校驗關鍵環境變數。  
2. 等待 `databaseReady`（`config/database.ts` 初始化）完成。  
3. `app.listen(PORT)` 啟動 HTTP。  
4. 啟動定時任務 `startJobs()`，並嘗試 `reconcileJobsRuntimeConfig()`。  

## 2.2 關閉序列

- 監聽 `SIGTERM/SIGINT`，停止 job、關閉 HTTP server、斷開 Prisma。
- 監聽 `unhandledRejection/uncaughtException` 進行錯誤記錄與退出策略。

---

## 3. 請求處理管線（`app.ts` 實際順序）

1. `helmet`（prod 與 dev 採不同 CSP）  
2. `requestId`（最早注入追蹤 ID）  
3. CORS 前置來源判斷（不合法 origin 直接 403）  
4. `cors(...)` 寫入 CORS header  
5. `localeMiddleware`（語言上下文）  
6. `compression`  
7. `express.json/urlencoded`  
8. JSON parse error 攔截（統一 `INVALID_JSON`）  
9. `performanceMonitor`  
10. `requestLogger`  
11. `opsMetrics`（HTTP 狀態碼指標）  
12. `responseFormatter`  
13. `/uploads` 靜態文件保護鏈（downloadLimiter -> optionalAuthenticate -> authorizeMedia）  
14. health/metrics 路由  
15. 其餘 API 路由  
16. 404 handler  
17. `errorHandler`（全局錯誤）

> 本順序是行為基礎，文檔與實作必須同口徑。

## 4. 路由分層與邊界

## 4.1 路由掛載（摘要）

- `/api/v1/auth`
- `/api/v1/sessions`
- `/api/v1/user`
- `/api/v1/interview`
- `/api/v1/psych-profile`
- `/api/v1/chat`
- `/api/v1/pairing`
- `/api/v1/cases`
- `/api/v1/judgments`
- `/api/v1/execution`
- `/api/v1/admin`
- 以及 `/api/v1` 下合併的 profile/reconciliation/content/notification

## 4.2 分層模式（現況）

- **Controller**：請求參數接收、調用服務、組裝輸出。
- **Service**：核心業務規則、並發控制、交易一致性。
- **Prisma 直連資料層**：無獨立 repository 目錄作為硬性分層。

因此本專案是「Controller -> Service -> Prisma」為主，而非嚴格四層 repository 架構。

---

## 5. 資料與一致性策略

## 5.1 資料庫初始化（`config/database.ts`）

- 支援啟動時遷移/同步（受 `RUN_MIGRATIONS` 與環境保護規則控制）。
- 連線帶重試、超時、診斷日誌。
- production 連線或遷移失敗會阻斷啟動，避免帶病運行。

## 5.2 併發控制

- 透過 `utils/lock.ts` 提供 `lockService.withLock`。
- 優先 Redis 分布式鎖；缺 Redis 時可降級記憶體鎖。
- production 預設不允許無 Redis 直接運行（除非 `ALLOW_SIMPLE_LOCK=true` 顯式接受風險）。

## 5.3 交易與幂等

- 核心流程（判決、訪談管線、部分案件流程）結合 transaction/鎖保護一致性。
- 若流程可重入，通常透過狀態機與唯一鍵/去重鍵控制重複寫入。

---

## 6. 安全與權限架構

- **用戶側**：JWT + Session ID 雙軌（視功能場景）。
- **後台側**：獨立 admin JWT，`adminAuth` + 權限點檢查（RBAC）。
- **中間件安全面**：
  - 限流（多場景差異化）
  - 請求驗證（Joi）
  - 媒體下載授權與方法限制
  - 統一錯誤格式化與翻譯

---

## 7. 可觀測性與運維

- `health` / `health/ready` / `health/live` 健康探針。
- `metrics` + `prom-client` 指標暴露。
- `opsMetrics` 寫入 Redis 分鐘桶（HTTP 狀態碼聚合）。
- `ops-alerts` 以健康檢查與比例閾值產生告警，可對接 Slack。
- 結構化日誌由 `winston` 統一輸出。

---

## 8. 與前端/外部系統集成位置

- 前端統一調用 ` /api/v1/* `。
- SSE 用於訪談與聊天即時事件。
- OpenAI 為主要 AI 外部依賴；Redis 為鎖與部分運維指標依賴；PostgreSQL 為主存儲。

---

## 9. 當前架構決策摘要

- 選擇模組化單體而非微服務：降低 MVP 複雜度，保留後續拆分空間。
- 將安全與觀測能力前置到 middleware 層：減少業務重複邏輯。
- 在高競態點引入鎖與狀態機：避免重複判決/重複處理。
- 支撐型能力（郵件、告警、部分指標）採降級策略：保主路徑可用性。
