# 報告 08：Staging / Preview 環境缺口報告

## 1. 環境定位

- 目標：在正式上線前，用「接近 production」配置做整合驗證與回歸測試。
- 現況：文件層有建議，但尚未形成獨立、可重複執行的制度化環境。

## 2. 目前現況（已存在）

- CI 有多層測試（unit/integration/flow/admin-e2e）。
- 生產配置驗證與保護機制完善（env strict checks、metrics 保護、DB fail-fast）。
- 前後台可獨立 build/run，具備落地 staging 的技術基礎。

## 3. 主要缺口

- 無獨立 `staging` pipeline 與 secrets 組。
- 無 staging 專屬 `DATABASE_URL` / `REDIS_URL` / OpenAI key 隔離策略文件。
- 無 staging 固定 smoke 套件（含 `/health/ready`、關鍵 API、管理台登入、flow 主鏈路）。
- 無 staging 發版前後「準入/回退」標準化清單。

## 4. 建議的最小 staging 契約

- 與 production 同版號 build artifact。
- 獨立 secrets（不可共用 production 密鑰）。
- 必備服務：
  - PostgreSQL（獨立 DB）
  - OpenAI（獨立配額或受限 key）
  - Redis（建議）
- 必開保護：
  - `NODE_ENV=production`
  - `METRICS_TOKEN` 或 `METRICS_ALLOWED_IPS`
  - `ADMIN_JWT_SECRET` 合規

## 5. 推進順序（可直接落地）

1. 建 `staging` secrets 與環境變量模板。
2. 新增 `staging-deploy` workflow（build -> deploy -> smoke）。
3. 將 `admin-e2e` 的核心檢查抽成 staging smoke（可 headless）。
4. 補 staging runbook（准入、回退、資料保護、監控）。

## 6. 成效預期

- 可顯著降低「CI 綠但 production 行為不一致」風險。
- 能在上線前提前暴露配置/遷移/權限類問題。
- 將環境治理從「人記憶」轉為「流程契約 + 可重複驗證」。
