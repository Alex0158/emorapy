# AI 流式回覆與思考動畫全面重構方案

## 1. 文件目的

本文件定義本項目所有 AI 回覆場景的統一目標架構、事件協議、前後端狀態機、可靠性要求、測試要求與最終驗收標準。

後續所有與 AI 回覆體驗相關的改造，均以本文件作為設計基準與驗收標準。

本文件覆蓋的核心目標不是「局部把某幾個頁面改成流式」，而是建立一套可長期演進的統一 AI Streaming Platform，讓所有 AI 場景具備一致、可恢復、可觀測、可驗證的體驗。

## 2. 現況總結

### 2.1 已存在的能力

目前倉庫內已存在兩套不同的流式能力：

1. `Interview/Chat` 直連 SSE token 流
2. `Chat Room` 房間事件流承載 AI token

此外還存在第三類 AI 體驗：

3. 長任務型 AI 頁面，以 polling + loading/動畫 呈現，而非真正流式內容或 phase stream

### 2.2 現有場景分類

#### A. 聊天型直連流式

代表場景：

- `我的故事 -> 繼續聊聊 -> Interview/Chat`

現況：

- 前端透過 `sseRequest` 接收 `token / metadata / safety_alert / complete / error`
- 後端 controller 直接輸出 SSE
- 後端 service 直接從 OpenAI 串流 token

問題：

- 首 token 到達前沒有穩定的「思考中」階段 UI
- 前端狀態機只區分 `isStreaming / streamingText`，過於粗糙
- 協議僅適用於 interview，不可直接複用到其他場景

#### B. 聊天室型事件流

代表場景：

- `聊天室`

現況：

- 使用者先送出訊息
- 另一條 `/rooms/:roomId/stream` SSE 接收房間事件
- AI 透過 `ai_start / ai_token / ai_end` 事件推送

問題：

- `ai_end` 後前端立即清空暫存 bubble，再 refresh 拉正式 message，交接不平滑
- 事件沒有 `seq`、沒有 replay、沒有 `streamId`
- 前端狀態是單例 `isAiStreaming / streamingAiText`，不可擴展
- 事件總線是單機記憶體模型，不適合完整可靠性要求

#### C. 長任務型 AI 頁面

代表場景：

- `QuickExperience/Result`
- 判決生成後等待頁
- 未來所有長任務 AI 生成頁

現況：

- 主要使用 polling
- 頁面播放本地動畫來模擬 AI 思考
- 沒有真實 phase stream
- 沒有正文流式能力

問題：

- 體驗與聊天場景割裂
- 使用者看不到真實進度
- 無法區分「真的在工作」還是「單純卡住」

### 2.3 根本問題

目前缺的不是單一場景的 SSE，而是全項目的統一 AI 回覆生命週期模型。

核心缺口有五個：

1. 沒有統一事件協議
2. 沒有統一前端狀態機
3. 沒有統一暫存回覆與正式消息的交接模型
4. 沒有統一可靠性與恢復機制
5. 沒有統一監控與驗證指標

## 3. 最終目標

本次重構完成後，項目內所有 AI 場景都必須接入同一平台，滿足以下目標：

1. 使用者送出內容後，必須立即獲得即時視覺回饋
2. 所有聊天型 AI 回覆都具備真實逐字流式顯示
3. 所有長任務型 AI 頁面都具備真實 phase stream
4. 任意 AI 回覆都具備可恢復能力
5. 任意 AI 回覆都具備一致的 UI 語義與錯誤恢復方式
6. 任意 AI 回覆都可被量化監控與驗收

## 4. 統一平台架構

建立統一的 `AI Streaming Platform`，由四層組成。

### 4.1 Contract 層

統一定義共享 stream 協議，放入共享 contracts。

所有前後端都使用同一套類型，不再讓 interview 與 chat 各自定義互不兼容事件。

### 4.2 Runtime 層

建立統一後端 stream publisher / stream session manager。

負責：

- stream 建立
- event sequence 編號
- event 發送
- heartbeat
- replay buffer
- reconnect 恢復
- stream 終態管理

### 4.3 Store 層

建立統一前端 `AI Stream Store / Hook`。

現況對齊：

- 已落地共享 hook：`frontend/src/hooks/useAIStreamSubscription.ts`
- `Interview/Chat`、`Chat Room`、`QuickExperience/Result` 已共用此 hook 處理 replay、重連、recovering 與事件歸併
- 當前剩餘尾差不是「沒有共享 hook」，而是尚未再提升成全局單例 store

負責：

- 管理所有進行中的 AI 回覆
- 處理事件累積與狀態轉移
- 管理暫存 bubble / phase card
- 管理恢復、重連、去重、落地交接

### 4.4 UI 層

建立統一 AI 回覆體驗元件系統。

包含：

- `AIResponseBubble`
- `AIThinkingIndicator`
- `AIStreamingText`
- `AIPhaseTimeline`
- `AIRecoveryBadge`
- `AIErrorState`

現況對齊：

- `AIThinkingIndicator`、`AIStreamingText`、`AIPhaseTimeline` 已落地為共享元件
- `AIRecoveryBadge`、`AIErrorState` 亦已補齊，`Interview` 與 `QuickExperience` 已接入共享恢復/錯誤殼層

聊天型 draft bubble 必須共用同一骨架元件，至少統一：

- `aria-live` / `aria-busy`
- streaming cursor
- `thinking/streaming/persisting` 的 DOM 狀態標記
- placeholder 與正式消息交接前的暫存 bubble 結構

## 5. 統一事件協議

### 5.1 事件類型

所有 AI 流式場景統一使用以下事件：

- `stream.created`
- `stream.queued`
- `stream.started`
- `stream.delta`
- `stream.phase`
- `stream.completed`
- `stream.persisted`
- `stream.failed`
- `stream.cancelled`
- `stream.heartbeat`

### 5.2 事件語義

#### `stream.created`

表示系統已建立本次 AI 回覆的 stream session。

#### `stream.queued`

表示本次請求已被接受，等待模型或工作流開始處理。

#### `stream.started`

表示模型或工作流正式開始執行。

#### `stream.delta`

表示可顯示給使用者的內容增量。

聊天型場景必須使用此事件承載 token / chunk。

#### `stream.phase`

表示長任務型流程進入某個業務階段。

例如：

- `collecting_context`
- `analyzing_emotion`
- `building_responsibility`
- `drafting_judgment`
- `finalizing`

#### `stream.completed`

表示模型輸出或工作流輸出邏輯已完成。

注意：這不表示最終內容一定已正式落庫或正式可見。

#### `stream.persisted`

表示最終結果已完成持久化，並可與正式資料模型對接。

這是前端把暫存內容替換成正式 message / judgment / result 的唯一依據。

#### `stream.failed`

表示本次 stream 失敗，並帶明確錯誤碼與錯誤語義。

#### `stream.cancelled`

表示本次 stream 被主動取消。

#### `stream.heartbeat`

表示 stream 仍存活，用於前端保活與 timeout 判定。

### 5.3 標準事件欄位

所有事件必須包含：

- `streamId`
- `requestId`
- `scopeType`
- `scopeId`
- `seq`
- `createdAt`
- `eventType`

可選欄位：

- `actorRole`
- `messageId`
- `phase`
- `deltaText`
- `fullText`
- `metadata`
- `error`

### 5.4 Scope 類型

`scopeType` 必須至少支持：

- `interview_session`
- `chat_room`
- `case_judgment`
- `judgment_detail`
- `generic_ai_task`

### 5.5 序號與恢復

每個 scope 內的每個 stream 都必須有單調遞增的 `seq`。

要求：

- 前端可用 `lastSeq` 恢復
- 後端可按 `lastSeq` replay 漏失事件
- 前端需能對重複事件去重

## 6. 統一前端狀態機

### 6.1 標準狀態

所有 AI 回覆統一使用以下狀態：

- `idle`
- `submitting`
- `queued`
- `thinking`
- `streaming`
- `persisting`
- `completed`
- `failed`
- `cancelled`
- `recovering`

### 6.2 狀態轉移規則

1. 使用者提交請求後立即進入 `submitting`
2. 收到 `stream.created` 或本地樂觀建立後進入 `queued`
3. 若在極短時間內尚未收到首字，也必須進入 `thinking`
4. 收到首個 `stream.delta` 後進入 `streaming`
5. 收到 `stream.completed` 後進入 `persisting`
6. 收到 `stream.persisted` 後進入 `completed`
7. 收到 `stream.failed` 後進入 `failed`
8. 收到 `stream.cancelled` 後進入 `cancelled`
9. 連線中斷但可恢復時進入 `recovering`

### 6.3 UI 顯示規則

#### `queued`

- 立刻顯示 AI placeholder
- 不可空白等待

#### `thinking`

- 顯示思考中動畫
- 可有文案與柔和動態
- 不得讓使用者誤以為卡死

#### `streaming`

- 顯示真實增量文本
- 顯示游標
- 不應因每個 token 重排造成抖動

#### `persisting`

- 保留最後文本
- 顯示「整理中 / 寫入中」的弱提示
- 不得清空暫存內容

#### `completed`

- 以正式 message 或正式結果替換暫存

#### `failed`

- 顯示錯誤狀態
- 提供重試或恢復行為

## 7. 聊天型場景最終規格

### 7.1 Interview

#### 目標體驗

使用者在 `Interview` 送出內容後，應立即看到 AI 已開始回應的 placeholder。

在首 token 到來前：

- 顯示思考中動畫
- 不得空白等待

首 token 到來後：

- 轉為逐字流式顯示

完成後：

- 臨時 bubble 平滑轉為正式 AI turn

#### 必須調整

1. 不再使用 `isStreaming && streamingText` 才渲染 bubble
2. `isStreaming && streamingText === ''` 時必須渲染 thinking bubble
3. metadata / safety / error 必須納入統一 stream envelope
4. 停止生成與中止流程必須映射到 `stream.cancelled`

### 7.2 Chat Room

#### 目標體驗

使用者送出公開訊息後：

- 立即出現 AI thinking placeholder
- token 到來後顯示逐字流
- `completed` 後仍保留暫存 bubble
- `persisted` 後由正式 message 無縫替換

#### 必須調整

1. `ai_end` 不再作為 UI 清空時機
2. 新增 `persisted` 事件
3. stream 狀態不再由單一 `isAiStreaming / streamingAiText` 表示
4. 改為 `streamId -> draftState` 模型
5. 支援重連與 replay

## 8. 長任務型場景最終規格

### 8.1 Quick Experience Result

目前的 loading 動畫不能作為最終方案。

最終要求：

1. 接入 phase stream
2. 真實顯示後端處理進度
3. 若未來後端支持正文流式，前端可無縫顯示

### 8.2 Judgment 類場景

判決生成類任務至少必須顯示：

- 已接收請求
- 正在整理上下文
- 正在分析責任
- 正在撰寫判決
- 正在保存結果

若正文可流式，則在 `drafting_judgment` 階段允許逐段展示。

## 9. 後端統一設計

### 9.1 新增統一服務

必須新增統一 `ai-stream.service`。

職責：

- 管理 stream session
- 生成 `streamId`
- 管理 `seq`
- 發送標準事件
- 生成 heartbeat
- 管理 replay buffer
- 處理 subscriber

### 9.2 統一事件分發

不得再依賴單機記憶體事件總線作為最終形態。

最終要求：

- 使用可跨實例的分發機制
- 使用可 replay 的短期事件存儲

推薦方向：

- Redis Pub/Sub + Redis Stream
- 或等價的可跨實例事件分發與補發方案

### 9.3 統一 SSE/Stream 出口

所有前端即時接入最終都應經過統一 stream gateway。

不再允許：

- interview controller 直接手工拼 token event
- chat routes 自己拼另一套事件語義

### 9.4 統一 AI Adapter

OpenAI token stream 必須經過同一 adapter 輸出標準事件。

要求：

- 聊天型模型輸出轉 `stream.delta`
- 長任務型流程輸出轉 `stream.phase`
- 最終內容保存完成後發 `stream.persisted`

## 10. 資料與持久化設計

### 10.1 Stream Session 記錄

建議新增持久化資料結構 `ai_stream_session`。

現況對齊：

- 已落地為 `ai_stream_sessions`
- 已由 `backend/src/services/ai-stream.service.ts` 在發送 `stream.*` 時同步 upsert
- 已在 `backend/prisma/schema.prisma` 與 `supabase/migrations/20260404193000_add_ai_stream_persistence.sql` 建模

至少包含：

- `id`
- `stream_id`
- `request_id`
- `scope_type`
- `scope_id`
- `status`
- `started_at`
- `completed_at`
- `persisted_at`
- `error_code`
- `error_message`
- `final_entity_type`
- `final_entity_id`

### 10.2 Event Buffer

事件不必永久存庫，但必須有短期可恢復存儲。

現況對齊：

- 短期 replay 已由 Redis buffer + snapshot 支持
- 非 heartbeat 事件已落地為 `ai_stream_events`，作為審計與排障事件表
- 已新增 `GET /api/v1/admin/reports/ai-streams` 作為治理查詢入口
- 已新增 `GET /api/v1/admin/reports/ai-streams/sessions` 與 `GET /api/v1/admin/reports/ai-streams/sessions/:streamId` 作為明細查詢入口
- 已新增 `cleanup_ai_stream_persistence` 定時任務，支持 archive-before-delete
- `ai_stream_session_archives` / `ai_stream_event_archives` 已落地作為長期歸檔層

要求：

- 支援按 `streamId` 或 `scopeId` 找最近事件
- 支援按 `lastSeq` replay
- 支援重連恢復

## 12. 可靠性要求

### 12.1 重連

所有 stream 客戶端必須支持重連。

### 12.2 Replay

後端必須支持按 `lastSeq` 補發。

### 12.3 去重

前端必須對事件去重。

### 12.4 多實例一致性

最終方案必須在多服務實例環境中仍保持可用。

現況對齊：

- 已於 2026-04-04 在本機 Docker/Colima 啟動 `redis:7-alpine`
- 已用 `TEST_REDIS_URL=redis://127.0.0.1:6380 npm run test:integration:ai-stream-redis -- --detectOpenHandles` 跑通跨實例分發、replay 與 snapshot 恢復驗證

### 12.5 暫存與正式資料交接

前端不得在 `persisted` 前清空暫存內容。

### 12.6 Timeout

需區分：

- 首 token timeout
- 連線 timeout
- persist timeout

三者錯誤語義必須不同。

## 13. 可觀測性要求

必須提供以下指標：

- submit -> queued
- submit -> started
- submit -> first delta
- first delta -> completed
- completed -> persisted
- reconnect count
- replay count
- stream failure rate
- cancel rate
- recover success rate

關鍵體驗指標：

- `TTFT`：time to first token
- `TTFP`：time to first phase
- `persist lag`

## 14. 測試要求

### 14.1 單元測試

必須覆蓋：

- 狀態機全部轉移
- 去重
- replay
- 亂序防禦
- `persisted` 交接
- cancel
- timeout

### 14.2 整合測試

必須覆蓋：

- Interview 完整流式鏈路
- Chat Room 完整流式鏈路
- 長任務 phase stream 鏈路
- 中途斷線重連恢復

現況對齊：

- Redis 多實例整合驗證已落地在 `backend/tests/integration/ai-stream-redis.integration.test.ts`
- 真實 Redis 實跑驗證已於 2026-04-04 完成

### 14.3 E2E 測試

必須覆蓋：

1. `Interview` 首 token 延遲時仍顯示 thinking
2. `Interview` token 到達後逐字顯示
3. `Chat Room` 在 `completed` 與 `persisted` 間不閃退
4. `Chat Room` 斷線後可恢復
5. `Quick Experience Result` 顯示真實 phase stream

## 16. 最終驗收標準

以下標準全部滿足，才算本方案驗收通過。

### 16.1 體驗一致性驗收

1. 所有聊天型 AI 場景在送出後 0 空窗
2. 所有聊天型 AI 場景首 token 前均有 thinking 狀態
3. 所有聊天型 AI 場景都是真實逐字流式
4. 所有場景的錯誤、取消、恢復體驗一致

### 16.2 技術一致性驗收

1. 所有 AI 回覆場景使用統一 stream contract
2. 所有 AI 回覆場景使用統一前端狀態機
3. 所有 AI 回覆場景使用統一 stream runtime
4. 不再保留 interview 與 chat 各自孤立的事件語義

### 16.3 可靠性驗收

1. Stream 可重連
2. Stream 可 replay
3. 多實例下可正常工作
4. 暫存內容與正式資料交接穩定

### 16.4 可觀測性驗收

1. TTFT 可量測
2. stream failure 可量測
3. reconnect / replay 可量測
4. persist lag 可量測

### 16.5 測試驗收

1. 單元測試覆蓋狀態機與事件處理
2. 整合測試覆蓋完整鏈路
3. E2E 覆蓋關鍵場景
4. 所有新增測試通過
