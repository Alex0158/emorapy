## 🧪 測試概述

### 測試目標

1. **功能正確性**：確保所有功能按預期工作
2. **穩定性**：確保系統穩定運行
3. **安全性**：確保安全機制有效
4. **性能**：確保響應時間和吞吐量達標

### 測試覆蓋率目標

- **單元測試**：80%+
- **集成測試**：60%+
- **關鍵路徑**：100%

**實現對齊**：單元測試位於 `backend/tests/unit/`，覆蓋 config、controllers、middleware、routes、services、utils、jobs；集成測試位於 `backend/tests/integration/`（含 smoke、quick-experience 流程）。無獨立 repository 層，service 測試通過 mock Prisma 或依賴注入完成。

## 📋 測試類型

### 1. 單元測試

**測試對象**：Service層、工具函數、業務邏輯

**測試框架**：Jest

**示例**：

```typescript
// tests/unit/services/auth.service.test.ts
import { AuthService } from '../../../src/services/auth.service';
import { UserRepository } from '../../../src/repositories/user.repository';
import bcrypt from 'bcrypt';

describe('AuthService', () => {
  let authService: AuthService;
  let userRepository: jest.Mocked<UserRepository>;
  
  beforeEach(() => {
    userRepository = {
      findByEmail: jest.fn(),
      create: jest.fn()
    } as any;
    
    authService = new AuthService(userRepository, emailService);
  });
  
  describe('register', () => {
    it('應該成功註冊新用戶', async () => {
      // Arrange
      const registerData = {
        email: 'test@example.com',
        password: 'password123',
        nickname: 'Test User'
      };
      
      userRepository.findByEmail.mockResolvedValue(null);
      userRepository.create.mockResolvedValue({
        id: 'uuid',
        email: registerData.email,
        nickname: registerData.nickname
      } as User);
      
      // Act
      const result = await authService.register(registerData);
      
      // Assert
      expect(result.user.email).toBe(registerData.email);
      expect(result.token).toBeDefined();
      expect(userRepository.create).toHaveBeenCalled();
    });
    
    it('應該拒絕已存在的郵箱', async () => {
      // Arrange
      userRepository.findByEmail.mockResolvedValue({
        id: 'uuid',
        email: 'test@example.com'
      } as User);
      
      // Act & Assert
      await expect(
        authService.register({
          email: 'test@example.com',
          password: 'password123'
        })
      ).rejects.toThrow('EMAIL_EXISTS');
    });
  });
});
```

### 2. 集成測試

**測試對象**：API端點、數據庫操作、外部服務集成

**測試框架**：Jest + Supertest

**示例**：

```typescript
// tests/integration/auth.routes.test.ts
import request from 'supertest';
import app from '../../src/app';
import { PrismaClient } from '@prisma/client';

const prisma = new PrismaClient();

describe('POST /api/v1/auth/register', () => {
  beforeEach(async () => {
    // 清理測試數據
    await prisma.user.deleteMany();
  });
  
  it('應該成功註冊用戶', async () => {
    const response = await request(app)
      .post('/api/v1/auth/register')
      .send({
        email: 'test@example.com',
        password: 'password123',
        nickname: 'Test User'
      })
      .expect(200);
    
    expect(response.body.success).toBe(true);
    expect(response.body.data.user.email).toBe('test@example.com');
    expect(response.body.data.token).toBeDefined();
  });
  
  it('應該拒絕無效的郵箱格式', async () => {
    const response = await request(app)
      .post('/api/v1/auth/register')
      .send({
        email: 'invalid-email',
        password: 'password123'
      })
      .expect(400);
    
    expect(response.body.success).toBe(false);
    expect(response.body.error.code).toBe('VALIDATION_ERROR');
  });
});
```

### 3. E2E測試

**測試對象**：完整用戶流程

**示例流程**：

```typescript
describe('完整案件流程', () => {
  it('應該完成從註冊到判決的完整流程', async () => {
    // 1. 註冊用戶
    const registerResponse = await request(app)
      .post('/api/v1/auth/register')
      .send({ email: 'user1@test.com', password: 'pass123' });
    
    const token = registerResponse.body.data.token;
    
    // 2. 創建配對
    const pairingResponse = await request(app)
      .post('/api/v1/pairing/create')
      .set('Authorization', `Bearer ${token}`);
    
    // 3. 創建案件
    const caseResponse = await request(app)
      .post('/api/v1/cases')
      .set('Authorization', `Bearer ${token}`)
      .send({
        pairing_id: pairingResponse.body.data.pairing.id,
        title: '測試案件',
        plaintiff_statement: '這是原告的陳述...',
        defendant_statement: '這是被告的陳述...'
      });
    
    // 4. 生成判決
    const judgmentResponse = await request(app)
      .post(`/api/v1/cases/${caseResponse.body.data.case.id}/judgment`)
      .set('Authorization', `Bearer ${token}`);
    
    expect(judgmentResponse.body.success).toBe(true);
    expect(judgmentResponse.body.data.judgment).toBeDefined();
  });
});
```

## v2.0 個人化判決系統測試章節

本節明確定義 v2.0 個人化判決系統的測試覆蓋範圍。

### InterviewService 單元測試

- **start**：建立 session；consent 由 `requireConsent` 中間件檢查，未同意時返回 403 CONSENT_REQUIRED
- **respond**：SSE 流式回應、turn 持久化、鎖獲取/釋放
- **skip**：跳過當前問題、流式回應
- **end**：結束訪談、觸發異步管線；僅 in_progress 可調用（非冪等）
- **resume**：返回未完成 session、供斷線恢復

### AsyncPipelineService 測試

- 管線階段順序（narrative → insight → snapshot → feedback_card）
- 重試邏輯（`processing_failed` 後從 `pipeline_step` 恢復）
- 部分成功（單步失敗標記、不影響已持久化步驟）

### SSE 流式集成測試

- EventSource / ReadableStream 接收 5 種事件：`token`、`metadata`、`safety_alert`、`complete`、`error`
- 當前實現：AI 回傳單一 JSON，後端解析後轉為 SSE 事件（`---METADATA---` 雙通道為設計方案，尚未實現）
- 斷線後後端仍完成寫入、前端 getSession 恢復

### ProfileSnapshotService 測試

- 凍結快照（freeze）：判決前凍結、不可變
- richness 計算：8 域加權、completeness 閾值

### JudgmentService 升級測試

- 畫像注入：richness &lt; 0.05 不注入；0.05–0.2 / 0.2–0.5 / ≥0.5 不同詳度注入
- `generateJudgment()` 內部從 live narratives/insights 即時組裝 context（無獨立 `buildProfileContext` 方法）與 prompt 拼接

### 安全測試

- **知情同意**：未同意時 `requireConsent` 中間件返回 403 CONSENT_REQUIRED
- **限流**：start：雙層限流（中間件每用戶每小時 3 次 + 業務邏輯每天 5 個 substantive session，僅計 ≥3 輪）、respond 間隔 3 秒與 25 輪上限
- **Session 互斥**：同一 session 併發 respond 返回 409、鎖在 turn 持久化與 SSE 關閉後釋放
- **安全偵測**：自傷等關鍵字 → `safety_flag`、SafetyAlert 展示

## 🧠 心理畫像測試策略（v2.0 新增）

### 單元測試

| 測試目標 | 覆蓋重點 | 模擬依賴 |
|---------|---------|---------|
| InterviewService | start（新 session + SEED_QUESTION）/ respond（callback-based SSE + AI JSON 解析）/ skip / end（非冪等，僅 in_progress）生命週期、turn_order 計算、狀態機轉換（in_progress→processing→completed/processing_failed/abandoned）、舊 session 處理（≥3 輪 processing+pipeline / <3 輪 abandoned） | Prisma、aiService |
| AsyncPipelineService | 5 步管線順序（extract→summarize→insights→richness→feedback）、resume(fromStep) 從指定步驟恢復、每步固定退避 2s/4s 重試 2 次、部分失敗恢復、processing_failed 標記 | Prisma、aiService |
| NarrativeService | extractNarratives（按域歸類 turns）+ summarizeNarratives（每域 AI 摘要，ANALYSIS_AI_CONFIG gpt-4o）、heuristic completeness 計算 | Prisma、aiService |
| DomainClassificationService | batchClassify（批量域分類 AI，INTERVIEW_AI_CONFIG gpt-4o-mini）、8 域枚舉校驗、無法分類 fallback | aiService |
| InsightExtractionService | extractInsights(userId, sessionId) 批次 AI（ANALYSIS_AI_CONFIG gpt-4o）、all-or-nothing 寫入、非法 insight_type 過濾 | Prisma、aiService |
| ProfileSnapshotService | createSnapshot（判決前凍結、返回 void）、getSnapshot(caseId, userId)、snapshot 含 narrative summary | Prisma |
| ProfileRichnessService | 8 域加權平均（權重總和 10.3）、completeness 閾值（<0.05/0.05-0.2/0.2-0.5/≥0.5）、域級 completeness 計算 | Prisma |
| PsychProfileService | getProfile（返回 narratives[]+insights[]）、giveConsent、deleteAllData（cascade: insights→narratives→turns→sessions→consent）、GET /feedback（history[]） | Prisma |
| lockService | acquireLock/releaseLock、timeout、重入檢測 | Redis / Prisma |

### 集成測試

| 測試場景 | 驗證點 |
|---------|--------|
| 完整訪談流程 | start → N×respond → end → poll result → feedback_card 生成 |
| SSE 流式回應 | EventSource 接收 5 種事件（token/metadata/safety_alert/complete/error）；當前實現為單一 JSON 解析（`---METADATA---` 為設計方案，未實現）；callback-based SSE（非 AsyncGenerator） |
| 併發控制 | 同一 session 同時發送 2 個 respond → 其中一個收到 409 |
| 知情同意 | 未同意用戶 → requireConsent 中間件返回 403 CONSENT_REQUIRED |
| 遺忘權 | DELETE /psych-profile → insights/narratives/turns/sessions 被刪除 + consent 重置。✅ ProfileSnapshot **保留**（代碼註釋 `// ProfileSnapshot 保留：判決紀錄完整性`）——設計與實現一致 |
| 安全偵測 | 包含自傷關鍵字的回答 → safety_flag: true → 前端顯示 SafetyAlert |
| 訪談恢復 | SSE 斷線後 → GET /resume → 返回 session → GET /:id 載入歷史（含 turns） |
| 判決注入 | richness ≥ 0.05 → generateJudgment 內部從 live data 組裝注入段落（無獨立 buildProfileContext） |
| processing_failed 重試 | session 狀態 processing_failed + pipeline_step=2 → 調用 POST /:id/retry → 讀取 pipeline_step → 從步驟 3 開始恢復 → 每步成功更新 pipeline_step → 最終 completed |
| abandoned 行為 | 用戶有 processing 中的 session 又 start 新 session → processing session 不變為 abandoned、兩者共存；僅 in_progress 的 session 被標為 abandoned |
| 鎖釋放時點 | 同一 session 發送 respond → turn 持久化 + SSE 流關閉後鎖才釋放 → 此時發送第二個 respond 可成功獲取鎖 |
| insight_type 枚舉校驗 | InsightExtractionService 接收到 AI 回傳非法 insight_type（如 "attachment_style"）→ 過濾並記 warn log，不寫入 DB |
| GET /psych-profile | 已完成訪談的用戶 → 返回 200 + narratives[] + insights[]；未訪談用戶 → 返回 200 + 空陣列 |
| GET /psych-profile/feedback | 已完成訪談 → 返回 history[]（含 feedback_card、domains_touched、updated_at）；無歷史 → 空 history[] |
| SSE error 事件 | AI call 失敗 → SSE 推送 `event: error` + `{code: "AI_CALL_FAILED", message: "..."}` → 客戶端收到 error 事件後可重試 |
| NOT_FOUND (404) | 使用不存在的 session_id 調用 respond/end/get → 返回 404 NOT_FOUND；使用其他用戶的 session_id → 同樣 404（不區分不存在與無權限） |
| SESSION_COMPLETED (409) | 已 completed/abandoned 的 session 調用 respond/end → 返回 409 SESSION_COMPLETED |
| MAX_TURNS_REACHED (422) | 第 26 輪 respond → 返回 422 MAX_TURNS_REACHED → 前端自動觸發結束流程 |
| TURN_TOO_FAST (429) | 兩次 respond 間隔 < 3 秒 → 返回 429 TURN_TOO_FAST；≥ 3 秒 → 正常通過 |
| RATE_LIMIT_EXCEEDED (429) | ① 中間件：同一用戶 1 小時內第 4 次 start → 429；② 業務：同一用戶 1 天內第 6 個 substantive session（≥3 輪）→ 429 RATE_LIMIT_EXCEEDED |
| 24h abandoned 清理 | in_progress 超 24 小時的 session → 定時任務標記 abandoned；若 ≥ 3 輪 → 先觸發 pipeline 保存數據再標記；< 3 輪 → 直接標記 abandoned |
| 域分類服務 | DomainClassificationService.batchClassify 正確將 turns 分類到 8 域；無法分類的 turn → 跳過或 fallback 至 personality |
