## 1. 決策摘要

CJ 前台產品能力不再按現有 `F01-F10` 全部同級對外呈現。用戶面向的信息架構收斂為四條主敘事：

1. 快速判斷
   - 單人快速體驗
   - 雙人協作填寫
2. 正式處理
   - 配對
   - 提交案件
   - 判決
   - 修復旅程
3. 先聊再判
   - 聊天室
   - 一鍵轉判決
4. 讓系統更懂你
   - 心理訪談
   - 個人 / 關係檔案

本決策不要求立刻刪除既有能力；重點是把產品入口、導航、頁面命名、文檔口徑與回歸用例從「多個同級能力堆疊」收斂成「四條主線 + 明確內部模式」。

## 2. 現狀問題

現碼與正式文檔中，前台主功能以 `F01-F09` 分布：

1. `F01` 快速體驗。
2. `F02` 協作聽證。
3. `F03` 正式案件管理。
4. `F04` 判決生成 / 查看 / 接受。
5. `F05` 和好方案與執行追蹤。
6. `F06` 心理訪談與心理畫像。
7. `F07` Chat 房間、邀請與轉判決。
8. `F08` 配對與關係檔案。
9. `F09` 身份認證與個人資料管理。

這些功能在工程上是合理拆分，但不適合全部作為用戶面向主入口。主要問題：

1. 「快速體驗」「協作聽證」「正式案件」「聊天室轉判決」都可能產生 `case / judgment`，容易讓用戶不清楚應從哪裡開始。
2. `F02` 協作聽證和 `F03` 正式案件都涉及雙方陳述，但差異其實是匿名同設備體驗 vs 登入後已配對正式處理，不是「單人 vs 雙人」。
3. `F04` 判決、`F05` 修復旅程是正式處理鏈路的後段，不應在產品導航中與「提交入口」混成同級。
4. `F06` 心理訪談是背景理解能力，會提升案件、判決和修復建議質量，但不應和建案入口搶第一層敘事。
5. `F07` 聊天室是自然溝通入口，產品上應表述為「先聊再判」，而不是另一個隱形案件入口。

## 3. 目標信息架構

### 3.1 第一層主入口

前台對用戶可見的一級主入口建議收斂為：

| 主入口 | 產品語義 | 承接能力 | 首屏/導航建議 |
| --- | --- | --- | --- |
| 快速判斷 | 不想註冊或不想正式處理時，先快速得到一次判斷 | F01 + F02 | 主導航 / 首頁主 CTA |
| 正式處理 | 和伴侶認真處理一件事，留下長期記錄並進入修復旅程 | F08 + F03 + F04 + F05 | 主導航 / 登入後主 CTA |
| 先聊再判 | 先開一個房間把事情說清楚，需要時一鍵轉判決 | F07 + F04 | 主導航 / 首頁次 CTA |
| 了解自己 | 讓系統更懂你們的背景、模式和關係脈絡 | F06 + F09 + F08 profile | Profile 區域 / 導航入口 |

### 3.2 第二層模式

「快速判斷」下保留兩個模式：

1. 單人快速體驗：現有 `/quick-experience/create`。
2. 雙人協作填寫：現有 `/quick-experience/collaborative`，產品文案從「協作聽證」降級為「雙人一起填」或「雙人協作填寫」。

「正式處理」下形成線性閉環：

1. 配對。
2. 提交正式案件。
3. 生成 / 查看 / 接受判決。
4. 生成和好方案。
5. 執行、打卡、重調、通知召回。

「先聊再判」下保留聊天室弱入口，但必須明確：

1. 聊天室是溝通空間，不是案件表單。
2. 只有當 A 方選擇「一鍵轉判決」時才進入 `case + judgment` 生成。
3. 判決詳情仍由正式的 `/judgment/:id` 承接。

「了解自己」下整合：

1. 心理訪談。
2. 我的故事 / 心理畫像。
3. 個人資料。
4. 關係檔案。

## 6. Brainstorm 挑戰結論（2026-05-03）

本節補充產品 / 後端 / QA 三個角度的挑戰結論。結論是：四主線方向成立，但不能先只改首頁和導航；必須先收斂 `case` 語義、授權邊界和回歸矩陣，否則會把現有隱性衝突放大。

### 6.1 產品心智與業務邊界

1. 四主線不能在首頁被設計成四個完全同級按鈕；用戶真正需要的是「我現在處於哪種狀態」的選擇：
   - 我只想快速知道大概誰對誰錯：快速判斷。
   - 我和對方想正式處理並留下記錄：正式處理。
   - 我們還沒準備好提交，只想先把話說清楚：先聊再判。
   - 我想讓系統更理解我的關係背景：了解自己 / 關係背景。
2. 「正式處理」不是單一功能，而是一條長鏈路：配對、提交、判決、接受 / 拒絕、修復計畫、執行、打卡、重調、通知召回。入口文案必須讓用戶知道它比快速判斷更重、更正式。
3. 「快速雙人協作填寫」容易被誤解成正式雙方流程。建議避免再使用「協作聽證」作對外名稱，可改為「雙方快速說明」或「雙人快速判斷」。
4. 「先聊再判」必須明確：聊天室本身是低壓溝通，不是默認把對方拉去被判決。轉判決必須是清晰且可感知的動作。
5. 「讓系統更懂你」涉及心理、個資和關係背景，信任成本高。建議在主導航中弱化為 profile / 關係背景能力，不宜和建案入口搶同等主 CTA。
6. 需要決定是否繼續把「案件」「判決」作為用戶可見核心詞。若保留，品牌感更強但壓力更高；若改為「這件事」「關係判斷」，心理負擔較低但法律感會減弱。

### 6.2 後端語義阻斷點

1. 目前 `Case.mode` 同時承載「工程流程 / 授權模式」與「產品入口」，這是最大風險。短期不應把四主線塞進 `CaseMode` enum。
2. `collaborative` 已經存在兩種語義：
   - `collaborative + session_id 有值`：匿名 / 快速雙人。
   - `collaborative + session_id = null`：登入 / 正式雙方或 chat-to-case。
3. 現有代碼已有不一致：
   - `CaseService.getCaseById` 使用正確 helper，只有 `quick` 或 `collaborative + session_id` 視為 session-bound。
   - `JudgmentService.generateJudgment` 仍把所有 `collaborative` 都視為 session-bound，會阻斷正式 collaborative 與登入 chat-to-case 的判決生成。
   - `EvidenceController` 上傳 / 刪除證據也把所有 `collaborative` 當 session-bound，會阻斷正式 collaborative 用 JWT 補證據。
4. `ChatToCaseLink` 是目前唯一能穩定識別「先聊再判」來源的資料。若 case list、analytics、修復旅程或通知不 join 這張表，就會把 chat-to-case 誤算成正式案件。
5. `claim-session` 目前主要覆蓋 quick case，不完整覆蓋快速雙人、匿名 chat、temp pairing、chat-to-case。四主線落地前要定義匿名轉登入後的目標狀態。
6. `mode=remote` 不能再等同於「正式處理」。正式 collaborative、登入 chat-to-case、pending pairing chat case 都可能進入判決 / 修復，但資格與 UI 語義不同。

### 6.3 必須補的工程規則

1. 建立單一 case classifier，不允許各服務各自用 `mode` 判斷：
   - `isSessionBoundCase = quick || (collaborative && session_id != null)`。
   - `isFormalCase = remote || (collaborative && session_id == null && !chat_to_case)`。
   - `isChatConvertedCase = exists ChatToCaseLink`。
2. 所有 judgment、evidence、AI stream、reconciliation、execution、notification follow-up、case list 都應使用同一分類規則。
3. 新增 additive source tracking，避免靠推斷：
   - `entry_point`: `quick_single | quick_collaborative | formal_remote | formal_collaborative | chat_to_case`。
   - `source_channel`: `home | nav | profile | chat_room | notification | unknown`。
   - 對既有 row 做 backfill，推斷不了的標 `unknown`。
4. API 路徑和舊 route 短期不要重命名。先加 hub / alias / tracking，再做文案與入口收斂。

### 6.4 QA / 回歸矩陣補充

四主線重構的最小 smoke 應至少覆蓋：

| 編號 | 場景 | 最小驗證 |
| --- | --- | --- |
| S1 | 舊路由兼容 | `/quick-experience/*`、`/case/*`、`/judgment/:id`、`/chat/room*`、`/profile/*` 不失效 |
| S2 | Protected redirect | 未登入直連 protected deep link，登入後回原目標 |
| S3 | Redirect 精準度 | 帶 `search/hash` 的 deep link 登入後不丟失 |
| S4 | 快速單人 | 建 quick case、result、補證據、session 回訪 |
| S5 | 快速雙人 | A/B 同設備提交、session 授權、錯 session 不可讀 |
| S6 | 匿名轉登入 | claim 成功 / 失敗都不阻斷 auth，成功後案件可回收 |
| S7 | 正式處理 | 配對 gate、提交、判決、和好方案、執行 |
| S8 | Mixed credential | JWT + stale `X-Session-Id` 不可錯誤覆蓋正式 JWT 授權 |
| S9 | 先聊再判 | 建房、邀請、發訊息、轉判決、未登入 judgment handoff |
| S10 | Notification deep link | 後端 CTA path 指向現有 route，過期登入後仍回目標 |
| S11 | AI stream | case/chat/interview/replan stream 的 reconnect、failed、persisted handoff |
| S12 | Mobile | 360 / 390 / 768 / desktop 下入口、底部導航、chat composer 不互相遮擋 |

### 6.5 需要拍板的決策

1. 正式處理是否必須 `active normal pairing` 才能提交？還是允許先建立草稿，再邀請對方？
2. 快速雙人結果是否可以 claim 到登入帳號？若可以，claim 後仍是 quick，還是升格成正式記錄？
3. 匿名聊天室轉判決是否可以 claim？claim 後是否建立 pairing？
4. chat-to-case 是否允許在 `pending pairing` 下進入修復旅程？若允許，UI 要標示為弱配對 / 單方視角。
5. 快速判斷是否能進修復旅程？若不能，判決後 CTA 只能引導升級正式處理；若能，修復旅程要支援 solo / anonymous 邊界。
6. 首頁第一屏要採「四入口」還是「情境選擇器」？建議採情境選擇器，四主線作內部結構。
7. 是否新增 DB tracking 欄位。若新增，必須同步建立 schema migration 待辦並標記 dev / release DB 都要更新。
