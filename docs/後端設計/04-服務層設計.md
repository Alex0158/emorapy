# 服務層設計（代碼對齊版）

**文檔版本**：v3.1  
**最後更新**：2026-03-06  
**對齊基準**：`backend/src/services/*.ts`

---

## 1. 實際架構特徵

- 服務層以「**單例導出**」為主（`export const xxxService = new XxxService()`）
- 非典型 DI 容器架構；服務之間直接 import 調用
- 交易與一致性主要靠：
  - Prisma transaction
  - `lockService` 分散鎖（含 Redis/DB fallback）
- 錯誤統一使用 `Errors.*` 與 `AppError` 體系

## 2. 服務分組（現況）

## 2.1 認證與身份

- `auth.service.ts`
- `session.service.ts`
- `profile.service.ts`
- `pairing.service.ts`

核心點：

- login 失敗計數與帳號鎖定

- 配對建立/加入/狀態管理

## 2.2 案件與判決

- `case.service.ts`
- `judgment.service.ts`

核心點：

- `case.service` 同時覆蓋 quick 與完整模式
- `judgment.service` 注入案件上下文 + 心理畫像上下文（受 env 治理開關控制）

## 2.3 訪談與心理畫像管線

## 3. 兩條關鍵服務鏈路

## 3.1 訪談鏈路

1. `interview.service.startSession`
2. `interview.service.respond/skip`（SSE token/metadata/safety_alert/complete）
3. `interview.service.endSession`
4. `async-pipeline.service.process`
5. `narrative -> insight -> richness -> feedback`
6. `psych-profile.service` 對外讀取畫像/回饋

治理要點：

- consent 必要
- turn 間隔控制
- max turns 限制
- pipeline 失敗可 retry

## 3.2 聊天室轉判決鏈路

1. `chat.service.createRoom/createInvite/acceptInvite`
2. `chat.service.sendMessage`
3. `chat.service.requestJudgment`
4. 內部組裝 case/judgment 上下文
5. `judgment.service` 生成判決
6. 返回 room judgment status

治理要點：

- 角色權限（A/B）
- 訊息可見範圍與歷史可見策略
- 房間狀態機（包含 `judgment_*` 狀態）

---

## 4. 一致性與防重策略

- `lockService.withLock(...)` 用於高衝突寫入路徑（如 quick case、interview respond、pipeline）
- Prisma transaction 用於多表寫入一致性
- chat judgment 有 in-flight promise gate，防止同房間重複發起

---

## 5. 與舊版設計差異（明確）

- 不是「介面 + 注入容器」風格，現況是單例 import 組裝
- 服務數量與責任已擴展到訪談畫像、chat 判決、ops 管理，不再是早期 MVP 的少量 CRUD 服務
- 部分行為由 runtime config + env 雙層控制，文檔須同時關注 `env.ts` 與 `system-config`
