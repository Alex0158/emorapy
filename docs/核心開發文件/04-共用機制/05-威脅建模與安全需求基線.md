## 1. 定位

本文把 Emorapy 的安全治理從「有哪些安全中間件」提升為「有哪些信任邊界、攻擊面、威脅類型、需求 ID、現有證據與缺口」。它對標 OWASP Threat Modeling、OWASP ASVS、OWASP MASVS、NIST SSDF 與 NIST CSF 2.0，但不宣稱 Emorapy 已通過 ASVS / MASVS 或第三方安全審計。

本文只治理需求與追溯口徑，不新增 runtime 行為。若安全需求尚無代碼或測試證據，狀態必須標為待承接或待建立基線。

## 3. 威脅建模方法

Emorapy 採用輕量化 STRIDE + 信任邊界矩陣。STRIDE 只作分類工具，不代表完整 threat model 已完成。

| 類型 | Emorapy 解讀 |
| --- | --- |
| Spoofing | 冒充 user、session user、admin、metrics caller、signed media URL 或 App device |
| Tampering | 篡改 request payload、`X-Session-Id`、case ownership、file path、AI downstream action、config 或 job trigger |
| Repudiation | 高風險操作沒有 request id、audit log、admin actor、release evidence 或 incident record |
| Information Disclosure | case/chat/evidence/psych/AI stream/log/metrics/Push/Deep Link 洩露 |
| Denial of Service | auth brute force、AI cost abuse、large payload/upload、SSE/AI retry storm、Redis/DB dependency failure |
| Elevation of Privilege | user 變 admin、普通 admin 獲得 `admin:all`、session-bound 越界讀 user-bound case、App 本地狀態繞過 backend |

## 6. 高風險流程 threat model 入口

| 流程 | 必須建模的威脅 | 最小證據 |
| --- | --- | --- |
| quick session / claim-session | session fixation、session/user ownership 混淆、直連弱入口 | session tests、auth docs、case ownership review |
| media upload / download | path traversal、unauthorized file read、public upload misconfig、large file DoS、malware | evidence controller tests、`authorizeMedia()` review、upload limiter |
| private chat -> shared mediation -> judgment | sender projection 越界、room event/stream/reply ID 側漏、mixed-context prompt、malicious echo、假摘要、A 代簽 B consent、selection race、hidden-control side channel | A/B audience matrix、per-owner model-request isolation、participant replay、prompt canary、capsule hash/revoke、B identity approval、shared-output counterfactual equality |
| psych interview / profile / snapshot -> judgment / repair | global consent 擴用、秘密證據、raw snapshot 保留、joint profile 單方覆寫、cross-case bleed | purpose-scoped consent、snapshot inventory/no-raw、joint version approval、Judgment/Repair context resolver tests |
| admin jobs / configs / alerts | privilege escalation、config tampering、job trigger abuse、audit gap | admin route permissions、audit logs、Admin e2e |
| metrics / release gate | metrics exposure、secret leakage、false green release、incident evidence 缺失 | metrics auth tests、release evidence、incident/postmortem record |
| App native auth / push / deep link / upload | insecure local storage、deep link hijack、push token mismatch、local file leakage | App adapter tests、MASVS mapping、App smoke/evidence |

## 9. 維護規則

1. 新增 `EMO-SEC-*` 時，若它構成共用 NFR，必須同步 `04-共用機制/02-信任安全與非功能需求.md` 與 `08-測試規範與驗收/04-需求驗證矩陣.md`。
2. 新增外部 provider、AI model、webhook、native capability、media storage、metrics endpoint 或 admin job 前，必須更新信任邊界。
3. Web 安全證據不得替代 App 安全證據；App 需獨立 adapter、smoke、evidence 與 MASVS 映射。
4. 安全需求不得因存在中間件而標為完整覆蓋；需追到測試、代碼審查、release evidence 或正式驗收。
5. 未經正式審計不得宣稱 ASVS level、MASVS 合規、SSDF attestation 或安全認證。
