## 1. 決策摘要

CJ 前台產品能力不再按現有 `F01-F10` 全部同級對外呈現。用戶面向的信息架構收斂為四條主敘事：

1. 快速判斷
   - 單人快速體驗
   - 雙人協作填寫
2. 正式處理
   - 配對
   - 提交案件
   - 判決
   - 修復旅程
3. 先聊再判
   - 聊天室
   - 一鍵轉判決
4. 讓系統更懂你
   - 心理訪談
   - 個人 / 關係檔案

本決策不要求立刻刪除既有能力；重點是把產品入口、導航、頁面命名、文檔口徑與回歸用例從「多個同級能力堆疊」收斂成「四條主線 + 明確內部模式」。

## 2. 現狀問題

現碼與正式文檔中，前台主功能以 `F01-F09` 分布：

1. `F01` 快速體驗。
2. `F02` 協作聽證。
3. `F03` 正式案件管理。
4. `F04` 判決生成 / 查看 / 接受。
5. `F05` 和好方案與執行追蹤。
6. `F06` 心理訪談與心理畫像。
7. `F07` Chat 房間、邀請與轉判決。
8. `F08` 配對與關係檔案。
9. `F09` 身份認證與個人資料管理。

這些功能在工程上是合理拆分，但不適合全部作為用戶面向主入口。主要問題：

1. 「快速體驗」「協作聽證」「正式案件」「聊天室轉判決」都可能產生 `case / judgment`，容易讓用戶不清楚應從哪裡開始。
2. `F02` 協作聽證和 `F03` 正式案件都涉及雙方陳述，但差異其實是匿名同設備體驗 vs 登入後已配對正式處理，不是「單人 vs 雙人」。
3. `F04` 判決、`F05` 修復旅程是正式處理鏈路的後段，不應在產品導航中與「提交入口」混成同級。
4. `F06` 心理訪談是背景理解能力，會提升案件、判決和修復建議質量，但不應和建案入口搶第一層敘事。
5. `F07` 聊天室是自然溝通入口，產品上應表述為「先聊再判」，而不是另一個隱形案件入口。

## 3. 目標信息架構

### 3.1 第一層主入口

前台對用戶可見的一級主入口建議收斂為：

| 主入口 | 產品語義 | 承接能力 | 首屏/導航建議 |
| --- | --- | --- | --- |
| 快速判斷 | 不想註冊或不想正式處理時，先快速得到一次判斷 | F01 + F02 | 主導航 / 首頁主 CTA |
| 正式處理 | 和伴侶認真處理一件事，留下長期記錄並進入修復旅程 | F08 + F03 + F04 + F05 | 主導航 / 登入後主 CTA |
| 先聊再判 | 先開一個房間把事情說清楚，需要時一鍵轉判決 | F07 + F04 | 主導航 / 首頁次 CTA |
| 了解自己 | 讓系統更懂你們的背景、模式和關係脈絡 | F06 + F09 + F08 profile | Profile 區域 / 導航入口 |

### 3.2 第二層模式

「快速判斷」下保留兩個模式：

1. 單人快速體驗：現有 `/quick-experience/create`。
2. 雙人協作填寫：現有 `/quick-experience/collaborative`，產品文案從「協作聽證」降級為「雙人一起填」或「雙人協作填寫」。

「正式處理」下形成線性閉環：

1. 配對。
2. 提交正式案件。
3. 生成 / 查看 / 接受判決。
4. 生成和好方案。
5. 執行、打卡、重調、通知召回。

「先聊再判」下保留聊天室弱入口，但必須明確：

1. 聊天室是溝通空間，不是案件表單。
2. 只有當 A 方選擇「一鍵轉判決」時才進入 `case + judgment` 生成。
3. 判決詳情仍由正式的 `/judgment/:id` 承接。

「了解自己」下整合：

1. 心理訪談。
2. 我的故事 / 心理畫像。
3. 個人資料。
4. 關係檔案。

## 6. Brainstorm 挑戰結論（2026-05-03）

本節補充產品 / 後端 / QA 三個角度的挑戰結論。結論是：四主線方向成立，但不能先只改首頁和導航；必須先收斂 `case` 語義、授權邊界和回歸矩陣，否則會把現有隱性衝突放大。

### 6.1 產品心智與業務邊界

1. 四主線不能在首頁被設計成四個完全同級按鈕；用戶真正需要的是「我現在處於哪種狀態」的選擇：
   - 我只想快速知道大概誰對誰錯：快速判斷。
   - 我和對方想正式處理並留下記錄：正式處理。
   - 我們還沒準備好提交，只想先把話說清楚：先聊再判。
   - 我想讓系統更理解我的關係背景：了解自己 / 關係背景。
2. 「正式處理」不是單一功能，而是一條長鏈路：配對、提交、判決、接受 / 拒絕、修復計畫、執行、打卡、重調、通知召回。入口文案必須讓用戶知道它比快速判斷更重、更正式。
3. 「快速雙人協作填寫」容易被誤解成正式雙方流程。建議避免再使用「協作聽證」作對外名稱，可改為「雙方快速說明」或「雙人快速判斷」。
4. 「先聊再判」必須明確：聊天室本身是低壓溝通，不是默認把對方拉去被判決。轉判決必須是清晰且可感知的動作。
5. 「讓系統更懂你」涉及心理、個資和關係背景，信任成本高。建議在主導航中弱化為 profile / 關係背景能力，不宜和建案入口搶同等主 CTA。
6. 需要決定是否繼續把「案件」「判決」作為用戶可見核心詞。若保留，品牌感更強但壓力更高；若改為「這件事」「關係判斷」，心理負擔較低但法律感會減弱。

### 6.2 後端語義阻斷點

1. 目前 `Case.mode` 同時承載「工程流程 / 授權模式」與「產品入口」，這是最大風險。短期不應把四主線塞進 `CaseMode` enum。
2. `collaborative` 已經存在兩種語義：
   - `collaborative + session_id 有值`：匿名 / 快速雙人。
   - `collaborative + session_id = null`：登入 / 正式雙方或 chat-to-case。
3. 現有代碼已有不一致：
   - `CaseService.getCaseById` 使用正確 helper，只有 `quick` 或 `collaborative + session_id` 視為 session-bound。
   - `JudgmentService.generateJudgment` 仍把所有 `collaborative` 都視為 session-bound，會阻斷正式 collaborative 與登入 chat-to-case 的判決生成。
   - `EvidenceController` 上傳 / 刪除證據也把所有 `collaborative` 當 session-bound，會阻斷正式 collaborative 用 JWT 補證據。
4. `ChatToCaseLink` 是目前唯一能穩定識別「先聊再判」來源的資料。若 case list、analytics、修復旅程或通知不 join 這張表，就會把 chat-to-case 誤算成正式案件。
5. `claim-session` 目前主要覆蓋 quick case，不完整覆蓋快速雙人、匿名 chat、temp pairing、chat-to-case。四主線落地前要定義匿名轉登入後的目標狀態。
6. `mode=remote` 不能再等同於「正式處理」。正式 collaborative、登入 chat-to-case、pending pairing chat case 都可能進入判決 / 修復，但資格與 UI 語義不同。

### 6.3 必須補的工程規則

1. 建立單一 case classifier，不允許各服務各自用 `mode` 判斷：
   - `isSessionBoundCase = quick || (collaborative && session_id != null)`。
   - `isFormalCase = remote || (collaborative && session_id == null && !chat_to_case)`。
   - `isChatConvertedCase = exists ChatToCaseLink`。
2. 所有 judgment、evidence、AI stream、reconciliation、execution、notification follow-up、case list 都應使用同一分類規則。
3. 新增 additive source tracking，避免靠推斷：
   - `entry_point`: `quick_single | quick_collaborative | formal_remote | formal_collaborative | chat_to_case`。
   - `source_channel`: `home | nav | profile | chat_room | notification | unknown`。
   - 對既有 row 做 backfill，推斷不了的標 `unknown`。
4. API 路徑和舊 route 短期不要重命名。先加 hub / alias / tracking，再做文案與入口收斂。

### 6.4 QA / 回歸矩陣補充

四主線重構的最小 smoke 應至少覆蓋：

| 編號 | 場景 | 最小驗證 |
| --- | --- | --- |
| S1 | 舊路由兼容 | `/quick-experience/*`、`/case/*`、`/judgment/:id`、`/chat/room*`、`/profile/*` 不失效 |
| S2 | Protected redirect | 未登入直連 protected deep link，登入後回原目標 |
| S3 | Redirect 精準度 | 帶 `search/hash` 的 deep link 登入後不丟失 |
| S4 | 快速單人 | 建 quick case、result、補證據、session 回訪 |
| S5 | 快速雙人 | A/B 同設備提交、session 授權、錯 session 不可讀 |
| S6 | 匿名轉登入 | claim 成功 / 失敗都不阻斷 auth，成功後案件可回收 |
| S7 | 正式處理 | 配對 gate、提交、判決、和好方案、執行 |
| S8 | Mixed credential | JWT + stale `X-Session-Id` 不可錯誤覆蓋正式 JWT 授權 |
| S9 | 先聊再判 | 建房、邀請、發訊息、轉判決、未登入 judgment handoff |
| S10 | Notification deep link | 後端 CTA path 指向現有 route，過期登入後仍回目標 |
| S11 | AI stream | case/chat/interview/replan stream 的 reconnect、failed、persisted handoff |
| S12 | Mobile | 360 / 390 / 768 / desktop 下入口、底部導航、chat composer 不互相遮擋 |

### 6.5 需要拍板的決策

1. 正式處理是否必須 `active normal pairing` 才能提交？還是允許先建立草稿，再邀請對方？
2. 快速雙人結果是否可以 claim 到登入帳號？若可以，claim 後仍是 quick，還是升格成正式記錄？
3. 匿名聊天室轉判決是否可以 claim？claim 後是否建立 pairing？
4. chat-to-case 是否允許在 `pending pairing` 下進入修復旅程？若允許，UI 要標示為弱配對 / 單方視角。
5. 快速判斷是否能進修復旅程？若不能，判決後 CTA 只能引導升級正式處理；若能，修復旅程要支援 solo / anonymous 邊界。

## 7. 第二輪深挖結論（2026-05-03）

本節補充安全、資料治理、狀態機、商業產品、發布運維五個角度的再挑戰。第二輪結論比第一輪更嚴格：四主線不是只要收斂信息架構，還必須建立安全硬 gate、資料生命週期、真實狀態機、轉化權益邊界與運維可排查性。

### 7.1 安全與濫用治理

1. 目前 repo 已有 `safety_support / crisis_support` 雛形，但主要在 AI 判決或修復生成階段起作用，尚未形成產品級硬 gate。
2. 必須明確哪些情況下不應：
   - 邀請對方。
   - 共同修復。
   - 聊天轉判決。
   - 顯示責任比例。
   - 發送伴侶召回通知。
   - 把敏感內容納入 AI prompt。
3. 需要全域風險等級，例如：
   - `standard`。
   - `sensitive`。
   - `high_risk_relationship`。
   - `imminent_crisis`。
   - `minor_or_suspected_minor`。
   - `illegal_or_nonconsensual_content`。
4. 高風險關係下，產品不得預設「雙方都有責任 / 雙方各退一步 / 邀請對方一起修復」。應強制走 solo safety plan。
5. 聊天轉判決不能只由 A 方單方發起後直接納入 B 方內容。B 方需要明示同意，並看到將被納入的訊息範圍。
6. Chat invite 需要反濫用規則：限速、短效碼、重複邀請冷卻、撤回、封鎖、檢舉、被拒後禁止立即再邀。
7. 未成年人與敏感證據目前邊界不足。正式處理、心理訪談、證據上傳前需要年齡 / 敏感內容 gate。
8. 建議新增 `SafetyAssessment` / `RelationshipRiskState` 或等價能力，讓 room、pairing、case、repair、notification 都能讀同一個安全狀態。

### 7.2 資料治理、同意與留存

1. 「快速判斷」現在不是短期資料，只是匿名 session 短期；`Case / Evidence / Judgment / temp Pairing` 仍可能長期存在。必須明確承諾的是 session 過期，還是資料過期。
2. `claim-session` 目前不是完整歸戶：只補 quick case 的 `plaintiff_id`，沒有處理 evidence owner、temp pairing、quick collaborative、anonymous chat、chat-to-case。
3. Chat `visibility_scope` 只是前台可見性，不等於資料刪除或 AI 使用同意；`ChatToCaseLink.conversion_snapshot` 會留下被納入判決的 message ids、摘要與風險旗標。
4. 心理資料 consent 目前偏單一 boolean，缺少 scope、version、撤回時間、用途邊界。應拆分：
   - 訪談同意。
   - 生成心理 / 關係檔案同意。
   - 用於判決個人化同意。
   - 用於修復建議個人化同意。
   - 撤回後不再注入 prompt。
5. Evidence 需要 retention / delete policy，不能只靠 file URL。刪 case、刪帳號、撤回 consent 後如何處理檔案必須定義。
6. Notifications payload 應最小化，避免長期存敏感 case title 或關係內容；通知偏好不等於資料處理同意。
7. Admin / support access 需要 purpose、欄位遮罩、存取審計與 export 限制。`reports:read` 看到 AI stream 明細時尤其要防止敏感原文外洩。
8. 需要建立 retention matrix：quick、formal case、chat messages、evidence、psych data、AI stream live/archive、notifications、audit logs 分別保留多久、如何刪除 / 匯出 / 匿名化。

### 7.3 狀態機與資料不變式

1. `quick -> formal` 目前沒有真實狀態機。註解提到轉正式，但實作主要只是把 quick case 補上 `plaintiff_id`，沒有轉 `mode`、建立正式 pairing、綁定 B 方。
2. 匿名 chat 登入後沒有 claim；`claimSession` 不處理 `ChatRoom.owner_user_id`、`ChatParticipant.user_id`、`ChatToCaseLink`。
3. `PairingStatus.temp` 是 quick 主線外的特殊狀態，且 quick temp pairing 可能 `expires_at = null`，需要納入正式狀態模型或明確標記為非正式 pairing。
4. `ChatRoomStatus.invite_accepted`、`ChatInviteStatus.expired`、`CaseStatus.cancelled` 等 enum 存在但 transition 不完整。要麼補 transition，要麼標記 deprecated。
5. Chat 轉判決有三套狀態：room、case、link。若 crash 在 `judgment_requested`，需要 recovery job。
6. RepairTrack、RepairParticipantState、ExecutionRecord 目前仍有新舊模型混用風險。長期應把 RepairTrack 作為修復旅程唯一 source of truth，ExecutionRecord 降級為 event / legacy log。
7. 必須建立跨表 consistency audit，先只報不自動修：
   - case / judgment。
   - chat room / chat-to-case link / judgment。
   - reconciliation plan / repair track / execution record。
   - pairing / case / user。
8. Pairing 已補服務層 invariant 與 migration precheck：一個 user 同時最多一個 normal `pending / active` pairing，join 前會檢查加入者沒有其他 active / pending pairing；DB-level partial unique / exclusion 約束仍需先跑 `precheck:pairing:normal-uniqueness` 並另開 migration 任務。

### 7.4 商業產品、轉化與留存

1. 首屏不應是「四入口平鋪」，而應是「現在你們在哪一步」：
   - 我先自己理清一次。
   - 我們一起把話說完。
   - 我們想正式處理這件事。
   - 我們先開個安全對話空間。
2. 「判決」可以保留為品牌內核，但首頁與 CTA 應降壓，可用「關係分析」「第三方整理」「下一步建議」等語言承接。
3. 匿名到註冊的最佳節點是用戶看到價值後：
   - quick result 後保存結果。
   - judgment 後轉修復旅程。
   - chat-to-case 後保存並查看結果。
   - repair 第一次小行動後開啟提醒 / 共同承諾。
4. 需要先定免費 / 註冊 / 付費權益，不急著定價格：
   - 免費：一次快速判斷與基本結果。
   - 註冊：保存歷史、claim 匿名結果、正式處理。
   - 付費或高階：深度修復旅程、重調、長期追蹤、深度個人化。
5. 不應一級展示：
   - 判決詳情。
   - 和好方案列表 / 執行看板。
   - 協作聽證。
   - Profile settings。
   - 心理訪談。
   - 通知中心。
   - Admin。

### 7.5 發布、運維、客服與管理後台

1. 四主線後，Admin 不能只看 cases / judgments 總量；目前 overview 已能返回 product-flow 分布，但仍需補「卡在哪一步、花了多少 AI 成本、是否應通知召回」。
2. 通知召回與排程清理語義已先補兩處後端漏口：修復旅程 choose-direction reminder 不再只按 `mode = remote`，改用 user-bound case scope，覆蓋正式 collaborative 與登入 chat-to-case；`cleanup_stale_draft_cases` 不再只清 remote draft，改用 stale formal draft scope 並排除 quick/session-bound/chat-to-case；其他 notification / analytics / scheduled cleanup 入口仍需逐步接入 `case-classifier`。
3. 需要 Admin 客服排查時間線：輸入 userId / email / sessionId / caseId / roomId 後，串起 auth、session、case、judgment、chat、interview、repair、notification、AI stream、job log、audit log。
4. 需要 AI request ledger：每次 AI request 記錄 flow、scope、model、prompt version、tokens、cost、retry、failure reason、stream_id / request_id。
5. 需要 notification 管理能力：查 dedup key、pending / sent / failed / read / snooze / acted、CTA path 校驗、取消 pending、重送 failed、批量召回，且所有操作寫 audit。
6. Production 必須禁止 seed / smoke 預設帳號污染。任何 seed / smoke admin 都要有環境前綴、過期策略、發布後掃描與停用流程。
7. 發布 gate 需要補四主線語義驗收：舊 URL、登入回跳、mixed credential、notification deep link、AI stream reconnect、chat-to-case、repair eligibility、Redis-backed runtime。

### 7.6 第二輪新增 P0 / P1

P0：

1. 全域 Safety Policy Engine：四主線共用風險分類與硬 gate。
2. Chat invite abuse 防護與 B 方轉判決明示同意。
