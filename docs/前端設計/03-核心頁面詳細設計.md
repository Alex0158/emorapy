# 熊媽媽法庭 - 核心頁面詳細設計

**項目名稱**：熊媽媽法庭（Mother Bear Court）  
**設計階段**：MVP開發階段  
**文檔版本**：v3.1

---

## 🎯 設計原則

### 核心設計理念

1. **零門檻體驗**：快速體驗模式無需註冊，立即使用
2. **溫暖親和**：視覺設計體現母熊法官的溫暖與包容
3. **清晰直觀**：流程簡單明了，降低認知負荷
4. **即時反饋**：每個操作都有明確的視覺反饋
5. **情感化設計**：通過動畫、插畫增強情感連接

- **暫停按鈕**：session 保持 `in_progress`，下次進入自動提示續聊
- **自動結束**：AI 設 `should_end: true` → 隱藏輸入框 → 自動調 POST /end → 轉場動畫 → polling 結果（用戶零操作）
- **連點防護**：發送後立即 disable 輸入框 + 按鈕，SSE 完成後恢復
- **續聊載入**：從 `/resume` 返回時先調 `GET /interview/:id` 載入完整對話（回應含 turns）

- 輸入框區域：固定在底部（position: sticky），上方 12px 陰影分隔

#### 4.5 SSE 處理說明

後端實現為**非逐字流式**：先完成整次 AI 呼叫取得完整回應，再由服務端解析後經 SSE 依序推送 `token`（可含整段文本）→ `metadata` → `safety_alert`（如有）→ `complete`。前端使用 `fetch` + `ReadableStream`（而非 `EventSource`，因需要 POST + 自定義 headers）。

```typescript
const response = await fetch(`/api/v1/interview/${sessionId}/respond`, {
  method: 'POST', headers: { Authorization: `Bearer ${token}`, 'Content-Type': 'application/json' },
  body: JSON.stringify({ message: userText }),
});
const reader = response.body!.getReader();
const decoder = new TextDecoder();
// 逐 chunk 解析 SSE 格式：event: token\ndata: {"text":"..."}\n\n
```

**事件處理**（與 [03-API設計](../後端設計/03-API設計.md) SSE 事件對齊）：
- `token` → 追加 `data.text` 到 pendingMessage buffer 並渲染（後端可能單次推送完整文本，前端可選擇一次性顯示或模擬逐字）
- `metadata` → 存儲 metadata 到當前輪；如 `should_end: true` 則設置 shouldAutoEnd 標記
- `safety_alert` → ✅ **已實現**：interviewStore 設置 `safetyAlert` 狀態，Chat 頁面渲染 SafetyAlert 組件（含 severity 分級、critical 時顯示危機資源熱線）。
- `error` → 顯示 Toast 錯誤提示
- `complete` → 將 pendingMessage 寫入 turns 陣列，清空 pendingMessage buffer，恢復輸入框

#### 4.6 錯誤與邊界處理

| 場景 | 處理 |
|------|------|
| 回應超時（30s 無 token） | ✅ **已實現**：`sseRequest.ts` 內建 30s TOKEN_TIMEOUT_MS，30 秒內無任何 token 事件 → 回調 `onError({ code: 'RESPONSE_TIMEOUT' })` → Chat 頁面顯示「AI 回覆超時，請重新發送」 |
| 連線超時（60s 無任何 SSE 事件） | ✅ **已實現**：`sseRequest.ts` 內建 60s CONNECTION_TIMEOUT_MS，60 秒完全無事件 → 回調 `onError({ code: 'CONNECTION_TIMEOUT' })` → Chat 頁面顯示「連線超時」 |
| 網絡斷開（navigator.onLine=false） | 顯示 Toast「網路中斷」→ 恢復後調 `GET /interview/:id` 載入完整對話 → 恢復輸入框。⚠️ **尚未實現**（無 `navigator.onLine` 監聽） |
| 403 CONSENT_REQUIRED | ⚠️ **實現差異**：訪談路由無 ConsentGuard 組件；consent 由 Profile 頁面進入前處理。後端 `requireConsent` 中間件返回 403，前端全局 request 攔截（非訪談專用處理） |
| 409 SESSION_COMPLETED | ⚠️ **實現差異**：無訪談專用的 409 處理或跳轉；由全局 request 攔截器顯示通用 Toast |
| 409 CONCURRENT_REQUEST | 顯示 Toast「正在處理中，請稍候」（前端已禁用按鈕，極少觸發）。⚠️ **實現差異**：由全局 request 攔截器處理，非訪談專用 |
| 429 TURN_TOO_FAST | ⚠️ **尚未實現**：無 3 秒後自動恢復輸入框的特殊處理，由全局 429 攔截器顯示通用 Toast |
| 空字串發送 | 前端攔截：trim 後為空則不發送。⚠️ **無 shake 動畫** |
| 超長文字（>2000 字） | ✅ **已實現**：InterviewInput 內建 MAX_CHARS=2000 限制，超過 80% 時顯示字數計數器，超限時禁用發送按鈕並 TextArea 顯示 error 狀態 |
| 訪談限流 | 雙層：①中間件每用戶每小時 3 次（POST /start）；②業務邏輯每用戶每天 5 個 substantive session（≥3 輪才計）；每 session 最多 25 輪、每輪間隔 ≥3 秒。見 [03-API設計](../後端設計/03-API設計.md) §限流策略 |
