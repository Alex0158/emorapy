# Emorapy F01 快速體驗、結果與升格測試設計與開發拆解

日期：2026-03-07  
對應功能：`F01 快速體驗、結果與升格`  
對應流程：`P01 快速體驗閉環`

## 1. 本次測試目標

本次屬於 Emorapy 測試開發的第一個 P0 專項。目標不是單純補案例數，而是先把 F01 真正做成一條可回歸、可揭露業務缺陷的主鏈路。

本次優先要驗證的業務規則如下：

1. 訪客進入 `/quick-experience/create` 時，無 `session_id` 應自動建立匿名 session。
2. 訪客提交 quick case 時，只要求原告陳述達標；被告陳述可為空字串，不得被過時測試錯誤阻斷。
3. quick case 建立後，結果頁必須能正確讀取案件、輪詢判決、展示責任比例與非阻斷內容推薦。
4. quick case 補傳證據失敗時，不得破壞主流程，且後續應可重試。
5. quick case 在登入/註冊後，`claim-session` 失敗不應阻斷 auth 成功態，但應有足夠測試護欄驗證其成功/失敗語義。
6. Session 類錯誤（`SESSION_EXPIRED`、`SESSION_ID_REQUIRED`、`INVALID_SESSION_ID`）必須遵守「清舊 -> refresh -> 重試」策略，且不能因舊 `caseSessionMap` 導致回訪失敗。

## 2. 代碼真實規則（以實際代碼為準）

### 2.1 功能、流程、頁面、接口基線

依據以下核心文檔：

1. `docs/核心開發文件/功能特性清單.md`
2. `docs/核心開發文件/業務流程整合.md`
3. `docs/核心開發文件/頁面清單.md`
4. `docs/核心開發文件/接口-功能-頁面-Mapping.md`
5. `docs/核心開發文件/接口描述/01-auth-session.md`

其中 F01 / P01 的操作級主鏈路為：

1. `POST /api/v1/sessions/quick`
2. `POST /api/v1/sessions/refresh`
3. `GET /api/v1/cases/by-session`
4. `POST /api/v1/cases/quick`
5. `POST /api/v1/cases/:id/evidence`
6. `GET /api/v1/cases/:id`
7. `GET /api/v1/cases/:id/judgment`
8. `GET /api/v1/content-items`
9. `POST /api/v1/auth/claim-session`（F01/F09 共用後置流程，登入/註冊後隱式）

### 2.2 前端 Create 頁的真實規則

核心位於：

1. `frontend/src/pages/QuickExperience/Create/index.tsx`
2. `frontend/src/store/sessionStore.ts`
3. `frontend/src/store/caseStore.ts`

當前代碼可確認：

1. 進頁時若本地沒有 `session_id`，會先建立 session。
2. quick case 送出前，只強制要求原告陳述達標。
3. 被告陳述實際提交為 `defendantStatement.trim() || ''`。
4. 若建案成功且回傳 `session_id`，前端會同步寫入：
   - `sessionStorage`
   - `caseSessionMap`
   - `useSessionStore`
5. 證據上傳失敗時，會記錄 `pending_evidence_<caseId>`，不阻斷跳轉結果頁。

### 2.3 後端 quick case 的真實校驗規則

核心位於：

1. `backend/src/utils/validation.ts`
2. `backend/src/services/case.service.ts`

當前 schema 明確規定：

1. `plaintiff_statement`：必填，長度 `30-2000`
2. `defendant_statement`：可選，長度 `10-2000`，且允許 `null` 與空字串 `''`
3. `evidence_urls`：可選，最多 3 個，且必須為 `https` URL

這表示：

1. 「被告陳述不能為空」不是當前事實規則
2. 若現有測試仍把空被告視為錯誤，應判定為測試過時，而不是產品需求

### 2.4 結果頁與 session 的真實規則

核心位於：

1. `frontend/src/pages/QuickExperience/Result/index.tsx`
2. `frontend/src/services/request.ts`
3. `frontend/src/utils/storage.ts`
4. `frontend/src/store/sessionStore.ts`

當前代碼可確認：

1. 結果頁優先用 `caseSessionMap.get(caseId)`，其次才是 `sessionStorage.get()` 與 `useSessionStore().session`。
2. `request.ts` 在收到 session 類 `400/401` 時，會：
   - 清理舊 session
   - 呼叫 `refreshSession(true)`
   - 之後再依情況給提示
3. `refreshSession()` 目前只更新全局 `sessionStorage` 與 store，不會同步更新既有 `caseSessionMap`。

這帶來一個高風險點：

1. 若某 quick case 仍綁舊 `caseSessionMap`，但後端 refresh 後已改綁新 session，結果頁可能仍優先帶舊 session 查詢，造成回訪失敗。

### 2.5 `claim-session` 的當前事實

核心位於：

1. `backend/src/services/auth.service.ts`
2. `backend/src/controllers/auth.controller.ts`
3. `backend/src/routes/auth.routes.ts`
4. `frontend/src/store/authStore.ts`

當前代碼可確認：

1. `login/register` 成功後，若本地存在 quick session，會非阻斷呼叫 `authApi.claimSession()`。
2. `claim-session` 目前實作主要是：
   - 找到 `quickSession.case_id`
   - 驗證 case 存在且 `mode === quick`
   - 若 `plaintiff_id` 尚未被占用，則寫入當前 `userId`
3. 當前實作沒有體現註釋中「轉 remote 模式、建立正式配對」的完整語義。

目前核心 SSOT 已把它明確定義為 `F01/F09` 共用後置流程。

因此本專項要把它視為兩件事：

1. 先補足當前真實行為的測試護欄
2. 同時把「升格後還有哪些產品語義要擴張」列為需持續觀察的高風險點

## 4. 已確認的缺口與疑點

### 4.1 測試過時

1. `backend/tests/integration/quick-experience.flow.test.ts` 中仍存在「被告陳述不能為空」的過時假設。
2. `frontend/src/pages/QuickExperience/Create/index.test.tsx` 中存在多個明顯對不上現行 UI 的斷言，如：
   - vertical 版型
   - `startNew`
   - 模板切換

### 4.2 業務/文檔風險

1. `claim-session` 的註釋、接口文檔與實作語義並不完全一致。
2. 結果頁優先使用 `caseSessionMap` 舊值，而 refresh 目前未同步更新 `caseSessionMap`，可能導致回訪缺陷。

### 4.3 測試層級缺口

1. 後端：
   - `auth.service.claimSession` 分支覆蓋不足
   - `POST /auth/claim-session` 路由測試不足
2. 前端：
   - `authStore` 對 `claimSession` 成功/失敗的非阻斷語義雖有基本測試，但仍需補更精確分支
   - Result 頁缺少 session refresh 與 `caseSessionMap` 聯動回歸
3. E2E：
   - 尚無 F01 專屬主鏈路 E2E

## 5. 本次開發範圍

本輪只處理 F01，並且按「先文檔、再後端、再前端、再失敗分流」落地。

### 5.1 後端

本輪優先補：

1. `auth.service.claimSession`
   - 無 session
   - session 無 case
   - case 非 quick
   - 已有 `plaintiff_id`
   - 正常 claim 成功
2. `POST /auth/claim-session`
   - controller / route 正常路徑
   - 未登入
   - request body 驗證

### 5.2 前端

本輪優先補：

1. 整修 `QuickExperienceCreate` 過時測試，移除對舊 UI 的錯誤假設。
2. 補 `Create` 頁「被告可留空」的真實規則測試。（已補）
3. 補 `Create` 頁「原告陳述不足 30 字時下一步 disabled」邊界測試（validateStatement 30-2000 規則）。（2026-03-07 已補）
4. 補 `Result` 頁 session refresh / session fallback / `caseSessionMap` 使用順序的回歸測試。
5. 補 `authStore` 的 `claimSession` 非阻斷語義測試。

### 5.3 E2E

本輪先建立最小 F01 主鏈路：

1. 訪客進入 quick create
2. 提交最小合法案件
3. 跳結果頁
4. pending / judgment 其中一種可驗證狀態成立

若需要先降低外部依賴，本輪也可先只建立文件與測試骨架，之後再擴展成完整真鏈路。

## 7. 與測試設計鐵律的對齊

本專項明確遵守 `Emorapy-測試設計鐵律.md`：

1. 先驗證真正的業務規則，不為了讓測試通過而保留過時假設。
2. 若文檔與代碼不一致，以代碼為當前事實來源，同時記錄不一致。
3. 對跨層流程（session、quick case、claim-session）優先做高價值閉環，而非堆積零散 UI case。
4. 若本輪測試確認 `caseSessionMap` / refresh 的真缺陷，必須立入 `04-業務缺陷報告`，不能修改測試去接受錯誤行為。
