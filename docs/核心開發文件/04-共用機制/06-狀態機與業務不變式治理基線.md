## 1. 定位

本文把 Emorapy 的狀態治理從「schema 裡有哪些 enum」提升為「哪些狀態可進入、由哪個 service 轉移、哪些不變式不可破壞、失敗後如何恢復」。頂級工程級 PRD / SRS 通常會把核心 domain state、happy path、failure path、retry path 與不可違反的 invariant 一起納入驗收。

本文對標 UML state machine、W3C SCXML、BPMN 與 ISO/IEC/IEEE 29148，但不宣稱 Emorapy 已具備完整 machine-readable statechart、BPMN 圖集或形式化驗證。

## 2. 外部基線參考

| 基線 | 採用原因 | Emorapy 採用方式 |
| --- | --- | --- |
| OMG UML 2.5.1 State Machines | 提供狀態、轉移、事件、guard、entry / exit action 的建模語義 | 用於要求狀態文檔說清 trigger、guard、side effect 與非法轉移 |
| W3C SCXML 1.0 | 提供通用 state-machine execution model | 用於校準「enum 不等於狀態機」；未有 machine-readable statechart 前只能標缺口 |

| ISO/IEC/IEEE 29148 | 要求需求可驗證、可追溯、可管理 | 用於要求每個狀態轉移能追到需求、接口、測試或恢復策略 |

## 4. Case 狀態機

| 狀態 | 進入條件 | 允許轉移 | 不得宣稱 |
| --- | --- | --- | --- |
| `draft` | formal remote / formal collaborative 尚未具備雙方陳述；session collaborative phase 1 | `draft -> submitted`，只允許 draft update / submit | 不代表 quick case 初始狀態；quick 建案直接 submitted |
| `submitted` | quick case 建立、formal case 雙方陳述完成、chat-to-case 建案、draft submit | `submitted -> in_progress` 由 judgment generation 開始；也可被 evidence upload 允許 | 不代表 judgment 已完成 |
| `in_progress` | `judgment.service.ts` 開始生成 judgment 時設置；允許崩潰恢復再次進入生成 | `in_progress -> completed` 或 `judgment_failed` | 不代表長期卡住可忽略；需 recovery / retry 口徑 |
| `judgment_failed` | judgment pipeline 失敗 | `judgment_failed -> in_progress` 經冷卻與重試；可由 chat retry 復用 case/link | 不代表永久失敗；不能重複建案替代重試 |
| `completed` | judgment 成功持久化，寫入 `completed_at` | 終態；後續走 repair / execution 而不是重寫 case | 不代表 repair journey 已完成 |
| `cancelled` | schema enum 存在；具體 runtime path 需按接口文檔核對 | 終態或人工治理，待補明確 transition | 不得假設所有流程都有 cancel API |

Case 不變式：

1. `mode=quick` 或 `mode=collaborative + session_id` 是 session-bound；`mode=remote` 或 `mode=collaborative + session_id=null` 是 user-bound formal case，判斷必須使用 `backend/src/utils/case-classifier.ts`。
2. `draft` 才允許 `updateCase()` 和 `submitCase()`；正式 remote 被告補陳述可由 `updateCase()` 自動轉 `submitted`。
3. evidence upload 只允許 `draft / submitted / in_progress`，以 `EVIDENCE_UPLOAD_ALLOWED_STATUSES` 為準。
4. judgment generation 允許 `submitted / judgment_failed / in_progress`，`judgment_failed` 有冷卻口徑；不得無限制重試。
5. chat-to-case 必須保留 `ChatToCaseLink`，不能只靠 `Case.mode` 判斷產品來源。

## 5. Chat 狀態機

| 狀態 | 進入條件 | 允許轉移 | 不得宣稱 |
| --- | --- | --- | --- |
| `solo_active` | room 建立、B 方離開或移除、pending invite 全部撤銷 | `solo_active -> invite_pending`、`solo_active -> judgment_requested` | 不代表一定只有一個 participant；AI mediator 也存在 |
| `invite_pending` | A 方建立 invite，pending invite 被寫入 | `invite_pending -> group_active`、`invite_pending -> solo_active`、`invite_pending -> judgment_requested` | 不代表 invite 永久有效；`expires_at` 需另判斷 |
| `group_active` | pending invite accepted 並建立/恢復 roleB participant | `group_active -> solo_active`、`group_active -> invite_pending`、`group_active -> judgment_requested` | 不代表 B 方訊息可無 consent 納入 judgment |
| `judgment_requested` | request judgment 通過 guard 並更新 room status | `judgment_requested -> judgment_completed` 或 `judgment_failed` | 不代表 case / judgment 已成功持久化 |
| `judgment_completed` | judgment 成功且 link 回寫 judgment_id | 可短時間 idempotent return；若有新訊息可再次 request 需重新判斷 | 不代表所有失敗路徑已清理 |
| `judgment_failed` | chat judgment 生成失敗 | 可復用最近 link / case 重試回 `judgment_requested` | 不得重複建案掩蓋失敗 |
| `archived` | enum 存在；具體 archive path 需按後續接口核對 | 終態 | 不得假設 archive 已形成完整產品流程 |

Chat 不變式：

1. active roleA 必須存在；發起 judgment 前檢查 roleA 正好一個、roleB 最多一個、AI mediator 最多一個。
2. 只有 roleA 可發 invite 與 request judgment。
3. judgment_requested / judgment_completed / judgment_failed / archived 不允許再發普通訊息；AI orchestrator 也不應對這些狀態繼續介入。
4. roleB 訊息納入 chat-to-judgment 前需要明示 consent；history visibility 會過濾可納入訊息。
5. `ChatInviteStatus.expired` 是 enum，但當前主要以 `expires_at` 做即時計算；若要持久化 expired 狀態，需新增 transition 與 cleanup / migration 口徑。
6. chat-to-judgment 在單人登入房建立 formal case / judgment 時，若 room owner 已有 pending / active normal pairing，必須復用該 live normal pairing；不得為同一 user 另建第二個 live normal pairing。此不變式與 `20260504182000_add_normal_pairing_uniqueness_trigger` 的 DB trigger 一致，也適用於 `P2002` race 後的重查復用。

## 6. Interview / AI Stream / Recovery 狀態不變式

| 狀態機 | 不變式 | 風險 |
| --- | --- | --- |
| InterviewSession | 只有 `in_progress` 可回覆 turn 或 end session；end session 根據 turns / content 進 `processing` 或 `completed`；pipeline 成功進 `completed`，失敗進 `processing_failed` | `abandoned` enum 與 cleanup path 需保持文檔同步 |
| AI Stream | `created / queued / started / streaming / completed / persisted / failed / cancelled` 必須與 event replay / snapshot / archive 一致 | 不得把 stream completed 等同 persisted |
| ProductStateRecoveryTask | `manual_review_required / in_review / resolved / dismissed` 有 timestamp side effect；Admin 可更新狀態 | 當前 service 允許任一 allowed status 更新，缺少嚴格 transition graph |
| Repair / Execution | 高風險 `safety_support / crisis_support` 不允許普通 repair / partner invite / co_active；execution status 不能替代 repair track status | 狀態分散在多個 service，後續需補完整 state diagram |

## 7. 新增或修改狀態的最低要求

| 要求 | 說明 |
| --- | --- |
| Enum source | schema enum、contract enum、前端 type 與 App type 需一致或明確標待承接 |
| Transition table | 列出 from / to / trigger / guard / side effect / retry / rollback |
| Ownership | 指明唯一轉移入口 service，不允許各 controller 手寫狀態分支 |
| Illegal transition | 指明應回哪個錯誤碼，例如 `CASE_NOT_EDITABLE`、`CONFLICT`、`PROCESSING_NOT_DONE` |
| Recovery | 指明卡住、失敗、重試、人工恢復或 cleanup 的規則 |
| Test evidence | 至少有 unit / integration / e2e / inspection / recovery task 證據之一 |
| Cross-platform impact | App 不得新增 App-only status；若 App 需要離線狀態，必須標為端側臨時狀態，不得污染 server contract |

## 9. 維護規則

1. 新增 schema enum 或 contract union 前，必須同步本文件或對應接口文檔。
2. 修改 Case / Chat / Interview / AI Stream / Repair / Recovery 狀態轉移時，必須同步 NFR、RTM、接口錯誤模型與測試。
3. 狀態變更不得只在 controller 層散落；應回到 service、classifier、policy 或 transition helper。
4. App 端只可新增端側 UI / offline / loading 狀態；涉及 server state 必須先進 shared contract / API / Parity。
5. 沒有測試或恢復策略的狀態不得標為已治理完成。
