## 1. 定位

本文定義跨 Web / App / Backend 的共用非功能需求。它不新增 runtime 行為；若現碼尚未覆蓋，對應狀態應記為待建立基線或待承接，不得把文檔需求寫成已完成能力。

## 4. 強制程度

| 強制程度 | 適用情況 | 例子 |
| --- | --- | --- |
| MUST | 若違反會造成安全、授權、資料歸屬、發布或跨端契約風險 | 高風險內容不得繞過 safety routing；App 不得建立 App-only 業務 DTO |
| SHOULD | 強建議，但允許在特定平台限制下有明確例外 | App 應優先承接 chat / notification / interview 的移動端價值 |
| MAY | 可選能力，不構成通用級產品承諾 | 某端可提供額外提示、快捷入口或本地提醒 |

MUST / SHOULD / MAY 的使用規則見 `00-跨端產品核心/05-工程級PRD對標與治理缺口台賬.md`。NFR 變更若使用 MUST，必須同步 RTM；若從 MUST 降級為 SHOULD，必須寫明平台限制或待裁決理由。

## 5. Web / App 差異要求

| 能力 | Web級要求 | App級要求 |
| --- | --- | --- |
| 身份恢復 | Browser route guard、local/session storage、auth 回跳需與 backend auth 一致 | SecureStore、cold start restore、Deep Link auth failure 需另行驗收 |
| 通知 | Web in-app notification 狀態需與 backend 同步 | Push token、登出清理、read sync、Deep Link landing 需回寫 Parity |
| 媒體 | Browser upload 不得繞過 evidence 授權 | ImagePicker / camera / upload adapter 需裁決資料歸屬與授權 |
| AI stream | Web 需處理 phase、snapshot、replay、timeout 和 fallback | App 需建立 native lifecycle 下的 stream / reconnect / background 策略 |
| 運維 | Admin Web 可治理 reports、costs、jobs、configs | 普通 App 不承接 Admin Web，只提供必要 telemetry |

## 7. 維護規則

1. NFR 若影響產品承諾，回寫 `00-跨端產品核心/`。
2. NFR 若影響 API、DB、shared enum 或 App adapter，回寫 `50-跨端Mapping與Parity/` 並建立待辦。
3. NFR 若進入 release gate 或測試 gate，回寫 `08-測試規範與驗收/04-需求驗證矩陣.md`。
4. 沒有量測來源的性能、可用性、成本或信任指標不得寫成已達標。
5. 安全標準映射只能提升審查嚴謹度；不得在未完成外部測試或審計時宣稱 ASVS / MASVS 合規等級。
