## 背景

後端已新增 migration：

- `backend/prisma/migrations/20260503235500_add_safety_assessment_models/migration.sql`

此 migration additive 新增：

- `SafetyRiskLevel`
- `SafetyAssessmentSubjectType`
- `SafetyAssessmentSource`
- `safety_assessments`
- `relationship_risk_states`
- `relationship_risk_states_one_active_scope_idx` partial unique index，確保每個 scope 同時最多一筆 active risk state

同時 `backend/src/utils/product-safety-policy.ts` 已新增 route / assertion snapshot helper，可把 `standard / safety_support / crisis_support` 以及合法 evidence/formal case assertion 映射為可直接寫入安全狀態表的欄位快照；`backend/src/services/safety-assessment.service.ts` 已提供單一讀寫入口，封裝 `SafetyAssessment` 建立、active `RelationshipRiskState` 刷新、active state 讀取與 route fallback snapshot。`ChatService.requestJudgment` 已接入 chat-to-judgment runtime：成功轉判決或 crisis 被攔截時，會 best-effort 寫入 room-level `SafetyAssessment`；高風險路由會刷新 active `RelationshipRiskState`，標準路由只記 audit，不覆蓋既有高風險 active state。`EvidenceController.uploadEvidence` 與 `CaseService.createCase` 也已把通過 gate 的 safety assertion best-effort 寫入 case/evidence scope `SafetyAssessment`。`repair-eligibility.service` 已新增 async resolver，`ReconciliationService` 與 `ExecutionService` 會優先讀 case scope active `RelationshipRiskState`，再 fallback 到 judgment route policy。`judgment-normalization.service` 已接管對外 judgment read path 的責任比例 visibility，優先讀 case scope active `RelationshipRiskState`，讀不到或失敗才 fallback 到 stored judgment route。

## Release 注意事項

- 已提交 migration 不代表 release DB 已套用。
- 發布 backend 前，必須確認 release DB migration 狀態。
- 若 release DB 尚未套用，寫入 `SafetyAssessment / RelationshipRiskState` 的 runtime 變更會被 chat-to-judgment best-effort 容錯記 warn，但不得把這視為已完成 release parity；發布前仍必須套用 migration。
- 此 migration 目前只新增表、enum 與 index，不改既有 API 行為；可先行部署 schema，再分批接入讀寫路徑。
