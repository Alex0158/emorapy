# 發起方（角色 A）偏誤風險分析

**日期**：2026-03-14  
**分析範圍**：Prompt 設計、責任比例算法、UI/API 語義、測試案例分布  
**結論**：存在多處可能導致偏向發起方（角色 A）的設計，需系統性檢視與緩解。

---

## 1. 核心映射關係

| 概念 | 對應 | 說明 |
|------|------|------|
| 發起方 | 角色 A | 快速體驗 / 協作流程中「先填寫」的一方 |
| 發起方 | plaintiff | 後端 API 與 DB 欄位 |
| 對方 | 角色 B | 後填寫或可留空的一方 |
| 對方 | defendant | 後端 API 與 DB 欄位 |

**結論**：發起方與「角色 A / plaintiff」為固定綁定，所有 prompt、算法、UI 皆以此為前提。若存在偏誤，會直接作用於發起方。

---

## 2. 偏誤風險點

### 2.1 敘事順序（首因效應）

**位置**：`ai.service.ts` 判決 prompt

```
1. **先對角色 A 說話**：
   用「角色 A」稱呼。描述他/她可能正在感受什麼...
2. **再對角色 B 說話**：
   同樣地理解角色 B...
```

**風險**：
- 角色 A 永遠先被處理、先被共情
- 讀者（含 AI 自身）易將 A 的敘事當成「主敘事」，B 當成「回應方」
- 首因效應可能讓 A 的框架主導後續責任歸屬的思考

**緩解建議**：
- 在 prompt 中明確加入：「先對誰說話不代表該方責任較輕或較重，僅為閱讀順序」
- 或隨機化順序（技術成本較高，且可能影響用戶體驗）

---

### 2.2 陳述呈現順序與篇幅

**位置**：所有 AI prompt（情感分析、案件類型、判決生成、責任評估）

```
角色 A 的描述：${plaintiffStatement}
角色 B 的描述：${defendantStatement || '（對方選擇暫時不發言）'}
```

**風險**：
- A 永遠在前，B 在後
- 快速體驗中，A 必填（≥30 字），B 可留空或僅 10 字
- B 留空時，AI 只能從 A 的敘述推測 B，易陷入 A 的詮釋框架
- 協作流程中，A 先寫、B 後寫，A 有更多準備時間

**緩解建議**：
- 在「B 未發言」情境的 prompt 中強化：「B 的沉默/未發言不代表同意 A 的敘述，分析時應保持對 A 敘事的主觀性警覺」
- 現有「歸因偏誤注意」已提及，可考慮擴充並置於更顯眼位置

---

### 2.3 歸因偏誤注意（已有，可加強）

**位置**：`ai.service.ts` 情感分析 prompt

```
歸因偏誤注意（特別是只有一方陳述時）：
- A 對 B 行為的描述是 A 的主觀詮釋，不是客觀事實。例如 A 說「他根本不在乎」，你應理解為「A 感覺不被在乎」，而不是「B 不在乎」。
- 在分析 B 的情感世界時，要跳出 A 的敘事框架，獨立推測 B 可能的感受——B 的沉默可能是「不堪負荷」而非 A 所詮釋的「冷漠」。
```

**現狀**：已有防護，方向正確。

**風險**：此段落在長 prompt 中可能被稀釋，且僅作用於情感分析，判決生成與責任評估 prompt 中未重複強調。

**緩解建議**：
- 在判決生成 prompt 的「自我檢查」段落中，加入：「若僅有一方陳述，是否過度採信該方視角？」
- 在 `assessResponsibilityRatio` 的 prompt 中，當 `defendantStatement` 為空時，加入額外提醒

### 2.4 Heuristic 算法的角色綁定

**位置**：`ai.service.ts` → `deriveRatioFromSignals`

```typescript
plaintiff += scorePattern(analysis.personA.communicationPattern);
plaintiff -= scorePattern(analysis.personB.communicationPattern);
plaintiff += statementPressure(plaintiffStatement);
plaintiff -= statementPressure(defendantStatement);
```

**邏輯**：`plaintiff` 越高 = 發起方（A）的「調整空間」越大。

**scorePattern 加權**：
- 批評、指責、翻舊帳、攻擊、蔑視、羞辱、控制、威脅 → +14
- 追逐、情緒化、逼問、高壓 → +6
- 迴避、沉默、退縮、敷衍、石牆 → +8（作用在 B 時會讓 plaintiff 減少）
- 承認、願意、修復、傾聽、同理 → -6

**statementPressure**：
- 總是、從不、每次都、你就是、你根本 → +6
- 我願意、我可以、我會改、我想理解 → -4

**風險**：
- 「追逐、情緒化、逼問」等常出現在發起方（因發起方往往是主動提出問題的人），會系統性提高 plaintiff 的調整空間
- 若發起方是「被動回應、冷淡」、對方是「追逐、逼問」，此 heuristic 可能反轉——但典型使用情境是「A 發起、A 追問、B 迴避」，故多數情況下會偏向 A 有較多調整空間
- 與「約會遲到」案預期一致（A 翻舊帳 → A 55–65%），但若測試案例多為「A 發起且 A 有溝通問題」，會強化此模式

**緩解建議**：
- 檢視測試案例分布：是否多數為「A 有較多調整空間」或「B 有較多調整空間」？若嚴重失衡，需加入反向案例
- 在 heuristic 註解中明確寫出「plaintiff = 發起方 = 角色 A」，避免後續維護者誤解

### 2.5 依附模板的 A/B 角色假設

**位置**：`judgment.service.ts` → `ATTACHMENT_PAIR_TEMPLATES`

```typescript
'anxious+avoidant': {
  prediction: 'A 的追求確認 + B 的需要空間 = 可能形成「追逐-撤退」循環：A 越追 → B 越退...',
  clinicalNotes: [
    '不要建議 A「不要想太多」「給對方空間」→ 會被感知為「你的感受不重要」',
    '不要建議 B「你應該多表達感受」→ 在情緒淹沒時強迫表達會適得其反',
    ...
  ],
},
```

**映射**：`labelA` = plaintiff 對應角色，`labelB` = defendant 對應角色。

**風險**：
- 模板假設「A = 追逐/焦慮型、B = 迴避型」是常見配對
- 若真實情境是「發起方（A）其實是迴避型、對方（B）是追逐型」，模板會錯配
- 例如：B 主動發起對話、追問，A 沉默——此時 A 仍是 plaintiff（因建案時 A 先填），模板卻可能給出「A 追求確認、B 需要空間」的預測，與事實相反

**緩解建議**：
- 依附配對應基於「雙方溝通模式」的實際分析結果，而非固定假設 A=追逐、B=迴避
- 若 insights 來自 profile，需確保 labelA/labelB 與 plaintiff/defendant 的對應在「誰先填」上有正確語義

### 2.6 原告/被告語義殘留

**位置**：`zh-TW.ts`、部分文檔

```
caseDetail.plaintiffStatement: "原告陳述"
caseDetail.defendantStatement: "被告陳述"
ratio.format: "原告 {plaintiff}% : 被告 {defendant}%"
```

**風險**：
- 「原告」語義帶有「提出控訴、主張權利」的框架
- 「被告」語義帶有「被指控、需回應」的框架
- 可能潛意識強化「發起方 = 有理方、對方 = 需辯護方」的認知

**現狀**：快速體驗與協作流程已改用「角色 A / 角色 B」，但案件詳情、比例顯示等處仍可能使用「原告/被告」。

**緩解建議**：
- 全面檢視 i18n，將對外展示的「原告/被告」改為「角色 A / 角色 B」或中性表述（如「一方 / 另一方」）
- 若需保留法律語境（如內部文檔），至少確保用戶可見處統一為「角色 A / B」

### 2.7 單方陳述的 clamp 邏輯

**位置**：`ai.service.ts` → `deriveRatioFromSignals`

```typescript
if (!defendantStatement || defendantStatement.trim().length === 0) {
  plaintiff = Math.max(45, Math.min(55, plaintiff));
}
```

**邏輯**：B 未發言時，將 plaintiff 限制在 45–55，避免過度偏斜。

**風險**：
- 此設計是合理的防護，避免「只聽一方就重判」
- 但 45–55 的區間本身是「略偏發起方」的（若真實情境是 B 責任極低，50:50 仍可能讓 A 覺得被冤枉）
- 反之，若真實情境是 A 責任極低（如 A 是受害者、B 是施暴者但未發言），45–55 會低估 B 的調整空間——但此情境應由 safety_support 處理，不應走一般比例邏輯

**結論**：單方陳述的 clamp 為合理設計，偏誤風險較低。

### 2.8 測試案例分布

**現有 5 案預期責任傾向**：

| 案例 | 預期 | 發起方（A）調整空間 |
|------|------|---------------------|
| 家務分工 | B 較多 (60–70%) | 30–40% |
| 約會遲到 | A 較多 (55–65%) | 55–65% |
| 金錢消費 | 均衡 (45–55%) | 45–55% |
| 婆媳關係 | B 較多 (60–70%) | 30–40% |
| 情感陪伴 | B 較多 (55–65%) | 35–45% |

**分析**：
- 5 案中僅 1 案（約會遲到）預期 A 較多調整空間
- 4 案為 B 較多或均衡
- 若 AI 或 heuristic 系統性偏向 A，會與多數案例的預期衝突，反而可能被測試捕捉到
- 但若測試僅有「約會遲到」一類「A 有問題」的案例，而缺少「A 明顯無辜、B 明顯有問題」的對照，則難以區分「模型正確」與「模型偏向 A 卻剛好符合約會遲到案」

**緩解建議**：
- 新增「角色反轉」測試案例：將現有某案（如家務分工）的 A/B 陳述對調，檢查責任比例是否合理反轉
- 若對調後比例未反轉，則存在角色順序偏誤

## 3. 總結：偏誤風險矩陣

| 風險點 | 嚴重度 | 決策 | 建議優先級 |
|--------|--------|------|------------|
| 敘事順序（先 A 後 B） | 中 | 需優化 | P0：前置梳理 + 順序中性原則 |
| 陳述篇幅與必填差異 | 中 | 合理（模擬自白模式） | — 不需優化 |
| Heuristic 角色綁定 | 中 | 合理，需監測 | — 以角色反轉測試驗證 |
| 歸因偏誤提醒 | 中 | 可加強 | P1：判決/責任評估 prompt 重複強調 |
| 依附模板 A/B 假設 | 低 | 待觀察 | P2 |
| 原告/被告語義殘留 | 低 | 可優化 | P2：統一對外文案為角色 A/B |
| 單方陳述 clamp | 低 | 已緩解 | — |

---

## 4. 建議行動（詳見敘事順序與Heuristic分析-20260314.md 第三節）

1. **P0**：判決 prompt 加入「前置梳理」（事件時序、因果鏈）+「順序中性原則」。
2. **P1**：歸因偏誤在判決、責任評估 prompt 中強化；新增角色反轉測試案例。
3. **P2**：原告/被告語義統一；依附模板與實際分析對應檢視。
4. **監測**：若可行，記錄 plaintiff 比例分布，長期偏離 50% 時觸發審計。
