本文件承接 `CJ` 的主站 admin 轉導入口、Admin Web、平台治理、運維觀測與後台配置能力。

本文只裁決 Admin Web 與平台治理。普通用戶 App 端不得承接 Admin Web 頁面、Admin token 或後台 RBAC；若治理能力需要對 App 產生可見影響，例如 notification deep link、telemetry、runtime gate、Push token 狀態或 App-visible health/status 提示，必須回查 [../20-App端/01-App導航與平台Adapter基線.md](../20-App端/01-App導航與平台Adapter基線.md) 與 [../50-跨端Mapping與Parity/01-App首輪能力與工程落點Mapping.md](../50-跨端Mapping與Parity/01-App首輪能力與工程落點Mapping.md)，並在未閉環時生成治理待辦。

## 1. 本子域回答什麼

本子域只回答以下問題：

1. Admin 端的身份建立、權限矩陣與頁面責任如何劃分。
2. 健康檢查、任務治理、配置治理、審計與報表屬於哪一組正式能力。
3. 什麼屬於後台治理，什麼不應混入前台產品正文。

本子域不回答：

1. quick、formal case、Repair Journey、interview、chat 的前台主流程正文。
2. 品牌、投放、對外敘事與營銷材料。
3. 共享 package 工程落點與 workspace 設計。
4. 普通用戶 App 端的 navigation、Deep Link、Push、SecureStore、upload 或 native lifecycle 實作。

## 2. 能力分組


| 分組 | 主要頁面 / 入口 | 核心責任 |
| --- | --- | --- |
| Admin 身份與 RBAC | `/admin/login`、`GET /admin/me` | admin token 建立、身份恢復、後台守衛 |
| 運維觀測 | `/admin/health`、`GET /admin/health/detailed`、`/health*`、`/metrics` | 健康檢查、就緒探針、Prometheus 指標 |
| 任務治理 | `/admin/ops/jobs`、`/admin/jobs`、`GET /admin/jobs*` | jobs 查詢、觸發、統計 |
| 配置治理 | `/admin/configs`、`GET /admin/runtime/interview`、`PUT /admin/configs` | runtime config、interview runtime config 讀寫 |
| 用戶與審計治理 | `/admin/users`、`/admin/audit-logs`、`/admin/admin-users*` | App user 狀態治理、審計查詢/導出、admin user CRUD |
| 報表治理 | `/admin/reports`、`GET /admin/reports/*` | overview、funnel、costs、AI stream sessions |
| 通知與產品狀態治理 | `GET/POST /admin/notifications*`、`GET/PATCH /admin/product-state/recovery-tasks*` | notification cancel/retry、產品狀態恢復任務查詢與狀態流轉 |
| 治理設定 | `/admin/settings`、`PUT /admin/alerts/rules`、`PUT /admin/feature-flags` | alert rules、feature flags、media provider config/test 與高權限設定 |


## 3. 入口與權限邊界

### 3.1 前台與 Admin 的邊界

主站 `frontend/` 內的 `admin/*` 只掛 `AdminRedirect`，並透過絕對 `VITE_ADMIN_LOGIN_URL` 把入口導向外部 Admin Web；真正的後台頁責由 `frontend-admin` 承接。

### 3.2 Admin 身份

後台身份與前台 JWT 分離：

1. 前台用戶登入不等於 Admin 登入。
2. `GET /admin/me` 是 Admin 身份恢復 API，不是頁面路由。
3. Admin token 不應用來承接前台產品流程。
4. 無 admin token 直連 `frontend-admin` 的 `/admin/*`（除登入頁）時，需回 `/admin/login`。

### 3.3 RBAC

後台權限按能力域分層，不以「是否是管理員」作粗糙判斷。當前正式能力至少包括：

1. `ops:read`
2. `ops:execute`
3. `config:read`
4. `config:write`
5. `users:read`
6. `users:write`
7. `reports:read`
8. `alerts:write`
9. `admin:all`

其中：

1. `/admin/audit-logs*` 需同時具備 `users:read` 與 `ops:read`。
2. `/admin/alerts/rules` 需同時具備 `alerts:write` 與 `ops:execute`。
3. `admin:all` 仍是最高權限覆蓋層，但不應拿來替代細分權限建模。

## 5. 固定規則

以下規則屬後台治理正文：

1. `/admin/login` 是獨立身份建立入口，不與前台 `/auth/*` 混用。
2. 主站的 `admin/*` 只負責外部轉導；真正的 Admin 正文頁只存在於 `frontend-admin`。
3. `health`、`ready`、`live`、`metrics` 屬運維探針，不等於前台功能頁。
4. jobs、configs、alerts、feature flags、reports、admin-users、admin notifications、product-state recovery tasks 都屬治理層能力，不應回寫到前台功能文檔作主流程描述。
5. `/admin/bootstrap`、`/admin/login`、`/admin/me` 這類接口不等於後台頁路由；頁面只看 `frontend-admin/src/router.tsx` 的真實註冊結果。
6. Admin 報表中的 AI stream 統計是治理能力，不直接替代前台對 AI stream 的產品語義。

## 6. 目前最重要的邊界

`03-管理端與平台治理/` 現階段同時承接 Admin 產品頁與平台運維治理，閱讀時要先分清兩個層面：

1. 如果你在改 `frontend-admin` 的實際頁面或操作流程，先看 Admin 身份、RBAC、對應 API。
2. 如果你在改健康檢查、報表、metrics、jobs、alerts，先把它視為平台治理而不是普通產品頁。

後續若這一層再變大，應優先在本目錄內分成「管理端產品域」與「平台治理域」，而不是把治理內容散回根層。
