## 1. 問題

Chat message 的 reply preview 目前用 `div role="button"` 加手動 `tabIndex` / `onKeyDown` 模擬按鈕。這種做法雖然可點，但語義上仍不是 native control，容易在 focus、disabled 狀態、測試與未來維護上出現偏差。

## 2. 證據

`frontend/src/pages/Chat/Room/components/ChatMessageItem.tsx`：

```tsx
<div className="chat-room-page__reply-preview" role="button" tabIndex={0} onClick={handleReplyPreviewClick} onKeyDown={handleReplyPreviewKeyDown}>
  <span className="text-xs text-muted-foreground">{t('chat.replyReference')}</span>
  <p className="chat-room-page__reply-preview-content text-xs">{replyTargetContent ?? t('chat.replyReferenceMissing')}</p>
</div>
```

同檔雖然已手寫 Enter / Space handler，但它仍不是 native `<button>`，也沒有獨立的 role/name 測試樣本來保證後續不退化。

## 3. 核心文件依據

1. `04-共用機制/07-可訪問性本地化與內容設計治理基線.md`：互動元件應優先採用語義化 HTML，複雜互動要有 role / name / keyboard pattern。
2. `08-測試規範與驗收/05-可訪問性本地化驗收基線.md`：Web 驗收必須覆蓋 semantic HTML、ARIA APG、keyboard / focus；不能只看 click 能不能觸發。
3. `07-待處理問題與治理/已處理/Web通知卡片互動語義待辦-2026-05-10.md` 已證明通知 open 行為不能靠 `div role=button` 長期承接，Chat 的 reply preview 需要同等語義基線。

## 4. 風險

1. screen reader 會把它當成自訂 widget，而不是標準按鈕。
2. keyboard 行為必須手工補 Enter / Space，未來容易漏掉焦點樣式、disabled、事件冒泡邊界。
3. 測試容易只驗證 onClick，不驗證 role / name / focus order。

## 5. 目標狀態

1. reply preview 改成 native `<button type="button">` 或語義更清楚的 explicit link / button。
2. open reply target 的可訪問名稱要明確，不能只靠上下文推斷。
3. 補 role/name / keyboard / focus 測試，驗證 Enter / Space 與 click 一致。
4. 不得把 nested interactive 再包進其他可點區域。
