## 1. 定位

本文定義通用級產品成功指標與健康信號。它不聲稱當前已具備全部埋點，也不憑空設定尚未量測的 KPI 數值。沒有穩定資料來源的項目一律標為「待建立基線」。

## 2. North Star

通用級 North Star 建議為：

| 指標 | 定義 | 當前狀態 |
| --- | --- | --- |
| 有效關係梳理閉環數 | 在安全 gate 允許的前提下，用戶完成一次可回訪、可理解、可進一步行動的梳理結果或修復節點 | 待建立基線 |

有效閉環不等於單純生成 AI 文本。至少應滿足：

1. 有合法身份或 session 歸屬。
2. 有明確 product flow 與 entry point。
3. 有可回訪結果或下一步行動。
4. 沒有繞過高風險安全分流。
5. 能在 API / 頁面 / 流程 / 證據中追溯。

## 5. 產品健康分類

| 類別 | 健康信號 | 不健康信號 |
| --- | --- | --- |
| 價值 | 用戶完成結果頁、接受下一步、回訪、升格或進入修復旅程 | 大量停在輸入頁、結果不可讀、無下一步 |
| 信任 | 安全分流有效、術語不過度承諾、失敗可解釋 | 危機內容被普通處理、用戶誤解為治療/法律服務 |
| 可靠性 | AI stream 可恢復、異步狀態可追蹤、version / health / metrics 可核驗 | 長時間 pending、不可重試、無 ledger 或無 evidence |
| 成本 | 高成本 AI 請求有 ledger、prompt version、product flow 與 pricing 來源 | 無歸因成本、失敗重試無上限 |
| 跨端 | Web / App / Backend / DB / shared 差異被記錄並可追溯 | App-only workaround、Web 已完成被誤判為 App 已驗收 |

## 5.1 App 版指標

| Metric ID | 主線 | 指標 | 說明 | 狀態 |

| CJ-MET-APP-001 | App Quick | App quick start -> result completion rate | App 未登入用戶從 quick 入口到結果頁完成比例 | 待建立基線 |
| CJ-MET-APP-002 | App Auth | App result -> auth claim success rate | App result 後登入/註冊並成功 claim session 的比例 | 待建立基線 |
| CJ-MET-APP-003 | App AI Stream | App stream reconnect / replay success rate | App 前台 stream 中斷後以 `after_seq`、snapshot 或 replay 恢復的比例 | 待建立基線 |
| CJ-MET-APP-004 | App Notification | Push / notification landing -> target action completion rate | Push 或 in-app notification 落地後完成目標 action 的比例 | 待建立基線 |

## 6. Web / App 指標邊界

| 層級 | 可記錄內容 | 不可替代內容 |
| --- | --- | --- |
| Web級 | Web route、頁面操作、Web/Admin smoke、Web 回歸證據 | App 原生 session restore、Push、Deep Link、native upload 驗收 |
| App級 | App screen、native adapter、Push / Deep Link / SecureStore / upload smoke | Web route guard、browser storage、Web 截圖 |
| Backend級 | API、DB、AI ledger、health、metrics、release gate | 用戶是否理解產品價值 |
| 產品級 | 漏斗、完成、回訪、信任安全、成本與留存 | 單個 endpoint 200 或單次截圖 |

## 7. 基線建立規則

1. 指標第一次接入時，先記為「待建立基線」，不得直接填寫主觀目標值。
2. 指標來源必須可追到 API、頁面、backend ledger、Admin report、metrics 或手動回歸證據。
3. App 指標只有在 App smoke / regression / CI / evidence 進入正式體系後，才能標為 App 已覆蓋。
4. 若指標影響發布判斷，必須在 `08-測試規範與驗收/04-需求驗證矩陣.md` 補對應驗收項。
5. 若指標來源只存在於 Admin report 或 Prometheus metrics，最多標為治理信號；要升為產品成功指標，還需能回到用戶流程或產品結果。
6. App 指標第一次進入 release 判斷前，必須回寫 `08-測試規範與驗收/04-需求驗證矩陣.md`，並能追到 App smoke、EAS build、simulator / device evidence 或 telemetry 來源。
