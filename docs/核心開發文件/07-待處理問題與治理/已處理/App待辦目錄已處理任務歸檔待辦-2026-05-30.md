## 2. 目標狀態

1. `待處理/` 只保留仍活躍的 App parity、外部 release sign-off 與已知風險入口。
2. 已完成 App 子任務移入 `已處理/`，並保持原始完成證據、邊界與驗證命令可追溯。
3. `00-活躍問題總覽.md` 明確列出本輪歸檔結果，同時修正 M6 active blocker 口徑：release DB parity 與 telemetry runtime 是已完成 pass evidence，不再是 current completion blocker。
4. `docs:check` 增加守衛：若 `待處理/` 內文件的 `**狀態**` 以 `已處理`、`已閉環` 或 `已完成` 開頭，直接失敗。
5. 逐文件代碼校驗總台賬中的路徑同步到 `已處理/`，避免 docs truth gate 與實際目錄分叉。

## 3. 業務場景

| 場景 | 預期行為 |
| --- | --- |
| Agent 選下一個 App 開發任務 | 只從真正活躍的 M6 外部 blockers 和 App parity 入口判斷，不被已完成子任務干擾 |
| 外部 release owner 看 `待處理/` | 看到的缺口仍是 EAS project id、Expo / Apple / ASC、EAS/TestFlight、真機、push provider delivery 與 native crash runtime |
| 工程 owner 查歷史 tooling 工作 | 到 `已處理/` 查看完整任務、結果與驗證命令 |
| 後續有人把已完成任務留在 `待處理/` | `npm run docs:check` 失敗，要求先歸檔或把狀態改回真實活躍狀態 |

## 4. 邊界與注意事項

1. 本任務不解除任何外部 release blocker，不填寫 secret，不產生 TestFlight / physical device / provider / native crash pass evidence。
2. 歸檔不是刪除；文件內容、驗證命令與完成結果必須保留。
3. 不把 `已知風險清單` 內的歷史已關閉風險誤判為已處理文件，守衛只讀文件頂部 `**狀態**` 欄。
4. 活躍入口仍保留 `App跨端Parity落地待辦` 與 `App外部ReleaseSignoff待辦`，因為它們承接尚未完成的 App strict completion。
