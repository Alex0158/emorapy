## 回饋決策規則

1. **Inline feedback**：登入、註冊、重設密碼等需要使用者修正／重試的阻塞錯誤，必須留在相關表單內，並提供 `role="alert"`／`aria-live`。
2. **Toast**：適合短暫成功確認、非阻塞警告及跨區域 action result；不得成為身份驗證等高風險失敗的唯一載體。同一 failure 只可由一層 emit：transport 已提示時，page catch 只更新 inline state。
3. **Page state**：初始 load failure、empty state、長時串流或離線狀態繼續使用既有 page-level state，不重複以全局 toast 取代。
4. **Truthful success**：clipboard、下載、分享、媒體等 browser API 必須等實際完成後才報成功；rejection 不得被當成 success。
5. **No blanket fallback**：storage、history mutation、optional telemetry 等不影響使用者主任務且已有明確 fallback 的 catch，不為了增加訊息數量而打擾使用者。

## 非本輪範圍

- App native 的 toast/banner contract；由 App parity 與 native accessibility 待辦獨立治理。
- 把所有 toast 機械轉成 modal 或 inline alert；這會製造訊息疲勞並破壞既有頁面狀態層級。
- optional browser capability、storage fallback、history mutation 或 telemetry failure 的全局報錯；只有影響使用者已發起任務時才升級為可見回饋。
