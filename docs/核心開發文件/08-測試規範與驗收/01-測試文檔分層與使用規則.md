`CJ` 的測試文檔固定分為四層：

1. `核心開發文件/08-測試規範與驗收/`
   - 長期有效的策略、分層、覆蓋與驗收口徑
2. `核心開發文件/測試/`
   - 與 SSOT 緊密配套的活躍案例、回歸包與執行入口
3. 根級 `歸檔/`
   - 歷史測試設計、失敗分析、缺陷報告與已退出活躍維護的專題稿
4. `核心開發文件/90-證據與盤點/`
   - 已核實的環境驗證、頁面快照與回歸證據

使用順序固定如下：

1. 先讀 `08-測試規範與驗收/`
2. 再讀 `核心開發文件/測試/`
3. 如需更細的歷史設計與缺陷分析，再讀根級 `歸檔/`
4. 如需已核實的證據鏈，再讀 `90-證據與盤點/`

現行自動化與門禁入口以 repo 內腳本與測試樹為準：

1. 前端頁面/狀態測試：`frontend/src/**/*.test.tsx`
2. 前端 chat 類 Playwright：`frontend/e2e/**/*.ts`
3. Admin 與跨站真服務 Playwright：`e2e/**/*.ts`
4. Web P0 true-service smoke：`npm run web:p0:true-service:smoke`（需 `RUN_WEB_P0_TRUE_SERVICE_SMOKE=true` 與 `WEB_P0_SMOKE_REPORT_PATH`，非本機 DB 需 `WEB_P0_SMOKE_ALLOW_REMOTE_DB=true`，artifact 需落在 `90-證據與盤點/環境與發版驗證/`）
5. 手動回歸門禁與匯總：`manual-regression:*`（發版前建議直接跑 `npm run manual-regression:gate`）
6. staging smoke / 關鍵 e2e 防回退：`npm run smoke:staging`、`npm run --workspace frontend test:e2e:critical-guard`
7. App route / feature / true-service / platform contract gates：`npm --prefix mobile run routes:check`、`features:check`、`true-service:check`、`platform:check`
8. App release / native readiness gates：`npm --prefix mobile run maestro:check`、`native:check`、`android:check`、`release:completion:audit`；完整 release 前由 `npm --prefix mobile run release:preflight` 串接

當前列表不代表 App release 已驗收。`mobile/` 已有 App route、feature coverage、true-service smoke harness contract、platform boundary、Maestro static、native readiness、Android readiness、release completion audit 與 release preflight 入口；這些入口用於證明 App 工程基線、contract wiring 與 release blocker 狀態，不等於 TestFlight / 真機 / provider delivery / production native crash runtime 已完成。新增 App 測試時，應先判斷其對應的是 App navigation、platform adapter、shared contracts / api-client、Backend API 還是 DB schema，並同步回寫 `20-App端/01-App導航與平台Adapter基線.md`、`50-跨端Mapping與Parity/01-App首輪能力與工程落點Mapping.md` 或對應待辦。

App 測試進入正式測試體系前，需先按 [03-App測試與證據接入基線.md](./03-App測試與證據接入基線.md) 判斷最小分層、證據落點與不得替代邊界。該文件裁決 local / dry-run / simulator / emulator / physical device / release target 的證明範圍，任何低層證據不得外推成高層 release sign-off。

若需求來自 `CJ-PRD-*` 或 `CJ-NFR-*`，需按 [04-需求驗證矩陣.md](./04-需求驗證矩陣.md) 回鏈到功能、流程、API、頁面、測試、證據或產品指標；Web 已覆蓋不得自動標記 App 已覆蓋。

Web P0 true-service smoke 是 mutating runner，只能在明確指定 backend / DB target 時執行。它可以使用 `AI_MOCK=true` 降低成本，但仍必須證明真 HTTP、auth/session、case list/detail、upload、chat request analysis、judgment detail 與 DB ownership sanity；若 target DB 不是本機 host，必須顯式設定 `WEB_P0_SMOKE_ALLOW_REMOTE_DB=true`。本機 artifact 只證明 local target；若 release gate 指定 non-local / release / production target，仍需另行產生該 target artifact。mock-backed Playwright、preflight blocker 或缺 artifact 的 raw output 不得替代這一層。

Admin credential-backed E2E 必須使用 `npm --prefix e2e run test -- --reporter=json`，避免 root Playwright runner 與 `e2e/` 內 `@playwright/test` 版本混用。正式 evidence 必須設定 `E2E_STRICT=true`、`E2E_BASE_URL`、`E2E_ADMIN_EMAIL`、`E2E_ADMIN_PASSWORD`，permission denied flow 還必須設定 `E2E_LIMITED_ADMIN_EMAIL`、`E2E_LIMITED_ADMIN_PASSWORD`；JSON artifact 需落在 `90-證據與盤點/環境與發版驗證/`，且 `stats.skipped=0`、`stats.unexpected=0`。本機 credential-backed artifact 不替代 non-local / release / production target artifact。

禁止反向使用：

1. 不要用根級 `歸檔/` 裡的歷史分析稿裁決當前產品真相
2. 不要用 `90-證據與盤點/` 的單次驗收結果替代正式測試規範
3. 不要把 dated 回歸執行記錄直接升格成正式規範
4. 不要用 Web/Admin 測試通過替代 App smoke / native adapter / Deep Link / Push / Upload 驗收
