本文件定義 `Emorapy` 只承認兩個正式版本形態：本機 Localhost 開發版，以及 Vercel/Railway 發布版。其他環境、一次性 smoke、CI test database、臨時 preview deployment 都只能作驗證輔助，不作產品版本口徑。

## 1. 版本形態

| 版本 | 用途 | 承載方式 | 是否面向正式使用 |
| --- | --- | --- | --- |
| 本機開發版 | 開發、調試、單元/集成測試、手動驗證 | Localhost；Supabase Dev DB；本機 Redis；不用 Docker 作日常開發前提 | 否 |
| 發布版 | 對外發布、正式驗收、回滾與運維 | Vercel 承載 Web；Railway 承載後端與 production runtime | 是 |

固定規則：

1. 本機開發版只要求在開發者機器可重現，不代表已發布。
2. 發布版只以平台實際部署狀態與 live endpoint 為準，不以本機 build 成功替代。
3. CI 是 gate，不是第三個版本；CI 通過只能說可發布，不能說已發布。
4. Preview、sandbox 或一次性遠端 smoke 只能作驗證輔助，不改變「本機版 / 發布版」的二分口徑，也不得保留為常設產品版本。

## 2. 本機開發版

本機開發版的固定特徵：

1. 後端以 `backend/` 下的 `npm run dev` 啟動。
2. 主站以前端 dev server 啟動，預設為 `localhost:5173`。
3. 管理端以前端 dev server 啟動，預設為 `localhost:5175`。
4. 日常開發不使用 Docker 作必要前提；Redis 固定優先使用本機 `redis-server`，目前標準為 `redis://127.0.0.1:6379` + `ALLOW_SIMPLE_LOCK=false`。
5. 本機 `.env` 只服務 local 開發，不應直接套用 production secrets。
6. 首選一鍵入口是 repo 根的 `./scripts/start-dev.sh`；它會確認/啟動本機 Redis，預設使用 `/tmp/emorapy-redis-dev` 作 Redis 資料目錄（可用 `EMORAPY_DEV_REDIS_DIR` 覆蓋），注入 `EMORAPY_COMMIT_SHA=$(git rev-parse HEAD)` 啟動 backend，並同時啟動主站與管理端。

本機版允許的差異：

1. 可以連 Supabase Dev DB / 本機 Redis。
2. 可以在明確知情時連 sandbox / 一次性遠端資源做聯調，但不得把它登記為第三個正式版本。
3. 未啟 Redis 時，部分 cache / lock / stream 能力可能降級，這不等同於發布版行為。
4. 可以使用 mock 或測試 API key，但測試結果不得聲稱代表 production。
5. Railway development Redis 不是本機開發版的必要條件，只能作可選 parity 增強。

本機版最小驗證：

1. 後端 `GET /health` 可用。
2. 主站與管理端能連到同一個本機 API base。
3. `GET /version`、主站 `/version.json`、管理端 `/version.json` 都應回報當前本機 `HEAD`；若 backend `commitSha=unknown`，代表啟動時未注入 `EMORAPY_COMMIT_SHA`，不得稱為本機版本對齊。
4. `/health` 在 development 可因 scheduled jobs disabled 顯示 `degraded`，但 database 與 lock checks 必須 healthy，且 lock 應顯示 Redis-backed runtime。
5. 快速體驗主鏈路至少能完成 session / case / judgment 的基本 smoke。
6. 涉及 schema 變更時，本機 Prisma migration 必須可跑通。

## 3. 發布版

發布版由兩類平台共同構成：

1. Vercel：主站 Web 與管理端 Web。
2. Railway：後端 API、jobs、health、metrics、production database runtime 連線。

發布版成立條件：

1. git commit 已推送到正式分支。
2. GitHub CI 對該 commit 通過。
3. Vercel 主站與管理端 production deployment 指向該 commit。
4. Railway 後端 production deployment 指向該 commit 或等價構建版本。
5. production database migration 已按發布計畫套用，且 migration state 可查。
6. live health / ready / version / 主鏈路 smoke 通過。

若其中任何一項缺失，只能說「部分已發布」或「代碼已推送」，不能說整個發布版已閉環。

## 4. 數據庫與 Migration 口徑

資料庫不是獨立第三個版本；它屬於發布版的一部分。

固定規則：

1. migration 檔案 commit 到 repo，只代表 migration 已進入代碼版本。
2. production DB 實際套用 migration，必須以 Railway / DB 連線下的 `prisma migrate deploy` 或等效平台紀錄確認。
3. production 不依賴應用啟動時自動 migration；正式 Railway service root 為 `/backend`，必須使用 `backend/railway.toml` 的 Dockerfile builder 與 `backend/Dockerfile`。`preDeployCommand` 在 service runtime environment 執行 `ops:production:predeploy`，失敗時不得啟動或切流至新 deployment；workflow 必須從 exact deployment manifest 重驗 builder / Dockerfile path。repo-root `railway.json` 只是非正式 fallback，不能作 Production 成功證據。`backend/src/config/database.ts` 的 `RUN_MIGRATIONS=true` 不作正式發布入口。
4. schema 變更發布前，應優先設計為前後兼容，讓 Web / API / DB 可以安全分步發布。
5. 不能用本機 database migration 成功，替代 production DB migration 成功。

發布時對資料庫的最低結論格式：

1. 無 schema 變更：明確記錄「本次無 migration」。
2. 有 schema 變更且已套用：記錄 migration 名稱、執行環境、執行時間與結果。
3. 有 schema 變更但未套用：不得標記發布版閉環。

## 5. 版本核驗方法

判斷本機版：

1. `git status -sb` 應清楚標示是否有未提交變更。
2. `git rev-parse HEAD` 是本機代碼版本。
3. 本機 API / Web endpoint 只證明 local runtime 可用。

判斷發布版：

1. Vercel 主站 `/version.json` 回報的 commit。
2. Vercel 管理端 `/version.json` 回報的 commit。
3. 後端 `/version` 或 `/api/v1/version` 回報的 commit。
4. Vercel inspect 顯示 production deployment `Ready`。
5. Railway deployment / logs / health 顯示 production 後端已更新。
6. DB migration state 與 repo migration 檔案一致。

標準回答口徑：

1. 「已推送」= commit 已到 `origin/main`。
2. 「前端已發布」= Vercel 主站 production 指向該 commit。
3. 「管理端已發布」= Vercel admin production 指向該 commit。
4. 「後端已發布」= Railway production 指向該 commit 且 health/version 正常。
5. 「資料庫已更新」= production migration state 已確認。
6. 「整體最新」= 以上全部一致。

## 6. 發布順序

標準順序：

1. 本機完成實作與必要測試。
2. commit 並推送正式分支。
3. 等 GitHub CI 通過。
4. 等 exact `main` SHA 的 GitHub CI push run 通過。
5. 觸發 GitHub Actions `Production Deploy and Verify`；流程在 Railway up、每次 Vercel promote 與最終 release gate 前都重驗 `origin/main == GITHUB_SHA`。Railway `preDeployCommand` 先在真正的 production service DB 執行 migration / release data gate，`/health/ready` 驗證 required migrations 後才切流，再部署 Vercel 主站 / 管理端。
6. 核對三個 version endpoint：主站、管理端、後端。
7. 跑 release gate、health / ready / smoke，並保存 workflow artifacts。
8. 記錄發布結論。

如前後端互相兼容，Vercel 與 Railway 的順序可按風險調整；但最終閉環必須同時確認 Web、API 與 DB。

### 6.1 部分發布失敗與自動恢復

`Production Deploy and Verify` 在任何 production deploy 前，必須先捕捉並保存以下 rollback baseline：

1. Railway 當前 production deployment id，以及 backend `/version` 的 `commitSha`；legacy endpoint 未提供 `deploymentId` 時允許為空，新版 endpoint 有值時必須與 Railway current deployment 一致。
2. 主站與 Admin 當前 Vercel production deployment id、deployment URL 與 `/version.json.commitSha`。
3. 平台 deployment identity 與公開 version endpoint identity 必須一致；baseline 不完整或不一致時，發布在切流前失敗。

Vercel 採 staged production deployment：先用 production build 與 `--skip-domain` 建立不接 production domain 的 deployment，驗證 exact `GITHUB_SHA` 後才 promote。Railway 已切流後，如 Vercel main / Admin promotion 或最終 release gate 任一失敗，failure-only rollback job 必須：

1. 對已改變的 Vercel surface 執行 Instant Rollback，恢復到 baseline deployment id。
2. 先把 Railway service 的 `EMORAPY_COMMIT_SHA` 恢復成 baseline SHA（`--skip-deploys`），再用 Railway Public GraphQL API `deploymentRollback(id)` 恢復 baseline backend deployment；account/workspace token 用 `Authorization: Bearer`，project token 用 `Project-Access-Token`。
3. Railway rollback 會以 baseline image / variables 建立新 deployment；流程必須讀取 mutation 回傳的新 rollback deployment id，輪詢其成為 `SUCCESS`，再核對三個 version endpoint 與 Vercel current deployment。backend endpoint 的 `deploymentId` 只在 baseline 版本已提供該欄位時強制比對，預期值為新 rollback deployment id；commit SHA 必須回到 baseline SHA。
4. 保存 `production-release-rollback-<run-id>` artifact；任何 surface 無法恢復時 rollback job 必須失敗，不得吞掉錯誤。

Production DB migration 固定採 expand-only 相容策略。自動恢復只回退 runtime 與 Web deployment，不執行 down migration；若新 schema 不是向後兼容，該變更不得進入此 production workflow。

## 7. 禁止混用

以下說法不再作正式判斷：

1. 「本機能跑，所以線上已更新。」
2. 「Vercel 已發布，所以後端也已發布。」
3. 「migration 檔案已 commit，所以 production DB 已更新。」
4. 「CI 通過，所以發布版已完成。」
5. 「preview 可用，所以 production 可用。」

任何版本結論都必須標明它指的是本機開發版，還是 Vercel/Railway 發布版。
