## 欄位說明

| 欄位          | 說明                                                            |
| ------------- | --------------------------------------------------------------- |
| PATH          | 請求路徑（與 METHOD 組合唯一識別）                              |
| METHOD        | HTTP 方法（GET/POST/PUT/DELETE 等）                             |
| API名字       | 中文語義名（對應後端 Controller 摘要或業務語義）                |
| 開發狀態      | 已使用 / 已確認廢棄 / 候選廢棄                                  |
| 對應功能/用途 | 詳細敘述（動作+對象+場景，可讀且具體）                          |
| 端別          | 學生端 / 僱主端 / 管理端 / 公共（可多選）；未知待確認（待釐清） |
| 級別          | JobPortal 級 / 平台級                                           |

## 狀態口徑說明

1. `已使用`：可在前端路由頁、子視圖或其直接依賴組件中追溯到實際調用。
2. `已確認廢棄`：代碼中有明確 `@deprecated` 或替代鏈路已落地且舊鏈路無調用。
3. `候選廢棄（前端未用）`：當前前端無調用證據，但後端仍提供能力；需業務確認後再下線。

## 維護規則

1. 新增/修改 API 時，先更新對應 `接口描述/`，再更新本清單。
2. 本清單以 `METHOD+PATH` 唯一，禁止重複條目。
3. 新增條目必須同時填寫 `PATH`、`METHOD`、`API名字`、`開發狀態`、`對應功能/用途`、`端別`、`級別`；並同步更新附錄 A、B。
4. 候選廢棄轉已確認廢棄，必須附替代證據（PR、註解、或代碼鏈路）。
5. 每次調整路由或頁內子分頁（如 `StudentProfile`、`ApplicationManagement`）需同步回寫附錄 A 關聯頁面索引。

## 一致性驗收結果

- 頁面清單主要 API 群組已覆蓋：認證註冊、職位、申請、企業、學生、簡歷、求職信、收藏、審核、統計、系統、通知、字典。
- `METHOD+PATH` 已按主表去重，候選廢棄不與已使用重複登錄。
- 已確認廢棄條目均具備代碼註解證據（`@deprecated`）。
- 已使用、已確認廢棄、候選廢棄三區均採用新欄位口徑（PATH | METHOD | API名字 | 開發狀態 | 對應功能/用途 | 端別 | 級別）。
- 候選廢棄區已表格化，與已使用/已確認廢棄同構。
- 附錄 A、B 僅覆蓋已使用與已確認廢棄共 128 條，不包含候選廢棄；驗收時不應以附錄暗示全量覆蓋。
