## 1. 路由架構總覽

前端採 `createBrowserRouter`，以三個布局分域：

- `AppLayout`：主站域（首頁、案件、判決、和好、執行、個人中心、訪談、聊天室、admin 轉導）
- `SimpleLayout`：快速體驗域（`/quick-experience/*`）
- `AuthLayout`：身份域（`/auth/*`）

全頁面均透過 `React.lazy + Suspense` 載入，fallback 為 `Loading`。

## 3. 路由守衛語義（實作細節）

## 3.1 `ProtectedRoute`

- 依 `authStore.isAuthenticated` 判斷是否允許進入。
- store hydration 未完成時顯示 loading（最多等待 5 秒）。
- hydration 逾時（5 秒）仍未完成時，一律視為未認證並導向 `/auth/login`，不渲染 children。
- 未登入時導向 `/auth/login`，並附帶 `state.from` 供登入後回跳。

## 3.2 `PublicRoute`

- 已登入用戶不應再停留在登入/註冊/忘記密碼頁。
- 預設導向目標：`/case/list`。
- 同樣具有 hydration 等待邏輯（最多 5 秒）。

## 3.3 重要邊界

- 前端沒有獨立 `ConsentGuard` 路由守衛；訪談 consent 由頁面流程觸發，後端 `requireConsent` 做最終保護。
- `chat/room*` 目前未包 `ProtectedRoute`，授權由頁內 API 調用與後端規則控制。
- `chat/room` 可作為弱入口，但接受 invite 已收斂為 `User only`；匿名 A 房主建立 invite 仍需 `room.session_id` 與 `canonical session_id` 匹配。
- chat 若進入 judgment ready，未登入用戶會先被 `ProtectedRoute` 送往 `/auth/login`，再回跳 `/judgment/:id`。

## 3.4 管理端（frontend-admin）登入守衛

- 所有 `/admin/*` 子路徑（除 `/admin/login`）經 `AdminSectionLayout` 渲染前，在 layout 內檢查 admin token（`useAdminToken` + `deriveAdminTokenStatus` 的 `tokenPresent` / `tokenReady`）。
- 無 token 或格式無效時：`<Navigate to="/admin/login" replace />`，不渲染側欄與子頁內容。
- `/admin/login` 為獨立 route，不經 `AdminSectionLayout`，故不檢查 token。
- 常用外部 admin 子頁包括：`/admin/health`、`/admin/jobs`、`/admin/ops/jobs`、`/admin/configs`、`/admin/users`、`/admin/audit-logs`、`/admin/settings`、`/admin/reports`。

## 4. `admin/*` 轉導規則

`AdminRedirect` 行為：

- 讀取 `VITE_ADMIN_LOGIN_URL`（必須為絕對 `http/https` URL）。
- 若有合法 URL：`window.location.assign(...)` 導向管理台。
- 若未配置或不合法：回到首頁 `/`。

這代表主前台不承載管理頁本體，只做入口轉發。

---

## 5. 參數命名映射規則

- 前端訪談頁參數是 `:sessionId`。
- 後端訪談 API 使用 `:id`。
- 映射在前端 API 層完成（例如 `getSession(sessionId) -> /interview/:id`）。

其他典型映射：
- 前端 `:planId` 對應 execution `plan_id` query/body
- 前端 `:judgmentId` 對應 reconciliation judgment 主鍵

---

## 6. 已知「不存在」路由（避免誤寫）

- 無獨立 `/500` 前端路由
- 無獨立 `/judgment/:id/result` 路由（由 `Judgment/Detail` 承載）
