# 報告 03：單元測試環境（Unit Test）

## 1. 環境定位

- 目標：快速驗證模組邏輯，不依賴完整基礎設施。
- 範圍：`backend test:unit` + `frontend test:run`（Vitest）。

## 2. 入口與執行方式

- 後端：`backend/package.json` -> `npm run test:unit`
- 前端：`frontend/package.json` -> `npm run test:run`（`vitest run`）
- CI 對應：`.github/workflows/ci.yml` 的 `backend` / `frontend` jobs

## 3. 後端關鍵變量與行為

- `backend/tests/setup.ts` 會固定 `NODE_ENV=test`。
- CI 常見注入：
  - `SKIP_DB_INIT=true`
  - `DATABASE_URL=file:./test.db`
  - `JWT_SECRET=test-*`
  - `OPENAI_API_KEY=sk-dev-test-placeholder`
- DB 初始化在 test 預設可被跳過（避免開連線與背景行為）。

## 4. 前端關鍵變量與行為

- `frontend/src/config/env.ts` 對 `VITE_API_BASE_URL` 有 localhost fallback。
- 單元測試以 jsdom 為主，不依賴真實後端服務。

## 5. 服務依賴（本環境）

- PostgreSQL：通常不強依賴（可 skip 或 mock）
- OpenAI：不打真實服務（placeholder/mocked path）
- Redis/SMTP/Slack：通常不需要

## 6. 驗證清單

- 後端：`npm run test:unit` 全綠
- 前端：`npm run test:run` 全綠
- 若要覆蓋率：後端 `npm run test:coverage`、前端 `npm run test:coverage`

## 7. 風險與改進

- backend `jest.config.js` 已有 coverage report，但沒有 `coverageThreshold`，目前不是硬 gate。
- frontend CI 使用 `test:run` 而非 `test:coverage`，覆蓋率門檻未在主 pipeline 強制。
- `frontend-admin` 尚無對應 unit test pipeline（僅 lint/build）。
