# CJ 維護規範（核心版）

**文檔版本**：v1.2  
**最後更新**：2026-03-05  
**目的**：確保核心文檔持續與代碼同步，避免「文檔看得到、代碼跑不到」。

---

## 1. SSOT 層級規則

- 一級（權威）：`docs/核心開發文件/*.md`。
- 二級（映射）：`docs/` 既有歷史文檔（如原 `功能特性清單.md`）。
- 發生衝突時以一級為準，二級文檔需回鏈並註記差異。

## 2. 變更觸發矩陣（最小同步集合）

| 代碼變更類型 | 必須同步的核心文檔 |
|---|---|
| 新增/刪除後端路由 | `全接口清單-主文檔.md`、`接口描述/*.md`、`接口-功能-頁面-Mapping.md` |
| 調整前端路由或守衛 | `頁面清單.md`、`接口-功能-頁面-Mapping.md` |
| 新增頁面業務流程 | `功能特性清單.md`、`業務流程整合.md`、`頁面清單.md` |
| 變更狀態機/分支邏輯 | `業務流程整合.md`、`術語表.md` |
| 新增候選或下線接口 | `全接口清單-主文檔.md`、`功能特性清單.md`、`接口描述/*.md` |
| 新增 admin 權限字串 | `術語表.md`、`接口描述/09-admin.md` |
| 新增或修改錯誤碼 | `全接口清單-主文檔.md`（全局錯誤碼字典）、對應 `接口描述/*.md` |
| 主流程步驟變更 | `業務流程整合.md`（回歸腳本編號區） |
| 高風險接口錯誤處置調整 | `全接口清單-主文檔.md`（反向追溯索引）、對應 `接口描述/*.md`（錯誤碼覆蓋矩陣） |

## 3. 編輯流程（強制）

1. 先改代碼，再改文檔，不允許先寫未落地能力為已實作。
2. 每條新增接口至少補齊：
   - 主註冊表一行；
   - 對應接口子文檔一行；
   - Mapping 一行；
   - 至少一個流程節點引用。
3. 每個新增頁面至少補齊：
   - 頁面清單主表一行；
   - 對應功能的小功能點一行；
   - Mapping 一行。
4. 候選廢棄轉已確認廢棄前，必須有替代方案與回歸證據。
5. 錯誤碼新增/改名時，必須同步更新：
   - 全局錯誤碼字典；
   - 相關模組「常見錯誤碼」欄位；
   - 前端錯誤顯示映射（如有）。
6. 任一 `P0` 流程若有 API、狀態機或權限調整，必須重跑對應 Script 並附結果。
7. 高風險 API 若調整錯誤處置，需同步更新「主文檔反向追溯索引」與「模組錯誤碼覆蓋矩陣」。

## 4. 狀態遷移規則（文檔層）

- `已使用 -> 候選廢棄`：前端主流程刪除調用，後端保留。
- `候選廢棄 -> 已確認廢棄`：後端路由移除或 feature flag 長期關閉且確認無回歸。
- `候選廢棄 -> 已使用`：前端重新接線並有頁面證據。

## 5. PR 自檢清單（提交前）

- [ ] `全接口清單-主文檔.md` 狀態統計數字已更新。
- [ ] `頁面清單.md` 的 GuardType、弱入口、候選保留一致。
- [ ] `功能特性清單.md` 新增功能包含「頁面 × API 小功能點」。
- [ ] `接口-功能-頁面-Mapping.md` 能從 API 追溯到頁面與流程節點。
- [ ] `接口-功能-頁面-Mapping.md` 的「前台完成度補丁表」與 `功能特性清單.md` 口徑一致。
- [ ] `業務流程整合.md` 的分支與失敗恢復已覆蓋。
- [ ] `術語表.md` 新名詞已定義。
- [ ] 若改動錯誤處理，`全局錯誤碼字典` 已同步。
- [ ] 若改動主流程，`回歸執行腳本` 的編號與步驟已同步。
- [ ] `高風險錯誤碼矩陣反向追溯` 的 API 清單與各模組矩陣一致。

## 6. 更新頻率建議

- 常規迭代：每次功能 PR 即時更新。
- 月度巡檢：核對候選廢棄清單是否需轉態。
- 發版前：做一次追溯閉環抽檢（至少抽 5 條高風險 API）。
