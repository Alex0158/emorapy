## 結論

- staging 已完成 Redis + metrics 收口，可用於回歸、手動驗證、公開狀態探活，以及 Redis-backed `AI Stream` / lock / metrics 驗證。
- staging 與 production 的當前主要差異已不在核心 runtime 形態，而在域名、流量與部署角色。
- staging 的 Redis 資源已清理收斂到單一有效實例，不再保留未接入 backend 的多餘 Redis 服務。
- `Staging Smoke Gate` 後續也應以 `STAGING_METRICS_TOKEN` 對 `/metrics` 做無 token / 有 token 兩種驗證，避免 staging 與 production 在機器抓取邊界上再次分叉。
- 倉庫現已新增 `Staging Deploy and Smoke` workflow，把 staging deploy 與 smoke gate 收成一次操作；`Staging Smoke Gate` 保留作為重跑驗證入口。
- staging smoke admin 已改用 validator-safe email（`example.com` 類型）並同步更新 GitHub secrets，避免 `.local` 類地址在 `/api/v1/admin/login` 的 Joi 驗證階段直接被拒。

## 建議後續

1. 後續每次 staging 變更 Redis / metrics / CORS 邊界時，先核 `/health`、`/api/v1/version`、`/metrics`（有無 token 兩種情況）、受保護 admin 接口，再更新本記錄。  
2. 若未來 staging 需要再次重建 Redis，務必先確認 backend 實際接入的是哪一顆 Redis，再做舊服務清理，避免產生孤兒資源。  
3. 若未來改用 `METRICS_ALLOWED_IPS` 取代 token，也需同步更新本記錄與 `docs/ENVIRONMENT.md` 的環境矩陣。
