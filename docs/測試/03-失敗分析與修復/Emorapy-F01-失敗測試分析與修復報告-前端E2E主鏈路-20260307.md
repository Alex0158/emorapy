# Emorapy F01 失敗測試分析與修復報告：前端 E2E 主鏈路

日期：2026-03-07  
對應功能：`F01 快速體驗、結果與升格`  
對應測試：`frontend/e2e/chat/quick-experience-flow.e2e.ts`

## 1. 失敗背景

本輪為 F01 建立了最小 Playwright 主鏈路，測試目標是驗證：

1. 訪客進入 `/quick-experience/create`
2. 填寫最小合法原告陳述
3. 進入第二步、第三步
4. 送出 quick case
5. 導航到 `/quick-experience/result/:id`

## 2. 執行方式

執行命令：

```bash
npx playwright test --config=e2e/playwright.config.ts e2e/chat/quick-experience-flow.e2e.ts
```

採用方式：

1. 沿用現有 chat E2E 的 `page.route()` 模式
2. 對 F01 相關 API 做 route stub
3. 不依賴真後端與真資料庫

## 3. 初次失敗現象

測試穩定失敗，且兩次重試結果一致。

已確認事實：

1. 第一階段輸入後，`Next` 已能進入後續步驟。
2. `POST /api/v1/cases/quick` 的 request 與 response 都已成功發生。
3. 但送出後頁面 URL 仍停留在 `/quick-experience/create`。
4. 預期的 `/quick-experience/result/case-e2e-1` 導航未發生。

## 4. 初次關鍵證據

Playwright 錯誤上下文顯示：

1. 第三步頁面仍停留在 create 頁，並可見 `Submit and Start Analysis` 按鈕。
2. `toHaveURL(/\\/quick-experience\\/result\\/case-e2e-1$/)` 在 8 秒內一直等不到跳轉。
3. 本次最終失敗已不是「按鈕不可點」問題，而是：
   - `cases/quick` 已被送出
   - 但成功後未跳轉

## 5. 根因定位

最終確認的真實根因是：

1. `frontend/src/hooks/useMountedRef.ts` 在 React 開發模式 `StrictMode` 下會被先 mount -> cleanup -> remount。
2. 原實作把 `ref` 初始值設為 `true`，但在 effect 中沒有重新設回 `true`。
3. 因此第一次 cleanup 後，`mountedRef.current` 會永久停在 `false`。
4. `QuickExperience/Create/index.tsx` 成功路徑中的：
   - `if (mountedRef.current) navigate(...)`
   
   在真瀏覽器開發模式下就不會成立，導致 `Create -> Result` 阻斷。

## 6. 修復內容

已完成修復：

1. `frontend/src/hooks/useMountedRef.ts`
   - 改為 `useRef(false)`
   - 在 `useEffect` 內明確設 `mountedRef.current = true`
   - cleanup 時再設回 `false`
2. 新增 `frontend/src/hooks/useMountedRef.test.tsx`
   - 驗證在 `StrictMode` 下重新掛載後仍保持 mounted 狀態正確
3. 重跑 F01 相關前端定向測試與 Playwright 主鏈路

## 7. 修復後驗證結果

已通過：

1. `vitest run src/hooks/useMountedRef.test.tsx src/store/authStore.test.ts src/pages/QuickExperience/Create/index.test.tsx src/pages/QuickExperience/Result/index.test.tsx`
2. `npx playwright test --config=e2e/playwright.config.ts e2e/chat/quick-experience-flow.e2e.ts`

注意：

1. Playwright 過程中另外修正了 E2E stub 的 API 回傳形狀，使其對齊真實 `getCase` / `getJudgmentByCaseId` 契約。
2. 這部分屬於測試腳本修正，不屬於業務缺陷。

## 8. 當前結論

本條初次失敗最終確認為**真實前端代碼缺陷**，且已修復；後續 E2E 剩餘異常屬於測試 stub 契約問題，已一併校正。

## 9. 與代碼的對照分析

理論上，`frontend/src/pages/QuickExperience/Create/index.tsx` 在 `handleSubmit()` 成功後應執行：

1. `localStore.remove('quick_case_draft')`
2. `navigate('/quick-experience/result/:id')`

但實際 E2E 表現說明：

1. 成功路徑中至少有一段邏輯沒有按預期完成
2. 或成功 response 雖已返回，但前端沒有進入導航分支

## 10. 當前處理結論

本條失敗已完成閉環：

1. `04-業務缺陷報告` 中的 `F01-BUG-001` 已更新為已修復
2. `05-修復方案與風險專題` 仍保留 session / 匿名升格相關風險，但已移除「Create -> Result 阻斷」作為未解決風險

## 11. 修復建議

建議下一步按以下順序處理：

1. 針對 `useMountedRef()` 的其他使用頁面做回歸：
   - `QuickExperience/Result`
   - `Interview/Chat`
   - `Interview/Result`
   - `Execution/CheckIn`
2. 繼續追 session refresh / `caseSessionMap` 的高風險鏈路。
