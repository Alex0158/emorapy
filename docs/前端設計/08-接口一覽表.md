#### 前端渲染防呆規則（Round15）

- 顯示成功率/失敗率前，必須先讀 `rateBase`，避免把 `total_runs` 與 `completed_runs` 混用。
- 若 `statsMeta.sampled=true`，圖表或表格需顯示「資料已採樣（僅最新 N 筆）」提示。
- 若 API 缺少 `rateBase`（舊版後端），前端回退按 `total_runs` 解讀。
- `dailyBuckets` 由後端保證補齊日期，前端直接按 `date` 渲染，不再二次補洞。
- 大視窗分析優先調整 `days`，不要無限制拉高 `maxRows`，避免前端繪製與記憶體壓力過大。
- 具體 TypeScript 接入模板見 `06-狀態管理與API設計.md` 的「管理員後台（運維）相關」章節。

## 🔄 接口依賴關係

### 核心流程依賴

**快速體驗流程**：
```
POST /sessions/quick (創建Session)
  ↓
POST /cases/quick (創建案件)
  ↓
GET /cases/:id/judgment (獲取判決)
  ↓
POST /judgments/generate/:id (判決重試，可選)
```

**完整模式流程**：
```
POST /auth/register (註冊)
  ↓
POST /auth/verify-email (驗證郵箱)
  ↓
POST /auth/login (登錄)
  ↓
POST /pairing/create (創建配對)
  ↓
POST /pairing/join (加入配對)
  ↓
POST /cases (創建案件)
  ↓
POST /cases/:id/submit (提交案件)
  ↓
GET /cases/:id/judgment (獲取判決)
  ↓
POST /judgments/generate/:id (判決重試，可選)
  ↓
POST /judgments/:id/reconciliation-plans (生成和好方案)
  ↓
POST /reconciliation-plans/:id/select (選擇方案)
  ↓
POST /execution/confirm (確認執行)
  ↓
POST /execution/checkin (執行打卡)
```

## 📋 接口文檔規範

### 請求格式

**認證頭**（需要認證的接口）：
```
Authorization: Bearer <token>
```

**Content-Type**：
```
Content-Type: application/json
```

### 響應格式

**成功響應**：
```json
{
  "success": true,
  "data": { ... },
  "message": "操作成功"
}
```

**錯誤響應**：

### 錯誤碼

- `400 Bad Request`：請求參數錯誤
- `401 Unauthorized`：未認證或Token過期
- `403 Forbidden`：無權限訪問
- `404 Not Found`：資源不存在
- `422 Unprocessable Entity`：業務邏輯錯誤
- `500 Internal Server Error`：服務器錯誤

**v2.0 訪談專用錯誤碼**（詳見 [03-API設計 § 訪談與心理畫像錯誤碼](../後端設計/03-API設計.md)）：

| 錯誤碼 | HTTP | 說明 |
|--------|------|------|
| `CONSENT_REQUIRED` | 403 | 用戶未同意知情同意 |
| `NOT_FOUND` | 404 | session 不存在或非本人 |
| `SESSION_COMPLETED` | 409 | session 已結束，不可繼續 |
| `CONCURRENT_REQUEST` | 409 | 同一 session 已有進行中的 respond（mutex） |
| `MAX_TURNS_REACHED` | 422 | 已達 25 輪硬上限 |
| `TURN_TOO_FAST` | 429 | respond 間隔不足 3 秒 |
| `RATE_LIMIT_EXCEEDED` | 429 | 雙層限流（中間件每小時 3 次 + 業務邏輯每天 5 個 substantive session） |
