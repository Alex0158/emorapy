## 1. 定位

本文建立 PRD 到工程驗收的追溯口徑。它不替代具體測試案例，也不聲稱 App 已驗收；它規定新增或修改需求時，必須能追到功能、流程、接口、頁面、測試或證據。

## 2. 矩陣欄位

| 欄位 | 說明 |
| --- | --- |
| PRD / NFR ID | `EMO-PRD-*` 或 `EMO-NFR-*` |
| 功能 / 流程 | 對應 `Fxx`、`Pxx` 或平台治理能力 |
| 平台層級 | 通用級、Web級、App級、共用機制級 |
| 工程落點 | API、頁面、DB、shared package、platform adapter 或運維腳本 |
| 驗證方式 | 單元、e2e、smoke、手動回歸、release gate、證據留存或待建立基線 |
| 指標 | 對應 `EMO-MET-*`，沒有穩定來源時標「待建立基線」 |
| 狀態 | 已覆蓋、部分覆蓋、待承接、待建立基線、待裁決 |

## 3. 驗證方式分類

| 驗證類型 | 定義 | 可用證據 |
| --- | --- | --- |
| Test | 由 automated unit / integration / e2e / smoke 驗證 | `backend/tests`、`frontend/src/**/*.test.tsx`、`frontend/e2e`、`e2e` |
| Analysis | 由代碼、schema、配置、日誌或指標分析驗證 | `backend/prisma/schema.prisma`、Admin reports、metrics、ledger |
| Inspection | 由文檔、設計、代碼審查或手動核查驗證 | PRD / NFR / Parity / API 文檔與 review 記錄 |
| Demonstration | 由手動回歸、release smoke 或可重放證據驗證 | `測試/`、`90-證據與盤點/`、release evidence |
| Baseline Pending | 有需求但尚未有穩定資料源 | `03-成功指標與產品健康.md` 中標為待建立基線 |

RTM 條目可以有多種驗證方式，但至少要有一種可追溯入口。App 條目在沒有 App screen / adapter / smoke / evidence 前，不得用 Web Test 替代。

## 6. 驗收狀態規則

| 狀態 | 定義 |
| --- | --- |
| 已覆蓋 | 需求已有可追溯代碼、測試或證據，且 docs 守衛可支撐 |
| 部分覆蓋 | 核心行為有實作，但指標、平台投影或證據仍不完整 |
| 待承接 | 通用需求成立，但某平台尚未落地 |
| 待建立基線 | 指標或驗收資料來源尚未穩定 |
| 待裁決 | 需要產品或工程決策，不能由測試文件自行判定 |

## 7. 變更規則

1. 新增 `EMO-PRD-*` 或 `EMO-NFR-*` 時，必須新增或更新至少一條 `EMO-RTM-*`。
2. 如果驗收只覆蓋 Web，不得標記 App 已覆蓋。
3. 如果只有單次手動證據，狀態最多為部分覆蓋；穩定真相需回寫正式規格。
4. 如果需求影響 API、DB、shared contract 或 App adapter，必須同步更新 Parity 文件或待辦。
5. 每次修改本文後需執行 `npm run docs:check`。
6. 使用外部標準作驗證依據時，必須標註「參考對照」或具體版本；未經審計不得宣稱合規等級。
7. AI / LLM 相關驗證必須同時區分 prompt injection 防護、output downstream gate、ledger 可追溯、prompt version 與安全路由；不得用單一單測覆蓋全部風險。
8. API contract 驗證必須區分 route truth、人工接口詳規與 OpenAPI/schema contract；不得用 `docs:check:truth` 通過替代 OpenAPI 完成。
9. 資料治理驗證必須區分資料分類、consent/delete、retention、archive、log/evidence 最小化與 App storage / telemetry；不得用單一 delete endpoint 代表完整隱私合規。
10. SLO / incident 驗證必須區分 health endpoint、metrics source、alert threshold、request id / logs、release gate、incident drill、postmortem 與長期監控；不得用 `/health` 200、`/metrics` 存在、request id 存在或單次 gate 通過代表長期 SLA / tracing / error budget。
11. 威脅建模驗證必須區分 trust boundary、threat category、安全需求、現有控制與缺口；不得用單一中間件或配置代表完整安全。
12. 架構決策驗證必須追到 ADR、影響面、替代方案、驗證方式與重審條件；不得用 repo 目錄說明替代架構取捨。
13. 狀態機驗證必須區分 enum source、transition guard、side effect、illegal transition、retry / recovery 與 App server-state 承接；不得用 enum 存在代表完整狀態機。
14. 錯誤契約驗證必須區分 HTTP status、`error.code`、request id、safe details、retry policy、Problem Details 與 OpenAPI error schema；不得用單一錯誤處理中間件代表完整 API failure contract。
15. 可訪問性與本地化驗證必須區分 role/focus 單測、keyboard-only flow、WCAG / axe 掃描、screen reader、i18n completeness、內容設計審查與 App native smoke；不得用 Web 局部測試代表完整可訪問性或 App 已覆蓋。
16. Schema migration 驗證必須區分 schema source、migration history、release DB parity、backfill / dual-read、contract compatibility、operational risk 與 App parity；不得用 migration commit、Dev DB 成功或 `db push` 代表 production-safe。
