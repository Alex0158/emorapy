## 1. 定位

CJ 的 Web 版與 App 版不是兩套產品，而是同一套產品核心在不同平台上的投影。

跨端產品核心固定回答四件事：

1. 用戶能做什麼。
2. 案件、聊天室、心理訪談、修復旅程之間如何流轉。
3. 角色、權限、狀態與資料歸屬如何判定。
4. Web / App / Backend 必須保持一致的契約是什麼。

平台文件只回答「這一端如何承載」。例如 Web 可以使用瀏覽器 route 與 local storage，App 可以使用 Expo Router、SecureStore、Push、Deep Link，但兩端不得分裂產品語義。App 進入實作時，平台承載規則必須回到 [../20-App端/01-App導航與平台Adapter基線.md](../20-App端/01-App導航與平台Adapter基線.md)，能力到 Backend / API / DB / shared package 的落點必須回到 [../50-跨端Mapping與Parity/01-App首輪能力與工程落點Mapping.md](../50-跨端Mapping與Parity/01-App首輪能力與工程落點Mapping.md)。

## 2. 主產品能力

| 能力組 | 跨端定義 | Web 現況 | App 原則 |
| --- | --- | --- | --- |
| 快速判斷 | 低門檻建立衝突描述，支持單人快速體驗與雙人協作填寫，產生可判斷材料 | 已由快速體驗與 session-bound quick case 承接 | 首版應復用同一 quick/session/case contract，不新建 App-only 案件語義 |
| 正式處理 | 登入用戶進入配對、正式案件提交、判決、修復旅程 | 已由 case / judgment / reconciliation / execution 主鏈路承接 | 首版應保留相同正式 case mode、pairing、judgment visibility 與 repair eligibility 規則 |
| 先聊再判 | 聊天室先沉澱雙方材料，再一鍵轉判決或案件材料 | 已由 chat room、message、judgment handoff 相關接口承接 | App 可優先成為高頻聊天入口，但不得改變 chat -> case/judgment 的資料歸屬規則 |
| 讓系統更懂你 | 心理訪談、個人/關係檔案、後續個性化判斷 | 已由 interview / psych profile / profile 能力承接 | App 可承接更適合移動端的訪談與通知節奏，但 profile schema 必須與後端一致 |
| 平台治理 | Admin、健康檢查、AI 成本、通知、恢復任務、發布治理 | Admin Web 與 backend 已承接 | App 不直接承接治理台，但其事件與錯誤必須能被後端觀測與追溯 |

## 3. 核心角色與資料歸屬

| 角色 / 主體 | 跨端規則 |
| --- | --- |
| 匿名 session 用戶 | 只能操作 session-bound quick flow；登入或 claim 後才可升格為 user-bound 資料 |
| 登入用戶 | 可建立正式案件、配對、讀取自己有權訪問的 judgment / reconciliation / profile |
| 雙方當事人 | 正式案件與雙人協作都必須保留雙方視角，不因平台不同改變授權判定 |
| Admin | 只能通過 Admin Web / backend admin API 管理平台，不作為 App 普通用戶能力 |
| Backend | 是身份、授權、狀態機、DB schema、AI ledger、通知與發布治理的裁決層 |

資料歸屬不得由前端路由或 App 本地狀態單獨裁決。前端和 App 只能提交上下文，最終以 backend auth/session/case/pairing/judgment 規則為準。

## 4. 跨端狀態機邊界

以下語義必須保持跨端一致：

1. quick case 與 formal case 的 mode / source / session 歸屬。
2. normal pairing 與 temp pairing 的授權邊界。
3. case submit gate、judgment generate gate、judgment visibility gate。
4. reconciliation / execution / repair journey 的 eligibility 與狀態轉移。
5. interview session、psych profile、chat room、notification 的狀態值與錯誤碼。
6. AI request ledger 的 product flow、source、prompt version、cost attribution。

若 App 因離線、推播、Deep Link、原生權限產生額外狀態，只能作平台適配狀態；不得把它寫成新的業務狀態機。

## 5. 一致性鐵律

1. API contract、DB schema、共享 enum、錯誤碼、授權 gate、狀態機只允許有一個核心定義。
2. 任何 Web / App 差異必須落到 [../50-跨端Mapping與Parity/00-跨端Parity總覽.md](../50-跨端Mapping與Parity/00-跨端Parity總覽.md)；若差異涉及 App 首輪 screen、API、DB、shared package、Deep Link、Push、SecureStore、upload 或 telemetry，還必須同步落到 [../50-跨端Mapping與Parity/01-App首輪能力與工程落點Mapping.md](../50-跨端Mapping與Parity/01-App首輪能力與工程落點Mapping.md)，不能散落在單端文件中。
3. 若某端暫未實現核心能力，記為 Parity 缺口；不得在文件中假裝已實現。
4. 若產品設計要求兩端最終一致，但現碼尚未閉環，必須新增待處理任務並掛回能力矩陣。
5. App 版本啟動後，Web 凍結基線仍是對照基準；但產品核心的最高裁決權在本目錄，而不是 Web 的 UI 現狀。
