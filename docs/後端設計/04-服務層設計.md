## 📋 服務層概述

### 設計原則

1. **單一職責**：每個服務只負責一個業務領域
2. **依賴注入**：通過構造函數注入依賴
3. **事務管理**：在Service層管理數據庫事務
4. **錯誤處理**：統一拋出自定義錯誤
5. **日誌記錄**：記錄關鍵業務操作

### 服務劃分

```
services/
├── auth.service.ts            # 認證服務
├── session.service.ts         # Session 服務（快速體驗）
├── pairing.service.ts         # 配對服務
├── case.service.ts            # 案件服務
├── judgment.service.ts        # 判決服務
├── reconciliation.service.ts  # 和好方案服務
├── execution.service.ts       # 執行服務
├── profile.service.ts         # 個人/關係檔案服務
├── ai.service.ts              # AI 服務
├── email.service.ts           # 郵件服務
├── file.service.ts            # 文件服務
├── content.service.ts         # 內容推薦服務
└── notification.service.ts    # 通知服務
```

## 🔐 認證服務 (AuthService)

### 職責

### 接口定義

```typescript
interface AuthService {

  // 發送驗證碼

  // 驗證郵件

  // 確認重置密碼
  confirmResetPassword(email: string, code: string, newPassword: string): Promise<void>;
  
  // 生成Token
  generateToken(user: User): string;
  
  // 驗證Token
  verifyToken(token: string): Promise<UserPayload>;

```

### 實現要點

**1. 註冊流程**：
```typescript
async register(data: RegisterDto): Promise<{ user: User; token: string }> {
  // 1. 驗證郵箱格式
  if (!this.isValidEmail(data.email)) {
    throw new AppError(400, 'INVALID_EMAIL', '郵箱格式錯誤');
  }
  
  // 2. 檢查郵箱是否已存在
  const existingUser = await this.userRepository.findByEmail(data.email);
  if (existingUser) {
    throw new AppError(409, 'EMAIL_EXISTS', '郵箱已存在');
  }
  
  // 3. 驗證密碼強度
  if (!this.isStrongPassword(data.password)) {
    throw new AppError(400, 'WEAK_PASSWORD', '密碼強度不足');
  }
  
  // 4. 加密密碼
  const passwordHash = await bcrypt.hash(data.password, 10);
  
  // 5. 創建用戶
  const user = await this.userRepository.create({
    email: data.email,
    password_hash: passwordHash,
    nickname: data.nickname,
    email_verified: false
  });
  
  // 6. 發送驗證郵件
  await this.sendVerificationCode(data.email, 'register');
  
  // 7. 生成Token
  const token = this.generateToken(user);
  
  return { user, token };
}
```

**2. 登錄流程**：
```typescript
async login(email: string, password: string): Promise<{ user: User; token: string }> {
  // 1. 查找用戶
  const user = await this.userRepository.findByEmail(email);
  if (!user) {
    throw new AppError(401, 'INVALID_CREDENTIALS', '郵箱或密碼錯誤');
  }
  
  // 2. 驗證密碼
  const isValid = await bcrypt.compare(password, user.password_hash);
  if (!isValid) {
    throw new AppError(401, 'INVALID_CREDENTIALS', '郵箱或密碼錯誤');
  }
  
  // 3. 檢查帳號狀態
  if (!user.is_active) {
    throw new AppError(403, 'ACCOUNT_INACTIVE', '帳號未激活');
  }
  
  // 4. 更新最後登錄時間
  await this.userRepository.updateLastLogin(user.id);
  
  // 5. 生成Token
  const token = this.generateToken(user);
  
  return { user, token };
}
```

**3. 驗證碼管理**：
```typescript
async sendVerificationCode(email: string, type: VerificationType): Promise<void> {
  // 1. 檢查發送頻率（每5分鐘一次）

  }
  
  // 2. 生成6位驗證碼

    code,

  });
  
  // 4. 發送郵件
  await this.emailService.sendVerificationCode(email, code, type);
}
```

```typescript
async createPairing(userId: string): Promise<Pairing> {
  // 1. 檢查是否已有配對
  const existingPairing = await this.pairingRepository.findActiveByUserId(userId);
  if (existingPairing) {
    throw new AppError(409, 'ALREADY_PAIRED', '已經有配對關係');
  }
  
  // 2. 生成唯一邀請碼（6位字母數字）

  });
  
  return pairing;
}
```

**2. 加入配對**：
```typescript
async joinPairing(userId: string, inviteCode: string): Promise<Pairing> {
  // 1. 查找配對記錄
  const pairing = await this.pairingRepository.findByInviteCode(inviteCode);
  if (!pairing) {
    throw new AppError(400, 'INVALID_CODE', '邀請碼無效');
  }
  
  // 2. 檢查是否過期
  if (pairing.expires_at < new Date()) {
    throw new AppError(400, 'CODE_EXPIRED', '邀請碼已過期');
  }
  
  // 3. 檢查是否已使用
  if (pairing.status !== 'pending') {
    throw new AppError(400, 'CODE_USED', '邀請碼已使用');
  }
  
  // 4. 檢查是否自己配對
  if (pairing.user1_id === userId) {
    throw new AppError(400, 'SELF_PAIRING', '不能與自己配對');
  }
  
  // 5. 更新配對記錄
  const updatedPairing = await this.pairingRepository.update(pairing.id, {
    user2_id: userId,
    status: 'active',
    confirmed_at: new Date()
  });
  
  // 6. 發送通知郵件
  await this.emailService.sendPairingNotification(pairing.user1_id, userId);
  
  return updatedPairing;
}
```

### 實現要點

**1. 創建快速體驗案件**：
```typescript
async createQuickCase(data: QuickCaseDto, sessionId: string): Promise<Case> {
  // 0. 驗證Session ID格式
  if (!sessionId || !sessionId.startsWith('guest_')) {
    throw new AppError(400, 'INVALID_SESSION_ID', '無效的Session ID格式');
  }
  
  // 1. 驗證Session是否存在且未過期
  const session = await this.sessionService.getSession(sessionId);
  if (!session) {
    throw new AppError(401, 'SESSION_EXPIRED', 'Session已過期或不存在');
  }
  
  // 2. 驗證陳述長度
  if (!data.plaintiff_statement || data.plaintiff_statement.trim().length < 50) {
    throw new AppError(400, 'INVALID_STATEMENT', '原告陳述長度必須至少50字');
  }
  
  if (data.plaintiff_statement.length > 2000) {
    throw new AppError(400, 'INVALID_STATEMENT', '原告陳述長度不能超過2000字');
  }
  
  if (!data.defendant_statement || data.defendant_statement.trim().length < 50) {
    throw new AppError(400, 'INVALID_STATEMENT', '被告陳述長度必須至少50字');
  }
  
  if (data.defendant_statement.length > 2000) {
    throw new AppError(400, 'INVALID_STATEMENT', '被告陳述長度不能超過2000字');
  }
  
  // 3. 驗證證據數量
  if (data.evidence_urls && data.evidence_urls.length > 3) {
    throw new AppError(400, 'TOO_MANY_EVIDENCES', '最多只能上傳3張圖片');
  }
  
  // 4. 驗證證據URL格式
  if (data.evidence_urls) {
    for (const url of data.evidence_urls) {
      try {
        new URL(url); // 驗證URL格式
      } catch {
        throw new AppError(400, 'INVALID_EVIDENCE_URL', '無效的證據URL格式');
      }
    }
  }
  
  // 5. AI自動判斷案件類型（帶錯誤處理）
  let caseType: string;
  try {
    caseType = await this.detectCaseType(
      data.plaintiff_statement,
      data.defendant_statement
    );
  } catch (error) {
    this.logger.error('Failed to detect case type', { error, sessionId });
    caseType = '其他衝突'; // 默認類型
  }
  
  // 6. 創建臨時配對（用於快速體驗）
  const tempPairing = await this.createTempPairing(sessionId);
  
  // 7. 生成案件標題（帶錯誤處理）
  let title: string;
  try {
    title = this.generateTitle(data.plaintiff_statement);
  } catch (error) {
    this.logger.warn('Failed to generate title', { error });
    title = '案件-' + new Date().toLocaleDateString(); // 默認標題
  }
  
  // 8. 創建案件記錄（使用事務確保數據一致性）
  let case_: Case;
  try {
    case_ = await this.caseRepository.create({
      pairing_id: tempPairing.id,
      title,
      type: caseType,
      plaintiff_id: null, // 快速體驗模式，無真實用戶
      defendant_id: null, // 快速體驗模式，無真實用戶
      plaintiff_statement: data.plaintiff_statement.trim(),
      defendant_statement: data.defendant_statement.trim(),
      status: 'submitted',
      mode: 'quick', // 標記為快速體驗模式
      session_id: sessionId, // 關聯Session ID
      submitted_at: new Date()
    });
  } catch (error) {
    this.logger.error('Failed to create case', { error, sessionId });
    throw new AppError(500, 'CASE_CREATION_FAILED', '案件創建失敗，請稍後再試');
  }
  
  // 9. 保存證據（如有，帶錯誤處理）
  if (data.evidence_urls && data.evidence_urls.length > 0) {
    try {
      for (const url of data.evidence_urls) {
        await this.evidenceRepository.create({
          case_id: case_.id,
          file_url: url,
          file_type: 'image',
          user_id: null // 快速體驗模式，無真實用戶
        });
      }
    } catch (error) {
      this.logger.error('Failed to save evidences', { error, caseId: case_.id });
      // 證據保存失敗不影響案件創建，記錄錯誤即可
    }
  }
  
  // 10. 更新Session，關聯案件ID
  try {
    await this.sessionService.addCaseToSession(sessionId, case_.id);
  } catch (error) {
    this.logger.warn('Failed to update session', { error, sessionId, caseId: case_.id });
    // Session更新失敗不影響案件創建
  }
  
  // 11. 異步觸發AI判決生成（不阻塞響應）
  this.judgmentService.generateJudgment(case_.id).catch(err => {
    this.logger.error('Failed to generate judgment', { caseId: case_.id, error: err });
    // 判決生成失敗不影響案件創建，用戶可以稍後重試
  });
  
  return case_;
}
```

**1. 生成判決**：
```typescript
async generateJudgment(caseId: string): Promise<Judgment> {
  // 1. 檢查是否已有判決
  const existing = await this.judgmentRepository.findByCaseId(caseId);
  if (existing) {
    return existing;
  }
  
  // 2. 檢查緩存
  const cached = await this.checkCache(caseId);
  if (cached) {
    return cached;
  }
  
  // 3. 獲取案件信息
  const case_ = await this.caseRepository.findById(caseId);
  if (!case_) {
    throw new AppError(404, 'CASE_NOT_FOUND', '案件不存在');
  }
  
  if (case_.status !== 'submitted') {
    throw new AppError(422, 'CASE_NOT_READY', '案件尚未準備好');
  }
  
  // 4. 調用AI服務生成判決
  let judgmentContent: string;
  let responsibilityRatio: { plaintiff: number; defendant: number };
  let summary: string;
  
  try {
    const response = await this.aiService.generateJudgment(
      case_.type,
      case_.plaintiff_statement,
      case_.defendant_statement || ''
    );
    judgmentContent = response.content;
    responsibilityRatio = response.responsibilityRatio;
    summary = response.summary;
  } catch (error) {
    this.logger.error('AI service error', { caseId, error });
    throw new AppError(503, 'AI_SERVICE_ERROR', 'AI服務暫時不可用');
  }
  
  // 6. 生成摘要（由 AIService 統一生成）
  // summary 已由 AIService 返回
  
  // 7. 保存判決
  const judgment = await this.judgmentRepository.create({
    case_id: caseId,
    judgment_content: judgmentContent,
    summary,
    responsibility_ratio: responsibilityRatio,
    ai_model: AI_CONFIG.model,  // 由環境變量驅動，默認 gpt-3.5-turbo（判決使用 AI_CONFIG，非 ANALYSIS_AI_CONFIG）
    prompt_version: 'v3.10'
  });
  
  // 8. 更新案件狀態
  await this.caseRepository.update(caseId, {
    status: 'completed',
    completed_at: new Date()
  });
  
  // 9. 緩存判決（24小時）
  await this.cacheJudgment(caseId, judgment);
  
  // 10. 發送通知郵件（僅完整模式）
  if (case_.mode !== 'quick' && case_.plaintiff_id) {
    await this.emailService.sendJudgmentNotification(case_.plaintiff_id, caseId);
    if (case_.defendant_id) {
      await this.emailService.sendJudgmentNotification(case_.defendant_id, caseId);
    }
  }
  
  // 11. 快速體驗模式：標記Session為已完成
  if (case_.mode === 'quick' && case_.session_id) {
    await this.sessionService.markSessionCompleted(case_.session_id);
  }
  
  return judgment;
}
```

**2. Prompt 規範（判決）**：

```typescript
async generatePlans(judgmentId: string, preferences?: PlanPreferences): Promise<ReconciliationPlan[]> {
  // 1. 獲取判決信息
  const judgment = await this.judgmentRepository.findById(judgmentId);
  if (!judgment) {
    throw new AppError(404, 'JUDGMENT_NOT_FOUND', '判決不存在');
  }
  
  // 2. 獲取案件信息
  const case_ = await this.caseRepository.findById(judgment.case_id);
  
  // 3. 構建Prompt
  const prompt = this.buildPlanPrompt(judgment, case_, preferences);
  
  // 4. 調用AI生成方案
  const plans = await this.aiService.generateReconciliationPlans(prompt);
  
  // 5. 計算難度等級
  const plansWithDifficulty = plans.map(plan => ({
    ...plan,
    difficulty_level: this.calculateDifficulty(plan),
    estimated_duration: this.estimateDuration(plan)
  }));
  
  // 6. 保存方案
  const savedPlans = await Promise.all(
    plansWithDifficulty.map(plan =>
      this.reconciliationPlanRepository.create({
        judgment_id: judgmentId,
        ...plan
      })
    )
  );
  
  return savedPlans;
}
```

**2. 計算難度等級**：
```typescript
private calculateDifficulty(plan: PlanContent): 'easy' | 'medium' | 'hard' {
  const totalScore = 
    plan.time_cost +
    plan.money_cost +
    plan.emotion_cost +
    plan.skill_requirement;
  
  if (totalScore <= 8) return 'easy';
  if (totalScore <= 12) return 'medium';
  return 'hard';
}
```

### InterviewService

**職責**：管理 AI 訪談生命週期

```typescript
interface InterviewService {
  startSession(userId: string, trigger?: InterviewTrigger): Promise<InterviewSession>;

  respond(id: string, userId: string, message: string, onSSE?: (event: SSEEvent) => void, isSkip?: boolean): Promise<void>;
  // ⚠️ 實際為 callback-based（onSSE），非 AsyncGenerator。Controller 層將 callback 轉為 HTTP SSE 推送。

  skip(id: string, userId: string, onSSE?: (event: SSEEvent) => void): Promise<void>;
  // 內部委派 respond(..., '', onSSE, true)

  endSession(id: string, userId: string): Promise<void>;

  getSession(id: string, userId: string): Promise<InterviewSession>;

  checkResume(userId: string): Promise<{
    has_pending: boolean;
    session_id?: string;
    last_ai_message?: string;
    turn_count?: number;
  }>;

  retryFailed(id: string, userId: string): Promise<void>;
}
```

> **實現對齊**：`startSession` 返回完整 session 對象（含 `turns[]`）。`getSession` 合併了原 `getResult` 和 `getHistory`，返回完整 session（含 status、feedback_card、richness_score、turns）。`retryFailed` 為獨立方法（對應 `POST /:id/retry`）。`endSession` 無返回值（`{ success, message }` 由 controller 包裝）。`respond` 參數名使用 `message`（非 `response`），與 API 請求體一致。

**SSEEvent 類型定義**（與 [03-API設計](./03-API設計.md) v2.1 對齊）：

```typescript
type SSEEvent =
  | { type: 'token'; data: { text: string } }
  | { type: 'metadata'; data: InterviewMetadata }
  | { type: 'safety_alert'; data: { message: string; severity: 'warning' | 'critical' } }
  | { type: 'error'; data: { code: string; message: string } }
  | { type: 'complete'; data: { session_id: string; status: string } };

interface InterviewMetadata {
  intent: string;
  target_domains: PsychDomain[];
  should_end: boolean;
  safety_flag: boolean;
}
```

**核心邏輯**：
- `startSession`：檢查 consent → 廢棄舊的 in_progress session（≥ 3 輪觸發 `asyncPipelineService.process()` 轉為 processing，< 3 輪標為 abandoned）→ 建立新 session → **所有 session 統一使用 SEED_QUESTION 常量**作為首條 AI 消息（零 AI 調用、零延遲），寫入 `turns[0].ai_message`。
  > **SEED_QUESTION 實際文本**：`'嗨！我想更了解你。先從輕鬆的開始——你知道自己的 MBTI 嗎？或者星座？'`（定義於 `interview.types.ts`）
  > **⚠️ 設計 vs 實現**：計畫中的「續聊 session 由 AI 根據已有洞察生成開場」尚未實現。
  - **注意**：start 只處理 `in_progress` 狀態的舊 session，不影響 `processing`/`completed`/`processing_failed`。
- `respond(id, userId, message, onSSE?, isSkip?)`：獲取 session 級 mutex lock（`lockService.withLock`，超時 30s）→ 呼叫 AI（`INTERVIEW_AI_CONFIG`：GPT-4o-mini，非流式 `openai.chat.completions.create`）→ 正則解析回應中的**單一 JSON**（含 text/intent/target_domains/should_end/safety_flag/safety_message）→ 通過 `onSSE` **callback** 依序推送 token → metadata → safety_alert（如有）→ complete → 存儲 turn → **釋放 lock**。AI 調用使用 `retryWithBackoff`（最多 **3 次重試**，指數退避，排除 429/401）。`skipTurn` 內部委派 `respond(..., '', onSSE, true)`。
  > **⚠️ SSE 實現**：實際使用 callback-based（`onSSE` 函數參數），非 `AsyncGenerator<SSEEvent>`。Controller 層將 callback 轉為 HTTP SSE 推送。
- `endSession`：僅允許 `status === 'in_progress'` → 設為 `processing` + 記錄 `ended_at` → 觸發 `asyncPipelineService.process(sessionId)`。**非冪等**：對非 in_progress 的 session 拋 VALIDATION_ERROR。
- `retryFailed`：僅對 `processing_failed` 的 session 有效，從 `pipeline_step + 1` 恢復管線執行。
- `getSession`：返回完整 session 含 turns、status、feedback_card、richness_score。前端用於 polling（每 3s 調一次直到 `status === 'completed'` 或 `processing_failed`）和對話歷史載入（斷線恢復時載入完整 turns）。
- Rate Limit：雙層限流——①中間件 `interviewStartLimiter`（每用戶每小時 3 次，`INTERVIEW_START_RATE_LIMIT`，防濫用）；②業務邏輯 `INTERVIEW_DAILY_SESSION_LIMIT`（每用戶每天 5 個 substantive session，≥ 3 輪才計數）。每 session 最多 25 輪（`INTERVIEW_MAX_TURNS`）；每輪間隔 ≥ 3 秒。

**錯誤與邊界處理**：

| 場景 | 行為 | 錯誤碼 |
|------|------|--------|
| respond/skip 時 session 非 in_progress | 拒絕 | `VALIDATION_ERROR` |
| respond 時未獲得 mutex lock | 拒絕，返回 409 | `CONCURRENT_REQUEST` |
| respond 時已達 25 turn（`INTERVIEW_MAX_TURNS`） | 拒絕 | `MAX_TURNS_REACHED` |
| respond 間隔 < 3 秒 | 拒絕，返回 429 | `RATE_LIMIT_EXCEEDED` |
| 用戶未 consent | `requireConsent` 中間件攔截，403 | `CONSENT_REQUIRED` |
| session 不存在或不屬於當前用戶 | 拒絕，返回 404 | `NOT_FOUND` |
| AI call 失敗 | `retryWithBackoff` 最多 3 次 → 仍失敗通過 onSSE 推送 error | `AI_CALL_FAILED` |
| endSession 對非 in_progress 的 session | 拒絕 | `VALIDATION_ERROR` |

### AsyncPipelineService

**職責**：session 結束後的異步處理管線

```typescript
interface AsyncPipelineService {
  process(sessionId: string): Promise<void>;
  resume(sessionId: string, fromStep: number): Promise<void>;
}
```

> **實現對齊**：`resume` 用於從指定步驟恢復（`InterviewService.retryFailed` 調用），非 `retryFailed`。

**管線步驟**（逐步 try-catch + 重試 2 次（固定 2s/4s）+ 部分成功策略。與 `async-pipeline.service.ts` 對齊）：

| 步驟 | 枚舉 | 操作 | AI 模型 | 可跳過 | 失敗影響 |
|------|------|------|---------|--------|---------|
| 1 | NARRATIVE_EXTRACTION | `domainClassificationService.batchClassify(sessionId)` → `narrativeService.extractNarratives(sessionId)` — 先用 AI 批次校驗域分類，再從 turns 按域歸類建立/更新 ProfileNarrative，heuristic 計算 completeness | INTERVIEW_AI_CONFIG（gpt-4o-mini，域分類） | 否 | 管線失敗 |
| 2 | NARRATIVE_SUMMARY | `narrativeService.summarizeNarratives(userId)` — **逐域** AI 摘要 | ANALYSIS_AI_CONFIG（gpt-4o） | 是 | 跳過，保留原始敘事 |
| 3 | INSIGHT_EXTRACTION | `insightExtractionService.extractInsights(userId, sessionId)` — **一次批次** AI call | ANALYSIS_AI_CONFIG（gpt-4o） | 是 | 跳過，保留舊 insight |
| 4 | RICHNESS_CALCULATION | `profileRichnessService.calculateRichness(userId)` | 無（純計算） | 否 | 管線失敗 |
| 5 | FEEDBACK_GENERATION | AI 生成 feedback card JSON → 寫入 session.feedback_card | ANALYSIS_AI_CONFIG（gpt-4o） | 是 | 使用預設 JSON 模板 |
| 6 | 狀態更新 | session → `completed` 或 `processing_failed` | — | — | — |

> **⚠️ 實現說明**：
> - **域分類已實現**：`domainClassificationService.batchClassify()` 在管線 step 1 中執行（使用 `INTERVIEW_AI_CONFIG`，gpt-4o-mini），校驗結果更新 `turns.ai_target_domains` 和 `session.domains_touched`。
> - **步驟順序**：richness（步驟 4）在 feedback（步驟 5）**之前**。
> - **模型分配**：域分類使用 `INTERVIEW_AI_CONFIG`（gpt-4o-mini）；敘事摘要、洞察提取、反饋卡片均使用 `ANALYSIS_AI_CONFIG`（gpt-4o）。
> - **敘事摘要**：逐域調用（非批次），AI 只回傳純文字摘要，completeness 由 heuristic 計算（`0.5 × min(1, wordCount/200) + 0.5 × min(1, turns/10)`）。

**部分成功策略**：
- 可跳過步驟（NARRATIVE_SUMMARY、INSIGHT_EXTRACTION、FEEDBACK_GENERATION）：重試 2 次後仍失敗 → 跳過並繼續下一步
- 不可跳過步驟（NARRATIVE_EXTRACTION、RICHNESS_CALCULATION）：重試 2 次後仍失敗 → 管線停止，session → `processing_failed`
- `resume(sessionId, fromStep)`：由 `InterviewService.retryFailed` 調用，設 session 回 `processing` 後從 `fromStep` 繼續執行
- 每步成功後更新 `pipeline_step`

### JudgmentService 升級（v2.0 改動）

**實現要點**：畫像上下文在 `generateJudgment` 內**內聯**組裝，無獨立 `buildProfileContext(userId, caseId)` 函數。上下文來自**即時**查詢的 narratives/insights，快照僅用於紀錄，不參與 prompt 組裝。

**流程概要**（`generateJudgment` 內，`case_.mode !== 'quick'` 且 `case_.plaintiff_id` 存在時）：

1. **Richness 與等級**：對每位參與者呼叫 `profileRichnessService.calculateRichness(userId)`，再以 `getRichnessLevel(richness)` 得到 `L0 | L1 | L2 | L3`（閾值：`< 0.05` → L0，`< 0.2` → L1，`< 0.5` → L2，`≥ 0.5` → L3）。
2. **快照僅作紀錄**：在組裝 prompt 前，對該使用者呼叫 `profileSnapshotService.createSnapshot(userId, caseId)`，用於判決紀錄完整性；失敗僅 log，不中斷生成。**不從 snapshot 讀取內容**，prompt 內容來自下方即時查詢。
3. **即時資料**：從 DB 查詢**即時**資料組裝心理畫像段落：
   - `ProfileNarrative`：`user_id`、`is_latest: true`，取 `domain`、`ai_summary`、`completeness`。
   - `ProfileInsight`：`user_id`、`is_active: true`，取 `domain`、`insight_type`、`key`、`value`、`confidence`。
4. **依等級注入**（每角色字串由內聯 helper `loadPsychProfile(userId, label)` 產出）：
   - **L0**：直接回傳空字串，不注入。
   - **L1**：僅**高信心洞察**（`confidence >= 0.7`），格式：`角色X的核心特質：key：value；...`。
   - **L2**：**敘事 + 洞察**。敘事：有 `ai_summary` 且 `completeness > 0.2` 的 narrative，格式 `角色X（domain）：ai_summary`；洞察：`confidence >= 0.5`，格式 `角色X的洞察：key=value；...`。
   - **L3**：**全量**。敘事：有 `ai_summary` 的皆納入，並帶 `domain`、完整度百分比；洞察：全部納入，格式含 `domain`/`insight_type`、`key`、`value`、信心百分比。
5. **原告/被告**：原告畫像（remote 與 collaborative 皆注入）；被告畫像僅在 **remote** 模式且 `case_.defendant_id` 存在時注入。
6. **v1.0 回退欄位**：在心理畫像段落之外，**一併**寫入 `UserProfile` 與 `RelationshipProfile`（若存在）：
   - **UserProfile**（原告必查，被告僅 remote 時查）：`mbti_type`、`communication_style`，避免重複時不重複推 MBTI 句。
   - **RelationshipProfile**（有 `case_.pairing_id` 時）：`relationship_duration_days`、`relationship_stage`、`is_long_distance`、`conflict_communication_style`。
7. 最終 `profileContext` 為上述 `profileParts.join('\n')`，傳入 `aiService.generateJudgment(..., { profileContext })`。

**案件模式與畫像使用規則**（與實現一致）：

| 案件模式 | 畫像使用 | 原因 |
|----------|----------|------|
| `quick`（快速體驗） | **不使用畫像** | 快速體驗無註冊用戶，無 userId，無法查詢畫像 |
| `remote`（遠程雙方） | **雙方畫像**（原告 + 被告各自注入） | 雙方都是註冊用戶，各自有獨立的畫像數據 |
| `collaborative`（協作模式） | **僅原告畫像** | 被告可能未註冊或未同意知情同意 |

**注入閾值策略**（`getRichnessLevel`，與 `interview.types` 一致）：

- `richness < 0.05`：**L0**，不注入心理畫像（UserProfile/RelationshipProfile 仍可注入）。
- `0.05 ≤ richness < 0.2`：**L1**，僅高信心洞察（confidence ≥ 0.7）。
- `0.2 ≤ richness < 0.5`：**L2**，敘事（completeness > 0.2）+ 洞察（confidence ≥ 0.5）。
- `richness ≥ 0.5`：**L3**，全量敘事與洞察。

> **✅ 已實現**：「關係互動層」（雙方 key/value 模板匹配，如「焦慮+迴避 → 追逐-逃避循環」）已在 `judgment.service.ts` 的 `buildInteractionLayer()` 實作。包含依附配對預測、文化差異、溝通風格衝突、臨床注意事項。  
> **✅ 已實現**：「心理畫像段落 token 預算 2500 + 5 級截斷優先順序」已在 `judgment.service.ts` 的 `truncateProfileParts()` 實作，優先順序為 ALWAYS_KEEP > HIGH > MEDIUM > LOW > FIRST_CUT。

### CaseContextService（v2.1 新增，v2.6 優化）

**文件**：`src/services/case-context.service.ts`

在 v2.0 的 `profileContext`（主判決 Prompt）基礎上，v2.1 新增 `CaseContextService`，為判決管線的其他子 Prompt 提供按案件類型智能選取的背景數據：

- **`loadCaseContext(caseId, preloadedCase?)`**：根據案件類型查找相關心理域（如「情感需求衝突」→ attachment + relationship_history），載入雙方用戶的高信心洞察（≥0.5）和關係資料。v2.6 新增 `preloadedCase` 可選參數，避免在 `JudgmentService` 中重複查詢已有的 case 數據。
- **`formatForEmotionalAnalysis()`**：輕量依附/溝通提示 → 注入 `analyzeEmotionalDynamics`
- **`formatForResponsibilityRatio()`**：關係歷史上下文 → 注入 `assessResponsibilityRatio`
- **`formatForSummary()`**：一句話背景 → 注入 `generateSummary`
- **`formatForReconciliationPlans()`**：完整個性化上下文（溝通偏好、底線、歷史執行率等）→ 注入 `generateReconciliationPlans`

**v2.6 域映射修正**：基於臨床心理學審查，每個案件類型的 secondary 域從 2 個擴展至 3 個。例如「生活習慣衝突」新增 attachment 為 secondary（控制/自主需求與依附相關），所有類型均含 family_origin（原生家庭影響具跨議題普遍性）。

**與 v2.0 profileContext 的關係**：
- `profileContext`（v2.0）：服務主判決 Prompt，含互動層分析和 token 預算控制
- `CaseContextService`（v2.1+）：服務其他 4 個子 Prompt，按各自需求選取不同層級的背景數據
- 兩者並存，互不衝突
